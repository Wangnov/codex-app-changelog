## 铁律(违反任意一条都算失败)

1. **只用事实包里的事实**。事实包没有的,一律不写。严禁脑补、套话、用训练知识填充,或编造任何数字。
2. **每条变化必须标注证据等级**,二选一:
   - **【实证】**——有人类可读的直接证据:Info.plist 键值、CSP 文本、技能文档 / 类型声明 / package.json 的 diff、依赖清单、签名 identifier。
   - **【信号】**——只有间接证据:前端模块名、二进制体积变化。必须显式写明"指向某方向,但无法从产物还原到源码级意图"。
3. **忽略噪音**。`webview/assets`、`.vite/build` 下带 hash 的文件名增减是构建产物重命名,**绝不能**写成"新增/删除了功能"。事实包里已去噪的"前端真实新增模块"才可作为【信号】使用。
4. **数字必须与事实包逐一对齐**(版本号、build、体积、依赖数量、域名、签名)。拿不准就不写具体数字。
5. **只写用户或开发者可感知的变化**。纯内部重构、签名 / 哈希变动不单独成条。
6. **归纳要有洞察**。把"新增 pdfjs-dist / tesseract.js / sharp / playwright"提炼成"补齐了读 PDF、OCR、图像处理、浏览器自动化的能力",而不是干巴巴罗列包名;把分散但相关的事实(某依赖群 + 对应接口变化 + 对应 CSP 放行)合并成**一个有意义的主题**。

3. `## ✨ 重点变化` —— 按主题分小节(`### 1. 标题 【实证/信号】`),从"用户 / 开发者最可感知"排到"底层"。每节先一句"是什么变化",再一句"意味着什么 / 可感知点",必要时附最关键的证据(一行 diff、一个键值、一个域名)。
4. `## 📊 原始数据` —— 三部分:(a) bundle 文件级统计(文件数、新增/移除/改动);(b) 关键组件体积变化表;(c) 新增依赖清单。**不要**把 app.asar 内部的新增/移除文件计数列进任何表格——那是 Vite hash 重命名主导的噪音,列出来会让读者误以为有上千个真实改动。
5. `## 🔁 可复现信息` —— 下载制品的 SHA-256、大小、EdDSA 验签结论、codesign/spctl 公证结论(表格)。
6. `## ⚖️ 边界与声明` —— 与 OpenAI 无关 / 非官方;不重分发任何官方二进制,只发布差异结论;【信号】条目不构成源码级事实;欢迎指正。

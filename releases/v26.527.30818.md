# Codex 桌面版 `26.527.30818`(build 3370)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版最值得关注的是桌面壳层的底座变化：包内从 Electron(跨平台桌面应用框架)相关路径切到 Codex Framework / Chromium(浏览器内核项目)形态，同时前端和插件层出现了面向 App 生成、浏览器侧栏、图表渲染、协作与宿主连接的新信号。  
One-liner: the app shell appears to move toward a Chromium-based Codex Framework packaging model, with broader browser/app-host and project workflow surfaces.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次是历史回填：两个版本均超出官方 Sparkle(应用自动更新框架)增量窗口，因此没有使用增量包重建，而是下载两个官方全量包后做结构化 bundle(应用包)差异。

对比范围是 `26.519.41501` build `3044`，发布于 `2026-05-23`，到 `26.527.30818` build `3370`，发布于 `2026-05-29`。可信度来自 Apple(苹果)公证与 OpenAI Developer ID(开发者身份签名)代码签名校验：上一版与最新版 `Codex.app` 均为 `accepted`，来源均为 `Notarized Developer ID`，签名来源均为 `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`。

本文使用两档证据等级：

- **【实证】**：来自 Info.plist(应用元数据文件)、package.json(项目依赖与脚本配置)、插件配置、可读 diff(差异文本)、签名与公证结果等直接证据。
- **【信号】**：来自前端模块名或二进制体积变化等间接证据。它们指向某方向，但无法从产物还原到源码级意图，也不代表功能一定已对用户开放。

## ✨ 重点变化

### 1. 桌面底座从 Electron 痕迹切到 Codex Framework / Chromium 形态 【实证】

包内出现了新的 `Contents/Frameworks/Codex Framework.framework/Versions/148.0.7778.179/`，其中主框架文件大小为 `248.3 MB`；同时旧的 `Contents/Frameworks/Electron Framework.framework/Versions/A/Electron Framework` 被移除，大小为 `181.5 MB`。

这意味着本版桌面应用的底层打包形态发生了明显变化。用户未必会直接看到“框架名称”变化，但启动、窗口、网页渲染、崩溃处理与系统集成的底层实现都可能受它影响。

关键证据：

```diff
/NSPrincipalClass: 'AtomApplication' → 'BrowserCrApplication'
```

同时 Info.plist 新增：

```text
/ChromiumBaseVersion = '148.0.7778.179'
/ChromiumBaseBundleVersion = '7778.179'
/CrProductDirName = 'com.openai.codex'
```

### 2. 新应用壳与宿主连接通道开始显形 【实证】

`package.json` 中的 `owl` 脚本被改为先执行 `build:dev-vite`，并新增 `build:dev-vite` 脚本；preload(预加载脚本)里新增了 `get-uses-owl-app-shell` 与 `connect-app-host` 相关通道。

这指向桌面应用正在引入或切换到新的应用壳路径。可感知层面，未来窗口壳、WebView(网页视图)与宿主进程之间的通信方式可能会更集中，也可能服务于更复杂的内嵌应用体验。

关键证据：

```diff
+ "build:dev-vite": "node ./scripts/build-dev-vite.mjs"
```

```text
codex_desktop:get-uses-owl-app-shell
codex_desktop:connect-app-host
usesOwlAppShell
```

### 3. 浏览器插件运行时路径从内置 app-server-runtime 转向外部注入 【实证】

Chrome(浏览器)插件安装脚本不再固定从插件目录下的 `app-server-runtime` 里拼出 `codex`、`node`、`node_repl`，而是允许通过 `appServerRuntimePaths` 传入运行时路径。

这说明浏览器自动化插件和主应用运行时之间的绑定方式变得更灵活。开发者可感知点是：浏览器宿主配置不再只依赖插件包内固定目录，后续更容易复用主应用提供的运行时。

关键证据：

```diff
- let o=a(t,"app-server-runtime");return $({codexCliPath:a(o,"codex"), ... nodePath:a(o,"node"),nodeReplPath:a(o,"node_repl"), ...})
+ return $({browserClientPath:v(t,"scripts","browser-client.mjs"),...e})
```

以及调用处：

```diff
- appServerHostConfig:C(t,{channel:i.channel})
+ appServerHostConfig:S(e,{channel:s.channel,...t.appServerRuntimePaths})
```

### 4. 浏览器与本地应用集成能力继续扩展 【实证】

Info.plist 新增了 `ASWebAuthenticationSessionWebBrowserSupportCapabilities` 下的三个能力键，包括 `CallbackURLMatchingIsSupported`、`EphemeralBrowserSessionIsSupported` 与 `IsSupported`，值均为 `True`。

这表明应用声明了对系统网页认证会话能力的支持。用户可感知点可能出现在登录、授权、回调 URL(URL，统一资源定位符)匹配或临时浏览会话相关流程中。

关键证据：

```text
/ASWebAuthenticationSessionWebBrowserSupportCapabilities/CallbackURLMatchingIsSupported = True
/ASWebAuthenticationSessionWebBrowserSupportCapabilities/EphemeralBrowserSessionIsSupported = True
/ASWebAuthenticationSessionWebBrowserSupportCapabilities/IsSupported = True
```

### 5. 文件类型识别范围明显扩展到网页、图片、媒体与 PDF 【实证】

Info.plist 新增了大量 UTI(统一类型标识符)声明，覆盖 HTML、XHTML、JavaScript、JPEG、PNG、SVG、GIF、WebM、WebP、PDF、AVIF、Ogg 音视频、MHTML、Chromium Extension 与 Chromium Shortcut。

这意味着系统层面知道 Codex 对更多内容类型有声明。可感知点可能体现在文件拖拽、打开方式、剪贴板内容、网页归档、图片与 PDF 文件处理等入口，但事实包不能证明每一种类型都已有完整用户功能。

关键证据：

```text
com.adobe.pdf → pdf / application/pdf
org.webmproject.webp → webp / image/webp
public.svg-image → svg / image/svg+xml
org.chromium.extension → crx / application/x-chrome-extension
```

### 6. App 生成、分享与设置相关前端模块出现 【信号】

去掉 Vite(前端构建工具)hash(哈希)噪音后，新增模块中出现 `appgen-page`、`appgen-settings-page`、`appgen-share-dialog`、`appgen-access-mode`、`appgen-url` 等名称。

这些模块名指向 App 生成、访问模式、设置与分享方向，但无法从产物还原到源码级意图，也不能证明相关功能已对所有用户开放。对开发者来说，它们是本版前端产品方向变化的可观察信号。

### 7. 图表、架构图与 Mermaid 渲染面明显扩大 【信号】

新增模块中出现 `mermaid.core`、`architectureDiagram`、`blockDiagram`、`c4Diagram`、`classDiagram`、`erDiagram`、`ganttDiagram`、`gitGraphDiagram`、`kanban-definition`、`mindmap-definition`、`sankeyDiagram`、`sequenceDiagram`、`stateDiagram`、`timeline-definition`、`treemap`、`xychartDiagram` 等。

这指向更丰富的图表与结构化可视化渲染方向，但无法从产物还原到源码级意图。可感知点可能是回答、预览、工件或文档中更复杂图表的展示能力。

### 8. 项目、PR 与 GitHub 工作流相关界面继续补强 【信号】

新增模块包含 `gh-cli-status-query`、`gh-pull-request-status-query`、`pull-request-status`、`pull-request-status-utils`、`pull-request-visual-state`、`pull-request-readonly-comment`、`local-projects`、`workspace-file-context-menu`、`workspace-dependencies-feature`。

这些名称指向 GitHub(代码托管平台)、PR(拉取请求)状态展示、本地项目与工作区依赖相关体验，但无法从产物还原到源码级意图。它们更像是开发者工作流界面在继续细化。

### 9. 协作、成员、@ 提及与侧栏线程结构有新迹象 【信号】

新增模块包含 `members`、`at-mention-list`、`use-workspace-users`、`sidebar-thread-section`、`sidebar-task-pr-chip-signals`、`profile-queries`、`connected-apps` 等。

这些模块名指向成员、用户、连接应用与线程侧栏方向，但无法从产物还原到源码级意图。可感知层面可能涉及团队空间、提及、个人资料或任务状态展示。

### 10. bundled 插件版本小幅更新 【实证】

Chrome 插件与 Browser(内置浏览器)插件版本从 `26.519.41501` 更新到 `26.527.30818`；LaTeX(排版系统)插件从 `0.2.0` 更新到 `0.2.2`。

这属于可读配置层面的直接变化。事实包只证明版本号变化，不能进一步证明具体功能或行为改动。

关键证据：

```diff
latex: 0.2.0 → 0.2.2

browser: 26.519.41501 → 26.527.30818
```

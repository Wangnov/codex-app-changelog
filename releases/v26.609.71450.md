# Codex 桌面版 26.609.71450 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

本批次两端并不同步：macOS 26.609.71450 的事实包变化很小，Windows 26.611.7849.0 则带来大量应用层与运行时信号。  
One-liner: macOS is mostly a package/version refresh, while Windows carries broader app, browser, remote-control, Sites, voice, and Computer Use changes.

## 关于这份报告

这份报告只基于两份官方签名分发包逆向重建后的机器差异事实：macOS arm64 从 26.609.71144 build 3952 到 26.609.71450 build 3965；Windows x64 MSIX 从 26.609.4994.0 到 26.611.7849.0。

证据分两档：【实证】表示来自 plist、manifest、CSP、技能文档、类型声明、package.json、依赖清单、插件清单、服务入口或新增可读字符串等确定进入包内的材料；【信号】表示仅来自前端模块名、体积变化、压缩 bundle 方向线索或同名前端模块改动线索。平台归属使用 `[macOS+Windows]`、`[仅 macOS]`、`[仅 Windows]`。

需要特别说明：两个平台版本号与发布时间可能错位，本报告分别引用各自事实，不把 Windows 中出现的功能变化自动套到 macOS。hash 命名的 webview/assets 与 .vite/build 变动按构建噪音处理。

## ✨ 重点变化

### 【实证】[macOS+Windows] 版本与包体批次推进，但两端变化规模差异很大

macOS 从 26.609.71144 build 3952 更新到 26.609.71450 build 3965，Info.plist 中 `CFBundleShortVersionString`、`CFBundleVersion` 与 `ElectronAsarIntegrity/Resources/app.asar/hash` 均变化。其 bundle 文件数保持 8298 → 8298，新增 0、移除 0、内容改动 61；app.asar 内部文件为 2870 → 2870，新增 1、移除 1、改动 11。

Windows 从 26.609.4994.0 更新到 26.611.7849.0，package.json 中应用版本从 26.609.41114 到 26.611.61049，`codexBuildNumber` 从 3888 到 3996。其文件数为 11972 → 11943，新增 7、移除 36、内容改动 92；app.asar 内部文件为 2815 → 3024，新增 1819、移除 1610、改动 93。Windows 的变化面明显更宽。

### 【实证】[仅 macOS] macOS 样本主要确认 Browser/Chrome 插件元数据版本推进

macOS 事实包中可读的插件差异集中在两个内置插件 manifest：`openai-bundled/plugins/chrome/.codex-plugin/plugin.json` 与 `openai-bundled/plugins/browser/.codex-plugin/plugin.json` 的 `version` 从 26.609.71144 更新到 26.609.71450。插件描述文本未显示语义变化。

这说明本 macOS 样本里能确认的是内置 Browser 与 Chrome 插件随应用版本号同步推进；事实包没有给出新增插件、服务入口、技能说明、默认提示或最大文件变化。

### 【实证】[仅 Windows] Browser/Chrome 插件能力文档新增 JavaScript 对话框 API，并收紧登录受阻时的浏览行为

Windows 中 Browser 与 Chrome 插件的技能文档新增同一条规则：当用户明确要求在浏览器中导航到某页面，但认证或登录阻止任务时，不应改用 web search、搜索引擎、其他站点或其他来源绕过登录；如果环境提供 secure browser authentication，则使用该流程，否则停止并请用户登录。

同一批文档还扩展了 Browser API：`Tab` 新增 `getJsDialog(): Promise<undefined | Dialog>`，并新增 `AlertDialog`、`BeforeUnloadDialog`、`ConfirmDialog`、`PromptDialog` 与 `Dialog` 联合类型。`clipboard` 描述从“this tab”调整为“browser session's clipboard”。这是插件公开文档层面的实证变化，指向浏览器会话级剪贴板语义与 JavaScript 弹窗处理能力。

### 【实证】[仅 Windows] 浏览器右键菜单新增“复制链接地址”，并进入多语言原生命令菜单

Windows 主进程/前端新增可读字符串包含 `Copy link`、`browserSidebar.contextMenu.copyLink`、`Open in browser`、`Open in external browser`、`codex.commandDescription.closeTab`、`codex.commandMenuTitle.closeTab` 等。多份 `native-menu-locales/*.json` 同步新增 `browserSidebar.contextMenu.copyLink`，例如中文为“复制链接地址”，日文为“リンクのアドレスをコピー”，韩文为“링크 주소 복사”。

这是高可见 UI 文案级实证：浏览器侧边栏上下文菜单新增复制链接地址能力；同时 `Close Tab` 与 close tab/window 命令字符串变化显示关闭标签页/窗口命令被拆分或重命名，但完整交互仍需运行时验证。

### 【实证】[仅 Windows] Computer Use 的 @oai/sky 从 0.4.10 到 0.4.13，并给 mac target 操作套上 policy 层

Windows 的 `@oai/sky/package.json` 从 0.4.10 更新到 0.4.13，移除了 `optionalDependencies` 中的 `objc-js` 1.5.0。多个 mac target 操作文件从直接调用 `getClient()`，改为通过 `withComputerUsePolicy` 包裹：`press_key`、`type_text`、`drag`、`set_value`、`scroll`、`click`、`perform_secondary_action`、`get_app_state`、`select_text`。`list_apps` 还新增 `setComputerUseResponseMeta(null)`。

类型声明也显示 mac client 从 `apple-event` 迁移到 `native-pipe`，`RequestOptions` 删除 service app path、bundle identifier、process identifier 字段，并新增内部 transports/transport 相关结构。新增文件中出现 `native-pipe.js`、`computer-use-policy.js` 及对应 `.d.ts`；移除文件包含 `service-app.js`、`apple-event.js` 及对应声明文件。主进程新增字符串也包含 `computer-use native pipe helper paths unavailable`，移除了 `computer-use native pipe transport module missing`。

### 【实证】[仅 Windows] Computer Use 总体体积几乎不变，但运行时与依赖结构有小幅重排

Windows 的 `cua_node` 总量为 285.5 MB → 285.6 MB，增幅按事实包口径显示为 +0 MB、+0%。新增依赖包数量为 0，但 `@oai/sky` 版本变化如上。另一方面，Windows 包体新增 `resources/codex` 271.2 MB 与 `resources/rg` 5.4 MB，同时 `resources/codex.exe` 增加 27.01 MB，`codex-windows-sandbox-setup.exe` 增加 0.45 MB，`codex-command-runner.exe` 增加 0.07 MB。

这些是文件级事实：Windows 分发包内 CLI/运行器/沙箱相关二进制明显变化，但事实包没有给出它们的源码语义，不能进一步断言具体功能行为。

### 【信号】[仅 Windows] 语音听写与悬浮头像交互继续增强

Windows 新增前端模块包括 `global-dictation-orb`、`use-dictation`、`use-dictation-hotkey`、`avatar-overlay-realtime-voice-button`、`use-avatar-overlay-selection`。同名前端模块 `avatar-overlay-native-frame` 新增字符串包括 `Dictation is not available on this device`、`Follow up`、`avatarOverlay.replyNotification`、`avatarOverlay.stopNotification`、`composer.dictation.unsupported`，并新增悬浮通知回复、停止运行任务、follow-up 输入框等可访问性文案。

主进程新增字符串还包含 `Preferred microphone input device for dictation`、`Failed to start realtime in the avatar overlay`、`Failed to read realtime session log for feedback.`。这些字符串确凿进入包内，但从前端模块线索推 UI 可见性仍属于信号：可以说语音、听写、悬浮头像通知与 realtime 相关路径在 Windows 包中扩展，不能断言所有用户已开放。

### 【信号】[仅 Windows] 远程控制设备配对、环境删除条件与远程文件读写路径调整

Windows 新增模块包括 `cloud-environments-settings-page`、`cloud-preferences`、`cloud-preferences-settings`、`remote-connections-settings` 同名模块也大幅变化。新增字符串包括 `Add a device`、`Pairing code`、`Enter pairing code generated by the device you want to control`、`Failed to add device`、`Disable this environment before deleting it`、`Remote-control client has been revoked`、`Failed to read remote file`、`Failed to write remote file`。

这组变化共同指向远程控制与云环境设置：手动配对码、添加设备、删除环境前必须禁用、远程文件读写失败路径与客户端撤销状态进入包内。删除条件从旧的“Only offline environments can be deleted”转向“Disable this environment before deleting it”也是明确文案变化。

### 【实证】[仅 Windows] Sites 插件去掉 auth-siwc starter 路径，改为确认平台认证路径

Windows 的 `sites-building/SKILL.md` 删除了直接使用 `templates/auth-siwc-enabled` 的建议，新增或改写为：公共或外部认证站点应先确认当前平台认证路径，不再从 starter 脚手架添加 app-owned auth stack。文件树也移除了 `resources/plugins/openai-bundled/plugins/sites/skills/sites-building/templates/auth-siwc-enabled/` 下的大量模板文件，包括 package.json、README、db/schema.ts、auth route、worker、D1 示例、`.openai/hosting.json` 等。

这是技能文档与文件树双重实证：Windows 包中的 Sites 构建流程从“内置 SIWC/OAuth starter”收敛为“确认平台认证路径后再实现”。

### 【信号】[仅 Windows] Sites 创建、部署、环境与版本状态进入会话 UI 文案

Windows 的 `local-conversation-thread` 新增字符串包括 `Creating Sites project`、`Created Sites project`、`Getting Sites environment`、`Getting Sites project`、`Getting Sites version`、`Deploying Sites version`、`Deployed Sites version`、`Getting Sites deployment`、`Got Sites deployment`。多个本地化包也出现 Sites 项目、版本、部署、访问组等翻译字符串。

这些是前端同名模块与本地化字符串的信号：会话线程 UI 开始承载 Sites 项目创建、环境获取、版本部署等进度状态。由于该证据来自前端 bundle 线索与可读字符串，不能断言 Sites 功能面向所有用户开放。

### 【信号】[仅 Windows] 设置页扩展到云环境、代码审查、浏览器资料导入与工作模式

Windows 新增前端模块包含 `cloud-environments-settings-page`、`cloud-preferences-settings`、`code-review-settings`、`browser-profile-import-dialog`、`browser-profile-import-query`、`browser-use-settings-visibility`、`settings-surface`。`settings-page` 新增字符串包括 `All settings`、`Cloud environment`、`Create cloud environment`、`Create environment`、`Add secret`、`Add variable`、`Additional editors`、`Automatic setup`、`Controls internet access after setup completes`。

`general-settings` 又新增 UI 字符串：`Adjust the base size used for code across chats and diffs`、`Adjust the base size used for the Codex UI`、`Accessible label for work mode settings options`、`Aria label for hold-to-dictate hotkey capture input`、`Aria label for one dictation dictionary entry`、`Accessible label for the ambient suggestions toggle`。这说明 Windows 设置面板新增或重排了云环境、变量/密钥、编辑器、网络访问、字体/界面尺寸、工作模式、听写热键和 ambient suggestions 相关入口；可见性仍需运行验证。

### 【信号】[仅 Windows] 自动化运行、动态工具与 CI 诊断路径进入包内

Windows 主进程新增字符串包括 `Failed to finish automation run`、`Failed to load dynamic tools for automation run`、`Failed to settle interrupted automation runs`、`Unknown dynamic tools response requestId`。`thread-side-panel-tabs` 新增字符串还包含一组 CI 处理指令：先检查用户要求的是哪个 CI，检查已安装 skills 或 tools 是否能访问该 CI，检查 credentials、permissions、tokens、MCP 或 tool access 是否可用，然后获取外部 logs、诊断并做最小修复。

这是较强的方向信号：Windows 包内新增自动化运行、动态工具加载、被中断自动化收敛，以及 CI 日志诊断工作流相关文案。但它们主要来自压缩 bundle 可读字符串与前端模块线索，不等同于完整源码事实。

### 【信号】[仅 Windows] 工作树初始化页从笼统状态改为分步进度与输出区域

Windows 的 `worktree-init-v2-page` 新增 `Creating worktree`、`Progress step for creating the worktree`、`Run environment setup`、`Progress step for running the worktree environment setup`、`Placeholder text before output starts streaming`、`worktreeInitV2.step.createWorktree`、`worktreeInitV2.step.runSetup`。旧字符串如 `Creating a worktree and running setup.`、`Fix worktree setup`、`Starting conversation.` 被移除。

这说明工作树初始化 UI 更偏向分步进度展示和输出流占位，而不是单一状态文案。证据来自同名前端模块改动线索，标为信号。

### 【信号】[仅 Windows] PDF 前端模块大幅增加，可能引入更完整的 PDF 渲染/注释基础

Windows `pdf` 前端模块体积从 0.0 MB 到 0.4 MB，新增大量 PDF 相关可读字符串和 CSS/ARIA 属性，例如 `PDFDataRangeTransport.requestDataRange`、`CMap name must be specified.`、`data-annotation-id`、`data-editor-rotation`、`data-l10n-id`、`aria-activedescendant` 等。

这是体积与字符串层面的强信号：PDF 相关前端代码从很小模块扩展为较完整的 PDF 处理模块。但事实包没有给出产品入口或用户可见流程，因此不能断言 PDF 功能已开放。

### 【实证】[仅 Windows] Webview CSP 未显示策略放宽，主要是入口脚本与 preload 列表变化

Windows `webview/index.html` 的主入口脚本从 `index-TowzspZU.js` 改为 `index-UPYMKhrc.js`，并移除了若干 `modulepreload` 项，如 `chunk-Cq_f4orQ.js`、`isEqual-PLfJaxty.js`、`src-B2PlTT4D.js`。CSP 仍显示 `default-src 'none'`，img、child、frame、worker、script、style、font、media、connect 的来源列表在事实包展示中未出现语义变化。

因此这里不能写成安全策略放宽；只能说入口构建产物与预加载图发生变化，CSP 展示片段保持严格默认拒绝的结构。

### [仅 Windows] 多语言包大面积同步新文案

多个语言资源新增自动代码审查、Sites 项目/版本、浏览器音频/摄像头/麦克风状态、appgen 编辑/分享、realtime voice、browser copy link、AGENTS.md 已存在时跳过 `/init` 等字符串。与此同时，`artifactDocxPreview.annotate`、`artifactDocxPreview.annotationMode`、`codex.localConversation.progressSidebar.title`、rate-limit reward/reset 相关键在多语言包中反复移除。

### [仅 Windows] 链接打开偏好与本地 URL 处理

### [仅 macOS] macOS 低层变化主要停留在完整性 hash 和插件 manifest 版本

macOS 的低层事实包括 app.asar integrity hash 更新、bundle 内容改动 61 个、app.asar 内部 11 个文件改动，以及 Browser/Chrome 插件 manifest version 更新。事实包没有提供可归并成 UI 或能力面的新增字符串、模块名、CSP、依赖或技能文档变化。

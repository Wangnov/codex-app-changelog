# Codex 桌面版 26.602.71036 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

这一批次的核心不是大规模文件增减，而是内置插件版本、Windows 运行时元数据，以及若干前端功能面的打包状态变化。

The key shift is not broad file churn, but bundled plugin versioning, Windows runtime metadata, and cross-platform frontend surface signals.

## 关于这份报告

本文基于两份官方签名分发包逆向重建后的机器差异事实：macOS arm64 对比 `26.602.40724` → `26.602.71036`，Windows x64 MSIX 对比 `26.602.4764.0` → `26.602.9276.0`。发布时间使用事实包给出的 `2026-06-07`。

证据分两档：【实证】来自 plist、manifest、插件清单、安装元数据、可读字符串、依赖清单和文件级 diff；【信号】来自同名前端模块、体积变化、样式类名和压缩 bundle 方向线索。平台归属使用 `[macOS+Windows]`、`[仅 macOS]`、`[仅 Windows]`。

需要特别说明：Windows 事实包的“版本与渠道”区块里平台/架构行写成了 `macOS arm64`，但输入分区和任务说明明确该段为 Windows x64 MSIX；本文按 `=== PLATFORM: Windows ===` 归属处理。没有事实支撑的 CSP、AppxManifest、新增资源或用户可见开关，不在正文中扩展推断。

## ✨ 重点变化

### 【实证】[macOS+Windows] Browser 与 Chrome 内置插件同步升到 26.602.71036

两端的 `browser` 与 `chrome` 插件 manifest 均只有版本号变化：`26.602.40724` → `26.602.71036`。描述文本仍指向同一能力面：Browser 用于应用内浏览器、本地页面、截图、点击、输入和 localhost 场景；Chrome 用于依赖用户现有 Chrome 状态的自动化。

平台差异是路径和包形态不同：macOS 位于 `Contents/Resources/plugins/openai-bundled/plugins/...`，Windows 位于 `resources/plugins/openai-bundled/plugins/...`。事实包没有显示这两个插件的服务入口、技能说明或权限文本发生实质差异，因此这里应理解为内置插件版本对齐，而不是能力面新增。

### 【实证】[仅 Windows] Computer Use 插件 manifest 版本随包更新

Windows 侧 `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` 显示 `computer-use` 插件版本从 `26.602.40724` 更新到 `26.602.71036`，描述为通过 Computer Use 从 Codex 控制 Windows 桌面应用。

macOS 事实包没有列出对应 `computer-use` 插件 manifest 改动，因此不能把这条扩展为双平台变化。此处是 Windows 样本中的明确插件清单变化。

### 【实证】[仅 Windows] owl shell 安装元数据刷新，运行时包未显示依赖新增

Windows 根部 `.installed-owl-shell.json` 的 `installedAt` 从 `2026-06-05T16:55:50.166Z` 更新为 `2026-06-08T23:05:30.351Z`。同一 diff 保留了 `runtimeInfo.arch: x64` 与 `archiveSha: 76a82b584f412bc890db1bfffdc24456a772f47cae7324a4e22eff7bb838c86f`。

事实包只给出安装元数据变化，没有给出运行时内容差异或行为差异。因此可以确认 Windows 包内记录的 owl shell 安装时间刷新，但不能据此断言 Computer Use 运行时能力发生改变。两平台的 `cua_node` 依赖变化均为“新增 0 个包”。

### 【信号】[仅 macOS] Appgen 分享对话框进入更完整的站点分享状态

macOS 的 `appgen-share-dialog` 同名前端模块体积增加 `19146 bytes`，新增可读字符串包括 `Add people or groups`、`Anyone in this workspace with the link`、`Anyone on the internet`、`Can visit`、`Copy link`、`Copied to clipboard`，以及复制 live site URL、外部打开 live site 的按钮标签。

这些是前端模块线索，因此属于【信号】而不是源码事实。它们共同指向站点分享对话框里的成员/群组搜索、链接权限、访问范围和复制链接交互。Windows 同名模块方向相反：移除了同一组分享相关字符串，体积减少 `19146 bytes`，所以这不是双平台共同新增。

### 【信号】[仅 macOS] @ mention 列表出现代理、MCP 与桌面应用分类线索

macOS 的 `at-mention-list` 模块体积增加 `22531 bytes`，新增字符串包括 `Computer use`、`Custom agents`、`Desktop apps`、`Live agents`、`MCP servers`、`Mac apps`、`No results`，并出现 `mcp-capability`、`mcp-capability-mentions`、`mcp-servers` 等类名。

这组信号指向提及列表中更细的能力分组：代理、MCP server、Computer Use 和桌面应用。事实包没有给出 Windows 同名模块变化，因此只能标为 macOS 样本中的前端信号，不能断言 Windows 同步具备或缺失该 UI。

### 【信号】[仅 Windows] 设置页新增权限、字体、环境、听写与弹窗窗口设置线索

Windows 的 `general-settings` 模块体积增加 `90992 bytes`，新增字符串覆盖默认权限开关、composer context usage indicator、work mode options、代码字体大小、Codex UI 字体大小、`Agent environment`，以及 `hold-to-dictate hotkey capture input`、`hotkey window hotkey capture input`、`dictation dictionary entry`、`Popout Window projectless default setting toggle`。

这说明 Windows 包内的设置页模块出现了权限、上下文用量提示、工作模式、字号、代理环境、听写热键/词典和弹窗窗口默认行为的界面线索。macOS 同名模块方向相反，移除了较短的一组设置字符串并减少 `93168 bytes`，因此不能写成双平台设置页新增。

### 【信号】[仅 Windows] 首页公告模块新增速率限制、Fast mode、多代理、远程连接和 ambient suggestions 横幅线索

Windows 的 `home-announcements` 模块体积增加 `29639 bytes`，新增字符串包含 Codex rate limit reset、Fast mode、backend-driven Codex app home banner、multi-agent composer banner、remote connections banner、`Add Connections`、ambient suggestions connected apps banner、禁用 ambient suggestions 的按钮，以及打开 rate limit resets modal 的按钮。

这组信号集中在首页横幅和公告：速率限制重置、Fast mode、多代理 composer、远程连接、连接入口和 ambient suggestions。macOS 同名模块移除了相应横幅字符串并减少 `29639 bytes`，所以该批次在两平台样本里呈现明显反向。

### 【信号】[仅 Windows] 调试面板新增应用状态、浏览器状态与会话信息线索

Windows 的 `debug-modal` 模块体积增加 `176718 bytes`，新增字符串包括 `(command unavailable)`、`Account ID`、`Active browser tab`、`Active conversation`、`Allow automatic worktree cleanup in this unpackaged build.`、`Allow host`、`Allow host starting`、`Already current`、`Ambient suggestion chats`、`App Actions`、`App Server`、`App session ID`。

这说明 Windows 包内调试面板出现更丰富的状态和诊断线索，涉及命令可用性、账号、浏览器标签、当前会话、host allowlist、应用服务和 session ID。macOS 同名模块反向移除部分字符串并减少 `178828 bytes`，因此只能说明两个样本在该模块上不同步。

### 【信号】[macOS+Windows] Worktree 初始化 v2 页面都移除了本地环境失败恢复文案

两平台的 `worktree-init-v2-page` 同名前端模块均移除与 worktree 创建和本地环境失败恢复相关的文案，包括 `Continue anyway`、打开 local environment settings、启动 repair thread、取消 worktree setup 并开始本地对话，以及 `Creating a worktree...` 系列状态文案。

这是双平台共同方向的前端信号。macOS 模块减少 `6963 bytes`，Windows 减少 `4964 bytes`。由于证据来自同名前端模块线索，不能直接断言 worktree 初始化流程被移除，只能说这些 UI 文案/状态在对应 bundle 差异中消失。

### 【实证】[仅 macOS] Info.plist 版本、build 与 asar 完整性 hash 更新

macOS `Info.plist` 显示 `/CFBundleShortVersionString` 从 `26.602.40724` 更新为 `26.602.71036`，`/CFBundleVersion` 从 `3593` 更新为 `3685`。`/ElectronAsarIntegrity/Resources/app.asar/hash` 从 `0d60982859a8e93d9df7bd629e63cf70b2d83c482420bee78c3af36fb0bf5e67` 更新为 `1eaacb8bd09c1647a501ac76f4718bc83badec23ad7160e1691ec161790790ab`。

Windows 事实包没有提供 AppxManifest 变化，因此本文不推断 Windows manifest 字段。

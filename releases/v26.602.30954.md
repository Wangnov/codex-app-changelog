# Codex 桌面版 `26.602.30954`(build 3575)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版的主线很清晰：Codex 桌面版把 Browser(内置浏览器自动化能力)、Chrome(谷歌浏览器集成) 和 Sites(网站构建与部署能力) 的工作流说明做了明显收敛，并同步更新了 Computer Use(让模型操作桌面应用的能力) 组件版本。  
One-liner: browser automation guidance is consolidated, Sites becomes more deployment-oriented, and Computer Use receives a bundled component update.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次为历史回填：两个版本均已超出官方 Sparkle(应用更新框架) 增量窗口，因此事实包采用两个官方全量包直接比较，而不是增量包重建路径。

对比范围是 `26.601.21317` build `3511`，发布于 `2026-06-03`，到 `26.602.30954` build `3575`，发布于 `2026-06-04`。平台为 `macOS arm64`。

可信度方面，两个对比对象均来自官方全量包，并以 Apple(苹果) 公证和 OpenAI Developer ID(开发者身份签名) 作为信任根。事实包显示，上一版与最新版的 `Codex.app` 均通过 `spctl` 公证验证，来源为 `Notarized Developer ID`，签名来源为 `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`。

本文使用两档证据等级：

- **【实证】**：来自人类可读的直接证据，例如 `Info.plist`、`package.json`、插件配置、技能文档 diff(差异文本) 或签名验证结果。
- **【信号】**：来自间接证据，例如前端模块名或二进制体积变化。它们只能说明“指向某方向”，但无法从产物还原到源码级意图。

## ✨ 重点变化

### 1. Browser 与 Chrome 自动化说明被大幅收敛到“运行时文档优先” 【实证】

Browser 和 Chrome 相关技能文档从长篇内置流程，改为更短的 Bootstrap(启动初始化) 说明，并要求通过 `agent.documentation.get("<name>")` 或 `browser.documentation()` 获取具体主题文档。

这意味着浏览器自动化的指导方式从“把大量规则写死在技能文档里”，转向“启动时读取当前运行时暴露的完整能力说明”。对开发者可感知的变化是：调试、截图、确认、Playwright(浏览器自动化接口) 等说明被拆到专门文档入口，主技能文档更像路由与初始化说明。

关键证据：

```diff
+Start with the directions in the Bootstrap section below. Use `await agent.documentation.get("<name>")` when you need information about the specific topic they cover:
+nodeRepl.write(await browser.documentation());
```

Chrome 侧还新增了更明确的选择边界：只有任务依赖现有 Chrome 状态，或用户明确要求 Chrome 时才使用；如果连接器、API(应用程序接口) 或 CLI(命令行工具) 认证失效，应先让用户修复认证，或明确批准 Chrome 作为 fallback(备选方案)。

```diff
+Prefer purpose-built connectors, APIs, or CLIs. If one fails due to missing or expired authentication, ask the user to reauthenticate or explicitly approve Chrome as a fallback.
```

### 2. Browser 插件新增多篇 API 与操作文档，旧的 CDP 文档被移除 【实证】

新增文件中出现了 Browser 和 Chrome 的 `docs/api.md`、`docs/playwright.md`、`docs/confirmations.md`，Chrome 还新增 `docs/chrome-troubleshooting.md`。同时 Browser 侧移除了 `docs/capabilities/tab/cdp.md`。

这指向一个更文档化、更分层的浏览器自动化接口：能力说明不再主要塞在 `SKILL.md` 中，而是分散到 API、Playwright、确认流程、故障排查等文档文件里。需要注意的是，事实包只证明文档与分发内容变化，不能证明底层浏览器能力本身新增或删除。

关键证据：

- 新增：`Contents/Resources/plugins/openai-bundled/plugins/browser/docs/api.md`
- 新增：`Contents/Resources/plugins/openai-bundled/plugins/browser/docs/playwright.md`
- 新增：`Contents/Resources/plugins/openai-bundled/plugins/chrome/docs/chrome-troubleshooting.md`
- 移除：`Contents/Resources/plugins/openai-bundled/plugins/browser/docs/capabilities/tab/cdp.md`

### 3. Sites 插件定位从“创建和托管站点”扩展为更完整的网站构建与部署工作流 【实证】

Sites 插件版本从 `0.1.10` 升到 `0.1.12`，描述改为 “Build and deploy websites with Sites”。关键词也从较窄的 `deploy`、`sites`、`games`，扩展到 `website`、`landing page`、`portfolio`、`dashboard`、`KPI dashboard`、`portal`、`tracker`、`hub`、`internal tool` 等使用场景。

这意味着 Sites 的产品表述更偏向“通用网站与内部工具构建 / 发布”，而不只是“创建、保存、部署、检查站点”。默认提示也从餐厅落地页、数独游戏、图片生成网站，改为新员工门户、数独排行榜、客服趋势仪表盘。

关键证据：

+"version": "0.1.12",

### 4. Sites 构建制品打包方式从脚本拼接转向 Vite 插件 【实证】

两个 Sites 模板的 `package.json` 都把 `build` 脚本从一长串 `vinext build && rm -rf dist/_appgen_meta ...` 简化为 `vinext build`。与此同时，两个模板的 `vite.config.ts` 都新增 `sites()` 插件，并从 `./build/sites-vite-plugin` 导入。

这说明 Sites 兼容部署所需的元数据打包逻辑，正在从模板脚本中的手工复制步骤，迁移到 Vite(前端构建工具) 插件中。对开发者的直接影响是模板构建命令更干净，`.openai/hosting.json` 与 `drizzle` 迁移文件的放置规则也在托管说明中同步调整。

关键证据：

```diff
-"build": "vinext build && rm -rf dist/_appgen_meta && mkdir -p dist/_appgen_meta && if [ -f .openai/hosting.json ]; then cp .openai/hosting.json dist/_appgen_meta/appgarden.json; fi && if [ -d drizzle ]; then cp -R drizzle dist/_appgen_meta/drizzle; fi",
+"build": "vinext build",
```

```diff
+import { sites } from "./build/sites-vite-plugin";
...
+    sites(),
```

托管文档中的制品路径也从 `_appgen_meta` 改为 `.openai`：

```diff
-   `dist/_appgen_meta/appgarden.json`
-   `dist/_appgen_meta/drizzle/**`
+   `dist/.openai/hosting.json`
+   `dist/.openai/drizzle/**`
```

### 5. Sites 部署流程更强调“不要只停在本地构建” 【实证】

`sites-hosting` 技能文档新增要求：除非用户明确要求只保留本地结果，否则不要在可部署站点工作中只完成本地构建；应继续部署，或报告具体阻塞点并请求下一步动作。

这会影响代理执行网站任务时的默认完成标准：对“可部署站点”而言，`build` 成功不再被文档视为自然终点，部署或明确阻塞说明才是终点。

关键证据：

```diff
+Do not finish deployable site work with only a local build unless the user explicitly asks to keep the work local. Deploy the site, or report the concrete blocker and ask for the action needed to continue.
```

### 6. Computer Use 打包组件升级，相关二进制体积同步增加 【实证】

Computer Use 插件版本从 `1.0.799` 升到 `1.0.809`。同时，多个 Computer Use 相关二进制体积上升：`SkyComputerUseService` 增加 `0.74 MB`，`CUALockScreenGuardian` 增加 `0.71 MB`，`SkyComputerUseClient` 增加 `0.18 MB`。

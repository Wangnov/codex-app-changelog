# Codex 桌面版 26.602.30954 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

这一批次的核心不是单点功能，而是把 Codex 的浏览器、站点构建、远程环境和协作界面往更完整的产品面推进。  
One-liner: Codex 26.602 tightens browser/plugin runtime docs, expands Sites workflows, and surfaces broader remote-control, sharing, task, and handoff UI signals across platforms.

## 关于这份报告

本文只使用输入中的官方签名分发包逆向 diff 事实：macOS 从 26.601.21317 build 3511 到 26.602.30954 build 3575，Windows 从 26.601.2237.0 到 26.602.3474.0。两个平台版本号与打包节奏不同，结论按平台分别标注。

证据分两档：【实证】来自 plist、manifest、插件清单、技能文档、package.json、依赖清单、类型声明、服务入口与新增可读字符串；【信号】来自前端模块名、同名模块可读字符串/样式类名变化和压缩 bundle 方向线索。平台归属使用 `[macOS+Windows]`、`[仅 macOS]`、`[仅 Windows]`。

hash 命名的 webview/assets 和 .vite/build 变化已视为构建噪音，不按功能增删解读。输入中未提供独立的内容安全策略变化；因此本报告不声称 CSP 有变化。

## ✨ 重点变化

### 【实证】[macOS+Windows] Browser 与 Chrome 插件改为“运行时文档优先”的控制面

Browser 与 Chrome 的技能文档被大幅改写：原先长篇内嵌操作说明被替换为 Bootstrap 流程与 `agent.documentation.get(...)` 文档入口。新增文档主题包括 `api-troubleshooting`、`confirmations`、`playwright`、`screenshots`，Chrome 还新增 `chrome-troubleshooting` 与 `file-management`。

两端都新增了 `browser/docs/api.md`、`chrome/docs/api.md`、`browser/docs/playwright.md`、`chrome/docs/playwright.md`、确认与故障排查文档，并移除了 `browser/docs/capabilities/tab/cdp.md`。这说明能力说明从单个长技能文件拆到独立文档面，而不是简单删减。

Chrome 插件描述也同步收紧：它仍用于依赖 Chrome 现有状态的任务，如标签页、登录会话、cookies、扩展，但新增明确边界：优先使用专门 connector、API 或 CLI；如果首选方式只是缺少或过期认证，应要求用户重新认证或明确批准 Chrome 作为 fallback。

### 【实证】[macOS+Windows] Sites 插件从“创建站点”转向更广的网站构建与部署工作流

Sites 插件版本从 0.1.10 升到 0.1.12，描述从 “Create, save, deploy, and inspect sites hosted by OpenAI” 改为 “Build and deploy websites with Sites”。关键词新增 `build`、`website`、`landing page`、`portfolio`、`dashboard`、`KPI dashboard`、`portal`、`tracker`、`hub`、`internal tool`，默认提示也从披萨餐厅落地页/图片生成网站转向新员工门户、数独排行榜和客服趋势仪表盘。

`sites-building` 技能新增 Fit Check：无约束项目使用 bundled vinext starter；已有 `.openai/hosting.json` 时必须使用 Sites 并保留项目结构。模板的 `vite.config.ts` 新增 `sites()` Vite 插件，`package.json` 的 build 脚本从手动复制 `_appgen_meta` 简化为 `vinext build`。`sites-hosting` 同步把部署元数据路径从 `dist/_appgen_meta/appgarden.json` / `dist/_appgen_meta/drizzle/` 改为 `dist/.openai/hosting.json` / `dist/.openai/drizzle/`，并要求可部署站点工作不能只停在本地 build，除非用户明确要求本地保留。

平台差异：两端都有上述清单和技能文档变化；Windows 的新增文件列表额外明确列出两个模板中的 `build/sites-vite-plugin.ts` 新文件，macOS 输入的新增文件节选未列出它们。

### 【实证】[macOS+Windows] 版本、构建号与插件清单同步进入 26.602 系列

macOS Info.plist 显示 `CFBundleShortVersionString` 从 26.601.21317 到 26.602.30954，`CFBundleVersion` 从 3511 到 3575，app.asar integrity hash 同步变化。根 `package.json` 版本也从 26.601.21317 到 26.602.30954，`codexBuildNumber` 从 3511 到 3575。

Browser 与 Chrome 插件版本两端均从 26.601.21317 更新到 26.602.30954。Computer Use 在 macOS 上从 1.0.799 到 1.0.809；Windows 上插件版本从 26.601.21317 到 26.602.30954。Windows 的 @oai/sky 包还从 `0.4.5-26566467949-4daa3290021d` 到 `0.4.9-26913854413-62c399f4cf2c`。

### 【实证】[仅 Windows] Computer Use 的 @oai/sky 依赖与截图数据类型发生可读变化

Windows 包内 @oai/sky 的 package.json 和 package-lock 显示版本升级到 `0.4.9-26913854413-62c399f4cf2c`，并移除了 `tslib` 直接依赖；文件树中也移除了多份 `node_modules/tslib` 文件。macOS 的 `cua_node 依赖变化`只给出“新增 0 个包”，未给出同等 @oai/sky package diff。

Windows 的 @oai/sky 文档和类型声明把 Screenshot 的 `data_url` 注释从 “Base64-encoded PNG data URL” 改为 “Base64-encoded JPEG data URL”，Linux screenshot 实现也把 `to_data_url` 调用改为带 `image/jpeg`。这只能证明 Windows 分发包内的跨平台 sky JS 代码与文档发生了这些变化，不等同于 macOS 原生截图链路也一定改变。

### 【实证】[macOS+Windows] 主进程/前端 bundle 新增了 Git、WSL、更新状态与 AppView RPC 相关字符串

两端新增可读字符串一致：`Expected Git object hash`、`diff --cc `、`diff --combined ` 指向 Git 对象校验与 combined diff 处理；`Whether Codex runs inside WSL on Windows` 指向 Windows/WSL 环境识别；`Failed to publish app update state to renderer` 与 `Failed to register AppView RPC services` 指向更新状态发布和 AppView RPC 注册错误路径；`Fetch timing` 指向请求计时观测；`api.md` 与新增插件文档文件互相印证。

这些字符串确凿进入包内，但它们来自压缩 bundle，不能还原完整源码语义，也不能断言所有用户界面已开放。

### 【信号】[macOS+Windows] 远程控制与移动配对 UI 更明确地区分 macOS 与非 macOS

`codex-mobile-setup-dialog` 同名模块两端都新增了刷新 remote-control pairing code、加载指示、`https://chatgpt.com/codex/pair`、`ChatGPT auth is required to load remote control environments.`，并把等待标题拆成 macOS 与 non-macOS 两套。用户可见文案还包括 “Approve on your device to control this Mac remotely” 与 “Approve on your device to control this computer remotely”。

这是高可见 UI 信号：远程控制配对流程在两端都被强化，并出现跨平台文案分支；但事实包没有给出后端开关或可用范围，不能断言所有账号可见。

### 【信号】[macOS+Windows] 本地/远程 handoff 状态文案从过程型转为结果型

`local-remote-dropdown` 两端新增 “Handing off to local/worktree”、“Handed-off to local/worktree”、“Hand-off ... failed/needs attention”，并移除 “Copying files”、“Creating a new worktree”、“Applying uncommitted changes” 等过程步骤文案。

这说明界面表达更聚焦 handoff 结果、失败和需注意状态。新增模块 `thread-handoff-step-row`、`local-remote-selection`、`get-move-to-local-targets` 与这一方向一致。

### 【信号】[macOS+Windows] 最近任务、云任务与任务更新入口进入头部区域

`header` 模块新增 “recent tasks menu”、“unread task updates”、“All tasks”、“Cloud tasks”、“All environments filter option”、“Clear search” 等字符串，并新增 aria-live/aria-atomic 相关样式状态。该模块只在 macOS 的同名模块列表中展开，但 Windows 的新增模块集合与多语言字符串同样包含 `codex.recentTasksMenu.triggerWithUnread`。

这是任务菜单和任务更新提示的前端信号；还不能证明具体任务系统的完整行为。

### 【信号】[macOS+Windows] 资料页新增活动资料卡预览和社交分享出口

`profile` 模块两端新增 profile share card preview dialog、保存、分享、LinkedIn、Reddit、X 分享按钮、生成预览图 alt text 等字符串，并新增 `aspect-[499/306]`、预览 hover target 等样式类名。移除 day streak 相关字符串。

这说明资料页更偏向活动卡片生成和外部分享，而不是单纯连续天数展示。事实包只能证明 UI 文案和样式进入 bundle。

### 【信号】[macOS+Windows] Remote connections 获得颜色选择器

`home-project-recent-chats-signals` 两端新增 remote connection color picker、hex color 输入、reset to default、change connection color 等字符串，移除开发用 active/resumed dots 和部分系统文件管理器打开文案。新增模块 `selectable-remote-connections-signal` 与此互相印证。

这属于远程连接列表可定制颜色的 UI 信号。

### 【信号】[macOS+Windows] 设置页出现更多工作模式、字体、dictation 与权限开关

macOS 的 `general-settings` 模块新增 disabled default permissions toggle、composer context usage indicator、work mode settings、Agent environment、UI/code base size、hold-to-dictate hotkey、dictation dictionary entry、Popout Window projectless default 等字符串。Windows 同名模块未在节选中展开，但两端新增模块集合包含 `use-is-dictation-supported`、`model-and-reasoning-effort-translations` 与 `platform`。

因此可确认 macOS 样本内设置页变化较清楚；跨平台只能作为共同前端方向信号，不能把 macOS 该模块的全部字符串直接套到 Windows。

### 【信号】[macOS+Windows] 用户附件、PR 检查与图片下载失败状态更细

`user-message-attachments` 两端新增图片附件加载失败 alt text、短文本、下载诊断 “started / URL resolved / bytes fetched / failed”，以及 PR merge conflicts、failing PR checks 的 pill 文案；旧的 chip 文案被移除。

这指向本地会话中的图片附件错误处理和 PR 相关附件展示细化。

### 【信号】[macOS+Windows] CI、Git 操作和浏览器设备工具栏的侧栏能力信号增强

`thread-side-panel-tabs` 两端体积增加约 0.3 MB，新增字符串包括 CI 日志诊断流程提示、凭据/权限/token/MCP/tool access 检查、`A pull request already exists for this branch`、`Accept edits to stage them`、取消 active git action、浏览器 device toolbar 宽/高/zoom 输入标签等。

这是一组较强的工作流信号：CI 修复引导、Git staging、浏览器设备模拟控制都进入侧栏相关 bundle。由于它来自同名前端模块改动线索，应标为【信号】。

### 【信号】[仅 Windows] 站点分享对话框出现 workspace 与公开链接权限文案

Windows 的 `appgen-share-dialog` 新增 “Add people or groups”、“Anyone in this workspace with the link”、“Anyone on the internet”、“Can visit”、“Copy link”、“Copied to clipboard”、“opening a live site externally”等字符串。macOS 的同名模块未在输入节选中列为变化；因此这里只标 Windows 独有信号。

这与 Sites 插件的构建/部署增强方向一致，但仍不能断言分享权限后端已经对所有用户开放。

### 【信号】[仅 Windows] @ mention 列表出现 Computer Use、Custom agents、MCP servers 与 Desktop/Mac apps 分组

Windows 的 `at-mention-list` 新增 “Computer use”、“Custom agents”、“Desktop apps”、“Live agents”、“MCP servers”、“Mac apps”、“No results” 以及 MCP mention scope 相关字符串和 `mcp-capability-mentions` 类名。macOS 输入未列出该模块变化。

这说明 Windows 前端包中 mention 选择器正在把文件、agent、MCP server 和桌面能力放进同一类入口信号中。

### 【实证】[macOS+Windows] 多语言资源同步出现 Business、Pro、worktree、计费与附件错误文案

两端大量 locale bundle 新增 Business Codex、Frontier Pro model、高级模型、自动代码和安全审查、Computer automation、worktree chat、merge conflicts、billing/help link、图片附件失败、recent tasks unread、Appgen announcement 等文案，同时移除 guided tour、required apps/browser extensions、raw/rich viewer、旧 worktree thread 与旧 upgrade modal 文案。

这是字符串实证：这些多语言文案确凿进入包内。它们代表产品文案准备和界面方向，但不保证对应套餐、入口或功能对所有用户可见。

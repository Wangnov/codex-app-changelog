# Codex 桌面版 26.616.32156 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

这批差异显示: macOS 是向 26.616.32156 前进的一组更新,Windows 样本虽然来自同一批镜像,但前端模块信号与 macOS 多处呈反向变化,需要按平台分别解读。

One-liner: this cross-platform batch bumps shared bundled plugin metadata, keeps Computer Use runtimes stable, and shows divergent frontend UI signals between macOS and Windows.

## 关于这份报告

本文只使用输入事实包中的机器提取事实。macOS 来源为官方全量包对比,版本为 26.616.31447 build 4133 到 26.616.32156 build 4157; Windows 来源为 x64 MSIX 全量包对比,版本为 26.616.3767.0 到 26.616.4196.0。

证据分两档:【实证】来自 plist、manifest、插件配置、依赖清单、服务入口、签名验证、可读字符串和关键文件 diff;【信号】来自同名前端模块、体积变化、样式类名和压缩 bundle 方向线索。平台归属用 `[macOS+Windows]`、`[仅 macOS]`、`[仅 Windows]` 标出。

对比范围覆盖版本与渠道、bundle 文件树、内置插件 manifest、Computer Use 运行时、cua_node 依赖、app.asar、同名前端模块、可读字符串、Info.plist 或 AppxManifest 变化、内容安全策略、新增组件与资源、关键文件具体改动。输入中未提供 AppxManifest、CSP、新增内置插件明细或新增组件资源明细,因此本文不扩展推断。

## ✨ 重点变化

### 【实证】[macOS+Windows] 内置 Browser 与 Chrome 插件版本推进到 26.616.32156

两个平台的 bundled plugin manifest 都显示 `browser` 与 `chrome` 插件版本从 26.616.31447 更新到 26.616.32156。Browser 插件说明仍围绕 in-app browser、localhost、file URL、截图、点击、输入和本地 web 目标检查; Chrome 插件说明仍指向依赖现有 Chrome 状态的自动化,包括标签页、登录会话、cookies、扩展和 Codex Chrome Extension setup。

平台差异是路径和宿主包不同: macOS 位于 `Contents/Resources/plugins/openai-bundled/plugins/...`,Windows 位于 `resources/plugins/openai-bundled/plugins/...`。Windows 额外出现 `computer-use` 插件 manifest 版本推进,macOS 事实包没有给出对应 diff。

### 【实证】[仅 Windows] Computer Use 插件 manifest 版本推进,描述限定为 Windows 桌面控制

Windows 的 `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` 显示版本从 26.616.31447 更新到 26.616.32156,描述为通过 Computer Use 从 Codex 控制 Windows 桌面应用。

这是一条 Windows 独有的 manifest 证据。macOS 输入只给出了 Computer Use 运行时总量,没有给出 computer-use 插件 manifest 变化。

### 【实证】[macOS+Windows] Computer Use 运行时体积与 cua_node 依赖没有增长

macOS 的 Computer Use 运行时总量为 365.9 MB → 365.9 MB,Windows 为 285.6 MB → 285.6 MB,两边都是 +0 MB、+0%。cua_node 依赖新增包数量两边均为 0。

这说明本批次输入样本中没有看到 Computer Use 运行时文件总量或 Node 依赖包数量扩张。两端绝对体积不同: macOS 365.9 MB,Windows 285.6 MB。

### 【实证】[仅 macOS] 应用版本、build 与 app.asar 完整性哈希更新

macOS 的 `Info.plist` 显示 `CFBundleShortVersionString` 从 26.616.31447 到 26.616.32156,`CFBundleVersion` 从 4133 到 4157。`ElectronAsarIntegrity/Resources/app.asar/hash` 也从旧哈希变为新哈希。

这确认 macOS 包元数据和 app.asar 完整性记录已更新。Windows 输入没有提供 AppxManifest 变化,因此不能在 Windows 侧写同类 manifest 结论。

### 【信号】[仅 macOS] Debug Modal 扩张,出现账户、会话、浏览器标签和 host 许可相关文案

macOS 的 `debug-modal` 从近 0 MB 增至约 0.2 MB,新增字符串包括 `(command unavailable)`、`Account ID`、`Active browser tab`、`Active conversation`、`Ambient suggestion chats`、`App Server`、`App session ID`、`Allow host`、`Allow host starting` 和 `Allow automatic worktree cleanup in this unpackaged build.`。

这些是已进入 macOS 前端 bundle 的字符串,但它们来自同名前端模块改动线索,所以只能作为 UI 和调试面变化信号。Windows 同一模块方向相反,移除了对应一组 debug-modal 字符串和样式类名。

### 【信号】[仅 macOS] Header 出现近期任务、云任务、设置入口和 Chrome 扩展线程入口信号

macOS 的 `header` 模块新增 `All tasks`、`Cloud tasks`、`App settings`、`Clear search`、`All environments filter option`,以及打开 recent tasks menu、从 header 打开 settings、在 Codex 桌面应用打开 active Chrome extension thread 的按钮标签。

这提示 macOS header 相关界面可能在强化任务菜单、环境过滤、设置入口和 Chrome 扩展线程入口。Windows 的 `header` 模块则移除了这些字符串,因此不能写成跨平台共同增强。

### 【信号】[仅 macOS] Thread App Shell 出现底部面板与侧边面板新标签动作信号

macOS 的 `thread-app-shell-chrome` 新增 `Bottom panel`,以及从 thread side panel 打开 side chat、browser tab、review tab、terminal、timeline tab、workspace file browser 的动作标签,并出现 `(max-width: 920px)`。

这是一组布局与导航信号:它指向线程界面中的侧边面板、底部面板和多类工作区入口。Windows 同名模块移除了同组字符串,所以该信号只归于 macOS。

### 【信号】[仅 macOS] Skills 页面出现安装技能、推荐技能、刷新与空状态文案

macOS 的 `skills-page` 新增创建新 skill、重新加载 skills list、新安装 skills 需要刷新后才能在 composer 使用、installed skills、recommended skills、搜索输入、加载状态和过滤空状态等文案。

这说明 macOS 包内出现了更完整的 Skills 页面 UI 文案信号。Windows 的 `skills-page` 对应移除了一组相同文案,包括创建、刷新、空状态、页面标题和 installed skills 标题。

### 【信号】[仅 Windows] Plugin Detail Page 扩张,出现插件分享、MCP 开关和 Skill 预览动作

Windows 的 `plugin-detail-page` 从近 0 MB 增至约 0.1 MB,新增 `Add plugin`、`Add people or groups`、workspace link 共享文案、MCP server enablement toggle、section navigation、`App unavailable`、skill preview modal more actions、skill card actions menu trigger 和返回按钮标签。

这些字符串是 Windows 前端模块的强 UI 信号,指向插件详情页、共享对象、MCP 开关和 skill 卡片操作。macOS 的同名模块则移除了 workspace share、MCP 开关和 Add plugin 等字符串,因此这里是 Windows 独有方向。

### 【信号】[仅 Windows] General Settings 扩张,出现权限、上下文用量、代码字号、听写、实时语音和 Popout Window 设置

Windows 的 `general-settings` 从近 0 MB 增至约 0.1 MB,新增 disabled default permissions toggle、composer context usage indicator、work mode settings、code across chats and diffs 的 base size、hold-to-dictate hotkey、realtime voice hotkey、wake word、dictation dictionary entry 和 Popout Window projectless default setting toggle。

这是一组设置页信号,覆盖权限、上下文显示、代码字号、语音输入和弹出窗口默认行为。macOS 同名模块方向相反,移除了这些设置文案中的一部分。

### 【信号】[仅 Windows] Browser Use Settings 扩张,出现历史访问、下载域名白名单和嵌入式浏览器设置

Windows 独有的 `browser-use-settings` 模块增至约 0.1 MB,新增 `Access history without asking`、embedded browser downloads、breadcrumbs、contact info settings、extension settings、password manager settings、site settings、data clearing options、unavailable plugins disabled toggle、`Add allowed domain` 与 `Add allowed download domain`。

这是一组高可见浏览器操作和隐私/权限设置信号。macOS 输入没有列出同名模块变化,因此不应套用到 macOS。

### 【信号】[macOS+Windows] Login Route 都移除了“Included with all ChatGPT plans”相关 onboarding 文案

macOS 与 Windows 的 `login-route` 模块都移除了 `Included with all ChatGPT plans`、`Badge text on the v2 desktop onboarding login page indicating Codex is included with ChatGPT plans` 和 `electron.onboarding.login.includedPlans.welcomeV2`。

这是一条两平台同向的前端信号:登录或 onboarding 页面中关于 ChatGPT plans 包含 Codex 的徽标文案被移出同名模块。因为来源是同名前端模块差异,不能进一步断言实际登录页一定删除了该展示。

### 【信号】[macOS+Windows] 权限模式样式出现 `not-thread`,服务层图标模块移除同名类

两平台的 `use-permissions-mode` 都新增样式/状态类名 `not-thread`;两平台的 `service-tier-icons` 都移除 `not-thread`。

这是一条较低层 UI 状态信号,可能与非线程上下文下的权限模式展示有关,但事实包没有提供可读文案支撑,因此不展开为功能结论。

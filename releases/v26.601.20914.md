# Codex 桌面版 `26.601.20914`(build 3497)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版最明确的变化集中在安装体验、浏览器控制策略、插件入口，以及更新/关于界面的本地化文案上。  
One-liner: installer handling, browser-control policy, plugin surface, and localized update UI all moved in this build.

## 关于这份报告

这不是 OpenAI(人工智能公司) 官方变更日志，而是一份基于官方签名分发产物的第三方逆向差异报告。事实包说明，本次是历史回填：两个版本都超出官方 Sparkle(桌面应用更新框架) 增量窗口，因此采用两个官方全量包直接比较，而不是增量包重建。

可信度方面，上一版与最新版的 Codex(代码代理桌面应用) 应用包均通过 Apple(苹果公司) 公证验证，结果为 `accepted`，来源为 `Notarized Developer ID`，签名主体为 `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`。报告中的 `codesign`(代码签名验证工具) / `spctl`(门禁策略验证工具) 结论均来自事实包。

证据等级分两档：

- **【实证】**：来自 Info.plist(macOS 应用元数据配置文件)、配置文件、技能文档、`package.json` 或可读差异文本的直接证据。
- **【信号】**：来自前端模块名或体积变化等间接证据；这类内容只说明“指向某方向”，但无法从产物还原到源码级意图。

对比范围为 `26.527.60818` build 3437，发布于 2026-06-01，到 `26.601.20914` build 3497，发布于 2026-06-02，平台为 macOS arm64。

## ✨ 重点变化

### 1. 启动阶段新增 macOS 安装迁移与 DMG 清理流程 【实证】

最新版在 `.vite/build/bootstrap.js` 中加入了一整段 macOS 启动前安装处理逻辑：当应用未位于 Applications(应用程序文件夹) 时，会尝试把 App Bundle(macOS 应用目录包) 移入或复制到 `/Applications`，显示安装窗口，并在后续处理来源 DMG(磁盘镜像安装包)。

这意味着用户从挂载镜像中直接启动时，应用可能会主动引导安装到 Applications，并在成功后打开安装后的应用；失败时也有对应状态文案。

关键证据包括：

```text
pending-source-dmg-cleanup.json
Installing ${t}
Moving ${t} to Applications…
hdiutil detach
shell.trashItem
```

### 2. Browser / Chrome 自动化的 CAPTCHA 策略变更 【实证】

Browser(内置浏览器插件) 与 Chrome(谷歌浏览器插件) 的技能文档都改了同一条规则：旧版禁止解决 CAPTCHA(验证码)，新版改为“每看到一个 CAPTCHA，都先询问用户是否要处理；只有用户确认后才解决”。

这对开发者和使用自动化浏览器能力的用户都很可感知：验证码不再是绝对禁止项，但必须逐次获得用户确认；绕过付费墙、安全拦截、年龄验证、最终密码修改步骤仍然被禁止。

证据来自两个技能文档的同向 diff：

```diff
- Do not solve CAPTCHAs ...
+ For each CAPTCHA you see, ask the user whether they want you to solve it. Solve that CAPTCHA only after they confirm.
```

### 3. 新增 `sites` 插件入口 【实证】

内置插件市场配置新增了一个名为 `sites` 的插件条目，来源为本地路径 `./plugins/sites`，安装策略为 `AVAILABLE`，认证策略为 `ON_INSTALL`，分类为 `Productivity`。

这说明官方分发包里已经出现了一个可安装的 Sites(站点/网站相关能力) 插件入口；但事实包只提供 marketplace 配置，不能进一步确认具体能力范围或是否已对所有用户开放。

关键证据：

```json
"name": "sites",
"path": "./plugins/sites",
"installation": "AVAILABLE",
"authentication": "ON_INSTALL",
"category": "Productivity"
```

### 4. 更新确认与关于窗口文案扩展到多语言包 【实证】

多份 `native-menu-locales` 本地化文件新增了两组用户界面文案：一组是安装更新前的确认文案，另一组是关于窗口中的版本、发布日期和构建信息。

这意味着更新流程会更明确地提示“Codex 将退出以安装更新，并中断本机正在进行的会话”；关于窗口也具备展示版本和发布日期的本地化文案。

中文包中的直接证据：

```json
"appHeader.installUpdate.confirmTitle": "现在更新 Codex？"
"appHeader.installUpdate.confirmSubtitle": "Codex 将退出以安装更新，这将中断此设备上正在进行的本地会话"
"codex.aboutDialog.versionLineWithDate": "版本 {version} • 发布于 {releaseDate}"
```

### 5. 计划、价格、升级和服务等级相关前端模块出现 【信号】

去掉 Vite(前端构建工具) hash 噪音后，新增前端模块中出现了 `plan-management-state`、`plan-pricing`、`upgrade-plan-dialog`、`get-codex-purchase-handoff-url`、`read-service-tier-for-request`、`service-tier-icons`、`use-codex-pricing-url` 等名称。

这些模块名指向订阅计划、价格展示、升级弹窗、购买交接和服务等级展示方向，但无法从产物还原到源码级意图，也不能判断这些入口是否已经对用户开放。

### 6. 浏览器侧栏、Appgen、线程渲染和编辑器集成出现新前端信号 【信号】

新增模块还包括 `browser-sidebar-webview`、`browser-sidebar-retained-webview`、`browser-sidebar-hidden-background-webview-host`、`appgen-access-state-messages`、`appshot-availability`、`thread-virtualizer`、`thread-resource-card`、`pull-request-fix-button`、`vscode-api` 等。

这些模块名指向浏览器侧栏状态保留、Appgen/Appshot(应用生成/应用截图相关能力) 可用性、线程虚拟化渲染、拉取请求修复入口和 VS Code(代码编辑器) 集成方向，但无法从产物还原到源码级意图，也不能证明这些能力已经完整上线。

### 7. Chromium 底层运行时升级 【实证】

`Info.plist` 显示 Chromium(浏览器内核) 基础版本从 `148.0.7778.179` 升至 `149.0.7827.54`，对应 `ChromiumBaseBundleVersion` 从 `7778.179` 升至 `7827.54`。

这属于底层运行时变化，可能影响内置网页视图、浏览器侧栏或渲染行为，但事实包没有提供具体用户可见修复项。

关键证据：

```diff
/ChromiumBaseVersion: '148.0.7778.179' → '149.0.7827.54'
/ChromiumBaseBundleVersion: '7778.179' → '7827.54'
```

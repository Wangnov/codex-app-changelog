# Codex 桌面版 `26.415.21839` build 1763

> 第三方变更日志 · 非官方 · 基于官方分发包逆向差异

这一版从 `26.415.20818` 升至 `26.415.21839`，主要变化集中在工作区文件树、文件打开方式，以及一个新增的原生 helper。  
English one-liner: this build adds a native Launch Services helper and refines workspace file-tree/open-with flows.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次是历史回填：官方 appcast 未对 `26.415.20818` → `26.415.21839` 提供增量包，因此改为下载两个官方全量包直接比较。

证据等级分两档：

- **【实证】**：来自 Info.plist、package.json diff、签名/公证结果、bundle 中新增的可读字符串、实际新增文件等人类可读证据。
- **【信号】**：来自同名前端模块的字符串、样式类名或体积变化。它们指向某方向，但无法从产物还原到源码级意图。

对比范围为 macOS arm64，上一版 `26.415.20818` build `1727`，最新版 `26.415.21839` build `1763`。

## ✨ 重点变化

### 1. 新增 macOS Launch Services 原生 helper 【实证】

`Contents/Resources/native/launch-services-helper` 是本版新增文件，大小约 `0.1 MB`。`package.json` 的 `dev` 与 `owl` 脚本也新增了 `pnpm exec node ./scripts/build-launch-services-helper.mjs`。

这意味着 Codex 桌面版开始随包携带一个面向系统 Launch Services 的原生辅助组件。结合前端同名模块中出现的“用指定 app 打开工作区文件”“用首选 app 打开工作区文件”“加载打开文件的 app 选项”等文案线索【信号】，这一组变化指向工作区文件打开体验的增强，但无法从产物还原到完整源码意图。

关键证据：

```diff
- "dev": "pnpm run rebuild:sqlite && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
+ "dev": "pnpm run rebuild:sqlite && pnpm exec node ./scripts/build-launch-services-helper.mjs && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
```

### 2. 工作区文件树被重新强调为独立侧栏能力 【信号】

同名前端模块 `index` 中新增了 `Toggle File Tree`、`Select file tree section`、`Label for showing all files in the thread file tree panel`、`Label for showing changed files in the thread file tree panel` 等文案线索，同时移除了多条以 `thread summary right panel` 为中心的旧文案。

这指向右侧面板中“文件树”与“摘要”之间的组织方式发生调整，文件树入口、筛选和空目录状态更明确。但这些线索来自同名前端模块改动，只能作为 UI、状态、布局或交互方向的【信号】。

补充证据：主进程/前端 bundle 新增可读字符串 `Toggle File Tree`【实证】。

### 3. 工作区文件浏览器加入更细的打开方式菜单线索 【信号】

`index` 模块新增了与工作区浏览器文件右键菜单相关的文案，包括“添加文件到当前聊天”“在指定 app 中打开”“在首选 app 中打开”“选择用于打开文件的 app 子菜单”等。

这说明工作区文件浏览器可能在上下文菜单和外部 app 打开流程上更完整。由于这些来自同名前端模块，仍只能判断为指向交互能力增强的【信号】，不能直接断言源码实现细节。

### 4. 主进程 bundle 出现 Windows Store 更新检查路径 【实证】

新版 bundle 新增了 `Checking Windows Store for package updates` 与 `Windows Store package update check completed` 两个可读字符串。

虽然本次分析对象是 macOS arm64 包，但这些字符串确凿存在于新版产物中，说明共享代码或打包逻辑中包含 Windows Store 包更新检查路径。它不代表 macOS 版本新增了对应可见界面。

### 5. 设置页相关前端模块大幅收缩，疑似设置项迁移或拆分 【信号】

`general-settings` 模块体积从约 `0.1 MB` 降至约 `0.0 MB`，减少 `93277 bytes`，并移除了多条设置页文案，例如 `Agent environment`、`Allow Codex to control the in-app browser for browser tasks. Restart Codex after changing this setting.`、`Allows other signed-in Codex clients to connect to this computer`、代码字号与 UI 字号调整说明等。

这指向设置页内部结构、加载边界或相关设置项呈现方式发生变化，但这些证据来自同名前端模块差异，无法确认是删除、迁移、懒加载还是重组。

### 6. 版本号、build 与 app.asar 完整性哈希更新 【实证】

`Info.plist` 中 `CFBundleShortVersionString` 从 `26.415.20818` 更新到 `26.415.21839`，`CFBundleVersion` 从 `1727` 更新到 `1763`。`ElectronAsarIntegrity/Resources/app.asar/hash` 也发生变化。

这确认本次是一次正式构建更新，而不是仅替换外层签名或元数据。

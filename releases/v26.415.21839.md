# Codex 桌面版 `26.415.21839`(build 1763)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这是一次很小的桌面端更新：可确认的用户 / 开发者可感知变化集中在新增一个原生 `launch-services-helper` 辅助组件，以及构建流程开始显式生成它。  
English one-liner: a small macOS arm64 update centered on a new native Launch Services helper.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次是历史回填：两个版本均已超出官方 Sparkle(应用更新框架) 增量窗口，因此事实包显示采用的是两个官方全量包直接比较，而不是增量包重建。

对比范围为 `26.415.20818` build 1727，发布于 2026-04-16，到 `26.415.21839` build 1763，发布于 2026-04-17。平台为 macOS arm64(苹果芯片架构)。

可信度方面，两个应用包均通过 Apple(苹果) 公证与 OpenAI Developer ID(开发者身份签名) 验证。上一版与最新版的 `spctl` 结果均为 `accepted`，来源均为 `Notarized Developer ID`，签名来源为 `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`。

本文使用两档证据等级：

- **【实证】**：来自人类可读的直接证据，例如 `Info.plist`、`package.json`、签名 / 公证结果、文件路径与哈希。
- **【信号】**：来自间接证据，例如前端模块名或二进制体积变化；这类信息只能说明“指向某方向，但无法从产物还原到源码级意图”。

## ✨ 重点变化

### 1. 新增原生 Launch Services 辅助程序 【实证】

`Contents/Resources/native/` 下新增了一个文件：`launch-services-helper`，体积约 0.1 MB。与此同时，`package.json` 中的开发与 `owl` 启动脚本都新增了 `pnpm exec node ./scripts/build-launch-services-helper.mjs` 这一步。

这意味着 Codex(代码智能体应用) 桌面版开始把一个与 Launch Services(macOS 应用启动与服务发现机制) 相关的原生辅助程序纳入构建与分发流程。事实包没有给出该 helper 的源码或调用点，因此不能进一步断言它具体负责打开文件、注册协议、唤起应用或其他系统集成行为。

关键证据：

```diff
+    "dev": "pnpm run rebuild:sqlite && pnpm exec node ./scripts/build-launch-services-helper.mjs && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
+    "owl": "pnpm exec node ./scripts/build-launch-services-helper.mjs && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development node ./scripts/owl-shell.mjs run",
```

新增文件：

```text
Contents/Resources/native/launch-services-helper (0.1 MB)
```

### 2. 版本号与构建号推进 【实证】

应用版本从 `26.415.20818` 更新到 `26.415.21839`，构建号从 `1727` 更新到 `1763`。这一点同时出现在 `Info.plist` 与 `package.json` 中。

这属于标准发布推进，但它也说明本次不是只替换签名或元数据：bundle(应用包) 文件树中确实出现了 1 个新增文件，并有 39 个内容改动。

### 3. app.asar 完整性哈希更新 【实证】

`Info.plist` 中 `ElectronAsarIntegrity/Resources/app.asar/hash` 发生变化，说明 Electron ASAR(Electron 应用资源归档) 内容完整性记录随本版重新生成。

这通常是应用层打包内容变化后的结果。事实包只给出 app.asar 内部有 6 个文件内容改动，以及大量 Vite(前端构建工具) hash 文件名增减属于噪音；因此这里不把 `webview/assets` 与 `.vite/build` 下的 hash 文件变化解读为新功能。

关键证据：

```text
ElectronAsarIntegrity/Resources/app.asar/hash:
5e8423d4df65bc7af56701e76fc28c6431d5dcaf63c54cc60708675e315e7d8d
→
21600e64c2c0b63f5f167117a0e65047c5cde8727b776d5a9671f63a742b0798
```

### 4. 依赖层没有新增 cua_node 包 【实证】

事实包显示 `cua_node` 新增依赖数量为 0。这意味着本次可见变化没有通过新增 `cua_node` 依赖包来扩展能力。

结合新增的 `launch-services-helper`，本版更像是加入一个很小的原生辅助分发单元，而不是引入一组新的大依赖能力。

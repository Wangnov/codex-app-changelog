# Codex 桌面版 `26.305.950` build 863

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版的主要变化集中在桌面端集成能力、设置页配置流和前端分包结构上，同时底层构建工具链有明确升级。  
One-liner: this build points to broader desktop integration and app-server driven UI surfaces, backed by a newer build pipeline.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次对比范围是 macOS arm64 的 `26.303.1606` build `806` 到 `26.305.950` build `863`。

事实来源包括官方全量包重建后的 bundle 文件树、`Info.plist`、`package.json`、`webview/index.html`、preload bridge、主进程 / 前端 bundle 中新增的可读字符串，以及去除 Vite hash 噪音后的前端模块名变化。

证据等级分两档：

- **【实证】**：来自人类可读配置、明确字符串、CSP、类型化入口、依赖 diff、签名 / 验证结果等直接证据。
- **【信号】**：来自前端模块名、同名前端模块的字符串 / 样式差异、二进制体积变化等间接证据。它们指向某方向，但无法从产物还原到源码级意图。

`webview/assets` 和 `.vite/build` 下带 hash 的文件名增减主要是构建产物重命名，本文不把这些文件级增删写成新增或删除功能。

## ✨ 重点变化

### 1. 前端入口改为大规模 modulepreload 分包 【实证】

`webview/index.html` 从单一主脚本加单一 CSS，变为显式预加载大量前端模块和多份样式文件。新增预加载项包括 `app-server-manager-hooks`、`playwright`、`automation-dialog`、`skills-page`、`settings-surface`、`general-settings`、`app-connect-modal`、`experimental-features-queries` 等。

这意味着桌面端前端 bundle 的加载结构发生了明显调整，更多功能面被拆成独立可加载模块；用户可能感知为相关页面或对话框的首次加载路径变化，但仅凭产物不能确认具体交互是否已对所有用户开放。

关键证据：

```diff
- <script type="module" crossorigin src="./assets/index-ByySpJCS.js"></script>
- <link rel="stylesheet" crossorigin href="./assets/index-A-iTIRzs.css">
+ <script type="module" crossorigin src="./assets/index-CMu6BCpo.js"></script>
+ <link rel="modulepreload" crossorigin href="./assets/app-server-manager-hooks-CAvrQB7n.js">
+ <link rel="modulepreload" crossorigin href="./assets/playwright-DGR8jDDG.js">
+ <link rel="modulepreload" crossorigin href="./assets/automation-dialog-BIAeyC_e.js">
+ <link rel="modulepreload" crossorigin href="./assets/skills-page-DeBxXSaK.js">
```

### 2. App Server 与本地环境配置入口浮现 【实证】

新版主进程 bundle 新增了 `codex app-server`、`environment.toml`、`codex.localEnvironmentConfigPath`、`features.unified_exec` 等可读字符串，前端真实新增模块中也出现 `app-server-manager-hooks`、`codex-api`、`local-environment-action-icon-options`、`worktree-paths`。

这组证据指向桌面端正在强化本地 App Server、统一执行路径和本地环境配置管理。由于这些字符串来自压缩后的产物，只能确认相关入口和文案存在，不能还原完整业务逻辑。

关键证据：`codex app-server`、`environment.toml`、`features.unified_exec`、`app-server-manager-hooks`。

### 3. 自动化与浏览器相关 UI 面扩大 【信号】

去噪后的新增前端模块包含 `automation-dialog`、`browser`、`playwright`，同时 `webview/index.html` 明确预加载 `automation-dialog` 和 `playwright` 相关 bundle。

这指向桌面端自动化对话和浏览器自动化相关界面在本版被纳入前端加载图，但无法从产物还原到源码级意图，也不能确认功能是否默认开放。

关键证据：新增模块 `automation-dialog`、`browser`、`playwright`。

### 4. 设置页新增外部 Agent 配置导入流程 【实证】

`agent-settings` 同名前端模块新增了多条与外部配置迁移相关的用户可见字符串，包括 `Apply selected`、`Import selected config. Restart Codex to apply all changes.`、`Imported external settings`、`Unable to import external settings`。

这意味着设置页中出现了导入 home-scoped 外部配置的流程文案，用户可感知点是选择配置项、导入后提示重启，以及失败 toast。

关键证据：

```text
Import selected config. Restart Codex to apply all changes.
Imported external settings
Unable to import external settings
```

### 5. MCP HTTP 配置支持 headers 与 bearer token 环境变量 【实证】

`mcp-settings` 新增了 `Add header`、`Add variable`、`Headers from environment variables`、`Bearer token env var`、`Label for MCP HTTP URL` 等字符串。

这表明 MCP server 详情页的 HTTP 配置面扩展到 headers、环境变量 headers 和 bearer token 环境变量。开发者可感知的变化是远程 MCP HTTP 接入配置更细。

关键证据：

```text
Headers from environment variables
Bearer token env var
Label for MCP HTTP URL
```

### 6. Usage 设置里的自动充值文案重新组织 【信号】

`usage-settings` 模块新增 `Auto-reload credit`、`Auto-reload minimum balance`、`Auto-reload target balance`，并移除了多条旧的 auto top-up 输入说明和 `Credits remaining` 文案。

这指向 Usage 设置页中自动充值表述和布局发生调整，但这些线索来自同名前端模块 diff，只能作为 UI / 状态 / 交互方向信号，不能当作源码级事实。

关键证据：`Auto-reload credit`、`Auto-reload minimum balance`、`Auto-reload target balance`。

### 7. 远程 SSH 与 GitHub CLI 探测相关错误路径增加 【实证】

主进程 bundle 新增了 `Failed to discover remote SSH connections.`、`failed to refresh discovered remote SSH connections`、`Failed to parse GitHub CLI response`、`Unexpected GitHub CLI response`、`gh auth login`、`not logged into any github hosts`。

这表明新版产物中存在远程 SSH 发现与 GitHub CLI 响应处理的错误路径。可感知点可能是连接发现、认证或失败提示更具体，但完整流程无法从压缩产物还原。

关键证据：`Failed to discover remote SSH connections.`、`Failed to parse GitHub CLI response`、`gh auth login`。

### 8. Windows 集成字符串进入主进程产物 【实证】

虽然本次比较平台是 macOS arm64，新版主进程 bundle 中新增了大量 Windows 相关字符串，包括 `Failed to register Windows folder context menu`、`Open project in Codex`、`File Explorer`、`Start Menu`、`Windows Terminal.lnk`、`wt.exe`、`devenv.exe`、`webstorm64.exe` 等。

这说明同一代码产物中加入了 Windows 文件夹右键菜单、开始菜单、终端和 IDE 发现相关路径。对 macOS 用户未必直接可见，但对跨平台桌面代码线是明确的新增证据。

关键证据：`Open project in Codex`、`Failed to register Windows folder context menu`、`File Explorer`。

### 9. preload bridge 增加 Fast Mode rollout 指标调用 【实证】

`.vite/build/preload.js` 暴露的 `electronBridge` 新增 `getFastModeRolloutMetrics`，通过 IPC 调用 `codex_desktop:get-fast-mode-rollout-metrics`。

这意味着渲染层获得了读取 Fast Mode rollout 指标的桥接能力；用户侧可能体现为 Fast Mode 公告、实验或展示逻辑更细，但这里只能确认 bridge API 存在。

关键证据：

```diff
+ getFastModeRolloutMetrics:async t=>e.ipcRenderer.invoke(`codex_desktop:get-fast-mode-rollout-metrics`,t)
```

### 10. Mermaid / 图表解析错误文案在多个图表模块中统一出现 【信号】

多个同名前端图表模块新增 `Parse error on line`、`end of input`，覆盖 gantt、block、sequence、c4、quadrant、sankey、journey、timeline、xychart、flow、mindmap、er、requirement、kanban 等模块。

这指向图表解析错误展示或解析器产物发生统一变化，但该结论来自同名前端模块差异，只能作为信号，不能说明具体解析逻辑如何改变。

关键证据：多个 diagram 模块共同新增 `Parse error on line`、`end of input`。

### 11. 构建工具链升级到 Electron Forge 7.11.1、Vite 8 beta、Vitest 4 beta 【实证】

`package.json` 显示 Electron Forge 相关包从 `^7.10.2` 升到 `^7.11.1`，`vite` 从 `^5.4.21` 改为 `8.0.0-beta.15`，`vitest` 从 `^1.6.0` 改为 `4.1.0-beta.5`，并新增运行时依赖 `tslib`。

这解释了前端产物分包结构变化的一部分背景，也说明此版构建链有较大跨度更新。

关键证据：

```diff
- "vite": "^5.4.21",
- "vitest": "^1.6.0"
+ "vite": "8.0.0-beta.15",
+ "vitest": "4.1.0-beta.5"
+ "tslib": "^2.8.1"
```

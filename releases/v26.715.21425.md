# Codex 桌面版 26.715.21425 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

本批次的共同主线是 Ultra 权限流程与多智能体入口；平台侧重点明显分化：macOS 偏向浏览器和线程面板，Windows 偏向插件、分享及权限管理。

Ultra/full-access handling and multi-agent entry points changed on both platforms, while macOS emphasizes browser and panel UI and Windows emphasizes plugins, sharing, and permissions.

## 关于这份报告

本报告比较两组官方签名分发产物：

- macOS arm64：`26.715.21316` build 5484 → `26.715.21425` build 5488。
- Windows x64 MSIX：`26.715.2236.0` → `26.715.2305.0`。

两平台版本号并不对齐，结论均按各自比较范围独立得出。Windows 事实包“平台/架构”行误写为 macOS arm64；本文依据输入所定义的 Windows x64 MSIX 样本归类，不据此改写任何版本数字。

这是历史全量包对比，不是增量包分析。macOS 前后包均通过 Apple 公证验证，签名来源为 OpenAI OpCo, LLC；事实包没有提供 Windows 签名验证结果。

证据分为两档：

- 【实证】：来自 plist、插件 manifest、签名验证及其他可直接读取的配置事实。
- 【信号】：来自同名前端模块的字符串、样式集合、模块名或压缩体积变化，只说明相关代码进入、移出或重组了对应 bundle，不代表功能已经向所有用户开放。

平台标记 `[macOS+Windows]`、`[仅 macOS]`、`[仅 Windows]` 表示证据所在样本，不表示其他平台未来或过去必然没有该能力。

本批次事实包未列出内置插件的目录级新增或移除，也未提供新增内置插件明细、CSP 差异、独立的主进程新增字符串、Windows AppxManifest 差异或新增 bundle 文件。因而本文不据缺失信息推导这些部分没有变化。

## ✨ 重点变化

### Ultra 可与 Full access 继续组合使用

【信号】[macOS+Windows]

两端都加入了 `Use Ultra with Full access?`、`Use Full access`，以及分别以受限权限或 Full access 使用 Ultra 的按钮文案。对应确认流程提供“继续使用审批”和“继续使用 Full access”两条路径。

旧文案所表达的流程是切换到 Ultra 时离开 Full access，或保留 Full access 并取消 Ultra；这些字符串在两端同时移除。macOS 还移除了 `Unavailable with Ultra reasoning` 及“Full access 在 Ultra 下不可用”的说明。

这是一组高度一致的交互方向信号：Ultra 与 Full access 的冲突处理从互斥切换转向警告后继续选择。证据来自前端模块差异，不能据此确认具体账户的权限策略或开放状态。

### 多智能体 composer 横幅进入两端首页代码

【信号】[macOS+Windows]

两端 `home-announcements` 都加入了多智能体 composer 横幅的正文和关闭按钮标签，其中明确出现：

- 将工作委派给并行运行的 subagents；
- 可能增加 token 使用量；
- `multi-agent` 状态标识和“已看过横幅”状态。

同一模块还加入 Fast mode、Codex gift credits、rate-limit reset 和后端驱动首页横幅的关闭标签，以及 gift credits 购买流程和 rate-limit reset 弹窗入口文案。这些内容已进入前端包，但是否展示仍可能受账户、后端配置或实验开关控制。

### 内置 Browser 与 Chrome 插件版本同步更新

【实证】[macOS+Windows]

两端内置 `browser` 与 `chrome` 插件 manifest 的版本均从 `26.715.21316` 更新为 `26.715.21425`，没有目录级新增或依赖新增。

manifest 中既有能力说明保持不变：

- `browser` 面向应用内浏览器，可打开、检查、导航、测试、点击、输入和截图本地 Web 目标，包括 localhost、环回地址、`file://` 和当前应用内浏览器标签。
- `chrome` 面向依赖用户现有 Chrome 状态的自动化，包括标签页、登录会话、Cookie、扩展和 ChatGPT Chrome Extension 设置；其说明仍要求优先使用专用 connector、API 或 CLI。

本次可确认的变化是插件版本元数据同步，不是这些能力在本批次首次加入。

### macOS 内置浏览器设置与历史、下载入口扩展

【信号】[仅 macOS]

`browser-use-settings` 增加约 98,875 字节，并出现一组相互印证的界面标签：

- 无需询问即可访问历史；
- 浏览器下载和历史入口；
- 后退、前进按钮；
- 联系信息、扩展、密码管理器和站点设置；
- 展开或隐藏各项浏览数据清理选项。

新增的 `aria-controls`、`aria-expanded`、`aria-labelledby` 等状态类也指向可展开的设置界面和更完整的无障碍标注。Windows 本批次事实包没有对应模块变化。

### 线程侧边面板与底部面板在两端呈相反重组方向

【信号】[macOS+Windows]

macOS 的 `thread-app-shell-chrome` 增加约 37,467 字节，并加入：

- `Bottom panel`；
- 关闭线程底部面板；
- 从底部或侧边面板打开另一标签页；
- 从任务侧边面板打开浏览器、代码审查、终端或工作区文件浏览器；
- 从聊天侧边面板打开 side chat；
- 当前聊天没有可用面板动作时的空状态；
- `max-width: 920px` 响应式条件。

Windows 的同名模块则减少约 34,866 字节，并移除上述响应式条件、`Bottom panel` 以及浏览器、审查、终端、文件浏览器和 side chat 的动作标签。

这不是简单的跨平台同步：在各自独立版本区间内，macOS 将相关面板字符串纳入该模块，Windows 则从该模块移出。它可能代表模块拆分、重新装载或平台 UI 分流，不能直接判定 Windows 删除了这些功能。

### Windows 插件详情页加入分享和 MCP 管理界面

【信号】[仅 Windows]

`plugin-detail-page` 增加约 156,015 字节，新增字符串覆盖多个高可见交互：

- 搜索 workspace 用户和群组；
- 添加或移除插件分享对象；
- “任何拥有链接的 workspace 成员”访问模式；
- MCP server 启用开关；
- 插件详情页分区导航和返回按钮；
- app tools、skill preview 与 skill card 的更多操作菜单；
- `App unavailable` 状态。

这组证据表明插件详情页 bundle 正在承载更完整的分享、技能操作和 MCP 启停界面，但不确认所有 workspace 均已开放相同选项。

### Windows 站点发布与分享流程扩展

【信号】[仅 Windows]

`appgen-share-dialog` 新增约 23,410 字节，并加入：

- 搜索 workspace 用户和群组；
- 添加、移除站点分享对象；
- 保存中的按钮状态；
- workspace 链接访问和 Internet 公开访问；
- `Can visit` 权限；
- 发布到公共 Internet；
- 复制 live site URL、在外部打开及 `Copied to clipboard` 状态。

这些文案共同指向站点发布后的访问控制和链接分发流程。与此同时，Windows 的 `appgen-settings-page` 移除了环境变量、secret、自定义域名、站点标题和确认删除 slug 等字符串；由于它们可能被迁移到其他 chunk，不能据此认定对应设置被产品删除。

### Windows 权限下拉菜单增加 Auto-review、Custom 与能力说明

【信号】[仅 Windows]

`permissions-mode-dropdown` 增加约 25,480 字节，出现：

- `Approve for me` 与 `Ask for approval`；
- `Change permissions`；
- `Custom (config.toml)`；
- Auto-review 选项及不可用原因；
- Full access 不可用原因；
- “始终询问后再编辑外部文件和使用 Internet”；
- “访问网站、发送数据并使用已启用插件”；
- Full access 确认对话框中的能力标题和说明。

这是权限模式呈现更加细分的信号，尤其把联网、外部文件、插件使用和自定义配置带入同一选择面。实际权限执行仍不能从前端字符串确认。

### Windows Computer Use 插件元数据随应用更新

【实证】[仅 Windows]

Windows 内置 `computer-use` 插件 manifest 版本从 `26.715.21316` 更新到 `26.715.21425`，说明仍为通过 Computer Use 从 ChatGPT 控制 Windows 桌面应用。

运行时总量保持 `287.5 MB`，`cua_node` 没有新增依赖包。因此本批次能够确认的是插件版本元数据更新，而不是 Computer Use 运行时扩容或依赖升级。

macOS 的 Computer Use 运行时同样没有体积或依赖变化，但事实包没有列出对应 `computer-use` manifest 改动。

### Windows 内嵌结账页加入返回 ChatGPT 的工具栏动作

【信号】[仅 Windows]

`checkout-webview-page` 增加约 12,752 字节，新增 `Back to ChatGPT`、返回 Codex 应用内 ChatGPT 的按钮说明，以及 `https://pay.openai.com` 和页面标题更新状态。

这说明支付 WebView 的返回导航和覆盖层状态进入了该前端模块，不等于结账入口已面向全部用户开放。

### 首页与历史导航代码在两端共同收缩

【信号】[macOS+Windows]

两端 `header` 模块均移除了相同的一组字符串，包括 `All chats`、环境筛选、应用设置、返回按钮、最近任务搜索清除、聊天历史、Chrome 扩展侧边栏更多操作和 `ChatGPT for Chrome`。

一致的移除更像共享前端代码的拆分或迁移。仅凭压缩模块差异，不能判断这些导航功能从产品中消失。

### 平台各自还有一组页面迁移信号

【信号】[仅 macOS]

macOS `skills-page` 移除了创建技能、重新加载技能、安装后刷新、空状态、页面标题和已安装技能分区等字符串；`plugins-settings` 移除了 `Manage plugins, skills, and MCPs`。这指向技能页或插件设置的信息架构重组，不能确认技能管理能力被移除。

【信号】[仅 Windows]

Windows `tasks` 模块减少约 306,482 字节，并移除连接应用、安装应用、准备 connected-app task、执行 onboarding 任务以及 CSV 图表示例等引导文案。这是 conversational onboarding 内容迁移或裁剪的强信号，但不是连接应用能力被删除的实证。

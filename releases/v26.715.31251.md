# Codex 桌面版 26.715.31251 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

本批次同时触及更新器、插件与技能管理、用量上限和拉取请求筛选；Windows 侧还集中扩展浏览器与站点设置，而若干插件详情和面板能力在两端呈相反方向。

Windows expands browser and site controls, while shared updater, plugin-management, usage-limit, and pull-request changes land amid substantial cross-platform UI divergence.

## 关于这份报告

本报告比较两组官方分发产物：

- macOS arm64：`26.715.21425`（build 5488）→ `26.715.31251`（build 5538）。
- Windows x64 MSIX：`26.715.2305.0` → `26.715.3651.0`。

两端版本号并不同步，因此所有结论分别依据各自事实包。插件内部版本号则在两端都出现 `26.715.21425` → `26.715.31251` 的变更，不能与 Windows 外层 MSIX 版本混为一谈。

证据等级：

- 【实证】：来自 manifest、Info.plist、配置文件、依赖统计、文件树统计及新增 bundle 字符串。字符串只能证明文案、错误路径或标识符已经进入包内，不代表所有用户都已可见。
- 【信号】：来自同名前端模块的字符串、样式类名或体积差异，以及前端 chunk 组合变化。它们说明 UI、状态或交互方向发生变化，但不能还原源码语义。

平台标记为 `[macOS+Windows]`、`[仅 macOS]` 和 `[仅 Windows]`。

两端 Bundle 顶层均无新增或移除文件，但 app.asar 内存在大量 Vite hash 重命名。本文已忽略这类 hash 噪音。事实包没有提供完整的内置插件清单、CSP 差异或 Windows AppxManifest 差异，因此不对这些项目是否变化作额外推断。

## ✨ 重点变化

### Windows Store 更新错误路径进入两端包内

【实证】[macOS+Windows] 两端新版主进程或前端 bundle 都新增了：

- `Downloaded MSIX has package identity `
- `Windows Store update operation failed`

这证明 MSIX 身份识别和 Windows Store 更新失败路径已经进入两端共同代码包。macOS 包中出现这些 Windows 专用字符串，只能说明共享代码被打包，不能据此断言 macOS 会执行该流程。

【信号】[仅 Windows] `resources/native/windows-updater.node` 从 1.2 MB 增至 1.8 MB，增加 0.56 MB，是本批次 Windows 侧最大的单组件体积变化。体积变化本身不能说明具体更新器实现。

### 浏览器与 Chrome 插件版本同步

【实证】[macOS+Windows] 内置 `chrome` 与 `browser` 插件 manifest 的版本均从 `26.715.21425` 更新到 `26.715.31251`。

两端 manifest 中可见的能力描述保持一致：

- `chrome` 面向依赖现有 Chrome 状态的自动化，包括标签页、登录会话、Cookie、扩展和 ChatGPT Chrome Extension 设置；描述要求优先使用专用连接器、API 或 CLI。
- `browser` 面向应用内浏览器操作，包括打开、检查、导航、测试、点击、输入和截图本地目标；其别名包括 `@browser`、`@browser-use` 和 `browser-use`，并明确覆盖 localhost、回环地址、`file://` 及当前应用内浏览器标签页。

【实证】[仅 Windows] `computer-use` 插件 manifest 的版本也从 `26.715.21425` 更新到 `26.715.31251`，描述仍明确为通过 Computer Use 从 ChatGPT 控制 Windows 桌面应用。

本批次没有提供新增内置插件明细，因此不能把这些版本更新解释为新增插件。

### Windows 浏览器设置面明显扩展

【信号】[仅 Windows] `browser-use-settings` 增加约 95,488 bytes，并出现一组高度关联的设置文案：

- `Access history without asking`
- 浏览器下载和历史记录入口
- 联系信息、扩展、密码管理器和站点设置入口
- 浏览器数据清理选项的展开与收起
- `Add site permission`
- 不可用插件的禁用开关

这些线索共同指向更完整的内置浏览器数据、权限和插件设置界面，但仍不能证明相关选项已向所有用户开放。

【信号】[仅 Windows] `header` 新增 `All chats`、`Cloud chats`、`Chat history`、`All environments filter option`、`App settings`、`ChatGPT for Chrome` 和 `Chrome computer use settings` 等字符串，显示聊天范围、环境筛选和 Chrome 入口正在汇入页头或导航区域。

### 插件、技能与 MCP 管理趋向统一

【信号】[macOS+Windows] 两端 `plugins-settings` 都新增 `Manage plugins, skills, and MCPs` 以及对应的组合设置区副标题，表明三类扩展入口正在被组织到同一设置区域。

【信号】[仅 macOS] `skills-page` 新增一套较完整的技能页面文案，包括：

- 页面标题、搜索框和加载状态
- 已安装技能与推荐技能分区
- 新建技能按钮 `New skill`
- 重载技能列表
- 新安装技能需要刷新后才能在 composer 使用的提示
- 无技能和筛选无结果状态

这是一组完整的技能发现、创建和刷新界面信号，不应只视为 chunk 体积变化。

### 插件详情页在两端反向变化

【信号】[仅 macOS] `plugin-detail-page` 增加约 158,456 bytes，并新增：

- MCP server 启用开关
- 插件详情页分区导航与返回按钮
- 工作区用户和群组搜索
- 移除已选共享对象
- `Add people or groups`
- `Anyone at workspace with the link`
- `Anyone in this workspace with the link`
- `App unavailable`
- 应用工具、技能预览及技能卡片的更多操作菜单

这些相互关联的文案和状态类名指向插件配置、共享、工具和技能操作面的扩展。

【信号】[仅 Windows] Windows 的同名模块反而缩小约 156,015 bytes，并移除了上述 MCP 开关、分区导航、共享对象和 `App unavailable` 等字符串。它只说明当前 Windows 构建中的该 chunk 收缩或重新拆分，不能独立证明功能被产品层面取消。

### 工作区月度用量上限可发起提升请求

【信号】[macOS+Windows] 两端均新增：

- `Button to request a workspace monthly usage limit increase`
- `Request limit increase`
- `settings.usage.limits.requestIncrease`

macOS 中线索位于共享应用/设置 chunk，Windows 中位于 `rate-limit-summary`。两端都指向工作区月度用量上限页面上的提升请求入口。

### 拉取请求增加显式关系筛选结果标题

【信号】[macOS+Windows] 两端 `pull-request-route` 均新增：

- `Heading for pull requests matching an explicit relationship filter`
- `codex.pullRequests.inbox.section.results`

这指向拉取请求收件箱针对显式关系筛选结果的独立标题或结果分区。

### 站点设置向 Windows 集中，分享界面两端收缩

【信号】[仅 Windows] `appgen-settings-page` 增加约 45,588 bytes，新增环境变量、secret、站点标题、删除确认 slug、自定义域名和 DNS 设置相关文案，包括 `Add custom domain`、`Add domain`、`Add secret` 与 `Add a domain to show DNS setup records`。

【信号】[仅 macOS] macOS 同名模块减少约 46,791 bytes，并移除同一组环境变量、secret、自定义域名、站点标题和删除确认输入的辅助标签。两端呈明显相反方向，但这仍可能包含代码拆分或迁移因素。

【信号】[macOS+Windows] 两端 `appgen-share-dialog` 都移除了站点分享对象搜索、保存状态、公开范围、复制站点 URL 和外部打开站点等字符串，包括 `Anyone on the Internet`。这说明该分享对话框在两个构建中共同收缩或迁移。

【信号】[仅 macOS] 另有 `Sites tools list failed` 进入共享应用 chunk，显示站点工具列表失败状态已进入包内模块；因为它来自同名前端模块差异，仍按信号处理。

### 任务侧栏与底部面板在 Windows 增强、macOS 收缩

【信号】[仅 Windows] `thread-app-shell-chrome` 新增约 34,866 bytes，并出现：

- 从任务侧栏打开浏览器、审查页、终端和工作区文件浏览器
- 从聊天侧栏打开 side chat
- `Bottom panel`
- 关闭线程底部面板
- 从底部面板或侧栏打开其他标签
- 当前聊天没有可用侧栏操作时的空状态

【信号】[仅 macOS] macOS 同名模块减少约 37,467 bytes，并移除了其中的 `Bottom panel` 以及浏览器、审查页、终端、文件浏览器和 side chat 操作标签。两端再次表现为相反的 chunk 方向。

### Windows Markdown、图片和文件引用界面扩充

【信号】[仅 Windows] `markdown` 增加约 354,706 bytes，新增：

- 带位置信息的文件引用 chip 辅助标签
- Markdown 图片无 alt 文本时的按钮、加载占位和失败回退标签
- `A follow-up message is already awaiting confirmation`
- 粗指针、hover 和深色模式媒体查询
- KaTeX display 相关字符串

同时移除了多组 Markdown/TextMate 语法作用域名称。整体更像 Markdown 渲染、图片加载、文件引用和交互状态的重组，不能仅凭 chunk 差异判断底层解析器已被替换。

### 事件总线连接与排队状态进入 macOS 前端模块

【信号】[仅 macOS] `esm` 新增约 23,833 bytes，并加入：

- `Connected to event bus`
- `Initializing event subscription for plugin`
- 向 client bus 和内部 event target 发出事件
- 总线未连接时排队、连接后发送
- client 禁用时不注册或不发送
- 重试耗尽和 shim event 分发失败

这些字符串形成了较完整的插件事件订阅、排队和故障状态面。旧的 `Callback Error`、`Context Cancel`、`Failed to deliver`、`Not ready` 和超时类字符串则被移除。

### 输入校验错误样式共同调整

【信号】[macOS+Windows] 两端 composer 或设置相关 chunk 都新增 `backdrop-blur-sm`，并将输入校验错误背景从 `/20` 调整为 `/70`。这是错误状态视觉强度或叠层表现变化的信号。

### 运行时与 Computer Use 依赖保持稳定

【实证】[macOS+Windows] 两端 `cua_node` 均无新增依赖包。

【实证】[仅 macOS] Computer Use 运行时总量保持 365.7 MB，变化为 0 MB、0%。

【实证】[仅 Windows] Computer Use 运行时总量保持 287.5 MB，变化为 0 MB、0%。

【实证】[macOS+Windows] 两端 `owl-electron-app.json` 的 `runtimeArchiveSha` 均发生变化：

- macOS：`2d21…dae3` → `b6cf…d690`
- Windows：`9d26…4cbe` → `e0e1…fd85`

这证明打包所用 runtime archive 内容改变，但事实包没有给出其内部语义差异。

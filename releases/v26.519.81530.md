# Codex 桌面版 26.519.81530 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

这一批包更像是同一代码线在两个平台上的错位落点：插件版本同步，但前端可见能力线索并不同步。  
One-liner: plugin manifests align across platforms, while UI-facing signals diverge sharply between macOS and Windows.

## 关于这份报告

本文只依据两份官方签名分发包逆向重建后的机器差异事实，不引用官方发布说明，也不推断未进入包内的行为。

证据等级分两档：【实证】来自 plist、插件 manifest、可读字符串、依赖清单、文件树与具体 diff；【信号】来自同名前端模块的体积、模块名、样式类名和压缩 bundle 字符串方向。平台标记为 `[macOS+Windows]`、`[仅 macOS]`、`[仅 Windows]`。

对比范围分别是 macOS arm64 `26.519.41501` build `3044` → `26.519.81530` build `3178`，以及 Windows x64 MSIX `26.519.5221.0` → `26.519.11010.0`。两个平台版本号和发布时间样本不同，下面不把某一平台结论套到另一平台。

本批事实包未提供内置插件清单整体变化、Computer Use 运行时总量、内容安全策略变化、新增组件与资源明细、AppxManifest 变化，相关部分只按已有事实处理。

## ✨ 重点变化

### 【实证】[macOS+Windows] Browser 与 Chrome 内置插件版本同步到 26.519.81530

两平台的 `browser` 和 `chrome` 插件 manifest 都把版本从 `26.519.41501` 更新到 `26.519.81530`。插件描述本身没有在 diff 中显示语义变化：Browser 仍指向 in-app browser、localhost、file URL、截图、点击、输入等用途；Chrome 仍指向远程 URL、已登录页面、现有标签页、cookie、扩展和 Chrome Extension 设置。

平台差异在于外层应用版本不同：macOS 应用本身也升到 `26.519.81530`，Windows 外层版本是 `26.519.11010.0`，但插件 manifest 内部版本同样落在 `26.519.81530`。

### 【信号】[macOS+Windows] 终端底部面板显著扩容，并加入崩溃恢复与可访问性相关文案

两平台同名模块 `thread-page-bottom-panel-state` 都从接近空模块扩到约 `0.3 MB`，新增字符串包括终端崩溃后重新渲染按钮、嵌入式终端崩溃说明、`Terminal.open` 前不能启用 accessibility、buffer 回收限制、mouse events 绑定、keypad 切换、链接打开拦截等。

这说明底部终端面板相关 UI 和状态处理在两端都有明显建设信号。证据仍是前端 bundle 线索，不能直接断言所有用户都能看到新终端面板。

### 【信号】[仅 macOS] 浏览器使用设置在 macOS 增强，Windows 样本反向移除同类线索

macOS 的 `browser-use-settings` 新增约 `47 KB`，出现 `Access history without asking`、添加允许/阻止域名、允许/阻止下载域名、允许/阻止上传域名、数据清理选项展开/收起的 accessibility label，以及不可用插件 toggle 的 disabled label。

Windows 同名模块则减少约 `47 KB`，并移除相同方向的字符串。也就是说，本输入样本显示浏览器权限/域名规则设置在 macOS 是新增信号，在 Windows 当前对比中是回退或未落入该样本的信号。

### 【信号】[仅 macOS] 线程侧栏与底部面板入口在 macOS 增强，Windows 样本移除同类入口线索

macOS 的 `thread-app-shell-chrome` 新增侧栏打开 side chat、browser tab、review tab、terminal、timeline、workspace file browser 的 action label，并出现 `Bottom panel`、`Browse project files`、底部面板关闭和新增 tab 按钮文案。

Windows 同名模块减少约 `21 KB`，移除这些入口字符串中的一组。平台差异很清楚：macOS 样本显示多面板导航入口增强，Windows 样本则没有同步呈现，反而是移除线索。

### 【信号】[仅 macOS] 技能页在 macOS 增强，Windows 样本移除同类页面线索

macOS 的 `skills-page` 新增创建技能、刷新技能列表、新装技能需刷新后才能在 composer 使用、过滤空状态、已安装技能、推荐技能、搜索输入、加载技能等文案。

Windows 的 `skills-page` 减少约 `9.9 KB`，移除同一批技能页文案中的一部分。结合插件 manifest 没有新增插件明细，本条只能说技能管理页面 UI 在 macOS 有增强信号，Windows 样本中没有同步出现。

### 【信号】[仅 macOS] worktree 初始化失败与恢复路径更具体

macOS 的 `worktree-init-v2-page` 新增创建 worktree、fork 当前会话、创建成功但 renderer 未能开始会话、创建失败、重试、编辑环境、取消并开始本地会话等文案。

这些字符串说明 macOS 前端对 worktree 初始化流程的状态、失败恢复和环境设置入口有更细颗粒度的 UI 信号。Windows 事实包没有对应模块变化条目。

### 【信号】[仅 Windows] 通用设置新增字体、工作模式、听写、环境与项目less窗口线索

Windows 的 `general-settings` 增加约 `100 KB`，新增字符串覆盖代码字号、Codex UI 基础字号、work mode settings、Agent environment、hold-to-dictate hotkey、hotkey window、dictation dictionary entry、ambient suggestions、Popout Window projectless default setting 等。

这些是高可见设置项信号，但证据来自同名前端模块改动线索，不能当成已发布功能开关的源码级事实。macOS 事实包没有列出同名变化。

### 【信号】[仅 Windows] Header 新增近期任务、云任务与环境筛选入口线索

Windows 的 `header` 模块新增 recent tasks menu、All tasks、Cloud tasks、All environments filter option、清空搜索、环境搜索失败、列出环境失败等字符串，并带有 `aria-live`、`aria-atomic` 等状态类名。

这显示 Windows 样本在顶栏/头部区域有近期任务菜单、云任务和环境筛选方向的 UI 信号。macOS 事实包没有对应 header 变化条目。

### 【信号】[仅 Windows] 线程侧栏标签页移除一批 CI/PR/编辑接收相关提示

Windows 的 `thread-side-panel-tabs` 从 `0.5 MB` 降到接近空模块，移除字符串包含外部日志诊断、CI 系统访问、凭证/权限/token/MCP/tool access 检查、已有 pull request、接受编辑并 stage 等。

这更像是某些侧栏提示或工作流说明从该模块退出。由于只来自压缩前端模块变化，不能断言相关能力整体被移除。

### 【信号】[macOS+Windows] Markdown 与 Alert 组件有共同瘦身/重排线索

两平台的 `markdown` 模块都减少约 `60 KB`，移除无 alt 文本图片/视频 fallback、placeholder、file citation chip accessibility label、KaTeX 相关字符串和若干样式类。`alert` 模块两端都减少约 `1.4 KB`，移除多类 validation 背景/边框、`data-testid`、若干布局类，同时新增 `http://www.w3.org/2000/svg` 样式/字符串线索。

这属于渲染组件和提示组件的低层重排信号，不足以单独判断用户可见功能变化。

### 【实证】[仅 Windows] Windows 包体结构移除 bwrap，并调整多个运行时二进制体积

Windows 文件树从 `832` 个文件变为 `831` 个文件，新增 `0`、移除 `1`、内容改动 `19`。被移除文件是 `resources/codex-resources/bwrap`，大小 `0.5 MB`。

同批次 Windows 关键二进制体积变化明显：`resources/codex.exe` 和 Chrome 插件 app-server-runtime 中的 `codex` 都从 `247.7 MB` 降到 `239.6 MB`，减少 `8.11 MB`；`resources/codex-windows-sandbox-setup.exe` 从 `8.3 MB` 降到 `6.1 MB`，减少 `2.21 MB`；`resources/codex` 从 `219.5 MB` 增至 `221.7 MB`，增加 `2.17 MB`；`resources/codex-command-runner.exe` 增加 `0.12 MB`。

### 【实证】[仅 macOS] macOS 主二进制与 Chrome 插件运行时小幅增大

macOS 文件树数量保持 `1335 → 1335`，新增 `0`、移除 `0`、内容改动 `61`。`Contents/Resources/codex` 与 Chrome 插件 app-server-runtime 中的 `codex` 都从 `192.1 MB` 增至 `193.8 MB`，增加 `1.70 MB`。

`app.asar` 体积显示为 `159.0 MB → 159.0 MB`，但 Info.plist 中 Electron asar integrity hash 从 `91bd...c5c8` 变为 `2cc5...c8b9`，说明应用层包内容确实变化。

## 🔎 低层信号索引

### 面板、终端与文件浏览

`thread-page-bottom-panel-state` 在两端共同扩容，`thread-app-shell-chrome` 在 macOS 增强、Windows 移除同类线索。证据边界：这是前端模块与可读字符串信号，说明代码包内出现或退出相关 UI 文案，不等于功能对所有用户可见。

### 浏览器权限与数据规则

`browser-use-settings` 在 macOS 新增允许/阻止域名、上传下载域名、history access、清理选项展开/收起；Windows 样本移除相同方向的字符串。证据边界：可读字符串进入或退出 bundle 是事实，但设置项是否启用仍需运行时验证。

### 技能管理

`skills-page` 在 macOS 新增创建、刷新、搜索、空状态、已安装/推荐技能标题；Windows 移除同类线索。证据边界：没有新增内置插件明细，不能写成新增技能能力面，只能写技能页 UI 信号。

### 设置、任务与环境

Windows `general-settings` 和 `header` 分别指向字号、听写、环境、projectless popout、近期任务、云任务、环境筛选。证据边界：这是 Windows 独有前端信号，macOS 事实包未提供对应变化。

### Markdown、Alert 与 Plugin Detail

两端 `markdown`、`alert` 都有共同瘦身信号；`plugin-detail-queries` 两端仅有 `+180 bytes` 并移除 `react.memo_cache_sentinel`。证据边界：这些更接近实现重排或构建输出变化，价值低于明确命令、设置和页面文案。

### Debug Modal

macOS `debug-modal` 减少约 `150 KB`，移除 `Account ID`、`App Actions`、`App Server`、`App session ID`、自动 worktree cleanup 等字符串。Windows 事实包没有同名变化条目。证据边界：只能说明 macOS 样本中 debug modal 模块可读内容退出或重排，不能断言调试能力整体移除。

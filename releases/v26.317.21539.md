# Codex 桌面版 `26.317.21539`(build 1088)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版从 2026-03-17 的 `26.313.41514` 更新到 2026-03-19 的 `26.317.21539`，变化集中在桌面启动兜底、构建工具版本，以及少量前端界面模块信号。  
English one-liner: a small desktop resilience and build-tooling update, with early UI module signals.

## 关于这份报告

这不是 OpenAI(人工智能公司) 官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次属于历史回填：两个版本均已超出 Sparkle(桌面应用更新框架) 增量窗口，因此采用两个官方全量包直接比较，而不是增量包重建。

方法论是：下载上一版与最新版官方全量包 → 校验 SHA-256(哈希摘要) → 解包并进行结构化 bundle(应用包) 差异分析 → 对关键可读文件、配置和体积变化进行归纳。事实包显示两个版本的公证验证结果均为 `CSSMERR_TP_CERT_REVOKED`，即部分早期签名证书后来被吊销；这影响系统安全安装判断，但不影响对 bundle 内容做差异分析。

本文使用两档证据等级：

- **【实证】**：来自 Info.plist、package.json、CSP(内容安全策略)、可读 diff(差异文本) 等直接证据。
- **【信号】**：来自前端模块名或二进制体积变化，只能说明“指向某方向，但无法从产物还原到源码级意图”。

对比范围：`26.313.41514` build `1043`，发布于 `2026-03-17`；到 `26.317.21539` build `1088`，发布于 `2026-03-19`。

## ✨ 重点变化

### 1. 启动失败时新增可操作兜底对话框 【实证】

桌面启动入口 `.vite/build/bootstrap.js` 从简单加载主进程文件，改为在主应用启动失败时弹出错误对话框，并根据更新状态提供操作按钮。

这意味着用户遇到主应用启动失败时，不再只是进程异常退出；新版至少能给出“安装更新”“检查更新”或“退出”的兜底路径。关键证据包括新增的按钮文案：

```diff
+"install-update":`Install Update`,"check-for-updates":`Check for Updates`,quit:`Quit`
```

同时，错误对话框文案包含：

```diff
+message:`${t.app.getName()} failed to start.`
+detail:r instanceof Error?r.message:`The main desktop app failed during startup.`
```

### 2. 启动流程接入 Sparkle 更新状态 【实证】

新版启动入口会初始化 `sparkleManager`，并在启动失败时根据 `getIsUpdateReady()` 与 updater(更新器) 状态决定展示“安装更新”或“检查更新”。

可感知点是：当桌面端损坏、启动主流程失败，且已有更新可用时，应用有机会引导用户直接安装更新，而不是只留下错误状态。关键证据：

```diff
+await a.initialize();
+let{desktopSentry:i,sparkleManager:a}=n;
+i.installUpdatesIfAvailable();
+a?.checkForUpdates();
```

### 3. 前端出现 loading-page 与 popover 模块 【信号】

去掉 Vite 构建 hash(内容指纹) 噪音后，事实包给出的真实新增前端模块为 `loading-page` 与 `popover`。这指向加载页与浮层 / 弹出层能力方向，但无法从产物还原到源码级意图，也不能证明功能已经对所有用户开放。

可感知点可能是某些等待、切换、弹层交互的界面基础设施在调整。关键证据来自去噪后的模块清单：

```text
新增模块: loading-page, popover
移除模块: arrow-left, regenerate, sentry-tracing.electron
```

其中 `webview/index.html` 也出现了对应预加载资源：

```diff
+<link rel="modulepreload" crossorigin href="./assets/popover-BZJE0yel.js">
+<link rel="modulepreload" crossorigin href="./assets/loading-page-Dr8JlrqP.js">
```

### 4. 构建工具 Vite 从 beta 升到稳定版 【实证】

`package.json` 显示开发依赖中的 Vite(前端构建工具) 从 `8.0.0-beta.15` 升级到 `8.0.0`。

这更像是底层构建链路稳定化，而不是直接面向用户的新功能；但它会影响前端资源打包形态，因此本次 `webview/assets` 下大量 hash 文件名变化应视为构建产物重命名噪音。关键证据：

```diff
-    "vite": "8.0.0-beta.15",
+    "vite": "8.0.0",
```

### 5. 主二进制体积明显增加 【信号】

`Contents/Resources/codex` 从 `106.4 MB` 增至 `116.9 MB`，增加 `10.51 MB`。这指向底层二进制内容有明显变化，但仅凭体积无法判断具体功能，也无法从产物还原到源码级意图。

可感知点需要结合实际运行行为进一步验证；在本事实包范围内，只能把它记录为二进制层面的变化信号。

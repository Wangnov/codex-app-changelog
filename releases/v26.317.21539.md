# Codex 桌面版 `26.317.21539` build 1088

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

本版从 `26.313.41514` 升至 `26.317.21539`，可见变化集中在启动失败处理、插件管理、本地环境设置、PR / GitHub 相关界面提示，以及代码高亮资源结构调整。  
One-liner: a small desktop update focused on startup fallback handling, plugin UI wording, environment setup affordances, and bundled grammar/resource reshuffling.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次为历史回填，官方 appcast 未对 `26.313.41514` → `26.317.21539` 提供增量包，因此改为下载两个官方全量包后直接比较 bundle 内容。

证据分为两档：**【实证】**表示可在人类可读的产物中直接看到，例如 `Info.plist`、`package.json`、HTML、启动脚本、可读字符串或签名验证结果；**【信号】**表示只能从前端模块名、同名模块字符串差异或体积变化推断方向，无法从产物还原到源码级意图。

对比范围是 macOS arm64：上一版 `26.313.41514` build `1043`，发布于 `2026-03-17`；最新版 `26.317.21539` build `1088`，发布于 `2026-03-19`。

## ✨ 重点变化

### 1. 启动失败时新增更新 / 退出兜底对话框 【实证】

新版 `.vite/build/bootstrap.js` 不再只是加载主入口，而是内置了主应用启动失败后的错误对话框逻辑。对话框文案包括 `Codex failed to start.`，并根据更新状态提供 `Install Update`、`Check for Updates` 或 `Quit`。

这意味着当主桌面应用在启动阶段失败时，用户可能会看到更明确的恢复路径，而不是静默退出或只留下日志。关键证据来自启动脚本新增逻辑：`install-update`、`check-for-updates`、`quit`，以及 `Desktop bootstrap failed to start the main app`。

### 2. 插件页从 Apps / Skills 混合入口转向更明确的插件管理 【信号】

`skills-page` 同名前端模块新增了 `All plugins`、`Choose a plugin marketplace`、`Enable plugin`、`Disable plugin`、`Install plugin`、`Failed to load plugins` 等字符串，同时移除了 `All Apps`、`All skills`、`New app`、`No apps found` 等旧文案。

这指向插件目录与插件启用状态的界面重整，但无法从产物还原到源码级意图。可感知点可能是插件页的分类、市场选择和启用开关文案更聚焦于 plugins，而不是 Apps / Skills / Apps 混合导航。

### 3. 本地环境设置新增 setup 环境变量弹出层线索 【信号】

`local-environments-settings-page` 新增 `Available environment variables`、`Setup script environment variables`、`Source workspace path`、`New worktree path`，以及 `Button label that opens the setup env vars popover`。

这指向本地环境 setup 脚本可查看或使用环境变量的 UI，但无法从产物还原到源码级意图。对开发者而言，可感知方向是 source workspace 与 worktree 路径可能被更明确地暴露给 setup 流程。

### 4. PR / GitHub 工作流的可访问性与失败路径被补强 【实证】

新版主进程 / 前端 bundle 新增可读字符串 `No git repository found for pull request board`、`gh.pr.board.failed`，`index` 模块还新增了 `Accessible label for a link that opens a pull request comment on GitHub`、`ARIA label for the hidden pull request board columns button` 等文案。

这说明 PR board、PR comment 链接和相关失败状态至少在产物文案层面有新增覆盖。与之相对，旧版移除了 `Failed to start PR comments workflow`、`Failed to start PR fix` 等字符串，表明相关错误路径或文案发生了调整。

### 5. Markdown、diff 与多语言代码高亮资源发生明显重排 【信号】

多个语言模块的体积和语法 token 字符串变化显著：`cpp` 从 `0.0 MB` 增至 `0.6 MB`，`javascript` 增至 `0.2 MB`，`php` 增至 `0.1 MB`，同时 `markdown` 从 `0.5 MB` 降至 `0.1 MB`，`browser` 从 `0.7 MB` 降至 `0.0 MB`。

这些线索指向语法高亮或编辑器语法资源的拆分、替换或重新打包，但无法从产物还原到源码级意图。新增 token 包括 `heading.1.markdown`、`markup.inserted.diff`、`comment.block.cpp`、`constant.other.description.jsdoc`、`comment.block.documentation.phpdoc.php` 等。

### 6. 前端新增 `loading-page` 与 `popover` 模块 【信号】

去掉 Vite hash 噪音后，前端真实新增模块为 `loading-page` 和 `popover`，移除模块为 `arrow-left`、`regenerate`、`sentry-tracing.electron`。

这指向加载页与弹出层组件的前端结构变化，但模块出现不等于功能已对所有用户开放，也无法从产物还原到源码级意图。`webview/index.html` 中也能看到新版预加载 `popover-BZJE0yel.js` 与 `loading-page-Dr8JlrqP.js`。

### 7. 崩溃 minidump 与 profiling 相关路径出现新增文案 【实证】

新版 bundle 新增 `Could not delete minidump`、`Failed to load minidump`、`Timed out waiting for minidump to stop being modified`、`No Sentry client available, profiling is not started`、`ProfilingIntegration is not available` 等字符串。

这说明桌面端崩溃转储与 profiling 相关错误路径在产物中出现了新的可读处理文案。可感知层面更可能体现在诊断、遥测或错误上报路径的行为变化，而非普通 UI 功能。

### 8. 构建工具从 Vite beta 切到稳定版本 【实证】

`package.json` 中 dev dependency 从 `vite: 8.0.0-beta.15` 变为 `vite: 8.0.0`，版本号从 `26.313.41514` 升至 `26.317.21539`，`codexBuildNumber` 从 `1043` 升至 `1088`。

这属于底层构建链调整，用户通常不会直接感知，但它解释了大量 `webview/assets` hash 文件名变化为何不能被当作功能增删。

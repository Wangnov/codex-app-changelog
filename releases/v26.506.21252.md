# Codex 桌面版 `26.506.21252`(build 2575)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版的主线是：Codex(代码代理桌面应用) 的浏览器自动化入口更正式，插件市场增加 Chrome(浏览器) 插件，同时系统层面开始声明可作为 CSV(逗号分隔值文件) 查看器。  
English one-liner: this build tightens browser automation, expands bundled plugins, and adds CSV document handling.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次为历史回填：两个版本均超出官方 Sparkle(应用更新框架) 增量窗口，因此采用上一版与最新版两个官方全量包直接比较，而不是增量包重建。

可信度方面，两个 App(应用程序) 均通过 Apple(苹果公司) 公证与 OpenAI(人工智能公司) Developer ID(开发者身份) 签名验证。`spctl` 结果均为 `accepted`，来源均为 `Notarized Developer ID`，签名来源均为 `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`。

证据分两档：

- **【实证】**：来自人类可读的直接证据，例如 `Info.plist`、插件 `plugin.json`、技能文档、`package.json`、签名与公证结果。
- **【信号】**：来自前端模块名或二进制体积变化，指向某方向，但无法从产物还原到源码级意图。

## ✨ 重点变化

### 1. 内置浏览器插件从 “Browser Use” 更名为 “Browser”，并升级到 `0.1.0-alpha2` 【实证】

`browser-use` 插件版本从 `0.1.0-alpha1` 升到 `0.1.0-alpha2`，展示名从 `Browser Use` 改为 `Browser`，别名新增 `@browser`。描述也从“建议测试”变成“显著前端改动后，在目标明确时使用 Browser 打开本地目标”。

这意味着 Codex 内置浏览器不再只是一个偏实验性的“使用浏览器”能力，而是被包装成更明确、更优先的本地网页检查与操作入口。

关键证据：

```diff
- "version": "0.1.0-alpha1"
+ "version": "0.1.0-alpha2"

- "displayName": "Browser Use"
+ "displayName": "Browser"
```

### 2. 浏览器自动化 API 入口调整为 `agent.browsers.*`，并强调使用 `iab` 浏览器 【实证】

浏览器技能文档从显式传入 `backend = "iab"`，改为先初始化运行时，再通过 `agent.browsers.get("iab")` 取得浏览器对象。文档中的调用示例也从 `agent.browser.*` 大量迁移到 `browser.*`。

可感知点是，Codex 内部对“浏览器实例”的抽象更清楚了：先选择内置浏览器，再围绕这个浏览器对象打开标签页、命名会话、截图和操作页面。

关键证据：

```diff
- await setupAtlasRuntime({ globals: globalThis, backend });
+ await setupAtlasRuntime({ globals: globalThis });
+ globalThis.browser = await agent.browsers.get("iab");
```

### 3. 新增本地 Chrome 插件入口，但可见能力仍以市场配置为准 【实证】

插件市场配置中新增 `chrome` 插件，来源为本地路径 `./plugins/chrome`，安装策略为 `AVAILABLE`，认证方式为 `ON_INSTALL`，分类为 `Productivity`。文件树中也出现 Chrome 插件相关的 `extension-host` 与 `browser-client.mjs`。

这说明 Codex 这一版开始把 Chrome 相关能力纳入内置插件体系，但事实包只证明“插件入口存在且可安装”，不证明它默认启用或具体向用户开放了哪些操作。

关键证据：

```json
  "source": { "source": "local", "path": "./plugins/chrome" },

```

### 4. Computer Use 插件升级，并开始声明技能目录 【实证】

`computer-use` 插件版本从 `1.0.770` 升到 `1.0.780`，同时新增 `"skills": "./skills/"`。Computer Use(让模型操作桌面应用的能力) 原本已经声明需要 `accessibility` 权限，本版新增的是技能目录入口。

这意味着桌面操作能力不只是底层服务升级，也开始通过技能文档进入 Codex 的插件描述与调度体系。

关键证据：

```diff
- "version": "1.0.770"
+ "version": "1.0.780"
+ "skills": "./skills/"
```

### 5. Codex 开始在系统层声明可查看 CSV 文件 【实证】

`Info.plist` 新增一组 `CFBundleDocumentTypes` 配置，声明文件扩展名 `csv`、MIME 类型 `text/csv`、内容类型 `public.comma-separated-values-text`，角色为 `Viewer`，优先级为 `Alternate`。

这表示 macOS 可以把 Codex 识别为 CSV 文件的备选查看器。事实包没有显示具体 CSV 预览界面，因此这里只能确认系统文件关联层面的变化。

关键证据：

```text
CFBundleTypeExtensions[0] = csv
CFBundleTypeMIMETypes[0] = text/csv
CFBundleTypeRole = Viewer
LSHandlerRank = Alternate
```

### 6. LaTeX 插件升级到 `0.1.1`，并改归到 Research 分类 【实证】

`latex-tectonic` 插件版本从 `0.1.0` 升到 `0.1.1`，分类从 `Engineering` 改为 `Research`，并新增 `composerIcon` 与 `logo`，均指向 `./assets/tex.svg`。

这更像是插件展示与分类上的完善：能力描述仍是使用内置 Tectonic 编译 LaTeX(排版系统) 与 TeX 文档，但在插件市场里的定位更偏研究场景。

关键证据：

```diff
- "version": "0.1.0"
+ "version": "0.1.1"
- "category": "Engineering"
+ "category": "Research"
+ "composerIcon": "./assets/tex.svg"
+ "logo": "./assets/tex.svg"
```

### 7. 前端模块名指向更完整的设置、技能、插件、差异查看与工作区界面，但不能视作源码级结论 【信号】

去掉 Vite(前端构建工具) hash(哈希) 噪音后，新增模块中出现 `account-settings`、`hooks-settings`、`skills`、`plugin-install-store`、`plugins-availability`、`diff-unified`、`file-diff`、`workspace-directory-tree`、`notebook-preview-panel`、`image-preview-dialog` 等名称。

这些模块名指向设置页、技能管理、插件安装状态、文件差异查看、工作区文件树与预览体验的扩展，但无法从产物还原到源码级意图，也不能证明这些界面都已对所有用户开放。

关键证据：事实包列出的去噪新增模块包含 `skills`、`plugin-install-store`、`diff-unified`、`file-tree-search-input`、`workspace-directory-tree`。

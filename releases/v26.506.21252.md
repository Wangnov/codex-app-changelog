# Codex 桌面版 `26.506.21252` build 2575

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版的核心变化是新增面向用户 Chrome 浏览器状态的自动化插件，同时重命名并强化内置 Browser 插件的调用规则。  
One-liner: Codex Desktop adds a bundled Chrome automation plugin and tightens the in-app browser automation surface.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方签名分发产物的逆向差异报告**。本次对比范围为 macOS arm64 的 `26.429.61741` build 2429 到 `26.506.21252` build 2575。

本版为历史回填：事实包说明官方 appcast 未对这一对版本提供增量包，因此改为下载两个官方全量包直接比较。差异结论来自 bundle 文件树、Info.plist、package.json、内置插件清单、技能文档、可读字符串和签名验证结果。

证据等级分两档：**【实证】**表示存在人类可读的直接证据，例如插件清单、技能文档、package.json diff、Info.plist 键值或新版 bundle 中新增字符串；**【信号】**表示只来自前端模块名、同名前端模块字符串差异或体积变化，指向某方向，但无法从产物还原到源码级意图。

## ✨ 重点变化

### 1. 新增内置 Chrome 插件 【实证】

`openai-bundled` 内置插件从 `browser-use`、`computer-use`、`latex-tectonic` 增加为 `browser-use`、`chrome`、`computer-use`、`latex-tectonic`。新增的 `chrome` 插件版本为 `0.1.7`，显示名为 `Chrome`，分类为 `Productivity`。

这意味着 Codex 开始把用户已有的 Chrome 状态纳入桌面自动化范围：插件描述明确覆盖远程 URL、已登录页面、现有标签页、cookies、扩展和 Codex Chrome Extension setup。关键证据：

```diff
+      "name": "chrome",
+      "source": {
+        "source": "local",
+        "path": "./plugins/chrome"
+      },
+      "policy": {
+        "installation": "AVAILABLE",
+        "authentication": "ON_INSTALL"
+      },
+      "category": "Productivity"
```

新增插件共有 345 个文件，总大小 8.6 MB；默认提示包括 `file my expenses for me`、`review my messages on social media`、`put in a PTO request in Workday`。

### 2. Chrome 自动化强调用户授权与扩展连通性 【实证】

`skills/chrome/SKILL.md` 明确把 Chrome 定位为需要用户 cookies、已登录会话、现有标签页、扩展或远程认证站点时的浏览器自动化入口。文档还要求在首个 Chrome-backed 任务中检查 open tabs，失败后重试，并检测 Chrome 是否安装、运行、扩展是否存在、native host manifest 是否有效。

可感知点是：Chrome 能力不是简单打开网页，而是通过 Codex Chrome Extension 接入用户现有浏览器环境；同时，若 Chrome 未运行必须询问用户是否启动，若扩展缺失则提示用户确认 Codex Chrome Extension 已安装并启用。技能文档还明确禁止在扩展通信失败后改用 applescript、bash 或其他脚本完成请求。

### 3. 内置 Browser 插件从 “Browser Use” 改名为 “Browser” 【实证】

`browser-use` 插件版本从 `0.1.0-alpha1` 升至 `0.1.0-alpha2`，显示名从 `Browser Use` 改为 `Browser`，描述也从 “browser built into Codex” 调整为 “in-app browser”。别名新增 `@browser`，并继续保留 `@browser-use`。

这意味着 Codex 更明确地区分两类浏览器入口：`Browser` 面向 Codex 内置浏览器和本地开发目标，新增 `Chrome` 面向用户已有 Chrome 状态。关键证据：

```diff
-    "displayName": "Browser Use",
+    "displayName": "Browser",
...
-  "version": "0.1.0-alpha1",
+  "version": "0.1.0-alpha2",
```

### 4. Browser 技能文档重写了运行时入口和操作规范 【实证】

`browser-use/skills/browser/SKILL.md` 改动较大：初始化示例从显式传入 `backend = "iab"` 变为 `setupAtlasRuntime({ globals: globalThis })` 后通过 `agent.browsers.get("iab")` 获取 browser；API 描述从 `agent.browser.*` 转向 `agent.browsers.*`。

这会影响 Codex 代理如何调用内置浏览器。文档还新增了截图必须以内联 Markdown 图片呈现、避免大页面逐元素读取、优先使用快照或定向检查等规则，指向更严格的浏览器自动化成本控制和用户可见输出规范。

### 5. Computer Use 插件小版本升级并开始声明技能目录 【实证】

`computer-use` 插件版本从 `1.0.770` 升至 `1.0.780`，并在插件配置中新增 `"skills": "./skills/"`。

这说明 Computer Use 不只是二进制组件更新，也开始通过技能文档参与能力路由。相关二进制也有小幅变化：`SkyComputerUseService` 从 14.5 MB 到 14.7 MB，`SkyComputerUseClient` 从 10.0 MB 到 10.2 MB。

### 6. Computer Use 设置出现 Chrome 扩展和下载/上传询问项 【信号】

`computer-use-settings` 同名前端模块新增了 `Always allow`、`Always ask`、`Any App`、`Ask before downloading files`、`Ask before uploading files`、`Browser extension not connected`、`manage Google Chrome settings`、`reinstall/remove the Google Chrome extension` 等字符串。

这些线索指向 Computer Use 设置页正在纳入 Chrome 扩展连接状态，以及网站下载、上传前是否询问的权限控制。但这是同名前端模块改动线索，只能说明 UI、状态或交互方向，无法从产物还原到源码级意图。

### 7. 插件市场能力从内置列表走向可添加来源 【实证】

package 与前端可读字符串共同显示，插件系统在这一版有更强的市场化管理迹象。`marketplace.json` 新增 `chrome` 为本地来源插件；`plugins-page` 同名前端模块新增 `Add marketplace`、`Add a plugin marketplace from a GitHub repo, Git URL, or local folder`、`Disable plugin`、`Built by OpenAI` 等文案。

其中 `marketplace.json` 是【实证】；`plugins-page` 是同名前端模块线索，只能作为【信号】。合并来看，变化指向插件页支持启用/禁用插件、添加更多 marketplace 来源和区分 OpenAI 构建插件，但源码级实现细节不可确认。

### 8. 外部代理迁移导入变得更具体 【实证】

`package.json` 新增依赖 `external-agent-migration`，`external-agent-import` 同名前端模块新增 `Import from other AI apps`、`Apps found`、`Claude Code`、`Claude Cowork`、`Bring over your setup, projects, and recent chats` 等文案。

可感知点是导入流程从泛化的项目/工具迁移，指向从其他 AI app 导入设置、项目和最近聊天，且点名 Claude Code 与 Claude Cowork。依赖新增是【实证】；具体 UI 流程文案来自同名前端模块，属于【信号】。

### 9. Markdown 渲染和预览能力继续扩展 【信号】

`markdown` 模块体积从 0.1 MB 增至 0.3 MB，新增 `ARIA label for rendered mermaid diagrams`、markdown image button、image fallback、file citation chip with line information 等字符串；前端真实新增模块还包括 `image-preview-dialog`、`notebook-preview-panel`、`diff-unified`、`file-diff`、`review-file-tree-pane` 等。

这些信号指向更丰富的 Markdown、图片预览、文件引用、diff 和 review 相关界面能力，但仅来自前端模块与同名模块差异，无法确认功能是否已完整开放。

### 10. macOS 开始声明 CSV 文档查看关联 【实证】

Info.plist 新增 CSV 文档类型：扩展名 `csv`，MIME type `text/csv`，UTI `public.comma-separated-values-text`，角色为 `Viewer`，`LSHandlerRank` 为 `Alternate`。

这意味着系统层面 Codex 可以被 macOS 识别为 CSV 的可选查看器。证据来自 Info.plist 新增键值。

### 11. LaTeX 插件调整定位和视觉资产 【实证】

`latex-tectonic` 插件版本从 `0.1.0` 升至 `0.1.1`，分类从 `Engineering` 改为 `Research`，并新增 `composerIcon` 与 `logo`，均指向 `./assets/tex.svg`。

这不是核心能力大改，但会影响插件在界面中的分类和呈现方式。

### 12. 自动化、远程连接、hooks、技能页与命令面持续扩展 【信号】

新增前端模块包括 `automation-schedule`、`automation-shared`、`hooks-settings`、`install-hook`、`skills-page`、`remote-connections-settings`、`thread-side-panel-browser-tab-state`、`open-config-toml-button`、`workspace-directory-tree`。同名模块还新增了 `Authorize remote control to load signed-in devices`、`Connect via SSH`、`Choose a discovered connection or add one manually`、`Button label for creating a new skill`、`codex.command.forceReloadSkills` 等字符串。

这些线索指向自动化、远程控制授权、SSH 连接、hooks 管理、技能刷新和侧边浏览器状态的界面持续扩展，但均属于前端信号，无法从产物确认完整产品意图。

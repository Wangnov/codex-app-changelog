# Codex 桌面版 26.623.31443 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

本批次看起来是一次以运行时基座、内置插件版本同步和前端交互迭代为主的跨平台更新,但两个平台暴露出的 UI 信号并不完全一致。  
One-liner: shared runtime/plugin refresh, with macOS showing broader route reshaping and Windows surfacing stronger side-panel, skills, and worktree-repair signals.

## 关于这份报告

本文只使用输入事实包中的机器提取事实。macOS 对比范围是 26.623.30605 build 4415 到 26.623.31443 build 4441; Windows 对比范围是 26.623.3245.0 到 26.623.3763.0。两者来自同一套 Codex 代码的不同打包,但版本号和发布节奏不完全对齐,因此本文分别引用各自事实。

证据分两档:【实证】来自 plist、manifest、插件清单、配置、签名验证、依赖清单、服务入口、技能文档或非“同名前端模块改动线索”的可读字符串;【信号】来自前端模块名、体积变化、样式类名和同名前端模块字符串差异。平台归属用 [macOS+Windows]、[仅 macOS]、[仅 Windows] 标注。

## ✨ 重点变化

### 【实证】[macOS+Windows] Chromium / Electron 基座推进到 149.0.7827.197

macOS 的 Info.plist 明确显示 `ChromiumBaseVersion` 从 149.0.7827.157 到 149.0.7827.197,`ChromiumBaseBundleVersion` 从 7827.157 到 7827.197。Windows 同批替换了 `149.0.7827.157.manifest` 为 `149.0.7827.197.manifest`,并且 Google Docs Reading Mode helper manifest 版本也从 149.0.7827.157 到 149.0.7827.197。

平台差异在包形态上很明显:macOS 是整个 `Codex Framework.framework/Versions/149.0.7827.157` 到 `149.0.7827.197` 的目录级替换;Windows 是 `chrome.dll` 等 Chromium 组件原地内容变化,其中 `chrome.dll` 从 308.2 MB 到 308.6 MB。

### 【实证】[macOS+Windows] Browser 内置插件版本同步到 26.623.31443

两平台的 Browser 插件 manifest 都把版本从 26.623.30605 更新到 26.623.31443。插件说明仍指向 Codex in-app browser / browser-use 能力,包括打开、检查、导航、测试、点击、输入、本地 localhost 或 file URL 截图,以及显式要求 @browser 时不应退回 macOS `open`、shell 或普通网页浏览。

平台差异:macOS 事实包只列出 Browser 插件版本变化;Windows 同时还列出 Computer Use 和 Chrome 插件版本变化。

### 【实证】[仅 Windows] Computer Use 与 Chrome 插件版本同步

Windows 的 `computer-use` 插件 manifest 从 26.623.30605 到 26.623.31443,描述为从 Codex 控制 Windows 桌面应用。Windows 的 `chrome` 插件 manifest 也同步到 26.623.31443,描述仍覆盖依赖现有 Chrome 状态的自动化场景,包括标签页、登录态、cookie、扩展和 Codex Chrome Extension 设置。

macOS 本输入样本没有对应的 Computer Use 或 Chrome 插件 manifest 变化条目,不能据此推断 macOS 没有相关能力,只能说本次 macOS 事实包未列出这些 manifest 改动。

### 【实证】[macOS+Windows] Computer Use 运行时总量稳定,未新增 cua_node 依赖包

macOS 的 Computer Use 运行时总量保持 377.4 MB → 377.4 MB,+0%;Windows 保持 298.9 MB → 298.9 MB,+0%。两平台 `cua_node` 依赖新增包数量均为 0。

平台差异主要是基线体积不同:macOS 的 runtime 总量大于 Windows,但本批次两者都没有可见依赖扩容。

### 【实证】[仅 Windows] Windows owl 运行时归档哈希变化

Windows 的 `resources/owl-electron-app.json` 中 `runtimeArchiveSha` 从 `c36221f...` 变为 `b631326...`,`runtimeName` 仍为 `owl`,`packagedFrom` 仍指向 Windows x64 Electron 输出目录。

这是 Windows 打包层的明确配置变化;macOS 输入样本没有同名配置项。

### 【信号】[仅 macOS] 前端路由和预加载图谱大幅重排,设置、远程连接、agent 设置与 appgen 相关路径更密集

macOS 的 `app.asar` 内部文件从 2656 到 2654,新增 992 / 移除 994 / 改动 12;去掉 Vite hash 噪音后,新增模块集中在 `settings-page`、`agent-settings`、`remote-connections-settings`、`remote-conversation-page`、`new-thread-panel-page`、`appgen-settings-page`、`appgen-library-page`、`worktree-init-v2-page`、`pull-requests-page`。`webview/index.html` 和 `avatar-overlay-composition-surface.html` 的 modulepreload 列表也从旧的 `plug~...` 路径转向新的 `new-~...`、`agent-settings`、`settings-page`、`remote-connections`、`plan-su` 等组合。

这些是路由、代码分包和预加载依赖的信号,不能等同于用户已经看到新入口;但它说明 macOS 包内前端结构在本批次有明显重排。

### 【信号】[仅 Windows] 线程侧栏、底部面板、浏览器面板和任务入口信号增强

Windows 的 `thread-side-panel-tabs` 新增字符串覆盖浏览器标签音频、摄像头/麦克风状态、下载行在系统文件浏览器中显示、地址栏建议、设备工具栏宽高和缩放、取消 git action、关闭设备工具栏、dismiss browser annotate coachmark。`thread-app-shell-chrome` 新增字符串覆盖从线程侧栏打开 side chat、browser tab、review tab、workspace file browser,以及底部面板打开 / 关闭 / 再开一个 tab。

这些都来自同名前端模块改动线索,因此是 UI 和交互方向信号。macOS 本输入样本没有同名 Windows 侧栏模块的对应新增线索。

### 【信号】[仅 Windows] Skills 页面出现创建、刷新、推荐和空状态相关文案

Windows 的 `skills-page` 新增字符串包括创建新 skill、刷新 skills list、新安装 skills 需要 refresh 才能在 composer 使用、过滤后空状态、installed skills、recommended skills、搜索输入、Loading skills 等。

这是技能页可见交互的强信号,但仍属于同名前端模块线索,不能单独断言功能已经对所有用户开放。

### 【信号】[macOS+Windows] 插件卡片“Plugin added”状态文案被移除

两平台 `plugins-page` 都移除了 `Plugin added`、`Status label for a plugin that is added to Codex`、`plugins.card.addedStatus`。这是跨平台一致的前端文案移除信号。

边界:它说明包内对应状态标签文案发生变化,但不能仅凭此判断插件安装状态模型被删除。

### 【信号】[仅 macOS] 插件详情页出现 MCP enablement、workspace 分享和 skill preview 相关线索

macOS 的 `plugin-detail-page` 新增字符串包括 MCP server enablement toggle、插件详情页 section navigation、workspace users and groups 搜索、移除 selected workspace share target、`Add people or groups`、`Anyone in this workspace with the link`、app tools modal / skill preview modal / skill card actions 的 more actions menu。

这是插件详情页向分享、权限和 MCP 开关管理扩展的信号。Windows 本输入样本没有同名模块的这些线索。

### 【信号】[仅 macOS] Appgen 站点分享文案进入包内

macOS 的 `appgen-share-dialog` 新增字符串包括搜索 workspace users and groups、保存按钮 saving 状态、复制 live site URL、外部打开 live site、发布到 public internet、`Anyone on the Internet`、`Can visit`、`Copied to clipboard`、`Copy link`。

这说明站点分享、公开发布和链接复制相关 UI 文案进入 macOS 前端包;仍不能据此断言所有用户可见。

### 【信号】[仅 macOS] Local conversation 内出现 MCP app UI host、CSP 与文档类型相关字符串

macOS 的 `local-conversation-page` 新增字符串包括 “App exposes MCP-style tools that the host can call.”、“CSS variable keys available to MCP apps for theming.”、“Content Security Policy configuration for UI resources.”、“Host supports receiving content messages (ui/message) from the view.”、“Host supports tool use via”。同时出现 `mentions/search`、`pages-document`、`powerpoint-deck`、`word-document`、`font-src`、`media-src` 等样式/协议线索。

这是 MCP app 嵌入、主题变量、iframe 内容、CSP 配置和文档类资源处理的强信号。macOS 的 `webview/index.html` 主 CSP 本身未显示策略域新增,仍是 `default-src 'none'` 并允许 self、app、blob、data、指定 web sandbox、OpenAI / Mapbox / ChatGPT websocket 等既有范围。

### 【信号】[仅 Windows] Worktree 初始化失败后的修复流程文案新增

Windows 的 `worktree-init-v2-page` 新增字符串包括 local environment setup 失败后继续启动线程、打开 local environment settings、启动 repair thread、取消 worktree setup 并开始 local conversation、`Continue anyway`、`Edit environment`、`Fix this project`、`Fix worktree setup`、repair thread 的 pending worktree label。

注意平台差异很尖锐:macOS 同名模块反而移除了这些字符串。因此本文只能说 Windows 本输入样本新增该修复流程信号,macOS 本输入样本则显示这些文案从同名模块中移除。

### 【信号】[仅 macOS] 浏览器任意网站访问确认文案从 pending request 面板移除

macOS 的 `pending-request-item-panel` 移除了 “Allow Codex to interact with any website without asking?”、“Allow any website”、取消 / 确认允许 Browser Use 访问任意网站的按钮文案,以及相关 warning 和 `composer.browserWebsiteAccess.confirm.allowAnyWebsite`。

这是权限确认 UI 文案调整信号。它不等于能力被移除,只能说明这些字符串不再出现在该同名前端模块差异中。

### 【信号】[macOS+Windows] Worktree / PR / new-thread 相关分包持续变化

macOS 新增和预加载列表中多次出现 `worktree-init-v2-page`、`pull-requests-page`、`new-thread-panel-page`、`remote-conversation-page` 的组合;Windows 侧栏字符串出现 “A pull request already exists for this branch”、“Accept edits to stage them”、“canceling an active git action”。这些共同指向线程、PR、git 操作和 worktree 工作流的持续迭代。

证据边界:macOS 主要是模块和预加载图谱信号,Windows 主要是前端可读字符串信号。

### 【信号】[仅 macOS] 自动化页出现 agent environment、setup import、archive scheduled task 与底部面板线索

macOS 的 `automations-page` 新增字符串包括 `Agent environment`、`Agent setup imported`、`Archive and remove`、`Archive chat and remove scheduled task?`、`Auto detect language option`、`Bottom panel`、`Bring over your setup, projects, and recent chats`。同时移除了多项 composer、cybersecurity banner、AGENTS.md init 和 accessibility 文案。

这说明自动化页和迁移 / 导入 / 归档任务状态相关 UI 正在重组。Windows 本输入样本没有 automations-page 改动线索。

### 【信号】[仅 macOS] Debug modal 扩展到 app actions、browser tab、conversation、allow host 与 worktree cleanup

macOS 的 `debug-modal` 从 0.0 MB 到 0.2 MB,新增字符串包括 `(command unavailable)`、`Account ID`、`Active browser tab`、`Active conversation`、`Allow automatic worktree cleanup in this unpackaged build.`、`Allow host`、`Allow host starting`、`Ambient suggestion chats`、`App Actions`。

这是内部调试 / 开发构建控制面的信号,不应解读为普通用户设置项已公开。

### 【信号】[仅 macOS] Header 出现 recent tasks、Cloud tasks、Chrome computer use settings 与 Codex for Chrome 文案

macOS 的 `header` 新增 recent tasks menu、unread task updates、All environments filter、All tasks、App settings、Chrome extension side panel header、Chrome computer use settings、Cloud tasks、Codex for Chrome 等字符串。

这说明 macOS 前端头部区域正在接入任务、环境过滤和 Chrome / cloud task 入口相关文案。Windows 本输入样本没有 header 模块对应线索。

### 【信号】[仅 macOS] 代码高亮 / 语法包重新分布

macOS 多个语言模块出现明显体积迁移:TypeScript、TSX、C、R、Haxe、Java、Turtle、GDScript shader、CMake 新增大量 grammar scope 字符串;JavaScript、JSX、Less、Markdown、Ruby 则移除大量 grammar scope 字符串。`diff` 模块新增 unified diff / git diff 相关 scope,`workbook` 从 0.0 MB 到 2.8 MB 并新增 range、theme color 与 Granola 数据属性线索。

这是编辑器 / 渲染器语法资源分包重排信号,不是用户功能结论。

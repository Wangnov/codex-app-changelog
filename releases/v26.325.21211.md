# Codex 桌面版 `26.325.21211` build 1255

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这次从 `26.324.21641` 到 `26.325.21211` 的变化不大，但方向很清楚：启动与更新链路开始面向多构建 flavor 和 Windows 更新器重构，远程控制、心跳自动化、权限模式和连接器设置也出现了可见推进。  
English one-liner: this build appears to reorganize startup/update plumbing while surfacing signals for heartbeat automation, remote control, permissions, and connector privacy flows.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次为历史回填，官方 appcast 没有为这一对版本提供增量包，因此对两个官方全量包做直接比较。

对比范围是 macOS arm64，上一版 `26.324.21641` build `1228`，最新版 `26.325.21211` build `1255`，两者发布日期均为 `2026-03-26`。

证据分两档：

- **【实证】**：来自可读配置、可读 diff、Info.plist、package.json、CSP、服务入口、签名 identifier、依赖清单、主进程 bundle 新增字符串等直接证据。
- **【信号】**：来自前端模块名、同名前端模块字符串变化、样式/状态类名或二进制体积变化。这类证据只指向某方向，但无法从产物还原到源码级意图。

已忽略 `webview/assets` 和 `.vite/build` 下 hash 文件名重命名造成的噪音。app.asar 内部 `新增 598 / 移除 600` 主要受 Vite 构建 hash 影响，不作为真实功能增删统计解读。

## ✨ 重点变化

### 1. 启动链路引入构建 flavor、独立 userData 路径与 Windows AppUserModelId 【实证】

新版 `.vite/build/bootstrap.js` 不再只按 macOS packaged 状态初始化，而是解析 product/build flavor，设置应用名，并按 flavor 计算 `userData` 路径；在 Windows 上还会设置 AppUserModelId。

这意味着同一个桌面代码库正在为不同运行形态做更明确的隔离，尤其是 `Agent` 运行态会根据 `CODEX_ELECTRON_AGENT_RUN_ID` 进入独立的 `agent/<run_id>` 数据目录。

关键证据：

```diff
+n.app.setName(e.n(s)),n.app.setPath(`userData`,i(s)),process.platform===`win32`&&n.app.setAppUserModelId(e.a(s));
```

### 2. 更新器从单一 Sparkle 脚本走向跨平台更新抽象 【实证】

`package.json` 移除了 `build:sparkle-native` 脚本，新增 `codexSparklePublicKey`，主进程新增字符串中同时出现 Sparkle addon、Windows updater、Windows update manifest、package identity、package URL、updater unavailable 等路径。

可感知点是：macOS 仍保留 Sparkle feed，但更新初始化、检查、安装失败路径正在被统一到更通用的 updater 抽象里；Windows 更新器相关错误文案已经进入产物。

关键证据：

```diff
-    "build:sparkle-native": "node ./scripts/build-sparkle-native.mjs",
+  "codexSparklePublicKey": "rhcBvttuqDFriyNqwTQJR3L4UT1WjIK4QxtwtwusVic="
```

新增字符串包括：

```text
Failed to load native Windows updater addon
Windows update manifest is missing required fields.
Windows update package URL must use https transport.
failed to initialize sparkle addon
```

### 3. 心跳自动化能力进入桌面端主进程与前端状态层 【实证】

主进程新增了多条 `Heartbeat automation` 相关字符串，覆盖 feature disabled、not due yet、thread missing、renderer/thread state blocked、timeout、busy、active heartbeat 等状态；前端 `index` 模块也新增了 heartbeat interval、automation type、pinned thread 等可访问性文案。

这指向一个围绕线程定期检查或提醒的自动化机制。直接字符串可证明相关状态和错误路径已进入产物，但具体触发条件和完整 UI 流程仍无法从压缩产物完整还原。

关键证据：

```text
Heartbeat automation skipped: not due yet
Heartbeat found something worth your attention.
That thread already has an active heartbeat.
Timed out waiting for heartbeat automation.
```

### 4. 远程控制环境连接能力扩展 【实证】

新版主进程新增了 remote control websocket payload、JSON-RPC transport、desktop app-server client authentication、ChatGPT login forwarding、VS Code remote open、apply patch helpers 等字符串；前端新增真实模块 `use-remote-connection-states`，设置页模块新增 `Remote control environments` 相关文案。

这意味着远程控制环境已经不只是 SSH 连接错误处理，而是开始出现远程控制环境列表、连接、认证和能力协商。产物明确包含“不支持 ChatGPT login forwarding / VS Code remote open / apply patch helpers yet”的路径，说明这些能力边界也被显式建模。

关键证据：

```text
connect remote control environments
load remote control environments
Remote control transport expected JSON-RPC objects.
Remote control transport requires a desktop app-server client for authentication.
```

### 5. 权限模式设置更细：默认、自定义与 Full access 【信号】

`external-agent-config-utils` 与 `agent-settings` 同名模块出现了权限模式相关新增文案，包括 `Default permissions`、`Custom (config.toml)`、`Full access`、`Guardian approvals`、`Full access is disabled by requirements.toml`。

这指向 agent 权限配置 UI 变得更明确，可能把全局默认、项目覆盖、自定义 `config.toml` 与 full access 控制区分开来。但这些线索来自同名前端模块变化，只能说明 UI、状态和配置方向，无法还原源码级意图。

关键证据：

```text
Default permissions mode
Custom config.toml settings
Enable full access?
Full access is disabled by requirements.toml
```

### 6. 连接器与应用设置页补上隐私、权限和风险提示 【信号】

`plugins-page` 模块体积从 `0.2 MB` 增至 `0.3 MB`，新增了连接器隐私、站点风险、授权范围、个性化 toggle、app actions menu、加载失败和空状态文案；同时 `control-group` 移除了部分相同文案，表明相关 UI 可能从通用控件移动到了插件页。

这指向连接器连接流程更强调隐私边界和风险提示，但仍属于同名前端模块改动线索，不能当作源码级事实。

关键证据：

```text
Connectors may introduce risk
ChatGPT is strictly limited to permissions you
By default, we do not train on your data. Data from this connector may be used to provide you relevant and useful information.
```

### 7. Markdown 与代码高亮资源重新分布，更多语言语法进入独立模块 【信号】

`markdown` 模块体积从 `0.0 MB` 增至 `0.7 MB`，`javascript`、`csharp`、`yaml`、`html`、`r`、`graphql`、`tex` 等模块新增大量语法 scope 字符串；与此同时 `wasm`、`typescript`、`less` 模块体积下降。

这指向 Markdown 渲染、数学公式和代码高亮资源有重新打包或能力扩展，但它主要来自前端模块体积与语法字符串变化，无法判断最终用户界面上具体新增了哪些语言或渲染特性。

### 8. SQLite rebuild 被纳入开发启动路径 【实证】

`package.json` 中 `dev` 脚本新增了 `pnpm run rebuild:sqlite`，此前该步骤已存在于 `test`、`forge:package`、`forge:make` 等路径中。

这意味着开发模式启动前也会重建 SQLite native 依赖，减少开发环境与打包/测试环境之间的 native module 差异。

关键证据：

```diff
-    "dev": "pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
+    "dev": "pnpm run rebuild:sqlite && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
```

### 9. 前端模块边界出现 VS Code、Windows sandbox 与 toast signal 【信号】

去掉 hash 噪音后，新版真实新增前端模块包括 `vscode-api`、`windows-sandbox-queries`、`toast-signal`、`use-platform`、`use-remote-connection-states`、`code-theme`；移除模块包括 `playwright`、`product-event-signal`、`use-os-info`、`with-window` 等。

这些模块名指向平台抽象、VS Code API、Windows sandbox 查询、远程连接状态与 toast 信号等方向，但模块出现本身不等于功能对用户开放，也无法从产物还原源码级意图。

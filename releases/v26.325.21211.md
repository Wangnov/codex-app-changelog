# Codex 桌面版 `26.325.21211`(build 1255)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版从文件规模看是一次较小更新，但启动入口和前端模块边界都有可读变化。  
One-liner: a small desktop update with clearer product/runtime bootstrapping and signals around platform and remote-connection surfaces.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次是历史回填：事实包说明两个版本均已超出官方 Sparkle(应用自动更新框架) 增量窗口，因此改为下载两个官方全量包直接比较，而不是通过增量包重建。

对比范围为 macOS arm64 平台，从 `26.324.21641` build `1228` 到 `26.325.21211` build `1255`，两个版本发布日期均为 `2026-03-26`。

可信度方面，两个制品都来自官方全量包地址，并提供了 SHA-256(文件哈希)。公证验证结果均返回 `CSSMERR_TP_CERT_REVOKED`。事实包特别说明：这会影响该版本能否被系统安全安装，但不影响对 bundle(应用包) 内容做差异分析。

本文使用两档证据等级：

- **【实证】**：来自 Info.plist(应用元信息文件)、package.json(应用包配置文件)、HTML(网页入口文件) / CSP(内容安全策略)、可读 diff(差异文本) 等直接证据。
- **【信号】**：来自前端模块名或二进制体积变化，只能说明“指向某方向，但无法从产物还原到源码级意图”。

## ✨ 重点变化

### 1. 启动入口开始按产品形态设置应用名与数据目录 【实证】

`bootstrap.js` 的启动逻辑不再只按当前平台和打包状态初始化，而是新增了产品名解析、应用名设置、用户数据路径设置，以及 Windows(微软桌面操作系统) 的 AppUserModelId(Windows 应用身份标识) 设置。

这意味着桌面端启动时开始更明确地区分不同运行形态或产品上下文；对用户可感知的结果可能体现在不同实例、数据目录或系统级应用身份的隔离上。

关键证据：

```diff
+n.app.setName(e.n(s)),n.app.setPath(`userData`,i(s)),process.platform===`win32`&&n.app.setAppUserModelId(e.a(s));
```

同时，新的用户数据路径逻辑还读取 `CODEX_ELECTRON_AGENT_RUN_ID`，并在 Agent(代理执行体) 场景下把路径落到 `agent/<run id>` 下：

```diff
+let a=process.env.CODEX_ELECTRON_AGENT_RUN_ID?.trim()||null;
+return r===e.u.Agent&&a!=null?(0,t.join)(i,`agent`,a):i
```

### 2. 更新流程配置收敛：移除 Sparkle 原生构建脚本，加入公钥配置 【实证】

`package.json` 中移除了 `build:sparkle-native` 脚本，同时新增了 `codexSparklePublicKey` 字段。开发脚本 `dev` 也开始先执行 `rebuild:sqlite`。

这说明更新相关配置从“本地构建 Sparkle 原生模块”的脚本形态，转向在包配置中携带 Sparkle 公钥；同时开发启动链路更强调 SQLite(轻量数据库) 原生模块的重建。

关键证据：

```diff
-    "build:sparkle-native": "node ./scripts/build-sparkle-native.mjs",
+  "codexSparklePublicKey": "rhcBvttuqDFriyNqwTQJR3L4UT1WjIK4QxtwtwusVic="
```

以及：

```diff
-    "dev": "pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
+    "dev": "pnpm run rebuild:sqlite && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
```

### 3. 前端模块信号指向平台适配、远程连接状态与 Windows sandbox 【信号】

去掉 Vite(前端构建工具) hash(内容指纹) 噪音后，真实新增模块包括 `use-platform`、`use-remote-connection-states`、`windows-sandbox-queries`、`vscode-api`、`toast-signal`、`code-theme`。

这些模块名指向平台识别、远程连接状态、Windows sandbox(Windows 沙盒环境)、VS Code(代码编辑器) API(应用程序接口)、提示消息与代码主题方向，但无法从产物还原到源码级意图，也不能证明相关功能已经对用户开放。

关键证据：

```text
新增模块: code-theme, toast-signal, use-platform, use-remote-connection-states, vscode-api, windows-sandbox-queries
```

### 4. 前端预加载入口从 `app-scope` 转向 `vscode-api` 【信号】

`webview/index.html` 的 modulepreload(模块预加载) 列表中，旧版的 `app-scope` 被移除，新版出现 `vscode-api`。这属于前端入口依赖形态的变化。

它指向 webview(内嵌网页视图) 侧可能在抽象应用上下文或编辑器集成边界，但无法从产物还原到源码级意图。

关键证据：

```diff
-    <link rel="modulepreload" crossorigin href="./assets/app-scope-BIXChNbS.js">
+    <link rel="modulepreload" crossorigin href="./assets/vscode-api-BtEtzXmw.js">
```

### 5. `sparkle.node` 原生文件从 unpacked 目录移除 【实证】

bundle 文件树显示移除了 `Contents/Resources/app.asar.unpacked/native/sparkle.node`，大小约 `0.1 MB`。结合 `package.json` 中移除 `build:sparkle-native` 脚本，这是更新相关原生组件布局变化的直接证据。

这不等同于“删除了自动更新功能”：同一份 `package.json` 仍保留 `codexSparkleFeedUrl`，并新增 `codexSparklePublicKey`。更准确的表述是，Sparkle 相关实现或打包方式发生了变化。

关键证据：

```text
移除: Contents/Resources/app.asar.unpacked/native/sparkle.node (0.1 MB)
```

### 6. 安全策略保持很窄的网络连接面 【实证】

新版 `webview/index.html` 中的 CSP 仍将 `connect-src` 限制在自身以及两个 OpenAI 相关域名：`https://ab.chatgpt.com` 与 `https://cdn.openai.com`。

这说明从这份入口文件看，webview 的显式网络连接白名单没有扩张到更多域名。

关键证据：

```html
connect-src 'self' https://ab.chatgpt.com https://cdn.openai.com;
```

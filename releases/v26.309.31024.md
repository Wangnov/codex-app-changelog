# Codex 桌面版 `26.309.31024`(build 962)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版从 `26.305.950` 升到 `26.309.31024`，可见变化集中在应用启动入口、前端预加载结构和 Electron(桌面应用运行框架) 桥接接口。  
One-liner: a packaging/bootstrap shift with a new renderer-to-main application menu bridge.

## 关于这份报告

这不是 OpenAI(人工智能公司) 官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次是历史回填：两个版本均已超出官方 Sparkle(应用自动更新框架) 增量窗口，因此采用两个官方全量包直接比较，而不是增量包重建。

对比范围为 macOS arm64(苹果芯片架构) 平台，从 `26.305.950`(build 863，2026-03-05) 到 `26.309.31024`(build 962，2026-03-12)。事实包给出了两个全量包的 SHA-256(文件哈希指纹) 和公证验证结果；两个版本的公证验证均返回 `CSSMERR_TP_CERT_REVOKED`。事实包同时说明，部分早期版本签名证书后来被吊销，这会影响系统安全安装判断，但不影响对 bundle(应用包目录) 内容做差异分析。

本文使用两档证据：

- **【实证】**：来自人类可读的直接证据，例如 `Info.plist`、`package.json`、预加载脚本 diff、CSP(内容安全策略) 文本。
- **【信号】**：来自模块名、体积变化等间接证据；这类信息只指向某方向，但无法从产物还原到源码级意图。

## ✨ 重点变化

### 1. 打包启动入口从 `main.js` 切到 `bootstrap.js` 【实证】

`package.json` 中的应用入口由 `.vite/build/main.js` 改为 `.vite/build/bootstrap.js`。

这意味着桌面端主进程启动链路至少在打包层面发生了调整；从产物只能确认入口文件变化，不能进一步判断源码中的启动职责如何重新分配。

关键证据：

```diff
-  "main": ".vite/build/main.js",
+  "main": ".vite/build/bootstrap.js",
```

### 2. 前端新增“显示应用菜单”的桥接接口 【实证】

预加载脚本中，暴露给前端的 `electronBridge` 新增了 `showApplicationMenu`，会通过 IPC(进程间通信) 调用 `codex_desktop:show-application-menu`，并传入 `menuId`、`x`、`y`。

这表示渲染层现在可以请求主进程在指定位置打开某个应用菜单。用户可感知层面，可能对应更原生的菜单弹出、上下文菜单或工具菜单体验；但事实包只能证明接口存在，不能证明具体入口已经对用户开放。

关键证据：

```diff
+showApplicationMenu:async(t,n,r)=>{await e.ipcRenderer.invoke(`codex_desktop:show-application-menu`,{menuId:t,x:n,y:r})}
```

### 3. 前端入口预加载清单大幅收敛，并引入消息总线、加载页与追踪相关模块 【实证】

`webview/index.html` 中的 `modulepreload`(模块预加载) 列表从大量细分资源变为更短的入口依赖集合，并出现 `trace`、`sentry-init.electron`、`react-dom`、`message-bus`、`loading-page` 等资源名。

这说明前端打包结构发生了明显变化：更多资源可能被重新分块或延迟加载。`message-bus` 指向前端消息分发方向，`loading-page` 指向启动或加载体验方向，`trace` 与 `sentry-init.electron` 指向追踪 / 错误上报初始化方向；但仅凭构建产物不能还原源码级意图。

关键证据：

```diff
+    <link rel="modulepreload" crossorigin href="./assets/trace-BQZGuJVA.js">
+    <link rel="modulepreload" crossorigin href="./assets/sentry-init.electron-eALzzrLX.js">
+    <link rel="modulepreload" crossorigin href="./assets/react-dom-D6EtqKiU.js">
+    <link rel="modulepreload" crossorigin href="./assets/message-bus-NQka-uRz.js">
+    <link rel="modulepreload" crossorigin href="./assets/loading-page-k_t1ZbLn.js">
```

### 4. 应用版本号与构建号更新 【实证】

`Info.plist` 与 `package.json` 均显示版本从 `26.305.950` 升至 `26.309.31024`，构建号从 `863` 升至 `962`。

这是本次差异的版本边界，也说明应用内部元数据与包级元数据一致更新。

关键证据：

```diff
-  "version": "26.305.950",
+  "version": "26.309.31024",
-  "codexBuildNumber": "863",
+  "codexBuildNumber": "962",
```

### 5. 主程序二进制体积明显增加 【信号】

`Contents/Resources/codex` 从 `88.0 MB` 增至 `104.3 MB`，增加 `16.30 MB`；`app.asar` 从 `36.6 MB` 增至 `41.4 MB`，增加 `4.78 MB`。

这指向桌面端主程序和应用层资源都有实质变化，但体积变化只能说明产物规模变化，无法从产物还原到源码级意图，也不能单独证明新增了某项用户功能。

### 6. 新增波斯语相关框架资源 【实证】

bundle 文件树新增的最大文件是 Electron Framework 下的 `fa.lproj/locale.pak`，同时出现 `fa_FEMININE.lproj`、`fa_MASCULINE.lproj`、`fa_NEUTER.lproj`，Sparkle 资源中也出现 `fa.lproj/Sparkle.strings`。

这说明底层框架资源中新增了 `fa` 相关本地化文件。它是否会在 Codex 应用内形成完整语言支持，事实包没有给出直接证据。

关键证据：

- `Contents/Frameworks/Electron Framework.framework/Versions/A/Resources/fa.lproj/locale.pak`：`1.0 MB`
- `Contents/Frameworks/Sparkle.framework/Versions/B/Resources/fa.lproj/Sparkle.strings`
- `Contents/Frameworks/Electron Framework.framework/Versions/A/Resources/fa_FEMININE.lproj/locale.pak`
- `Contents/Frameworks/Electron Framework.framework/Versions/A/Resources/fa_MASCULINE.lproj/locale.pak`
- `Contents/Frameworks/Electron Framework.framework/Versions/A/Resources/fa_NEUTER.lproj/locale.pak`

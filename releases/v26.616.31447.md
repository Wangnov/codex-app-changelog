# Codex 桌面版 26.616.31447 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

这一批次最值得注意的是：两端都更新了 Browser / Chrome 内置插件版本标记，Windows 还把 Computer Use 的 Windows Sky 运行时改为按运行时 node_modules 路径动态加载。  
One-liner: this batch is mostly a packaged-runtime and UI-surface alignment update, with a Windows-only Computer Use loader fix and clear cross-platform baseline skew.

## 关于这份报告

本文只依据两份官方签名分发包逆向重建后的机器提取事实：macOS arm64 从 `26.616.30709 build 4108` 到 `26.616.31447 build 4133`，Windows x64 MSIX 从 `26.616.3309.0` 到 `26.616.3767.0`。两端版本号和发布时间基线不同，因此每条结论均单独标注平台归属。

证据分两档：  
【实证】来自 plist、manifest、插件清单、脚本源码 diff、依赖清单、签名验证、文件统计，以及事实包中不属于“同名前端模块改动线索”的可读配置或源码差异。  
【信号】来自同名前端模块的新增/移除字符串、样式类名、模块名和体积变化，只说明包内 UI/交互方向发生变化，不等于确认功能已经对所有用户可见。

平台标记含义：`[macOS+Windows]` 表示两端事实包都有同向证据；`[仅 macOS]` / `[仅 Windows]` 表示只在对应平台事实包中出现。

## ✨ 重点变化

### 【实证】[macOS+Windows] Browser 与 Chrome 内置插件版本同步到 26.616.31447

两端都显示 `openai-bundled/plugins/chrome/.codex-plugin/plugin.json` 的 `version` 从 `26.616.30709` 更新到 `26.616.31447`。两端也都显示 `openai-bundled/plugins/browser/.codex-plugin/plugin.json` 的 `version` 从 `26.616.30709` 更新到 `26.616.31447`。

插件描述文本本身没有显示语义变化：Chrome 仍围绕用户现有 Chrome 状态、标签页、登录会话、Cookie、扩展和 Codex Chrome Extension 设置；Browser 仍强调 Codex 内置浏览器、localhost / file URL、截图、点击、输入和本地目标检查。

平台差异：macOS 事实包只列出 Browser 与 Chrome 两个插件清单版本变更；Windows 除 Browser 与 Chrome 外，还出现 Computer Use 插件和脚本改动。

### 【实证】[仅 Windows] Computer Use 插件版本更新，并修正 Windows Sky 运行时加载方式

Windows 的 `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` 显示 `computer-use` 插件版本从 `26.616.30709` 更新到 `26.616.31447`，描述为通过 Computer Use 从 Codex 控制 Windows 桌面应用。

更关键的是 `computer-use-client.mjs` 的源码 diff：它不再顶层静态导入 `WindowsComputerUseClientBase`，而是在 `platform() === "win32"` 时，通过 `NODE_REPL_NODE_MODULE_DIRS` 环境变量定位 node_modules，再动态导入 `@oai/sky/.../computer_use_client_base.js`。若找不到该路径，会抛出 `Windows Computer Use Sky runtime is unavailable`。

这说明 Windows 包内 Computer Use 运行时加载边界发生了实质调整：从固定依赖解析，改为依赖特权 Node REPL 暴露的模块目录。macOS 事实包没有对应脚本 diff，不能把这个结论外推到 macOS。

### 【信号】[macOS+Windows] @ mention 列表正在扩展为能力与对象入口

两端 `at-mention-list` 都新增同一组可读字符串，包括 `Computer use`、`Custom agents`、`Desktop apps`、`Files and chats`、`Live agents`、`MCP servers`、`Mac apps`，以及 MCP server mention scope、用户分组、agents/chats 加载状态文案。样式类名也新增 `mcp-capability`、`mcp-capability-mentions`、`mcp-servers`。

这是强 UI 信号：composer 的 @ mention 候选不只是文件或聊天对象，也在面向 Computer Use、自定义 agents、MCP servers、桌面应用等能力入口组织。由于证据来自同名前端模块改动线索，这里不能断言所有入口已开放或后端已完全启用。

### 【信号】[仅 macOS] 插件详情页增加分享、启用和技能预览相关 UI 线索

macOS 的 `plugin-detail-page` 体积增加 `133332 bytes`，新增字符串包括插件分享对话框中的工作区用户/群组搜索、移除分享目标、`Add people or groups`、`Anyone at workspace with the link`、`Anyone in this workspace with the link`、`Add plugin`、`App unavailable`，以及 MCP server enablement toggle、section navigation、skill preview modal、skill card actions menu 等无障碍标签。

这是一组很集中的插件管理信号：插件详情页可能在强化工作区分享、可见性、启用开关、技能预览和更多操作菜单。Windows 事实包没有 `plugin-detail-page` 对应改动，不能写成跨平台实证。

### 【信号】[仅 macOS] worktree 初始化失败后的修复路径更明确

macOS 的 `worktree-init-v2-page` 新增 `Creating worktree`、`Continue anyway`、`Edit environment`、`Fix this project`、`Cancel button for worktree creation`，以及本地环境设置失败、创建 local task 失败、启动修复线程失败等文案。

这说明 worktree 初始化流程在 macOS 包内新增或改写了失败后的继续、编辑环境、启动修复线程、取消并转本地对话等路径。证据仍是前端模块信号，因此只能说明 UI 文案和状态进入包内。

### 【信号】[仅 Windows] Markdown 渲染增加文件引用、媒体 fallback、表格复制和选中文本加入聊天线索

Windows 的 `markdown` 模块增加 `71991 bytes`，新增字符串覆盖 file citation chip、无扩展名文件引用、markdown image/video fallback、image bytes loading placeholder、`Add to chat`、选中文本 overlay 加入聊天、`Copy table`。样式类名新增 `data-markdown-copy`、`data-markdown-image-preview-trigger`、`data-markdown-table`、`cursor-zoom-in`、`data-hex-color-swatch`。

这是 Windows 侧高可见阅读体验信号：Markdown 内容中的文件引用、图片/视频加载失败状态、表格复制、图片预览和选中文本加入聊天都可能被重做或加入。macOS 事实包没有同名 `markdown` 改动。

### 【信号】[仅 Windows] Header、最近任务、设置入口和云任务相关 UI 前进；macOS 同名线索则被移除

Windows 的 `header` 新增 `All tasks`、`All environments filter option`、`App settings`、最近任务菜单无障碍标签、打开设置按钮、打开 Chrome extension thread、清空搜索、`Chrome computer use settings`、`Cloud tasks`。  
macOS 的 `header` 则移除了部分同类字符串，包括最近任务菜单、`All tasks`、`App settings`、返回按钮和设置入口。

这不是“一个平台有功能、另一个平台没有”的直接结论，而是典型平台基线错位信号：Windows 从较早基线补入这些 header 文案，macOS 在本次相邻版本 diff 中反向移除了部分同名线索。

### 【信号】[仅 Windows] Home announcements 增加多类首页横幅；macOS 同名线索减少

Windows 的 `home-announcements` 新增 Codex rate limit reset、Fast mode、backend-driven home banner、multi-agent composer banner、remote connections banner、`Add Connections`、ambient suggestions、rate limit resets modal、`Codex app` 等文案。  
macOS 同一模块则移除这些横幅相关文案的一部分。

因此，Windows 侧可以说首页公告/横幅能力在该版本差异中进入包内；macOS 侧只能说本输入样本显示同名横幅文案减少，不能推断 macOS 永久取消这些能力。

### 【信号】[仅 Windows] Thread shell 的侧栏/底栏 tab 操作进入包内；macOS 同名线索减少

Windows 的 `thread-app-shell-chrome` 新增侧边栏打开 side chat、browser、review、terminal、timeline、workspace file browser 的动作标签，也新增 bottom panel、关闭底栏、从底栏或侧栏打开另一个 tab、空状态文案，以及 `(max-width: 920px)` 响应式线索。  
macOS 同名模块移除了类似侧栏/底栏动作字符串。

这说明 Windows 的 thread shell 包内 UI 正向增加侧栏/底栏导航与响应式布局线索；macOS 在本次 diff 中则是同名内容减少。

### 【信号】[仅 Windows] Skills 页面增加创建、刷新、推荐、搜索和加载状态；macOS 同名线索减少

Windows 的 `skills-page` 新增创建 skill、刷新 skills list、新安装 skill 需要刷新、过滤空状态、installed / recommended skills 标题、搜索输入、`Loading skills...` 等文案。  
macOS 的 `skills-page` 移除创建、刷新、空状态、页面标题、installed skills 标题等文案。

这组信号说明 Windows 侧 Skills 页面 UI 更完整地进入包内；macOS 侧本次相邻版本 diff 显示这些同名线索减少。

### 【信号】[macOS+Windows] Browser use settings 与 RPC 调试文本减少

两端 `browser-use-settings` 都移除 embedded browser downloads、breadcrumbs、contact info、extension settings、password manager settings、`Access history without asking` 等设置文案；Windows 额外移除 site settings、隐藏清理选项相关标签。  
两端 `rpc` 都移除一组 RPC stub、ReadableStream/WritableStream hook、promise export ID 复用等错误字符串。

这更像打包结构、调试暴露、设置页拆分或压缩产物变化的信号。它不等于功能删除实证，但说明这些可读错误/设置文本不再出现在对应同名前端模块中。

### 【信号】[仅 macOS] Browser annotation 离开拦截提示进入包内

macOS `thread-side-panel-tabs` 新增 `Send annotations or cancel to exit annotation mode`、`Toast shown when the user tries to leave browser annotation mode while annotations are pending`、`browserSidebarCommentRuntime.toast.tweaksExitBlocked`，并移除 `thread.browser.tweaks.submitBanner.dismiss`。

这是浏览器 annotation 模式下未发送标注时阻止离开的 UI 信号。Windows 事实包没有对应条目。

### 【信号】[仅 macOS] Debug modal 与 Browser settings 的大量可读调试/设置文本被移除

macOS `debug-modal` 大幅减少 `194063 bytes`，移除 `(command unavailable)`、`Account ID`、`Active browser tab`、`Active conversation`、`All types`、`Allow automatic worktree cleanup in this unpackaged build.`、`Allow host` 等字符串。  
同平台 `browser-use-settings` 也减少 `108280 bytes`，移除 embedded browser 多项设置文案。

这说明 macOS 包内调试 modal 和 browser settings 的可读表面显著收缩。是否是功能迁移、拆分、压缩或开关条件变化，事实包不足以确认。

### 【信号】[仅 Windows] General settings 中权限、字体、语音热键等文案减少

Windows `general-settings` 减少 `117154 bytes`，移除 disabled default permissions toggle、composer context usage indicator、work mode options、code base size、hold-to-dictate hotkey capture、hotkey window capture 等文案。

这是 Windows 设置页可读表面的减少信号，涉及权限、上下文用量指示、工作模式、代码字号和语音/热键捕获。但由于只来自同名前端模块线索，不能直接写成设置项被移除。

# Codex 桌面版 26.519.22136(macOS / Windows 对照)

> 第三方跨平台变更日志 · 非官方

本批次的核心差异是:macOS 版进入 26.519.22136(build 3003)并带来多处 Chrome(谷歌浏览器)、Browser(浏览器自动化)与应用壳层变化；Windows(微软桌面系统) 26.519.2736.0 的 MSIX(微软应用安装包格式)对照包没有任何文件级变化。

One-liner: macOS moved forward with Chrome/profile and app-shell changes, while the Windows package in this comparison stayed byte-for-byte unchanged.

## 关于这份报告

本报告基于两份官方签名分发包的逆向 diff(差异对比):macOS 来自 Sparkle(应用更新框架)相关全量包对比，Windows 来自 MSIX 包对比。macOS 对比范围为 26.513.31313(build 2867) → 26.519.22136(build 3003)；Windows 对比范围为 26.519.2736.0 → 26.519.2736.0。

可信度分两档:【实证】表示来自 plist(应用元数据)、AppxManifest(Windows 应用清单)、CSP(内容安全策略)、技能文档、类型声明、package.json(包配置文件)、依赖清单或可读脚本 diff；【信号】表示只来自前端模块名或体积变化，只能说明方向，不能还原源码意图或确认功能已开放。

平台归属标记含义:`[macOS+Windows]` 表示两平台事实包都体现；`[仅 macOS]` 表示只在 macOS diff 中出现；`[仅 Windows]` 表示只在 Windows diff 中出现。本批次尤其要注意:Windows 对照包 SHA-256 完全相同，文件数、内容改动和 app.asar(应用代码归档)内部文件均无变化。

## ✨ 重点变化

### Chrome 扩展安装检测改为多配置档扫描

【实证】[仅 macOS] Chrome 插件的 `check-extension-installed.js` 不再只检查一个选中的 Chrome profile(用户配置档)，而是会扫描每个可用的 `Default` 或 `Profile X` 目录，并输出每个 profile 的安装、注册、启用、禁用原因和版本信息。JSON(结构化数据格式)输出新增 `profiles` 数组和 `selectedProfileDirectory` 提示。

平台差异:macOS 包中脚本和技能文档都已更新；Windows MSIX 对照包没有任何内容改动，因此没有证据显示 Windows 这批已经跟上。

### Chrome native messaging host 安装清单开始写入 app-server 配置

【实证】[仅 macOS] Chrome 插件的 `installManifest.mjs` 新增 `channel:"prod"`、`extension-host-config.json` 写入逻辑，以及 `codexCliPath`、`browserClientPath`、`nodePath`、`nodeReplPath`、`proxyHost`、`proxyPort` 等 app-server-runtime(应用服务运行时)配置字段。新增文件列表也显示 Chrome 插件下出现 `app-server-runtime/codex`、`app-server-runtime/node`、`app-server-runtime/node_repl`。

平台差异:macOS 已把 Chrome native messaging host(浏览器本地通信宿主)从单纯 manifest(清单文件)安装扩展为带运行时配置的安装；Windows 事实包显示 0 个新增文件、0 个内容改动。

### Chrome 插件版本号与桌面版本对齐

【实证】[仅 macOS] `plugins/chrome/.codex-plugin/plugin.json` 中 Chrome 插件版本从 `0.1.7` 改为 `26.519.22136`，与 macOS 桌面短版本一致。

平台差异:此变化只在 macOS diff 出现；Windows 对照包未发生任何插件文件变化。

### 浏览器标签接管与保留语义调整

【实证】[仅 macOS] Chrome 技能文档更新了 tab(浏览器标签页)接管说明:接管已打开的 Chrome 标签后，不再描述为移动到 agent tab group(智能体标签组)，而是描述为当前浏览器会话获得控制权。`finalize` 清理语义也调整为:deliverable(交付物)和 handoff(交接)标签释放会话控制后保持打开，而不是统一移动到 `✅ Codex` 标签组。

平台差异:macOS 的 Chrome 技能文档已更新；Windows 无对应文件变化证据。

### Browser 与 Chrome 技能文档放宽 waitForTimeout 禁令，并澄清 keep 字段

【实证】[仅 macOS] Browser 与 Chrome 两份技能文档都删除了“不要使用 `tab.playwright.waitForTimeout(...)`”这一条，并把 `FinalizeTabsOptions.keep` 从“Tabs to keep open”改为“清理后明确保留的标签处置”，同时把 `FinalizeTabsKeep.tab` 描述为要保留的 tab object(标签对象)。

平台差异:只在 macOS 包中出现文档变更；Windows 对照无变化。

### 前端应用壳层、工作区、Git 与插件页面出现大规模模块重组信号

【信号】[仅 macOS] 去除 Vite(前端构建工具) hash(哈希)噪音后，macOS app.asar 中新增大量未混淆模块名，包括 `app-shell-state`、`app-shell-tab-controller`、`app-window`、`apps-availability`、`browser-sidebar-manager`、`git-branch-switcher`、`plugin-auth`、`plugin-config-edits`、`thread-side-panel-tabs`、`worktree-environment-dropdown` 等。与此同时也移除了 `global-settings`、`plugins-page-section`、`plugins-page-selectors`、`remote-control-connections-visibility`、`vscode-api` 等模块。

平台差异:这批前端重组信号只在 macOS app.asar 中出现。Windows app.asar 内部文件为 1530 → 1530，新增 0、移除 0、改动 0，因此没有同类信号。

### macOS 启动引导增加额外启动门控信号

【信号】[仅 macOS] `.vite/build/bootstrap.js` 中的可读差异显示，主应用启动前新增了一层条件判断，只有 `await t.g()` 成立后才初始化 Sparkle 并导入 main 模块。由于这是打包后的前端/启动代码片段，事实包无法还原该条件的业务含义。

平台差异:仅 macOS 包出现该启动路径变化；Windows 包无内容改动。

### Windows 对照包没有实质更新

【实证】[仅 Windows] Windows 侧上一版与最新版均为 `26.519.2736.0`，两个全量包 SHA-256 都是 `bf284a70f0c62da1911a62700465c31ce21f849278cd9a8290428c9b43accda1`。文件数 832 → 832，新增 0、移除 0、内容改动 0；app.asar 内部文件 1530 → 1530，新增 0、移除 0、改动 0。

平台差异:这意味着在本次跨平台对照中，Windows 包没有体现 macOS 侧的 Chrome、Browser、前端壳层和启动路径变化。

# Codex 桌面版 `26.429.20946`(build 2312)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版最明确的变化集中在 Browser Use(内置浏览器控制插件) 的能力说明和接口暴露上，同时前端产物出现了多组指向应用连接、自动化、远程会话与热键窗口的模块信号。  
One-liner: Browser automation gets more explicit DOM-level controls, while the UI bundle points to broader app and workflow surfaces.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次是历史回填：两个版本均超出官方 Sparkle(应用更新框架) 增量窗口，因此采用两个官方全量包直接比较，而不是增量包重建。

对比范围是 `26.422.71525`(build 2210，2026-04-29) → `26.429.20946`(build 2312，2026-04-30)，平台为 macOS arm64。信任根来自 Apple(苹果) 公证与 OpenAI Developer ID(开发者身份签名)：两个解包后的 `Codex.app` 均通过 `spctl`(macOS 安全评估工具) 验证，结果为 `accepted`，来源为 `Notarized Developer ID`，签名主体为 `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`。

本文证据分两档：**【实证】**表示来自可读配置、技能文档、类型声明、`package.json`、`Info.plist` 或签名验证等直接证据；**【信号】**表示仅来自前端模块名或二进制体积变化，指向某方向，但无法从产物还原到源码级意图。

## ✨ 重点变化

### 1. Browser Use 增加 DOM 级浏览器操作接口 【实证】

Browser Use 的技能文档新增了 `dom_cua` 接口，包含按可见 DOM(文档对象模型) 节点点击、双击、滚动、输入、按键，以及读取可见 DOM 快照的能力。

这意味着 Codex 在内置浏览器里做页面测试时，不只依赖坐标或 Playwright(浏览器自动化工具) 快照，也开始暴露更接近“按页面元素操作”的控制面。关键证据是 `Tab` 接口新增 `dom_cua: DomCUAAPI`，并定义了 `get_visible_dom()`、`click({ node_id })`、`type({ text })`、`scroll({ node_id?, x, y })` 等方法。

### 2. Browser Use 的定位从“本地目标”扩展为更完整的内置浏览器测试工具 【实证】

`browser/SKILL.md`、`agents/openai.yaml` 和 `plugin.json` 都调整了描述：新版明确写到可用于打开和控制 Codex 内置浏览器，执行导航、检查、点击、输入、截图等测试任务，主要面向本地开发页面和文件。

用户可感知点是插件说明更像一个完整的浏览器测试能力，而不只是针对 `localhost`、`127.0.0.1`、`::1`、`file://` 的窄范围入口。插件默认提示也从两个示例收敛为 `"Test my checkout flow on localhost"`，品牌色从 `#0F766E` 改为 `#013B7B`。

### 3. 截图回传规则更明确：用户需要看到的截图应内嵌到回复中 【实证】

Browser Use 技能文档新增一条操作规范：如果截图是给用户看的，应当在 Markdown(标记文本格式) 回复中内嵌图片。

这会影响 Codex 使用内置浏览器做可视化验证时的输出方式：不只是“我截图了”，而是更倾向于把截图作为结果的一部分展示出来。证据来自 `browser/SKILL.md` 新增行：`If you take a screenshot that the user should see, include the image inline in your Markdown response.`

### 4. Browser Use 的市场分类从 Engineering 调整到 Research 【实证】

内置插件市场清单中，Browser Use 的 `category` 从 `Engineering` 改为 `Research`。

这不一定代表底层能力改变，但会影响它在插件市场或分类入口中的呈现方式。证据来自 `Contents/Resources/plugins/openai-bundled/.agents/plugins/marketplace.json`：`"category": "Engineering"` → `"category": "Research"`。

### 5. Computer Use 插件版本升级，并新增 Apple Events 权限说明 【实证】

Computer Use(让模型操作桌面 / 浏览器的能力) 插件版本从 `1.0.758` 升到 `1.0.770`，同时主应用 `Info.plist` 新增 `NSAppleEventsUsageDescription`：`Codex uses Apple Events to control Mac apps on your behalf.`

这说明桌面应用控制能力在权限说明层面更明确，macOS 在相关授权场景中可能展示这段用途说明。与此同时，Computer Use 相关二进制也有体积变化：`SkyComputerUseService` 增加 `+0.38 MB`，`SkyComputerUseClient` 减少 `-0.48 MB`，这些体积变化只作为辅助信号，不单独推断具体功能。

### 6. 前端模块指向应用连接、OAuth、自动化、远程会话和热键窗口等新界面方向 【信号】

去掉 `webview/assets` 和 `.vite/build` 的 hash(构建指纹) 噪音后，真实新增模块中出现了 `app-connect-modal`、`app-connect-oauth`、`automations-page`、`remote-connections-page`、`hotkey-window-*`、`pull-requests-page`、`plugin-detail-page`、`file-preview-page`、`editor-diff-page`、`scratchpad-page` 等名称。

这些模块名指向应用连接、OAuth(开放授权)、自动化页面、远程连接、热键窗口、Pull Request(代码合并请求) 看板、插件详情、文件预览和差异编辑等方向，但无法从产物还原到源码级意图，也不能证明这些界面已对所有用户开放。

### 7. Webview 预加载层新增 MCP 应用沙箱消息通道 【实证】

`.vite/build/preload.js` 新增了 `codex_desktop:mcp-app-sandbox-host-message` 相关监听，并通过 `window.postMessage` 把消息转发到页面上下文。

这表明应用前端与某种 MCP(模型上下文协议) 应用沙箱宿主消息之间新增了桥接路径。由于证据来自压缩后的预加载脚本，只能确认通道存在，不能进一步推断具体产品形态。

### 8. 构建脚本从单一 SQLite 重建扩展为 native modules 重建，并出现 Windows Computer Use 打包脚本 【实证】

`package.json` 中多个脚本从 `rebuild:sqlite` 改为 `rebuild:native-modules`，对应脚本文件名也从 `rebuild-sqlite.mjs` 变为 `rebuild-native-modules.mjs`。同时新增 `computer-use:windows:bundle`，指向 `scripts/windows-computer-use-plugin-bundle.ts`。

这说明构建流程不再只围绕 SQLite(嵌入式数据库) 原生模块重建命名，而是扩展为更通用的原生模块重建；新增 Windows Computer Use 打包脚本则指向跨平台插件打包工作，但这只是构建配置层面的实证，不能说明 macOS 版用户直接获得了 Windows 能力。

### 9. 多语言菜单文案有小幅修订 【实证】

多个 `native-menu-locales` 文件发生了文案调整，包括 `fa.json`、`mr-IN.json`、`ar.json`、`zh-TW.json`、`hr-HR.json`、`so-SO.json`、`tl.json`。

这些改动主要影响浏览器侧边栏上下文菜单、Intel / Apple Silicon 启动提示等本地化表达。例如繁体中文 `browserSidebar.contextMenu.forward` 从“前進”改为“下一頁”。这类变化属于可见文案打磨，不代表新功能。

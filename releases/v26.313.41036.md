# Codex 桌面版 `26.313.41036`(build 1032)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这是一次从 `26.311.21342` 到 `26.313.41036` 的小版本更新，变化集中在底层可执行文件、前端模块编排，以及 macOS(苹果桌面操作系统) 权限声明。  
English one-liner: a small desktop update with a larger core binary, several frontend direction signals, and a Bluetooth permission key change.

## 关于这份报告

这不是 OpenAI(人工智能公司) 官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。

本次事实包说明：这是一版历史回填，两个版本均已超出官方 Sparkle(应用自动更新框架) 增量窗口，因此改为下载两个官方全量包直接比较。常规方法论是：取官方上一版全量包与官方增量包，做 EdDSA(一种数字签名算法) 验签，再通过 BinaryDelta(二进制差分重建机制) 重建新版，最后做结构化 diff(差异比较)。本次实际使用的是两个官方全量包的直接比较。

可信度方面，两个全量包均给出了 SHA-256(文件哈希校验值)。公证验证结果均为 `CSSMERR_TP_CERT_REVOKED`，事实包明确说明：这类早期版本的证书吊销会影响系统安全安装判断，但不影响对 bundle(应用包目录) 内容做差异分析。

证据等级分为两档：

- **【实证】**：来自 Info.plist(苹果应用配置文件)、package.json(应用包配置文件)、CSP(内容安全策略) 文本、依赖清单、签名标识等人类可读直接证据。
- **【信号】**：来自前端模块名或二进制体积变化等间接证据，只能说明“指向某方向，但无法从产物还原到源码级意图”。

对比范围：`26.311.21342` build `993`，发布于 `2026-03-13`；对比到 `26.313.41036` build `1032`，发布于 `2026-03-16`。

## ✨ 重点变化

### 1. 版本号与构建号推进到 `26.313.41036` / build `1032` 【实证】

本版将应用版本从 `26.311.21342` 提升到 `26.313.41036`，构建号从 `993` 提升到 `1032`。这说明这是一次正式打包发布，而不是单纯资源重签或本地构建差异。

关键证据：

```diff
- "version": "26.311.21342",
+ "version": "26.313.41036",

- "codexBuildNumber": "993",
+ "codexBuildNumber": "1032",
```

Info.plist 中也同步记录了：

```diff
CFBundleShortVersionString: 26.311.21342 → 26.313.41036
CFBundleVersion: 993 → 1032
```

### 2. 蓝牙权限声明键发生调整 【实证】

macOS 权限声明从 `NSBluetoothPeripheralUsageDescription` 切换为 `NSBluetoothAlwaysUsageDescription`，说明应用仍声明“需要访问 Bluetooth(蓝牙)”这一能力，但配置键发生了变化。

可感知点是：如果系统触发相关权限提示，应用包内对应的权限说明仍是 `This app needs access to Bluetooth`，但新版使用的是另一个 Info.plist 键。

关键证据：

```diff
+ /NSBluetoothAlwaysUsageDescription = "This app needs access to Bluetooth"
- /NSBluetoothPeripheralUsageDescription = "This app needs access to Bluetooth"
```

### 3. 前端新增 `alert`、`banner`、`app-scope`、`product-event-signal` 模块信号 【信号】

去掉 Vite(前端构建工具) hash(构建指纹) 噪音后，事实包识别出真实新增前端模块：`alert`、`app-scope`、`banner`、`product-event-signal`。这些模块名指向提示、横幅、应用作用域、产品事件信号等方向，但无法从产物还原到源码级意图，也不能证明对应功能已经对用户开放。

可感知点是：本版前端层可能在补充通知展示、范围控制或产品事件信号相关的界面基础设施；但这仍只是模块命名层面的方向信号。

关键证据：

```text
新增模块: alert, app-scope, banner, product-event-signal
移除模块: pencil, product-events, vscode-api
```

### 4. `vscode-api`、`pencil`、`product-events` 模块不再出现在去噪后的前端模块清单中 【信号】

去噪后的前端模块清单显示，`vscode-api`、`pencil`、`product-events` 被移除。这里不能写成“移除了 VS Code 集成”或“删除了某个功能”，因为事实包只证明模块名层面发生变化。

可感知点是：前端内部的模块边界或命名可能在调整，尤其是 `product-events` 到 `product-event-signal` 的变化，指向事件相关实现的重组，但无法从产物还原到源码级意图。

关键证据：

```text
移除模块: pencil, product-events, vscode-api
新增模块: product-event-signal
```

### 5. 底层 `codex` 可执行文件体积增加约 `1.64 MB` 【信号】

`Contents/Resources/codex` 从 `104.8 MB` 增至 `106.4 MB`，增加 `1.64 MB`。这是本次体积变化最大的组件，指向底层核心程序发生了实质变化，但仅凭体积变化无法判断具体功能、性能或行为变化。

可感知点是：如果本版有运行时行为变化，更可能来自这一底层组件或其配套前端更新；但这里仍属于体积信号，不构成源码级事实。

关键证据：

```text
Contents/Resources/codex: 104.8 MB → 106.4 MB, +1.64 MB
```

### 6. 应用前端包 `app.asar` 小幅增大，完整性哈希更新 【实证】

`Contents/Resources/app.asar` 从 `41.6 MB` 增至 `41.7 MB`，增加 `0.11 MB`；同时 ElectronAsarIntegrity(Electron 应用包完整性校验) 中的 `app.asar` 哈希发生变化。结合前端模块清单，可确认应用层资源确实重新打包。

可感知点是：前端资源有变更，但事实包也明确提醒，`webview/assets` 与 `.vite/build` 下大量 hash 文件名增减主要是构建产物重命名，不能据此判断新增或删除功能。

关键证据：

```diff
ElectronAsarIntegrity/Resources/app.asar/hash:
9d34e603d40fdc8decaca6f37ff874c2fa9d65aa135ea427c3e9edc65bcfdf3b
→

```

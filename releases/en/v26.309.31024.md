# Codex Desktop `26.309.31024` build 962

> Third-party changelog В· unofficial В· based on reverse diffs of official update artifacts

This release is less about one isolated feature and more about a broader expansion of startup flow, plugin/MCP paths, settings permissions, and diagnostics.

## About This Report

This is not an official changelog. It is a **reverse diff report based on official distribution artifacts**. This entry is a historical backfill: the official appcast did not provide a Sparkle delta for `26.305.950` вҶ’ `26.309.31024`, so the comparison used two official full packages instead.

The comparison covers macOS arm64: `26.305.950` build `863`, released on `2026-03-05`, versus `26.309.31024` build `962`, released on `2026-03-12`.

Evidence is split into two levels:

- **[Confirmed]**: direct human-readable evidence, such as `package.json`, `preload.js`, `webview/index.html`, `Info.plist`, newly added main-process bundle strings, file-tree data, and signature verification output.
- **[Signal]**: indirect evidence from frontend module names, changed strings/styles inside same-name frontend modules, or size deltas. These point in a direction but cannot be reconstructed into source-level intent.

Hashed filenames under `webview/assets` and `.vite/build` are treated as build artifact renames. They are not interpreted as feature additions or removals.

## вңЁ Key Changes

### 1. Main process entry moved to bootstrap, making startup flow more explicit [Confirmed]

The Electron entry in `package.json` changed from `.vite/build/main.js` to `.vite/build/bootstrap.js`. The main-process bundle also gained strings such as `Desktop bootstrap failed to start the main app`, `bootstrap handoff complete`, `main app.whenReady resolved`, and `Session Crashed`.

This means desktop startup now has a more explicit bootstrap phase, with new paths for startup failure, session crash handling, and handoff. Key evidence: `"main": ".vite/build/main.js"` вҶ’ `"main": ".vite/build/bootstrap.js"`.

### 2. Application menu and update actions are exposed through the frontend bridge [Confirmed]

`preload.js` added `showApplicationMenu`, which invokes the main process through `codex_desktop:show-application-menu`. The main-process bundle also added `Check for Updates`, `Install Update`, and `application menu refreshed`.

The user-visible implication is that desktop menu and update actions are more likely to be triggered or refreshed from the frontend, instead of relying only on static native menu paths.

### 3. Plugin marketplace and MCP runtime evidence expanded substantially [Confirmed]

The main-process bundle added `Failed to fetch plugins from app server`, `plugin.json`, and `marketplace.json`, along with many `mcp.*` fields such as `mcp.tool.name`, `mcp.resource.uri`, `mcp.prompt.result`, `mcp.progress.percentage`, and `mcp.session.id`.

This shows that plugin marketplace metadata, plugin descriptors, and MCP lifecycle data are moving into a more central runtime path. The fact package does not provide a built-in plugin inventory change, so this is not evidence of specific new bundled plugins.

### 4. Agent settings show signals for admin config, network access, and approval policy [Signal]

The same-name `agent-settings` frontend module added strings including `Admin config`, `Allow network access`, `Approval policy`, `Approval policy is restricted by this installation.`, and `Can edit files outside this workspace`.

These clues point to more granular permission, network access, and administrator-controlled states in settings. Because they come from same-name frontend module diffs, they cannot be reconstructed into source-level intent.

### 5. Usage settings point toward auto top-up and payment management changes [Signal]

The same-name `usage-settings` frontend module added `Automatically add credit when you reach your minimum balance.`, `Unable to open payment settings right now. Please try again.`, and `manage-payment`, while removing several older minimum-balance and auto-top-up strings.

This points to UI and error-handling changes around auto top-up configuration and payment management, but it remains a frontend signal rather than proof of a complete billing-flow change.

### 6. Remote SSH messaging shifted from GitHub CLI discovery toward Codex-managed SSH [Confirmed]

The main-process bundle added `Codex-managed SSH block is malformed`, `failed to read Codex-managed SSH connections`, and `failed to refresh remote SSH connections`; it also removed `Failed to parse GitHub CLI response`, `Unexpected GitHub CLI response`, `gh auth login`, and `github.com`.

This shows a visible messaging shift in remote connection error paths: away from GitHub CLI login/parsing prompts and toward Codex-managed SSH connection read/write and refresh paths.

### 7. Thread steer and app terminal states gained new error paths [Signal]

The same-name `app-server-manager-hooks` frontend module added `Failed to read the app terminal for this thread.`, `No app terminal session is attached to this thread yet.`, and `note: output is truncated to the latest terminal buffer kept by the app.` The main-process bundle also added `Missing thread follower steer-turn response` and `Unknown thread follower steer-turn response requestId`.

These clues point to more detailed handling around thread terminal snapshots, truncated terminal output, and steer acceptance flow. The frontend-module portion remains a UI/state signal.

### 8. Observability, debugging, and automatic instrumentation expanded [Confirmed]

The main-process bundle added `Desktop Sentry test error (debug button)`, `No Sentry client available, profiling is not started`, `Automatic instrumentation of `, `auto.http.otel.connect`, `auto.db.supabase`, `auto.function.mcp_server`, and `The statsigIntegration() can only be used in the browser.`, among other strings.

This shows broader runtime coverage for diagnostics, profiling, feature-flag integrations, and automatic instrumentation. These strings alone do not prove actual collection policy or default enablement.

### 9. Webview first-load structure was rearranged [Confirmed]

`webview/index.html` still points to a new `index` chunk, but many older business chunk preloads were removed. The new preload set includes startup-oriented modules such as `trace`, `sentry-init.electron`, `react-dom`, `message-bus`, and `loading-page`.

The visible direction is a frontend load path split into startup, message bus, loading page, and main app phases. Performance gains cannot be inferred from the artifact alone.

### 10. Frontend packages for syntax highlighting, math, and browser rendering were reshaped [Signal]

After hash-noise filtering, new modules include `shiki-default-theme`. Same-name modules such as `tsx`, `python`, `shellscript`, `go`, `html`, `postcss`, `angular-html`, and `json` show many syntax-scope string changes. `katex` grew by `263700 bytes`, and `browser` grew by `651436 bytes`.

These signals point to reorganized or expanded resources for code highlighting, math rendering, and browser rendering, but they do not reconstruct source-level intent.

### 11. New `fa` localization resources appeared [Confirmed]

The bundle file tree added `Contents/Frameworks/Electron Framework.framework/Versions/A/Resources/fa.lproj/locale.pak`, along with `fa_FEMININE.lproj`, `fa_MASCULINE.lproj`, and `fa_NEUTER.lproj` resources. Sparkle also added `fa.lproj/Sparkle.strings`.

This confirms that `fa` localization resources are now present in the distribution package. It does not prove that full product language switching is exposed to users.

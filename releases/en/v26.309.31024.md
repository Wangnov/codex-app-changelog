# Codex Desktop `26.309.31024` (build 962)

> Third-party changelog · Unofficial · Based on reverse-engineered diffs of official update packages

This release upgrades from `26.305.950` to `26.309.31024`, with visible changes concentrated around the application startup entry point, frontend preload structure, and Electron bridge interfaces.  
One-liner: a packaging/bootstrap shift with a new renderer-to-main application menu bridge.

## About This Report

This is not an official OpenAI changelog, but a **reverse-engineered diff report based on official distribution artifacts**. This comparison is a historical backfill: both versions have fallen outside the official Sparkle delta update window, so the analysis uses direct comparison of two official full packages rather than reconstructing delta packages.

The comparison covers the macOS arm64 platform, from `26.305.950` (build 863, 2026-03-05) to `26.309.31024` (build 962, 2026-03-12). The evidence package provides SHA-256 hashes and notarization verification results for both full packages; notarization verification returned `CSSMERR_TP_CERT_REVOKED` for both versions. The evidence package also notes that signing certificates for some earlier versions were later revoked, which affects system security installation decisions but does not affect diff analysis of the bundle contents.

This report uses two evidence levels:

- **[Confirmed]**: Human-readable direct evidence, such as `Info.plist`, `package.json`, preload script diffs, and CSP text.
- **[Signal]**: Indirect evidence from module names, size changes, and similar indicators. These point in a direction, but cannot recover source-level intent from artifacts.

## ✨ Key Changes

### 1. Packaged startup entry point changed from `main.js` to `bootstrap.js` [Confirmed]

The application entry point in `package.json` changed from `.vite/build/main.js` to `.vite/build/bootstrap.js`.

This means the desktop main-process startup chain changed at least at the packaging layer. From the artifact alone, we can confirm only the entry file change; we cannot determine how startup responsibilities were redistributed in the source code.

Key evidence:

```diff
-  "main": ".vite/build/main.js",
+  "main": ".vite/build/bootstrap.js",
```

### 2. New frontend bridge interface for showing the application menu [Confirmed]

In the preload script, the `electronBridge` exposed to the frontend now includes `showApplicationMenu`, which invokes `codex_desktop:show-application-menu` via IPC and passes `menuId`, `x`, and `y`.

This indicates that the renderer layer can now ask the main process to open a given application menu at a specified position. At the user-visible level, this may correspond to a more native menu popup, context menu, or tool menu experience; however, the evidence package can only prove that the interface exists, not that a specific entry point has already been exposed to users.

Key evidence:

```diff
+showApplicationMenu:async(t,n,r)=>{await e.ipcRenderer.invoke(`codex_desktop:show-application-menu`,{menuId:t,x:n,y:r})}
```

### 3. Frontend entry preload list significantly consolidated, with message bus, loading page, and tracing-related modules introduced [Confirmed]

In `webview/index.html`, the `modulepreload` list changed from many granular resources to a shorter set of entry dependencies, with resource names such as `trace`, `sentry-init.electron`, `react-dom`, `message-bus`, and `loading-page`.

This indicates a clear change in the frontend packaging structure: more resources may have been repartitioned or deferred. `message-bus` points toward frontend message dispatching, `loading-page` points toward startup or loading experience, and `trace` plus `sentry-init.electron` point toward tracing / error reporting initialization; however, build artifacts alone cannot recover source-level intent.

Key evidence:

```diff

+    <link rel="modulepreload" crossorigin href="./assets/sentry-init.electron-eALzzrLX.js">
```

`Contents/Resources/codex` grew from `88.0 MB` to `104.3 MB`, an increase of `16.30 MB`; `app.asar` grew from `36.6 MB` to `41.4 MB`, an increase of `4.78 MB`.

This points to substantial changes in both the desktop main executable and application-layer resources, but size changes only indicate changes in artifact scale. They cannot recover source-level intent from the artifact, nor can they independently prove that a specific user-facing feature was added.

### 6. New Persian-related framework resources [Confirmed]

The largest newly added file in the bundle file tree is `fa.lproj/locale.pak` under Electron Framework. Additional entries include `fa_FEMININE.lproj`, `fa_MASCULINE.lproj`, and `fa_NEUTER.lproj`, and Sparkle resources also include `fa.lproj/Sparkle.strings`.

This indicates that `fa`-related localization files were added to underlying framework resources. The evidence package does not provide direct evidence that this results in full language support within the Codex application itself.

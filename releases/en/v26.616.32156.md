# Codex Desktop 26.616.32156 macOS / Windows Comparison

> Third-party cross-platform changelog · unofficial

This diff shows macOS moving forward to 26.616.32156, while the Windows sample comes from the same mirror batch but has several frontend module signals moving in the opposite direction from macOS, so each platform must be read separately.

One-liner: this cross-platform batch bumps shared bundled plugin metadata, keeps Computer Use runtimes stable, and shows divergent frontend UI signals between macOS and Windows.

## About This Report

This report only uses machine-extracted facts from the supplied fact packages. The macOS comparison uses official full packages from 26.616.31447 build 4133 to 26.616.32156 build 4157. The Windows comparison uses x64 MSIX full packages from 26.616.3767.0 to 26.616.4196.0.

Evidence has two levels: [Confirmed] comes from plist, manifest, plugin config, dependency lists, service entries, signature verification, readable strings, and concrete file diffs; [Signal] comes from same-name frontend modules, size changes, CSS class names, and minified bundle directionality. Platform scope is marked as `[macOS+Windows]`, `[macOS only]`, or `[Windows only]`.

The comparison covers versions and channels, bundle trees, bundled plugin manifests, Computer Use runtime totals, cua_node dependencies, app.asar, same-name frontend modules, readable strings, Info.plist or AppxManifest changes, content security policy, added components/resources, and concrete key-file diffs. The input does not provide AppxManifest, CSP, newly bundled plugin details, or added component/resource details, so this report does not infer beyond that.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Bundled Browser and Chrome Plugins Bump to 26.616.32156

Both platforms show `browser` and `chrome` bundled plugin manifests moving from version 26.616.31447 to 26.616.32156. The Browser plugin description still covers the in-app browser, localhost, file URLs, screenshots, clicking, typing, and local web target inspection. The Chrome plugin description still targets automation that depends on the user's existing Chrome state, including tabs, logged-in sessions, cookies, extensions, and Codex Chrome Extension setup.

The platform difference is path and host package layout: macOS uses `Contents/Resources/plugins/openai-bundled/plugins/...`, while Windows uses `resources/plugins/openai-bundled/plugins/...`. Windows additionally shows a `computer-use` plugin manifest bump; the macOS fact package does not include the corresponding diff.

### [Confirmed] [Windows only] Computer Use Plugin Manifest Bumps, With Windows Desktop Control Description

Windows `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` shows version 26.616.31447 changing to 26.616.32156, with the description stating that it controls desktop apps on Windows from Codex through Computer Use.

This is Windows-only manifest evidence. The macOS input provides Computer Use runtime totals but no computer-use plugin manifest change.

### [Confirmed] [macOS+Windows] Computer Use Runtime Size and cua_node Dependencies Do Not Grow

The macOS Computer Use runtime total stays at 365.9 MB → 365.9 MB, and Windows stays at 285.6 MB → 285.6 MB. Both are +0 MB and +0%. The number of newly added cua_node dependency packages is 0 on both platforms.

This means the supplied samples do not show expansion in Computer Use runtime file totals or Node dependency package counts. Absolute sizes differ by platform: 365.9 MB on macOS and 285.6 MB on Windows.

### [Confirmed] [macOS only] App Version, Build, and app.asar Integrity Hash Update

macOS `Info.plist` shows `CFBundleShortVersionString` changing from 26.616.31447 to 26.616.32156, and `CFBundleVersion` changing from 4133 to 4157. `ElectronAsarIntegrity/Resources/app.asar/hash` also changes from the old hash to the new one.

This confirms updated macOS package metadata and app.asar integrity records. The Windows input does not provide AppxManifest changes, so no equivalent Windows manifest conclusion is made.

### [Signal] [macOS only] Debug Modal Expands With Account, Session, Browser Tab, and Host-Allowance Strings

### [Signal] [macOS only] Header Shows Signals for Recent Tasks, Cloud Tasks, Settings, and Chrome Extension Thread Entry

The macOS `header` module adds `All tasks`, `Cloud tasks`, `App settings`, `Clear search`, `All environments filter option`, and button labels for opening the recent tasks menu, opening settings from the header, and opening the active Chrome extension thread in the Codex desktop app.

This suggests header-related UI work around task menus, environment filtering, settings entry, and Chrome extension thread entry on macOS. The Windows `header` module removes these strings, so this cannot be described as a shared cross-platform enhancement.

### [Signal] [macOS only] Thread App Shell Shows Bottom Panel and Side Panel New-Tab Action Signals

The macOS `thread-app-shell-chrome` module adds `Bottom panel`, plus action labels for opening side chat, browser tab, review tab, terminal, timeline tab, and workspace file browser from the thread side panel. It also adds `(max-width: 920px)`.

This is a layout and navigation signal pointing to side panels, bottom panels, and multiple workspace entry points in the thread UI. The Windows same-name module removes the same group of strings, so the signal is macOS-only.

### [Signal] [macOS only] Skills Page Adds Installed Skills, Recommended Skills, Refresh, and Empty-State Copy

The macOS `skills-page` module adds copy for creating a new skill, reloading the skills list, newly installed skills requiring refresh before composer use, installed skills, recommended skills, search input, loading state, and filtered empty states.

### [Signal] [Windows only] Plugin Detail Page Expands With Plugin Sharing, MCP Toggle, and Skill Preview Actions

### [Signal] [Windows only] General Settings Expands With Permissions, Context Usage, Code Size, Dictation, Realtime Voice, and Popout Window Settings

The Windows `general-settings` module grows from near 0 MB to about 0.1 MB and adds strings for disabled default permissions toggle, composer context usage indicator, work mode settings, base size for code across chats and diffs, hold-to-dictate hotkey, realtime voice hotkey, wake word, dictation dictionary entry, and Popout Window projectless default setting toggle.

### [Signal] [Windows only] Browser Use Settings Expands With History Access, Download Domain Allowlisting, and Embedded Browser Settings

The Windows-only `browser-use-settings` module grows to about 0.1 MB and adds `Access history without asking`, embedded browser downloads, breadcrumbs, contact info settings, extension settings, password manager settings, site settings, data clearing options, unavailable plugins disabled toggle, `Add allowed domain`, and `Add allowed download domain`.

### [Signal] [macOS+Windows] Login Route Removes “Included with all ChatGPT plans” Onboarding Copy on Both Platforms

Both macOS and Windows `login-route` modules remove `Included with all ChatGPT plans`, `Badge text on the v2 desktop onboarding login page indicating Codex is included with ChatGPT plans`, and `electron.onboarding.login.includedPlans.welcomeV2`.

This is a same-direction frontend signal on both platforms: login or onboarding badge copy about Codex being included with ChatGPT plans is removed from the same-name module. Because the source is a same-name frontend module diff, it does not prove the live login page necessarily removed the display for every user.

### [Signal] [macOS+Windows] Permissions Mode Gains `not-thread`, While Service Tier Icons Removes the Same Class

Both platforms add the `not-thread` class/state name in `use-permissions-mode`; both platforms remove `not-thread` from `service-tier-icons`.

This is a lower-level UI state signal that may relate to permissions-mode display outside thread contexts, but the fact package provides no readable copy to support a stronger functional conclusion.

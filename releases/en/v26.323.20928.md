# Codex Desktop `26.323.20928` build 1173

> Third-party changelog · Unofficial · Based on reverse diffs of official update artifacts

This build moves from `26.318.11754` build 1100 to `26.323.20928` build 1173, with the most visible changes around startup experience, settings structure, plugin entry points, and code rendering.  
English one-liner: this build adds a native startup loader and shows early restructuring around plugins, settings, remote connections, and syntax rendering.

## About This Report

This is not an official changelog. It is a **reverse-diff report based on official distribution artifacts**. This entry is a historical backfill: the official appcast did not provide a Sparkle delta package for this version pair, so the fact pack compares two official full packages directly.

The comparison covers macOS arm64: previous version `26.318.11754` build 1100, released on `2026-03-19`; latest version `26.323.20928` build 1173, released on `2026-03-24`.

Evidence is split into two levels: **[Confirmed]** means the change is backed by human-readable evidence such as Info.plist, readable configuration, HTML/CSS, main-process bundle strings, signing data, or verification output. **[Signal]** means the evidence only comes from frontend module names, changed strings in same-name frontend modules, or size shifts; it points in a direction but cannot recover source-level intent from the artifact.

## ✨ Highlights

### 1. Inline Codex Logo Startup Loader Added [Confirmed]

`webview/index.html` now includes a full inline startup style and DOM block before the main React bundle loads: `startup-loader`, `startup-loader__logo`, `startup-codex-logo-fade-in`, and `startup-codex-logo-shimmer`, with an SVG mask for the Codex logo.

This means the app can show a lightweight startup placeholder earlier in the blank-screen phase, without waiting for the main frontend bundle to finish loading. The evidence comes from the readable `webview/index.html` diff, including `color-scheme: light dark`, `prefers-reduced-motion: reduce`, and `startup-codex-logo-shimmer 2200ms`.

### 2. Plugin Settings Are Being Split Into Dedicated Frontend Modules [Signal]

After removing hash-noise, the real new modules include `plugins-page`, `plugins-settings`, `settings-content-layout`, and `settings-sections`. The `settings-surface` module also adds `Title for plugins settings section` and `plugins-settings`.

This points to plugin-related settings being separated from the previous page structure, but the evidence comes from frontend module names and same-name module diffs, so it cannot establish source-level intent. Supporting signals include removals from `skills-page`: `All plugins`, `App enabled`, `App disabled`, and `Accessible label for the plugin enablement toggle`.

### 3. Remote Connection and Remote Project Path Flows Continue to Change [Confirmed]

New readable strings in the main-process/frontend bundle include `Remote command exec connection is unavailable` and `resolveRemoteSshCodexHome timed out`. The `remote-connections-settings` module adds `Add new connection` and `Button label to manually add a remote connection`.

This shows explicit readable traces for remote command execution availability, remote Codex home timeout handling, and manually adding remote connections. The `index` module also adds `Accessible label for the disabled input showing the current remote folder path` and `Add your remote project from this menu`, pointing to UI copy changes around adding remote projects.

### 4. General Settings Add Enter-Key Behavior Copy [Signal]

`general-settings` adds `settings.general.enterBehavior.label`, `settings.general.enterBehavior.description`, `Label for the enter key behavior toggle`, and `Description for the enter key behavior toggle`.

This points to an Enter-key behavior setting in General Settings, but the evidence comes from same-name frontend module diff signals, so it cannot recover the source-level design or confirm the exact configurable behavior.

### 5. Merge Conflict Display and Actions Show New UI Traces [Signal]

`general-settings` adds `(Current Change)`, `(Incoming Change)`, `merge conflict actions`, plus state/class names such as `data-merge-conflict-action`, `data-merge-conflict-conflict-index`, and `merge-conflict-action`.

This points to enhanced UI around merge conflict display or actions, but the evidence remains at the level of bundled frontend strings and class names, so the full interaction behavior cannot be confirmed.

### 6. Diff and Multilingual Syntax Highlighting Rules Were Heavily Reworked [Signal]

The `diff` module adds rules such as `markup.changed.diff`, `markup.deleted.diff`, `markup.inserted.diff`, `meta.diff.header.git`, and `meta.diff.range.unified`. The `jsx`, `php`, `less`, `python`, `java`, `shellscript`, and `ruby` modules grow and add many syntax-scope strings.

This points to repackaging or expansion of code display, diff rendering, and language highlighting capabilities, but the artifact cannot confirm source-level implementation. At the same time, `cpp`, `fortran-free-form`, `stylus`, `scss`, `vue`, `postcss`, `katex`, and `tex` show size reductions or removed strings, indicating syntax resources were also trimmed or reorganized.

### 7. Main Process Adds Window and Git Startup Error Copy [Confirmed]

The new main-process bundle contains readable strings `New Window` and `Failed to start git process`.

This confirms that the build includes copy for a new window entry point and a Git process startup failure path. Since the evidence is a string literal in a minified bundle, it confirms presence only, not the exact trigger conditions.

### 8. Bluetooth Permission Wording Moves From Always to Peripheral [Confirmed]

`Info.plist` removes `/NSBluetoothAlwaysUsageDescription` and adds `/NSBluetoothPeripheralUsageDescription = 'This app needs access to Bluetooth'`.

This is a macOS permission declaration change, and users may see a different Bluetooth permission category in system prompts or privacy settings. The version fields also move from `26.318.11754` / `1100` to `26.323.20928` / `1173`.

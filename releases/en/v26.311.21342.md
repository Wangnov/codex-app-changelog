## ✨ Key Changes

### 1. Appearance Settings Begin Integrating System Theme Variants [Confirmed]

`preload.js` adds bridge support related to system theme variants: `codex_desktop:get-system-theme-variant`, `codex_desktop:system-theme-variant-updated`, and exposes `getSystemThemeVariant` and `subscribeToSystemThemeVariant` to the frontend.

This means the frontend can now read and subscribe to changes in the system theme variant. The perceptible direction is more complete integration between appearance settings and the system theme, though the exact UI presentation still needs to be verified at runtime.

Key evidence:

```diff
+ getSystemThemeVariant
+ subscribeToSystemThemeVariant
+ codex_desktop:get-system-theme-variant
+ codex_desktop:system-theme-variant-updated
```

### 2. Theme Assets and Appearance Settings Modules Increase Notably [Signal]

After removing Vite hash-renaming noise, the real newly added modules include `appearance-settings`, `use-resolved-theme-variant`, along with multiple theme-name modules: `absolutely-dark`, `absolutely-light`, `linear-dark`, `linear-light`, `lobster-dark`, `matrix-dark`, `notion-dark`, `notion-light`, `proof-light`, `sentry-dark`, and `temple-dark`.

These module names point to expanded appearance settings and theme variants, but they are only directional signals. They cannot reconstruct source-level intent from the artifact, nor can they prove that all themes are available to users.

### 3. Application Layer Adds the `ssh-config` Dependency [Confirmed]

`package.json` adds the dependency `ssh-config`, with version range `^5.0.3`.

At minimum, this shows that the application layer has started bundling an SSH configuration-related library. Combined with remote-connection-related module names, it points toward remote environment configuration, but the dependency itself does not mean the feature is already available.

Key evidence:

```diff
+    "ssh-config": "^5.0.3",
```

### 4. New Frontend Modules Appear for Remote Connections and App Server Status [Signal]

New modules include `remote-connections-settings`, `app-server-connection-state`, `onboarding-login-content`, and `use-is-apps-enabled`; the entry preload list also includes `app-connect-modal` and `app-server-manager-hooks`.

These names point to ongoing organization around remote connection settings, app server connection state, and login onboarding interfaces. However, this remains module-name signal only: it cannot reconstruct source-level intent from the artifact and cannot prove that the feature entry points are stably available.

### 5. Module Signals Appear for Diff Viewing, Editing Entry Points, and VS Code Interfaces [Signal]

New modules include `diff-view-mode`, `pencil`, `arrow-left`, and `vscode-api`. Here, `diff-view-mode` points to a diff viewing mode, `pencil` points to editing actions, and `vscode-api` points to a VS Code interface layer.

These changes may relate to code viewing, editing, or external editor integration, but the evidence only comes from module names and should not be written as confirmed functionality.

### 6. Local Development Startup Script Adjusted [Confirmed]

In `package.json`, the `dev` script removes the leading `pnpm run rebuild:sqlite &&`, while `forge:package` and `forge:make` still retain `rebuild:sqlite`.

The developer-visible point is that running the `dev` script no longer executes that rebuild command first.

Key evidence:

```diff
- "dev": "pnpm run rebuild:sqlite && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
+ "dev": "pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
```

## ✨ Highlights

### 1. New macOS installation migration and DMG cleanup flow during startup [Confirmed]

The latest version adds a substantial block of macOS pre-start installation handling logic in `.vite/build/bootstrap.js`: when the app is not located in Applications, it attempts to move or copy the App Bundle into `/Applications`, displays an installation window, and later handles the source DMG.

This means that when users launch the app directly from a mounted image, the app may actively guide installation into Applications and then open the installed app after success; failure states also have corresponding status copy.

Key evidence includes:

```text
pending-source-dmg-cleanup.json
Installing ${t}
Moving ${t} to Applications…
hdiutil detach
shell.trashItem
```

### 2. CAPTCHA policy change for Browser / Chrome automation [Confirmed]

The skill documentation for both Browser and Chrome changed the same rule: the old version prohibited solving CAPTCHAs, while the new version says to ask the user whether they want each CAPTCHA solved, and only solve it after the user confirms.

This is noticeable for developers and users of browser automation capabilities: CAPTCHAs are no longer an absolute prohibition, but they still require user confirmation each time; bypassing paywalls, security blocks, age verification, and final password-change steps remains prohibited.

Evidence comes from matching diffs in the two skill documents:

```diff
- Do not solve CAPTCHAs ...
+ For each CAPTCHA you see, ask the user whether they want you to solve it. Solve that CAPTCHA only after they confirm.
```

### 3. New `sites` plugin entry point [Confirmed]

The bundled plugin marketplace configuration adds a plugin entry named `sites`, sourced from the local path `./plugins/sites`, with installation policy `AVAILABLE`, authentication policy `ON_INSTALL`, and category `Productivity`.

This shows that an installable Sites plugin entry has appeared in the official distribution package; however, the fact package only provides the marketplace configuration, so it cannot further confirm the concrete capability scope or whether it is available to all users.

Key evidence:

```json
"name": "sites",
"path": "./plugins/sites",
"installation": "AVAILABLE",
"authentication": "ON_INSTALL",
"category": "Productivity"
```

### 4. Update confirmation and About window copy expanded into localization bundles [Confirmed]

Multiple `native-menu-locales` localization files add two groups of UI copy: one for confirmation before installing an update, and another for version, release date, and build information in the About window.

This means the update flow can more explicitly warn that “Codex will quit to install the update, interrupting local sessions in progress on this device”; the About window also gains localized copy for displaying version and release date.

Direct evidence from the Chinese bundle:

```json
"appHeader.installUpdate.confirmTitle": "现在更新 Codex？"
"appHeader.installUpdate.confirmSubtitle": "Codex 将退出以安装更新，这将中断此设备上正在进行的本地会话"
"codex.aboutDialog.versionLineWithDate": "版本 {version} • 发布于 {releaseDate}"
```

### 5. Frontend modules related to plans, pricing, upgrades, and service tiers appeared [Signal]

After removing Vite hash noise, new frontend modules include names such as `plan-management-state`, `plan-pricing`, `upgrade-plan-dialog`, `get-codex-purchase-handoff-url`, `read-service-tier-for-request`, `service-tier-icons`, and `use-codex-pricing-url`.

These module names point toward subscription plans, pricing display, upgrade dialogs, purchase handoff, and service-tier display, but the artifact cannot restore source-level intent, nor can it determine whether these entry points are already open to users.

### 6. New frontend signals for browser sidebar, Appgen, thread rendering, and editor integration [Signal]

New modules also include `browser-sidebar-webview`, `browser-sidebar-retained-webview`, `browser-sidebar-hidden-background-webview-host`, `appgen-access-state-messages`, `appshot-availability`, `thread-virtualizer`, `thread-resource-card`, `pull-request-fix-button`, `vscode-api`, and others.

These module names point toward retained browser sidebar state, Appgen/Appshot availability, virtualized thread rendering, a pull request fix entry point, and VS Code integration, but the artifact cannot restore source-level intent and cannot prove that these capabilities have fully shipped.

### 7. Chromium runtime upgrade [Confirmed]

`Info.plist` shows the Chromium base version moving from `148.0.7778.179` to `149.0.7827.54`, with `ChromiumBaseBundleVersion` moving from `7778.179` to `7827.54`.

This is a lower-level runtime change that may affect the built-in web view, browser sidebar, or rendering behavior, but the fact package does not provide specific user-visible fixes.

Key evidence:

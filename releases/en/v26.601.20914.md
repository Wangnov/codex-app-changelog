# Codex Desktop 26.601.20914(macOS / Windows Comparison)

> Third-party cross-platform changelog · Unofficial

This batch shows the Codex desktop app moving into the 26.601 series on both macOS and Windows, but the packages are not perfectly lockstep: the shared app layer is closely aligned, while each platform exposes different shell and bundled-component changes.  
One-liner: Codex 26.601 refreshes the shared app layer, browser/control policies, and packaged runtime, with macOS adding installer migration logic and Windows exposing more DevTools/Sites payload.

## About This Report

This report is based on reverse diffs of official signed distribution packages: the macOS package was taken from Sparkle full downloads, and the Windows package from MSIX packages. The macOS comparison is 26.527.60818(build 3437) → 26.601.20914(build 3497); the Windows comparison is 26.527.7698.0 → 26.601.1994.0. The release date field is set to 2026-05-30 as provided by the input fact pack.

Evidence is split into two levels: [Confirmed] means readable evidence from plist, AppxManifest, skill docs, package.json, dependency lists, or readable diffs; [Signal] means only module names, filenames, or size changes are visible, which can indicate direction but cannot reconstruct source intent or confirm user-visible behavior.

Platform labels mean: [macOS+Windows] has evidence on both platforms; [macOS only] appears only in the macOS package; [Windows only] appears only in the Windows package. This report deliberately ignores webview/assets and .vite/build hash-renaming noise.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Versions and Runtime Base Move Into the 26.601 / Chromium 149 Batch

macOS moves from 26.527.60818(build 3437) to 26.601.20914(build 3497). Info.plist shows Chromium moving from 148.0.7778.179 / 7778.179 to 149.0.7827.54 / 7827.54. Windows moves from 26.527.7698.0 to 26.601.1994.0, and the file layer also shows removal of `148.0.7778.179.manifest` plus size changes in `chrome.dll` and `resources.pak`, indicating that the Windows package also went through a browser-runtime update.

The platform difference is that macOS provides explicit Info.plist fields and an SCMRevision change; the Windows fact pack mainly exposes file-tree and binary/resource size changes, without equivalent manifest fields.

### [Confirmed] [macOS+Windows] app.asar Grows Slightly, With Highly Similar Internal File Churn

Both platforms increase `resources/app.asar` by about +1.32 MB. On macOS, app-layer internal files move from 2774 to 2804, with 1001 added, 971 removed, and 77 changed. On Windows, the same layer moves from 2722 to 2752, with 1001 added, 971 removed, and 76 changed. Both fact packs explicitly note that large changes under webview/assets are Vite build-hash noise and should not be used to infer feature changes.

The main platform difference is the baseline file count and changed-file count: macOS has 52 more internal app.asar files than Windows and one more changed file, but the add/remove counts match exactly, suggesting the shared app layer is largely synchronized.

### [Confirmed] [macOS+Windows] Browser and Chrome Control Policy Now Allows CAPTCHA Solving After User Confirmation

On macOS, both the in-app browser skill document and the Chrome control skill document changed from “do not solve CAPTCHAs” to asking the user for confirmation for each CAPTCHA and solving only that confirmed CAPTCHA. The policy still forbids bypassing paywalls, browser/web safety interstitials, age verification, and the final password-change submission step.

The Windows fact pack does not include the same readable diff text, but the Windows app.asar add/remove shape closely matches macOS, and Windows adds `resources/plugins/openai-bundled/plugins/browser/.codex-plugin/control-in-app-browser-multi-tab.md`, showing browser-control plugin updates as well. Strictly speaking, the readable CAPTCHA policy evidence comes from macOS; the cross-platform label is based on the shared app-layer and plugin-change evidence across both packages.

### [Confirmed] [macOS+Windows] The Sites Plugin Enters the Packaged/Marketplace Surface, With More Visible Payload on Windows

On macOS, `marketplace.json` adds a `sites` plugin, sourced from `./plugins/sites`, with installation set to `AVAILABLE`, authentication set to `ON_INSTALL`, and category set to `Productivity`. On Windows, newly added files include `resources/plugins/openai-bundled/plugins/sites/skills/sites-building/templates/auth-siwc-enabled/package-lock.json` and `vinext-starter/package-lock.json`.

The platform difference is clear: macOS provides readable evidence that Sites entered the plugin marketplace index; Windows visibly ships Sites plugin skill template files. The Windows fact pack does not include a `marketplace.json` diff, so it cannot confirm that the Windows marketplace entry is identical.

### [Confirmed] [macOS only] New DMG-to-Applications Install Migration Flow

The macOS `bootstrap.js` readable diff adds `pending-source-dmg-cleanup.json`, detects whether the current app is running from `/Volumes/`, locates the source DMG, copies the app bundle into `/Applications`, opens the installed app, detaches the DMG, and moves it to Trash. It also adds an installer progress window with Installing, Moving to Applications, Opening, failed, and openFailed states.

The Windows package has no corresponding DMG or Applications-folder concept and no comparable evidence. This is the clearest macOS-only platform-shell change in this batch.

### [Confirmed] [macOS only] Update Confirmation, About Dialog, and Build Info Strings Expand Across Many Locales

macOS native menu locale files add keys such as `appHeader.installUpdate.*` and `codex.aboutDialog.*`, including strings for “Update Codex now?”, “the update will interrupt active local sessions,” “About {appName},” “Version {version},” and build information. The Simplified Chinese locale also adds the matching text.

The Windows fact pack does not include these locale diffs, so this report marks them as macOS-confirmed only. Given the close app.asar alignment, Windows may contain similar app-layer strings, but the provided input is insufficient to mark them as confirmed on Windows.

### [Signal] [macOS+Windows] Settings, Billing, Plan Management, Thread Virtualization, and PR Fix Entry Points Shift at Module Level

After removing hash noise, macOS adds modules such as `agent-settings.search`, `computer-use-settings.search`, `mcp-settings.search`, `personalization-settings.search`, `plan-management-state`, `plan-pricing`, `upgrade-plan-dialog`, `pull-request-fix-button`, `thread-virtualizer`, and `thread-resource-card`; removed modules include `plugin-auth`, `pull-request-board-model`, and `virtualized-turns`. Windows does not list module names, but its app.asar add/remove counts exactly match macOS.

These are directional signals, not proof that any feature is enabled. The platform difference is that macOS exposes module names, while Windows only provides matching app.asar statistics.

### [Signal] [macOS+Windows] Browser Sidebar, Hidden WebView, and Multi-Tab Browser Control Direction

macOS adds modules including `browser-sidebar-webview`, `browser-sidebar-retained-webview`, `browser-sidebar-hidden-background-webview-host`, and `browser-sidebar-hidden-browser-use-webview-host`. Windows adds `control-in-app-browser-multi-tab.md`. Together, these point toward updates around browser sidebar behavior, retained/hidden WebView hosts, and multi-tab browser control.

This remains a [Signal]: the fact packs do not provide source semantics or interaction paths, so they cannot confirm which capabilities are actually exposed in the UI.

### [Signal] [Windows only] DevTools / Lighthouse Payload Changes Significantly

On Windows, `gen/third_party/devtools-frontend/.../lighthouse-dt-bundle.js` grows from 2.1 MB to 6.3 MB, a +4.23 MB increase and the largest size change in the Windows package. Additional added/changed paths mention GreenDevAgent, WebMCPView, Puppeteer, AI assistance, and HeapSnapshot under DevTools frontend paths. The macOS fact pack does not include comparable file-level evidence.

This does not prove a new Codex user-facing DevTools feature. It only shows a substantial Chromium/DevTools resource update inside the Windows MSIX.

### [Signal] [macOS only] The owl Development Command Adds a Mock Keychain Flag

In macOS `package.json`, the `owl` script adds `--use-mock-keychain`. This is a development/test command-layer change, not a shown user feature. The Windows fact pack does not include a package.json diff, so it is marked macOS only.

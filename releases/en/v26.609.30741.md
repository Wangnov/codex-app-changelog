This is not an official changelog, but a **reverse-engineered diff report based on official distribution artifacts**. The method was to take the official Sparkle full package for the previous version and the official delta package, verify them with EdDSA, reconstruct the latest bundle with BinaryDelta, then perform structured diffs across files, configuration, dependencies, and readable text.

In terms of confidence, both the previous full package and the official delta package passed EdDSA verification with the official Sparkle public key. The reconstructed `Codex.app` passed `codesign --deep --strict`, and `spctl` reported it as `accepted`, with source `Notarized Developer ID` and signing subject `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`.

Evidence is divided into two levels: **[Confirmed]** means there is directly readable evidence such as Info.plist, CSP, skill documentation, type declarations, package.json, dependency lists, or signing identifiers; **[Signal]** means the evidence comes only from frontend module names or binary size changes, pointing in a certain direction but not allowing source-level intent to be reconstructed from the artifact.

## ✨ Highlights

### 1. Computer Use Runtime Adds PDF, OCR, Image Processing, and Browser Automation Dependencies [Confirmed]

`cua_node` adds 29 npm packages covering PDF parsing, OCR, image encoding/decoding and processing, Canvas, pixel-level comparison, browser automation, and related areas. The most direct change is that the runtime grows from `213.5 MB` to `365.8 MB`, an increase of `152 MB`, or `71%`.

This means the local Computer Use capability boundary shipped with Codex Desktop has expanded significantly: `pdfjs-dist` points to PDF reading, `tesseract.js` / `tesseract.js-core` point to OCR, `sharp` / `@img/sharp-libvips-darwin-arm64` / `@napi-rs/canvas` point to image processing and drawing, and `playwright` / `playwright-core` point to browser automation.

Key evidence: newly added dependencies include `pdfjs-dist@5.4.624`, `tesseract.js@7.0.0`, `sharp@0.34.5`, `@napi-rs/canvas@0.1.91`, and `playwright@1.57.0`.

### 2. `@oai/sky` Moves from a Prerelease Build to `0.4.10`, with Configurable Screenshot Mouse Size [Confirmed]

In Computer Use, `@oai/sky` is updated from `0.4.8-202606020558-pr-981870-f570e0604110` to `0.4.10`. Its type declarations add `mouse_size_px?: number`, indicating that the mouse pointer size in Linux target screenshots can be adjusted by pixel value; `0` disables it, and the default is `12`.

The user-visible implication is more controllable screenshot output, especially for automation flows that rely on visual positioning or comparison. Multiple Linux operation wrappers also now pass `options` into the underlying `sky_linux` calls, including click, drag, scroll, move, type_text, press_key, and get_screenshot.

Key evidence: `Options.d.ts` adds `mouse_size_px?: number`; `sky_linux.js` adds `--mouse-size-px` argument construction; `@oai/sky/package.json` changes to version `0.4.10`.

### 3. macOS Computer Use Service Path Now Reads from an Environment Variable [Confirmed]

The Computer Use client for macOS changes how it resolves the service App path: instead of reading `globalThis.nodeRepl.computerUse.serviceAppPath`, it now reads `globalThis.nodeRepl.env.SKY_CUA_SERVICE_PATH`, while still falling back to inferred paths for the bundled `Codex Computer Use.app` when unavailable.

This means local or debugging environments can specify the Computer Use service location through `SKY_CUA_SERVICE_PATH`. From the artifact, the confirmed fact is the changed read path; external configuration strategy cannot be inferred further.

Key evidence: in `mac/client.js`, the path source changes to `nodeRepl.env.SKY_CUA_SERVICE_PATH`.

### 4. New Dock Tile Plugin and Multiple Icon Assets [Confirmed]

`Info.plist` adds `NSDockTilePlugIn = CodexDockTilePlugin.plugin`, and the bundle adds `Contents/PlugIns/CodexDockTilePlugin.plugin`, with signing identifier `com.openai.codex.dock-tile-plugin` and signing subject `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`.

This shows that the app now ships with a macOS Dock Tile plugin mechanism. Newly added resources also include `icon.png`, `icon-dev-dark.png`, `icon-chatgpt.png`, and `icon-dev-light.png`; `package.json` adds `codexAppBrand: "codex"`, and `Info.plist` adds `CodexAppIconBaseName = icon`.

Key evidence: `NSDockTilePlugIn`, `CodexDockTilePlugin.plugin`, `codexAppBrand = codex`, and `CodexAppIconBaseName = icon`.

### 5. Sites Plugin Updates from `0.1.14` to `0.1.15`, with Hosting Flow More Reliant on the Connector as the Source of Truth [Confirmed]

The Sites plugin version moves from `0.1.14` to `0.1.15`. The `sites-hosting` skill documentation is extensively rewritten: its scope narrows from “creating or reusing hosting configuration, saving versions, deploying, checking status, and managing permissions” to “locally preparing deployable Sites source code and build artifacts,” and it explicitly states that the Sites connector and tool descriptions are the source of truth for site creation, environment variables, source credentials, versions, deployments, and access control.

This means Codex’s local responsibilities in the Sites workflow are now described more clearly: local validation, source preparation, and artifact preparation; remote projects, versions, deployments, environment variables, and permission management are handled according to connector descriptions.

Key evidence: the new `sites-hosting/SKILL.md` text states “Use this skill for local preparation...” and “Use the Sites connector and tool descriptions as the source of truth...”.

### 6. Sites Building Documentation Adds Cloudflare D1 Prepared Statement Constraints [Confirmed]

The `sites-building` skill documentation adds D1 guidance: application queries and runtime initialization should use prepared statements on the raw D1 binding; multi-line single SQL statements can use `prepare(...).run()`; each `prepare()` should receive only one SQL statement; multi-statement operations should use `batch([...])`.

The developer-visible impact is that when Codex generates or modifies D1-related Sites projects, it will be more likely to favor precompilable, single-statement, batchable SQL structures and avoid passing newline-separated input to `exec()`.

Key evidence: `sites-building/SKILL.md` adds examples such as `await env.DB.prepare(schemaSql).run()`, `do not combine multiple semicolon-delimited statements`, and `d1.batch([...])`.

### 7. Webview CSP Adds Mapbox Connection Domains [Confirmed]

Two webview pages add `https://api.mapbox.com` and `https://events.mapbox.com` to `connect-src`: both `webview/index.html` and `webview/avatar-overlay-composition-surface.html` contain the same allowlist additions.

This indicates that the frontend runtime environment now permits network connections to the Mapbox API and event endpoints. CSP alone cannot confirm that a specific UI feature has shipped, but this is a direct runtime permission change.

Key evidence: `connect-src` adds `https://api.mapbox.com` and `https://events.mapbox.com`.

### 8. Frontend Signals for Audio, Approvals, Plugin Details, Window Tabs, Announcements, and More [Signal]

After removing Vite hash noise, newly added frontend module names include `transcribe-audio`, `use-realtime-waveform`, `automatic-approval-review`, `plugin-detail-queries`, `windows-tabs-open`, `announcement-atoms`, `workspace-message-announcement-state`, `appgen-gating`, and others.

These module names point toward areas such as audio transcription / real-time waveform, automatic approval review, plugin detail queries, window tab state, announcement state, and app generation gating. However, these are only module-name signals; source-level intent cannot be reconstructed from the artifact, and they do not mean these features are already available to users.

Key evidence: the real newly added frontend module list contains the unobfuscated module names above.

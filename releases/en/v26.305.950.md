# Codex Desktop `26.305.950` build 863

> Third-party changelog · Unofficial · Based on reverse diffs of official update artifacts

The main changes in this build cluster around desktop integration, settings flows, and frontend chunking, with a clear build-toolchain upgrade underneath.  
One-liner: this build points to broader desktop integration and app-server driven UI surfaces, backed by a newer build pipeline.

## About This Report

This is not an official changelog. It is a **reverse diff report based on official distribution artifacts**. The comparison covers macOS arm64 from `26.303.1606` build `806` to `26.305.950` build `863`.

Sources include the reconstructed bundle file tree, `Info.plist`, `package.json`, `webview/index.html`, the preload bridge, newly observed readable strings in main-process / frontend bundles, and frontend module-name changes after filtering out Vite hash noise.

Evidence is split into two levels:

- **[Confirmed]**: direct evidence from readable configuration, explicit strings, CSP, typed entry points, dependency diffs, signing / verification output, or similar artifact-level facts.
- **[Signal]**: indirect evidence from frontend module names, readable string / class diffs inside same-name frontend modules, or binary size movement. These point in a direction, but cannot reconstruct source-level intent.

Hash-named file churn under `webview/assets` and `.vite/build` is treated as build artifact renaming. This report does not interpret those file additions or removals as feature additions or removals.

## ✨ Highlights

### 1. The frontend entry now uses large-scale modulepreload chunking [Confirmed]

`webview/index.html` moved from one main script and one CSS file to explicit preloading of many frontend modules and several stylesheets. Newly preloaded entries include `app-server-manager-hooks`, `playwright`, `automation-dialog`, `skills-page`, `settings-surface`, `general-settings`, `app-connect-modal`, and `experimental-features-queries`.

This means the desktop frontend loading structure changed materially, with more surfaces split into independently loadable chunks. Users may notice different first-load behavior for related pages or dialogs, but the artifact alone cannot confirm whether each surface is enabled for all users.

Key evidence:

```diff
- <script type="module" crossorigin src="./assets/index-ByySpJCS.js"></script>
- <link rel="stylesheet" crossorigin href="./assets/index-A-iTIRzs.css">
+ <script type="module" crossorigin src="./assets/index-CMu6BCpo.js"></script>
+ <link rel="modulepreload" crossorigin href="./assets/app-server-manager-hooks-CAvrQB7n.js">
+ <link rel="modulepreload" crossorigin href="./assets/playwright-DGR8jDDG.js">
+ <link rel="modulepreload" crossorigin href="./assets/automation-dialog-BIAeyC_e.js">
+ <link rel="modulepreload" crossorigin href="./assets/skills-page-DeBxXSaK.js">
```

### 2. App Server and local environment configuration entry points surfaced [Confirmed]

The new main-process bundle contains readable strings such as `codex app-server`, `environment.toml`, `codex.localEnvironmentConfigPath`, and `features.unified_exec`. The filtered frontend module additions also include `app-server-manager-hooks`, `codex-api`, `local-environment-action-icon-options`, and `worktree-paths`.

Together, these point to stronger desktop support for a local App Server, unified execution paths, and local environment configuration management. Because the strings come from compressed artifacts, they confirm the presence of related entry points and messages but not the full source-level logic.

Key evidence: `codex app-server`, `environment.toml`, `features.unified_exec`, `app-server-manager-hooks`.

### 3. Automation and browser-related UI surfaces expanded [Signal]

After hash-noise filtering, newly observed frontend modules include `automation-dialog`, `browser`, and `playwright`. `webview/index.html` also explicitly preloads `automation-dialog` and `playwright` bundles.

This points to automation dialogs and browser automation-related UI becoming part of the frontend loading graph in this build, but it cannot reconstruct source-level intent or confirm whether the feature is enabled by default.

Key evidence: added modules `automation-dialog`, `browser`, `playwright`.

### 4. Settings gained an external Agent config import flow [Confirmed]

The same-name `agent-settings` frontend module added several user-visible strings related to external config migration, including `Apply selected`, `Import selected config. Restart Codex to apply all changes.`, `Imported external settings`, and `Unable to import external settings`.

This means the settings surface now contains copy for importing home-scoped external config. The user-visible touchpoints are selecting items, being told to restart after import, and success / failure toasts.

Key evidence:

```text
Import selected config. Restart Codex to apply all changes.
Imported external settings
Unable to import external settings
```

### 5. MCP HTTP configuration now covers headers and bearer token env vars [Confirmed]

`mcp-settings` added strings including `Add header`, `Add variable`, `Headers from environment variables`, `Bearer token env var`, and `Label for MCP HTTP URL`.

This shows the MCP server detail surface expanded HTTP configuration to include headers, environment-variable-backed headers, and bearer token environment variables. For developers, remote MCP HTTP setup appears more granular.

Key evidence:

```text
Headers from environment variables
Bearer token env var
Label for MCP HTTP URL
```

### 6. Auto-reload wording in Usage settings was reorganized [Signal]

The `usage-settings` module added `Auto-reload credit`, `Auto-reload minimum balance`, and `Auto-reload target balance`, while removing several older auto top-up input descriptions and `Credits remaining`.

This points to wording and layout changes in the Usage settings auto-reload flow. Because it comes from same-name frontend module diffs, it is a UI / state / interaction signal, not a source-level fact.

Key evidence: `Auto-reload credit`, `Auto-reload minimum balance`, `Auto-reload target balance`.

### 7. Remote SSH and GitHub CLI discovery error paths increased [Confirmed]

The main-process bundle added `Failed to discover remote SSH connections.`, `failed to refresh discovered remote SSH connections`, `Failed to parse GitHub CLI response`, `Unexpected GitHub CLI response`, `gh auth login`, and `not logged into any github hosts`.

This confirms artifact-level error paths for remote SSH discovery and GitHub CLI response handling. The user-visible result may be more specific connection, auth, or failure messaging, but the full flow cannot be reconstructed from compressed output.

Key evidence: `Failed to discover remote SSH connections.`, `Failed to parse GitHub CLI response`, `gh auth login`.

### 8. Windows integration strings entered the main-process artifact [Confirmed]

Although this comparison is for macOS arm64, the new main-process bundle includes many Windows-related strings: `Failed to register Windows folder context menu`, `Open project in Codex`, `File Explorer`, `Start Menu`, `Windows Terminal.lnk`, `wt.exe`, `devenv.exe`, `webstorm64.exe`, and more.

This is clear evidence that the shared desktop code line gained paths related to Windows folder context menus, Start Menu, terminal discovery, and IDE discovery. It may not be directly visible to macOS users.

Key evidence: `Open project in Codex`, `Failed to register Windows folder context menu`, `File Explorer`.

### 9. The preload bridge added Fast Mode rollout metrics [Confirmed]

`.vite/build/preload.js` exposes a new `electronBridge.getFastModeRolloutMetrics` function that invokes `codex_desktop:get-fast-mode-rollout-metrics` over IPC.

This gives the renderer a bridge API for reading Fast Mode rollout metrics. On the user side, it may support more granular Fast Mode announcement, experiment, or display logic, but the confirmed fact is the presence of the bridge API.

Key evidence:

```diff
+ getFastModeRolloutMetrics:async t=>e.ipcRenderer.invoke(`codex_desktop:get-fast-mode-rollout-metrics`,t)
```

### 10. Mermaid / diagram parse-error text appeared across many diagram modules [Signal]

Many same-name diagram modules added `Parse error on line` and `end of input`, spanning gantt, block, sequence, c4, quadrant, sankey, journey, timeline, xychart, flow, mindmap, er, requirement, and kanban modules.

This points to a shared change in diagram parse-error display or parser output, but the evidence comes from same-name frontend module diffs, so it remains a signal rather than a source-level statement about parser behavior.

Key evidence: multiple diagram modules jointly added `Parse error on line` and `end of input`.

### 11. The build toolchain moved to Electron Forge 7.11.1, Vite 8 beta, and Vitest 4 beta [Confirmed]

`package.json` shows Electron Forge packages moving from `^7.10.2` to `^7.11.1`, `vite` changing from `^5.4.21` to `8.0.0-beta.15`, `vitest` changing from `^1.6.0` to `4.1.0-beta.5`, and a new runtime dependency on `tslib`.

This helps explain part of the frontend chunking change and confirms a sizable build pipeline update in this release.

Key evidence:

```diff
- "vite": "^5.4.21",
- "vitest": "^1.6.0"
+ "vite": "8.0.0-beta.15",
+ "vitest": "4.1.0-beta.5"
+ "tslib": "^2.8.1"
```

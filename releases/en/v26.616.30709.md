# Codex Desktop 26.616.30709 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

This batch looks like an expansion of the automation surface: browser, Sites, realtime voice, downloads, plugin directories, projects/cloud chats, and PR workflows all moved.  
One-liner: automation, browser downloads, deployment safety, realtime voice, and plugin/workflow surfaces all moved forward, with macOS adding Record & Replay.

## About This Report

This report uses only machine-extracted diffs from reconstructed official signed distribution packages: macOS arm64 from `26.611.62324` build `4028` to `26.616.30709` build `4108`, and Windows x64 from `26.611.8604.0` to `26.616.3309.0`. The platform version numbers do not fully align, so each conclusion is tied to its own platform facts.

Evidence levels:

- [Confirmed]: plist, manifest, CSP, skill docs, type declarations, package.json, dependency lists, plugin lists, service entrypoints, signing data, and readable strings outside the “same-name frontend module change clues” section.
- [Signal]: frontend module names, module size changes, readable strings and CSS/state-class diffs from same-name frontend modules. These show packaged UI/state/interaction direction, but are not source-level proof and do not guarantee user visibility.

Platform tags use `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`. Hash renames, Vite build-output churn, and isolated asset-name changes are not treated as functional additions or removals.

## ✨ Highlights

### [Confirmed][macOS only] Record & Replay Is Now a Bundled Plugin

macOS adds a new bundled plugin, `record-and-replay`. The bundled plugin list expands from `browser`, `chrome`, `computer-use`, `latex`, and `sites` to include `record-and-replay`. The plugin has `168` files, total size `56.8 MB`, version `1.0.829`, and display name `Record & Replay`.

This is not just package growth. The plugin manifest, marketplace entry, service entrypoint, and skill instructions together describe a complete capability surface: recording mouse clicks, typed text, and interacted window content on the user’s Mac for up to `30` minutes, then turning the event stream into a reusable skill. Default prompts include “Record my workflow and turn it into a reusable skill”. The MCP service is `event-stream`, with an entrypoint at `SkyComputerUseClient`.

The platform split is clear: macOS adds the `plugins/record-and-replay` directory and a full Computer Use app runtime set. The Windows fact package has no corresponding added plugin detail, and its marketplace diff only shows the LaTeX category change.

### [Confirmed][macOS+Windows] Browser / Chrome Safety Confirmation Rules Are Clearer

Both platforms update the `browser` and `chrome` confirmation docs by restructuring the first mode from “Hand-Off Required” into “Unsupported Actions”, explicitly requiring the agent to find an alternative instead of bypassing browser or web safety barriers. HTTPS security interstitial bypass and paywall bypass are listed under this safety-boundary area; the final step of submitting a password change remains a user hand-off case.

Both browser skill docs also now emphasize that when the user explicitly asks to navigate to a page and authentication blocks the task, the agent should not switch to web search, a search engine, another site, or another source to work around login. If secure browser authentication is available, use that flow; otherwise ask the user to log in.

The Chrome skill also adds session naming: at the start of each Chrome browser task, after setup and before opening or claiming tabs, call `browser.nameSession("...")` with a short task name.

### [Confirmed][macOS+Windows] Browser API Adds Media Download, and Download Settings Reach Main Process/UI Text

Both platforms add `downloadMedia(options: LocatorDownloadMediaOptions)` to `browser/docs/api.md` and `.codex-plugin/api-multi-tab.md`, described as triggering a media download for the first matched element. `LocatorDownloadMediaOptions` is also added.

Main-process strings show download settings entering the package: `Choose Downloads Folder`, `Folder where files downloaded by the in-app browser are saved`, `Whether manual browser downloads ask where to save each file`, and `Failed to publish downloads state to renderer`. Frontend settings clues also include `Download history`, `Ask where to save downloads`, `System Downloads folder`, `Unable to change the downloads folder`, and `Browser download history cleared`.

The main platform difference is in debug surfaces. macOS `debug-modal` explicitly adds `Cancel download`, `Open download`, `Pause download`, and `Resume download`. Windows `debug-modal` changes much more broadly, with many App Server, experiment, host-permission, and environment-state strings.

### [Confirmed][macOS+Windows] Sites Deployment Now Prefers Private Deploy and Requires Approval for Shared/Public Deploy

Both platforms add `Deployment Selection` and `User-Facing Handoff` to `sites-hosting/SKILL.md`. After saving a site version, the skill should deploy unless the user explicitly asked to keep the work local. If `deploy_private_site_version` is available, the normal Sites flow should call it first, with server-side verification of owner-only access. If private deploy is unavailable, or fails because the site is shared, public, or cannot be verified as owner-only, the skill must not automatically call `deploy_site_version`; it should ask the user for approval.

The handoff is also clearer: after successful deployment, return the deployed URL and a concise result summary, and do not attach local source, build output, upload archive, or individual code files unless the user explicitly asks. Windows also shows the `sites` plugin moving from `0.1.15` to `0.1.16`; the macOS fact package does not list the corresponding sites plugin.json version diff, but the skill doc change is shared.

### [Confirmed][macOS+Windows] LaTeX Plugin Upgrades and Moves Category

Both platforms update the `latex` plugin from `0.2.2` to `0.2.3`, and its category changes from `Research` to `Education & Research`. macOS marketplace also adds the `record-and-replay` entry and changes LaTeX’s marketplace category; Windows marketplace diff only shows the LaTeX category adjustment.

### [Confirmed][macOS+Windows] Realtime Voice Adds Wake Word, Hotkey, and Floating Avatar Controls

Main-process strings on both platforms include `codex.command.realtimeVoice`, `Failed to apply realtime voice hotkey`, `Failed to start realtime voice wake word`, `Realtime voice wake word detected`, `Realtime voice wake word needs microphone access`, and wake-word helper listening/ready/error states. This shows realtime voice hotkey and wake-word helper paths are packaged.

Frontend module clues also show floating avatar controls for microphone and output: `Mute microphone`, `Unmute microphone`, `Mute realtime voice`, and `Unmute realtime voice`. macOS additionally adds a native `Contents/Resources/native/realtime-voice-wake-word` file. Windows additionally shows realtime voice sending messages to other threads, creating threads, and delegating work in `use-avatar-overlay-selection`, giving this Windows sample stronger avatar/thread delegation clues.

### [Confirmed][macOS+Windows] Plugin Directory and Connector Permission Copy Become More Granular

Both platforms’ `plugins-page` clues include `By OpenAI`, `By your workspace`, `Created by you`, `OpenAI internal`, and workspace-provided plugin labels, replacing some imported-plugin and marketplace wording. `plugins-availability` adds connector modal copy for admin approval, personalization, privacy, site data-stealing risk, connector risk, app load failure, and admin-disabled states.

This area mixes evidence types: the frontend wording is a UI signal, while the LaTeX and record-and-replay plugin manifest/marketplace entries are configuration-level confirmed evidence.

### [Signal][macOS+Windows] Composer Attachments Shift Toward Multi-File, Photos, Remote Files, and Appshot

Both platforms’ `composer` module adds `Add files`, `Add photos`, `Add remote files`, `Attach appshot`, plus state classes such as `pick-files`, `pick-local-files`, `pick-remote-files`, and `multi-file`. Older screenshot/annotation attachment wording appears to move into the `cube` module clues.

This suggests the composer attachment entry point is being split or reorganized around local files, remote files, photos, and Appshot capture states. Because the evidence comes from same-name frontend module diffs, this remains a signal.

### [Signal][macOS+Windows] Sidebar, Tasks, Cloud Chats, and Project Entry Points Are Converging

Both platforms’ `app-main` module adds `Cloud chats`, `ChatGPT project`, `Menu item that creates a ChatGPT project from the unified sidebar`, `List label for cloud chats in the sidebar`, `Manual order`, and `Get to know Codex`. In the macOS sample, `header` also adds `All tasks`, `Cloud tasks`, `All environments filter option`, recent task menu, settings entry, and Chrome extension thread entry strings.

These point to sidebar/header task and cloud-chat consolidation, but without source semantics they remain UI signals. The shared `app-main` items are treated as cross-platform; the `header` details are kept in the low-level index because the Windows fact package does not list that module block.

### [Signal][macOS+Windows] PR / CI Workflows Gain Comment Handling, Check Fixes, and Missing-Info States

Both platforms’ `local-conversation-thread` adds strings such as `Address all actionable attached PR feedback.`, `Address every actionable comment without asking the user which ones to handle.`, and instructions to run the narrowest relevant verification, commit, and push after resolving conflicts. State classes include `missing-branch-info`, `missing-pr-info`, and `pull-request`. New frontend modules also include `pull-request`, `pull-request-checks-summary`, `pull-request-status`, and `pull-request-visual-state`.

Windows additionally shows reviewer groups, `Awaiting approval`, `Changes requested`, and `Per-check action label for fixing a single failing pull request check` in `pull-requests-page`. So the base PR direction is cross-platform, while this Windows sample exposes richer PR-page detail.

### [Signal][macOS+Windows] Automation History Moves Toward Archive, Bulk Archive, and Read State

Both platforms’ `automations-page` adds `Archive all`, `Context menu item to archive an automation history row`, `Dropdown menu item to mark all automation runs as read`, `Action label to unarchive an automation history thread`, and `Custom schedule`, while removing pause/resume/delete automation menu copy. This points to an automation history information-architecture shift from direct automation control toward archive and read-state management.

### [Signal][macOS+Windows] Remote Connections Add WSL Errors and Offline-Only Deletion Rules

Both platforms’ `remote-connections-settings` adds `WSL connection failed`, `settings.remoteConnections.details.wslDistro`, `settings.remoteConnections.deviceConnections.wslSubtitle`, `Only offline environments can be deleted`, and `SSH config entry not found`. The main process also adds `Only offline remote control environments can be deleted.`. This combines UI/error-path signals with confirmed packaged strings for remote environment management and WSL behavior.

### [Confirmed][macOS only] macOS Adds a Desktop Permission Usage Description

macOS `Info.plist` adds `NSDesktopFolderUsageDescription = Codex needs access to your Desktop for the task you selected`. Combined with onboarding module strings around loading Desktop access configuration and requesting Desktop access, macOS has a clear system permission declaration change. The Windows package provides no corresponding AppxManifest change block.

### [Confirmed][macOS+Windows] Native Runtime and Dependency Loading Paths Update

Both platforms’ `package.json` updates the product version to `26.616.30709`, `codexBuildNumber` from `4028` to `4108`, and `e2e:computer-use-native-pipe` from direct `node` to `node --import tsx`. Both `owl-electron-app.json` files change `runtimeArchiveSha`, showing the owl runtime archive was replaced.

macOS additionally shows `node-gyp-build` moving from `4.6.0` to `4.8.4`, adding imports mappings and changing native addon resolution, tag matching, and Windows shell spawn behavior. The Windows fact package does not list this node-gyp-build file diff, but Windows package components `resources/codex.exe`, `resources/codex`, and `resources/app.asar` all grow substantially.

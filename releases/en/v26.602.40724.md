## About This Report

This report uses only machine-extracted facts from reverse-reconstructed official signed distribution packages: macOS arm64 from `26.602.30954` build `3575` to `26.602.40724` build `3593`, and Windows x64 from `26.602.3474.0` to `26.602.4764.0`. Evidence has two levels: `[Confirmed]` means the fact comes from plist, manifest, plugin metadata, service entries, dependency lists, readable strings, or key file diffs that are clearly inside the package; `[Signal]` means it only comes from frontend module names, style classes, size changes, or compressed bundle direction. Platform ownership is marked as `[macOS+Windows]`, `[macOS only]`, or `[Windows only]`.

The comparison covers versions and channels, bundle file trees, cua_node dependencies, app.asar summaries, same-name frontend module change clues, newly readable main-process/frontend strings, Info.plist or AppxManifest changes, content security policy changes, new components/resources, and concrete key-file diffs. The fact packages do not provide separate details for newly bundled plugins, total Computer Use runtime size, AppxManifest changes, content security policy changes, or new component/resource inventories; this report does not invent missing numbers.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Bundled Browser and Chrome plugins aligned to 26.602.40724

On both platforms, bundled `browser` and `chrome` plugin metadata updates the version from `26.602.30954` to `26.602.40724`. The plugin descriptions do not show content changes in the provided diffs, so the confirmed change is version metadata alignment, not a change in described capabilities.

On macOS, the paths are `Contents/Resources/plugins/openai-bundled/plugins/browser/.codex-plugin/plugin.json` and `Contents/Resources/plugins/openai-bundled/plugins/chrome/.codex-plugin/plugin.json`. On Windows, the paths are `resources/plugins/openai-bundled/plugins/browser/.codex-plugin/plugin.json` and `resources/plugins/openai-bundled/plugins/chrome/.codex-plugin/plugin.json`.

### [Signal] [macOS+Windows] Profile share card gains custom pet preview controls

The same-name `profile` frontend module on both platforms shows the same newly readable strings and style changes pointing to custom pet controls in the profile share-card preview: `Custom pet`, `Show pet`, `Hide pet`, `Next custom pet`, `profile.shareCard.preview.hidePetLabel`, `profile.shareCard.preview.nextPet`, plus accessibility labels for switching pets.

This comes from frontend module change clues, so it cannot prove the feature is visible to all users. Still, the identical string set appearing on both macOS and Windows shows that the related UI, state, or interaction entered both platforms’ frontend bundles.

### [Signal] [Windows only] Thread side/bottom panel entry clues appear

The Windows `thread-app-shell-chrome` module adds many panel-entry strings, including `Bottom panel`, `Browse project files`, actions to open side chat, browser tab, review tab, terminal, timeline tab, workspace file browser, plus button labels for closing the thread bottom panel and opening another tab from the bottom or side panel.

These are high-visibility UI signals, but the source is still same-name frontend module change clues. They show that Windows package contents now include directions for thread side/bottom panel entries and new-tab actions, but they are not source-level proof that the full feature is complete or enabled. The macOS fact package has no corresponding `thread-app-shell-chrome` change clue.

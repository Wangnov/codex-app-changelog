# Codex Desktop 26.715.61943 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

The shared theme is a reworked thread workspace, while the Windows package contains denser browser-operation, Computer Use, live Office-control, and file-rendering signals; settings, site sharing, and home announcements are notably out of phase between platforms.

Shared thread-workspace changes are joined by Windows-heavy browser, Computer Use, Office control, and file-rendering signals, while settings, sharing, and home announcements move in opposite packaging directions.

## About This Report

This report compares two sets of official distribution artifacts: macOS arm64 from 26.715.52143 (build 5591) to 26.715.61943 (build 5628), and Windows x64 MSIX from 26.715.7063.0 to 26.715.8383.0. The version windows do not align, so every conclusion is attributed independently and must not be projected from one platform onto the other.

The macOS comparison uses historical full packages. Both samples passed Apple notarization validation and identify OpenAI OpCo, LLC as the signing origin. The Windows facts provide SHA-256 hashes for both full packages but do not include a separate signature-verification result that can be cited.

Evidence is divided into two levels:

- [Confirmed] covers plist data, plugin manifests, dependency inventories, service entry points, signature validation, and readable package facts outside same-name frontend-module clues.
- [Signal] covers strings, style classes, and size changes inside same-name frontend modules. These indicate UI, state, or interaction changes but do not prove rollout.

Platform tags are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`.

The evidence inventory covers versions and channels, bundle trees, the Computer Use runtime, `cua_node` dependencies, `app.asar`, same-name frontend modules, Info.plist, and concrete file diffs. The input does not provide standalone built-in-plugin additions, main-process string additions, a CSP diff, a Windows AppxManifest diff, an added-component/resource inventory, or a hash-noise-filtered list of true frontend file additions and removals. No extra conclusions are drawn for those areas.

## ✨ Highlights

### Thread side and bottom panels form the shared change theme

[Signal] [macOS+Windows]

On both platforms, `thread-app-shell-chrome` gains the same panel text and responsive condition:

- `Bottom panel`, plus controls for closing the thread bottom panel and opening another tab from either the bottom or side panel.
- Actions for opening the browser, Review, terminal, and workspace file browser from the task side panel.
- An action for opening a side chat from the chat side panel and Detail from the side-panel New Tab page.
- An empty state for chats without available side-panel tab actions.
- A `(max-width: 920px)` condition and new panel layout, background, border, and focus classes.

This is the clearest shared UI signal in the batch. On Windows, related changes extend into a fuller browser and task-control surface; the macOS evidence supports only the shared shell layer.

### Windows thread panels gain broad browser-control and task-operation signals

[Signal] [Windows only]

`thread-side-panel-tabs` grows by approximately 343,826 bytes and gains strings covering:

- Browser tabs playing audio or using the camera or microphone.
- Download action menus, download rows, and revealing files in the system file browser.
- Browser address-bar suggestions.
- Device-toolbar width, height, zoom, and close controls.
- Canceling active Git actions, accepting edits to stage them, and the state where a Pull Request already exists for the branch.

These strings confirm that the text and states entered the Windows frontend module. They do not prove that complete device emulation or Git workflows are visible to every user.

### Windows @ mentions connect Computer Use, live Office control, and agent types

[Signal] [Windows only]

`at-mention-list` grows by approximately 170,819 bytes and gains a mutually reinforcing group of entry-point strings and classes:

- `Computer use` and its feature label.
- Live Microsoft Excel workbook control.
- Live Microsoft PowerPoint presentation control.
- `Custom agents`, `Live agents`, and `Desktop apps`.
- `ChatGPT conversation(s)` and `Files and chats`.
- The classes `microsoft-excel-plugin`, `microsoft-powerpoint-plugin`, `mcp-capability-mentions`, and `mcp-servers`.

This is a Windows signal for new routing and mention surfaces, not confirmation that the capabilities are enabled for every account.

### Windows Markdown rendering gains file-citation and image-state signals

[Signal] [Windows only]

The `markdown` module grows from approximately 0.1 MB to 0.4 MB and gains text for:

- A file-citation chip with location information.
- Markdown image buttons when alternative text is absent.
- Placeholders while image bytes are loading.
- Fallbacks when image loading fails.
- A state where a follow-up message is already awaiting confirmation.

The module also adds numerous accessibility states and removes a set of Markdown/TextMate-style syntax scopes. The evidence supports a rendering and interaction reorganization but does not confirm a specific editor implementation change.

### Browser and Chrome plugin metadata is synchronized on both platforms

[Confirmed] [macOS+Windows]

The bundled `browser` and `chrome` plugin manifests both move from version `26.715.52143` to `26.715.61943` on macOS and Windows.

The `browser` manifest continues to define aliases such as `@browser` and `@browser-use`, covering opening, inspecting, navigating, testing, clicking, typing, and screenshotting local addresses, `file://` targets, and the current in-app browser tab. The `chrome` manifest continues to cover automation that depends on existing Chrome tabs, signed-in sessions, cookies, extensions, and ChatGPT Chrome Extension setup.

The confirmed change is the version field. The descriptions themselves are unchanged, so these capabilities must not be presented as newly introduced in this batch.

### The Windows Computer Use plugin synchronizes its internal version

[Confirmed] [Windows only]

The bundled Windows `computer-use` plugin manifest moves from version `26.715.52143` to `26.715.61943`. Its description remains “Control desktop apps on Windows from ChatGPT through Computer Use.”

This is a metadata update to an existing plugin, not evidence of a newly added plugin. The Windows `cua_node` total remains 287.5 MB, with zero added dependency packages.

### macOS shows a combined plugins, skills, and MCP settings surface

[Signal] [macOS only]

`plugins-settings` gains:

- `Manage plugins, skills, and MCPs`
- A subtitle for the combined plugins settings section
- The `plugins-settings` class

This points to a combined organizational surface for plugins, skills, and MCP management. The input contains no new plugin directory or manifest addition, so it does not support claiming that a new bundled plugin was added.

### Home announcements move in opposite packaging directions

[Signal] [macOS+Windows]

On macOS, `home-announcements` grows by approximately 28,328 bytes and adds:

- Multi-agent composer banner body and dismissal labels.
- `Delegate work to subagents that work in parallel. Note: may increase token usage.`
- Dismissal labels for Fast mode, Codex gift credits, rate-limit resets, and backend-driven home banners.
- Entry points for the gift-credit purchase flow and available rate-limit-reset modal.
- State classes such as `has-seen-multi-agent-composer-banner`.

On Windows, the same module shrinks by approximately 26,379 bytes and removes the corresponding multi-agent, Fast mode, gift-credit, rate-limit-reset, and backend-banner strings and state classes.

This demonstrates opposite packaging directions across the two version windows. It does not prove that an announcement was permanently launched or withdrawn on either platform.

### General settings move in opposite directions across platforms

[Signal] [macOS+Windows]

On Windows, `general-settings` grows by approximately 92,351 bytes and gains settings text for:

- Default permissions and full-access visibility.
- The composer context-usage indicator.
- Base sizes for code across chats and diffs, and for the ChatGPT UI.
- Hotkey capture.
- Ambient suggestions.
- Colored diff markers.
- The default projectless behavior for Popout Window.
- ChatGPT and Codex macOS Dock icon selection.

On macOS, the same module shrinks by approximately 95,930 bytes and removes an overlapping set of those strings and styles.

The presence of “macOS Dock icon” labels in the Windows bundle only confirms that shared frontend strings entered that build. It does not indicate that Windows exposes macOS-specific settings.

### Site publishing and sharing signals enter Windows and leave macOS

[Signal] [macOS+Windows]

On Windows, `appgen-share-dialog` grows by approximately 23,410 bytes and adds:

- Adding people or groups, searching for users, and removing workspace users.
- “Anyone in this workspace with the link” and “Anyone on the Internet” visibility.
- Publishing a site to the public internet.
- Copying a live-site URL and opening a live site externally.
- `Can visit`, `Copied to clipboard`, and `Copy link`.

On macOS, the same module shrinks by approximately 24,643 bytes and removes overlapping user-selection, link-scope, copying, and external-opening text.

These are opposite module signals from different version windows. They do not prove that site publishing is Windows-only.

### The Windows temporary-chat UI path is reduced

[Signal] [Windows only]

`chatgpt-temporary-chat-ui` shrinks by approximately 13,539 bytes and removes:

- Buttons and accessibility labels for turning temporary chat on and off.
- The temporary-chat header indicator.
- The onboarding primary button and associated explanatory text.

This shows that temporary-chat text left this standalone UI module. It does not by itself confirm removal of temporary chat as a capability.

### macOS workbook code is substantially re-chunked

[Signal] [macOS only]

`workbook-from-markdown` grows by approximately 108,179 bytes and gains errors related to document closing, serialization, plugin lists, find-and-replace operations, paths, and asynchronous processing.

At the same time, `workbook` drops from approximately 2.6 MB to nearly 0 MB and removes named-range, theme-color, and numerous `data-granola-*` style or node strings. Together, these changes resemble workbook code re-chunking or dependency movement, but the facts do not contain source code or a true added-file inventory that would support a specific architectural conclusion.

### macOS gains plugin event-bus connection and queuing signals

[Signal] [macOS only]

The `esm` module gains strings for plugin event subscriptions, a client bus, and an internal event target, including:

- Connecting to the event bus.
- Queueing while the bus is unavailable and pushing after connection.
- Emitting queued events.
- Not emitting or registering when the event-bus client is disabled.
- Reaching the maximum retry count.
- Failing to dispatch a shim event.

It also removes older errors such as `Failed to deliver`, `Not ready`, and `Promise timed out`. This points to changes in plugin event delivery and reconnection paths.

### The debug modal is heavily reduced on both platforms

[Signal] [macOS+Windows]

`debug-modal` shrinks by approximately 230,423 bytes on macOS and 226,002 bytes on Windows. Both platforms remove strings including `Account ID`, `Active browser tab`, `Active conversation`, `Activity grouping version`, process and type filters, automatic worktree cleanup, and numerous modal styles.

This supports a direction of splitting, trimming, or relocating the debug UI. It does not confirm that the underlying diagnostics were completely removed from the application.

## ⚖️ Boundaries and Disclaimer

This is an unofficial third-party analysis based on reverse-reconstructed diffs of official distribution artifacts. It is not an OpenAI release note and does not redistribute application binaries.

Every [Signal] comes from module names, strings, classes, or size directions in compressed frontend modules. Signals are not source-code facts and do not prove that a capability is available to every user. Removed strings likewise do not independently prove feature removal: code may have been re-chunked, lazy-loaded, or moved elsewhere.

The macOS and Windows comparisons use different outer version ranges. This report describes only the content changes present in each supplied sample. Evidence-based corrections are welcome.

## About This Report

This is not an official changelog. It is a **reverse diff report based on official distribution artifacts**. This entry is a historical backfill: the official appcast did not provide a Sparkle delta for `26.311.21342` → `26.313.41036`, so the comparison uses two official full packages and summarizes structured diffs across the bundle, app.asar, Info.plist, frontend modules, and readable strings.

This report uses two evidence levels:

- **[Confirmed]**: directly visible facts from Info.plist, package.json, CSP, readable configuration, newly present main/frontend bundle strings, signatures, or verification results.
- **[Signal]**: indirect evidence from same-name frontend module changes, module names, or size shifts. These point in a direction but cannot reconstruct source-level intent.

Scope: macOS arm64, comparing `26.311.21342` build `993`, released on `2026-03-13`, against `26.313.41036` build `1032`, released on `2026-03-16`.

## ✨ Highlights

### 1. Local environment lifecycle gains cleanup script surfaces [Confirmed]

This build introduces strings and settings related to cleanup scripts, including `Cleanup script`, `Cleanup script failed`, `This script will run before a worktree is deleted.`, and `No cleanup script configured.`.

This suggests the local environment surface is no longer limited to setup; the artifact now includes a capability surface for running cleanup logic before a worktree is deleted. The main-process strings also remove `Failed to run setup script` and `Setup script canceled`, while keeping `Setup script` and adding cleanup failure paths.

Key evidence:

```text
Cleanup script
Cleanup script failed
This script will run before a worktree is deleted.
Overrides the default cleanup script for specific OSes.
```

### 2. Remote connection state becomes more explicit in the thread UI [Signal]

The `app-server-connection-state` module adds connection-status label strings for connected, connecting, disconnected, error, and unauthed states, plus `Login required`. It also adds green, yellow, red, and gray state classes along with `animate-pulse`.

This points to a more visible remote connection status indicator in the UI, but the evidence comes from a same-name frontend module diff, so it cannot be treated as source-level intent. The user-visible effect may be clearer remote connection badges on thread pages.

Key evidence:

```text
threadPage.remoteConnectionStatusBadge.connected
threadPage.remoteConnectionStatusBadge.connecting
threadPage.remoteConnectionStatusBadge.disconnected
threadPage.remoteConnectionStatusBadge.error
threadPage.remoteConnectionStatusBadge.unauthed
Login required
```

### 3. MCP server elicitation response paths appear in the artifact [Confirmed]

The new main/frontend bundles add `Missing thread follower submit-mcp-server-elicitation response`, `Unknown thread follower submit-mcp-server-elicitation response requestId`, `Failed to forward MCP server elicitation response`, and `Unexpected MCP server elicitation request method`.

This shows that a path for MCP servers to initiate a request and receive a user or thread-side response is present in the shipped artifact. The frontend module also contains the `mcp-server-elicitation` state class, which corroborates the main-process strings.

Key evidence:

```text
Failed to forward MCP server elicitation response
Missing thread follower submit-mcp-server-elicitation response
Unexpected MCP server elicitation request method
```

### 4. Git/PR workflows add draft PR and bulk comment-handling signals [Signal]

The `git-settings` module adds strings such as `Create draft pull requests`, `Display PR status icons on thread rows in the sidebar`, and related save-failure labels. The `index` module adds `Address all`, `Address every actionable comment without asking the user which ones to handle.`, and `Address requested changes before merging.`.

This points toward PR workflows with more emphasis on draft PR creation, sidebar PR status, and bulk handling of review comments. Because these strings come from same-name frontend module changes, they are UI/interaction signals only.

Key evidence:

```text
Create draft pull requests
Display PR status icons on thread rows in the sidebar
Address all
Address every actionable comment without asking the user which ones to handle.
```

### 5. Agent speed tiers and a realtime voice experiment surface appear [Signal]

The `general-settings` module adds `Default speed`, `1.5x speed, 2x plan usage`, `About 1.5x faster, with credits used at 2x`, `Realtime voice`, and `Talk to Codex in real time from the thread composer. Restart Codex after changing this setting.`.

This points to settings-page work around agent speed tiers and a realtime voice experiment toggle. Since the evidence comes from same-name frontend module changes, it does not prove the feature is enabled for all users.

Key evidence:

### 6. Plugins and Apps page gains load/uninstall error states [Signal]

The `skills-page` module adds `Unable to load apps`, `Retry button label for the apps list error state`, `Failed to uninstall plugin`, and `Toast shown after successfully uninstalling a plugin`.

This points to added UI states for Apps list loading errors, retry, plugin uninstall success, and plugin uninstall failure. Since the evidence comes from a same-name frontend module diff, it confirms the UI text direction but not backend behavior.

### 7. Syntax-highlighting resources were reshuffled [Signal]

Several language modules changed size materially: `typescript`, `jsx`, and `tsx` each grow by roughly 175 KB to 181 KB, while `javascript` shrinks by roughly 175 KB. `java`, `go`, `r`, `gdscript`, and others add grammar scope strings, while `c`, `haxe`, `stylus`, `scss`, and `shellscript` remove groups of scope strings.

This points to a reorganization of built-in code display or syntax-highlighting resources, which may affect code snippet highlighting details. These are same-name frontend module and size-change signals only.

### 8. macOS Bluetooth permission key updated [Confirmed]

`Info.plist` removes the old `/NSBluetoothPeripheralUsageDescription` key and adds `/NSBluetoothAlwaysUsageDescription`, with the same value: `This app needs access to Bluetooth`.

This is a permission-declaration change in the macOS bundle. The fact package does not provide a matching feature entry point, so this report records only the permission key change and does not infer its purpose.

Key evidence:

```text
Added /NSBluetoothAlwaysUsageDescription = 'This app needs access to Bluetooth'
Removed /NSBluetoothPeripheralUsageDescription
```

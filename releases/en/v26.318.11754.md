## ✨ Highlights

### 1. Automatic approval appears to gain a reviewer flow [Signal]

The main `index` frontend module added several UI strings around automatic approval review, including `Automatic approval review`, `A carefully prompted reviewer agent is reviewing this request before Codex runs it.`, `reviewed this request`, and `stopped reviewing this request before Codex ran it.`.

This points to an experience where a reviewer agent may run before Codex executes a request. However, these clues come from same-name frontend module changes, so they only indicate UI and state direction, not complete source-level intent. Related state classes include `high-risk`, `medium-risk`, and `bg-token-danger/15`.

### 2. Plugin and skill discovery continues to take shape [Signal]

The `skills-page` module added strings such as `All plugins`, `New plugin`, `Hide extra results`, `skills.appsPage.createPlugin`, and `skills.pluginsPage.section.discover.showExtraSearchResults`, along with visible entries for Adobe Acrobat, Data Science, Google Calendar, Google Docs, Google Drive, Google Sheets, and Google Slides.

This points to continued work on plugin/skill discovery, search, filtering, creation entry points, and app categories. Because the evidence comes from same-name frontend module changes, it remains a UI and interaction signal.

### 3. A question notification setting appears in settings [Signal]

The `general-settings` module added `Enable question notifications`, `Show alerts when input is needed to continue`, `notifications.questions.label`, and `notifications.questions.description`.

This points to a settings control for alerts when user input is needed. The evidence comes from a same-name frontend module change, so it indicates settings UI and notification wording direction, not the full notification implementation path.

### 4. Conversation fork and session error paths gained new prompts [Signal]

The `index` module added `Aria label for the button that forks a conversation from this user message`. The `app-server-manager-hooks` module added `Cannot fork a conversation while it is in progress.`, `Forked conversation state not found.`, `Source conversation not found.`, `Target turn not found.`, and `Target turn must not be after the latest turn.`.

These clues point to a flow for forking a conversation from a user message, plus boundary checks for that operation. Since they come from same-name frontend module changes, they indicate feature direction and user-visible error states, but not backend or state-machine implementation details.

### 5. Main-process strings add GitHub PR and child process snapshot error paths [Confirmed]

The new main-process/frontend bundle contains readable strings `Failed to list pull requests`, `Failed to collect child process snapshot`, and `fsUtils.readdir requires AppServerClient`.

These strings are confirmed to exist in the new bundle, showing new artifact-level error paths related to listing pull requests, collecting child process snapshots, and AppServerClient-backed filesystem reads. The fact package only provides compressed bundle strings, so the full call chain cannot be recovered.

### 6. Syntax highlighting and math rendering assets changed noticeably [Signal]

Several language and rendering modules changed in size or token sets: `katex` grew by `263700 bytes`, `fortran-free-form` by `87114 bytes`, `c` by `72082 bytes`, `css` by `48992 bytes`, and `go` by `46744 bytes`, while `jsx`, `tsx`, `javascript`, `php`, and `markdown` shrank substantially.

This points to changes in editor/rendering syntax resources, MathML/KaTeX handling, or related rendering assets. All of these clues come from same-name frontend modules and size deltas, so they cannot confirm source-level refactoring intent.

### 7. Bluetooth permission text moved to a newer plist key [Confirmed]

`Info.plist` added `/NSBluetoothAlwaysUsageDescription = 'This app needs access to Bluetooth'` and removed `/NSBluetoothPeripheralUsageDescription`, whose old value was also `This app needs access to Bluetooth`.

This means the macOS Bluetooth permission text remains present, but the plist key moved to `NSBluetoothAlwaysUsageDescription`. This is a directly readable Info.plist change.

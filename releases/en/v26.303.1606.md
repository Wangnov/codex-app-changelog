# Codex Desktop `26.303.1606` build 806

> Third-party changelog · Unofficial · Based on reverse diffs of official update artifacts

This release does not add bundle files or cua_node dependencies, but it introduces several user-visible workflow signals across main-process strings, settings copy, and frontend modules.  
One-liner: this build points to deeper Git/worktree automation and deeplink handling, while shrinking the bundled `codex` binary.

## About This Report

This is not an official changelog. It is a **reverse diff report based on official distribution artifacts**. The comparison covers macOS arm64 from `26.228.1430` build 760 to `26.303.1606` build 806.

The source facts come from structured bundle diffs after unpacking official full ZIP packages. The fact pack states that the official appcast did not provide a delta package for this version pair, so this historical backfill compares the two official full packages directly.

Evidence levels:

- **[Confirmed]**: Direct human-readable evidence such as Info.plist, package.json diffs, readable strings, service entries, signatures, or verification results.
- **[Signal]**: Indirect evidence from same-name frontend module diffs or binary size changes. It points in a direction, but cannot recover source-level intent from the artifact.

## ✨ Highlights

### 1. Clipboard-based `codex://` deeplink opening appears in the app [Confirmed]

The new main-process bundle adds readable strings including `Open Deeplink from Clipboard`, `Clipboard does not contain a valid codex:// deeplink.`, `Copy a codex:// URL to the clipboard and try again.`, and `Invalid Deeplink`.

This means the desktop app now contains at least a clipboard-open path and error handling around `codex://` links. The strings are confirmed in the artifact; the exact entry point, menu placement, and full interaction flow cannot be reconstructed from the minified bundle alone.

### 2. PR merge flows start showing up in the desktop app [Confirmed]

The new build adds main-process strings such as `Failed to merge pull request`, `No git repository found for PR merge`, `No workspace available for PR merge`, `Pull request number is required`, and `You have uncommitted local changes. Commit or stash them first.`

Together, these point to added or expanded PR merge handling, including checks for repository context, workspace availability, PR number, and uncommitted local changes. The `git-settings` frontend module also adds copy such as `Pull request merge method`, `Choose how Codex merges pull requests`, `Always force push`, and `Saved pull request merge method`, but that module diff is a [Signal] and only indicates the direction of the settings UI.

### 3. Worktree auto-deletion appears to replace or rename auto-cleanup semantics [Signal]

The same-name `worktrees-settings-page` frontend module adds strings such as `Automatically delete old worktrees`, `Automatic deletion enabled`, `Automatic deletion disabled`, `Auto-delete limit`, and `Disable automatic worktree deletion?`, while removing strings such as `Auto-cleanup limit` and `Saved auto-cleanup limit`.

This points to a settings-page shift from auto-cleanup wording toward automatic deletion, but the frontend artifact cannot prove whether the underlying policy changed. The main process also adds `Failed to clean up archived worktree`, confirming that an archived-worktree cleanup failure path exists in the new artifact.

### 4. Network request approval gains scoped allow-rule language [Signal]

The `index` same-name frontend module adds `Approve a network request and save a host allowlist rule`, `Approve a network request for the current conversation`, and `Approve a network request only for the current attempt`.

This points to a network approval UI that distinguishes host allowlist rules, current conversation scope, and current attempt scope. Because the evidence comes from a same-name frontend module diff, it is not a source-level feature fact.

### 5. Connection and Agent environment UI continues to move [Signal]

The `index` module adds `Add connection`, `A connection with this display name already exists.`, and `Agent environment`. After hash-noise filtering, the real removed frontend module list contains `remote-connections-settings`.

This points to changes around connection management and Agent environment UI, but module names and frontend strings cannot prove that a specific feature was added, removed, or exposed to users.

### 6. WSL mode detection strings are present in a macOS artifact [Confirmed]

The new main-process bundle adds `WSL mode is enabled but no eligible distro is available` and `wsl.exe`.

This confirms that strings related to WSL mode and distro eligibility checks are present in the artifact. Since the compared platform is macOS arm64, this is best read as cross-platform code entering the same bundle, not as evidence that WSL functionality is available on macOS.

### 7. Development and diagnostics scripts add metadata probe/path commands [Confirmed]

`package.json` adds two scripts:

```diff
+    "metadata-path": "pnpm exec tsx ./scripts/dev-metadata.ts path",
+    "metadata-probe": "pnpm exec tsx ./scripts/dev-metadata.ts probe",
```

The `playwright:agent:repl` script also changes to launch via `node --import tsx`. This is a readable configuration change for development, diagnostics, or testing workflows, and may not be visible to ordinary users.

### 8. The main binary shrinks significantly while app.asar grows slightly [Signal]

`Contents/Resources/codex` changes from 84.4 MB to 77.1 MB, a decrease of 7.37 MB. `Contents/Resources/app.asar` changes from 36.4 MB to 36.5 MB, an increase of 0.12 MB.

This shows a clear reduction in the packaged core binary and a small increase in the application-layer bundle. Size changes only point to build, dependency, or code organization changes; they cannot recover source-level intent from the artifact.

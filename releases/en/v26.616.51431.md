# Codex Desktop 26.616.51431 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

The substantive changes in this batch are concentrated in the Windows MSIX: the Computer Use runtime and `@oai/sky` were updated, adding browser backend dependencies, app policy approval, tool-level telemetry, and app-specific instruction injection; the macOS arm64 input sample compares the same package against itself and has no content changes.

Windows carries the substantive runtime changes in this batch, while the macOS sample is byte-identical.

## About This Report

This report uses only the machine-extracted facts in the input packages. The macOS comparison is `26.616.51431 build 4212` to the same `26.616.51431 build 4212`; the Windows comparison is `26.616.5445.0` to `26.616.6631.0`. Platform version numbers and release timing may be offset, so every conclusion is marked by platform.

There are two evidence levels: [Confirmed] comes from manifests, plugin manifests, dependency lists, type declarations, skill or documentation diffs, service entrypoints, and readable strings outside the “same-name frontend module change hints” category; [Signal] comes only from module names, size direction, or compressed bundle hints. This input does not provide standalone sections for real frontend module additions/removals, same-name frontend module hints, new readable strings in the main process or frontend bundles, CSP changes, or newly added components/resources. The high-value changes that can be expanded here mainly come from Windows `cua_node`, dependency manifests, and key file diffs.

Platform markers mean: [macOS+Windows] has evidence on both platforms; [macOS only] appears only in the macOS fact package; [Windows only] appears only in the Windows fact package.

## ✨ Highlights

### [Confirmed] [Windows only] Computer Use Runtime Updated to `cua-node 0.0.2`

On Windows, `resources/cua_node/manifest.json` changes naming from `win32/x64` to `windows/x64`, and `target` from `win32-x64` to `windows-x64`. `node_repl_archive_path` moves from `20260609.1` to `20260619.1`, and the runtime archive moves from `cua-node-0.0.1-20260609232601-73ffe8b8261e-win32-x64.zip` to `cua-node-0.0.2-20260620010411-120c650ffb83-windows-x64.zip`.

The runtime total grows from `285.6 MB` to `298.9 MB`, an increase of `13 MB`, about `+5%`. `node_repl.exe` grows from `13.5 MB` to `14.0 MB`.

In the macOS input sample, `cua_node` remains `377.4 MB → 377.4 MB`, with no file content changes.

### [Confirmed] [Windows only] New CDP Browser Backend Package

Windows adds `@oai/cdp-browser-backend@0.2.0`, described as “Browser Use backend that talks to a Chrome DevTools Protocol endpoint.” The largest added files include:

- `@oai/cdp-browser-backend/dist/main.cjs`, `4.2 MB`
- `dist/skill/scripts/browser-client.mjs`, `1.0 MB`
- multi-platform `classic-level` native prebuilds, including `darwin-x64+arm64`, `win32-x64`, `linux-x64`, `linux-arm64`, and `android-arm64`

This is a directory-level added dependency set, stronger evidence than package size growth alone. It shows that Browser Use backend capability is now inside the Windows distribution package, but the fact package does not provide a UI entrypoint or visible toggle, so this does not prove availability to all users.

### [Confirmed] [Windows only] `@oai/sky` Updated from `0.4.13` to `0.4.19`

On Windows, `@oai/sky/package.json`, `.modules.yaml`, and `.pnpm/lock.yaml` all show `@oai/sky` moving from `0.4.13` to `0.4.19`. The new version adds a dependency on `@statsig/js-client`, and the lockfile adds `@statsig/js-client@3.33.2` plus `@statsig/client-core@3.33.2`.

The older optional dependency chain `objc-js@1.5.0`, `node-addon-api@8.8.0`, and `node-gyp-build@4.8.4` is removed from related lockfile entries. The fact package proves dependency list changes only; it does not establish telemetry toggles, experiment assignment behavior, or runtime exposure to users.

### [Confirmed] [Windows only] Computer Use Adds App Policy Lookup and Risk-Aware Approval

The Windows `MacComputerUseClient` type declaration adds `getAppPolicy(appOrArgs)`, returning `MacAppPolicyResult` with:

- `decision: "allowed" | "denied" | "forbidden"`
- `allowPersistentApproval`
- `target.bundleIdentifier`
- `target.displayName`
- `target.appPath`
- `target.risk: "high" | "low"`
- `warningSubtitle`

`client.js` adds `ComputerUseIPCAppPolicyRequest`, and the default `apiVersion` changes from a hash-like value to `CodexComputerUseIPC-1`. `computer-use-policy.js` now calls `getAppPolicy` before running a tool and generates approval behavior from the result: `denied` reports an organization policy block, while `forbidden` reports a safety-based prohibition. Approval metadata now displays by `bundleIdentifier` and `displayName`, and uses `allowPersistentApproval` to decide whether `always` persistence is allowed.

This shows the Computer Use approval chain inside the Windows package moving from simple app-string confirmation to an approval model with policy results, risk level, persistence limits, and safety rejection paths.

### [Confirmed] [Windows only] Computer Use Tool Calls Now Carry Tool Names and Telemetry Hooks

The Windows `withComputerUsePolicy` type signature changes from receiving only an input object to receiving `ComputerUseToolName` plus the input object. Multiple tool call files now pass explicit tool names into the approval wrapper, including:

- `get_app_state`
- `click`
- `drag`
- `perform_secondary_action`
- `press_key`
- `scroll`
- `select_text`
- `set_value`
- `type_text`

`list_apps.js` adds `logComputerUseToolCalled({ toolName: "list_apps" })`. `create_client.js` adds `logComputerUseClientCreated()` and creates a `Set` passed into the `get_app_state` path. `computer-use-policy.js` also contains `logComputerUseApprovalRequested` and `logComputerUseApprovalResolved`.

These are package-level confirmed facts: tool-level names, approval requests, approval results, and client creation events are now present in the Windows runtime code.

### [Confirmed] [Windows only] App-Specific Instructions Can Be Injected into Computer Use Window State Text

The Windows `AppState` type comment changes from accessibility text describing visible elements and indexes to text prefixed with app-specific guidance on first access when available. `MacWindowAppState` adds `appSpecificInstructions?: string | null`, and the `get_app_state` and `window_result` signatures add `appsWithDeliveredInstructions: Set<string>`.

`window_result.js` reads `appSpecificInstructions`; when present and not already delivered for that app, it wraps the content in `<app_specific_instructions>` tags and prefixes it to the window text. The code also contains an exception set with `com.apple.iWork.Numbers`, for which injection is skipped.

This shows a runtime mechanism inside the Windows package for delivering per-app instructions on first access. The fact package does not include the actual instruction text, so it cannot support a fuller claim about any specific app’s behavior.

### [Confirmed] [Windows only] Computer Use Error Surface Expanded

Windows `errors.d.ts` and `errors.js` add four server error codes:

- `couldNotGetSenderPID: -10017`
- `ambiguousApp: -10018`
- `couldNotGetBootstrapPort: -10019`
- `screenLocked: -10020`

These error paths relate to process identification, ambiguous app selection, bootstrap port retrieval, and locked-screen state, showing that the Computer Use server error surface was expanded inside the Windows package.

### [Confirmed] [Windows only] Bundled Plugin Versions Synchronized to `26.616.51431`

In Windows bundled plugin manifests, `computer-use`, `chrome`, and `browser` `.codex-plugin/plugin.json` versions move from `26.616.41845` to `26.616.51431`. The diff does not show changed description text as new capability text.

The confirmed change is manifest version synchronization; the version bump alone does not prove equivalent functional changes across the plugins.

### [Confirmed] [Windows only] Browser / Chrome / Computer Use Bundled Plugins Remain in the Package

The Windows diff explicitly touches three openai-bundled plugins:

- `computer-use`: described as controlling Windows desktop apps through Computer Use
- `chrome`: described as automation using the user’s existing Chrome state, including tabs, logged-in sessions, cookies, extensions, and Codex Chrome Extension setup
- `browser`: described as the Codex in-app browser for opening, inspecting, navigating, testing, clicking, typing, or screenshotting localhost, `127.0.0.1`, `::1`, `file://`, and the current in-app browser tab

In this batch, their confirmed change is version synchronization; the added browser backend capability is evidenced by the `@oai/cdp-browser-backend` dependency, not by rewritten manifest descriptions.

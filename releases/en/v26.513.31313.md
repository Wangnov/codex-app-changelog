# Codex Desktop 26.513.31313 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

This batch shows a clear built-in plugin rename and LaTeX/Browser capability reshuffle on Windows; macOS looks like a narrower landing from the same code line, centered on Computer Use runtime changes and frontend interaction signals.

One-liner: Windows gets the heavier plugin/runtime reshuffle, while macOS shows a narrower Computer Use and UI-signal update.

## About This Report

This report uses only the supplied reverse-diff facts from officially signed distribution packages: macOS arm64 moves from `26.513.20950` build `2816` to `26.513.31313` build `2867`; Windows x64 MSIX moves from `26.506.3741.0` to `26.519.2736.0`.

Evidence levels:

- [Confirmed]: plist, manifest, package.json, plugin manifests, skill docs, service entry points, dependency lists, explicit config diffs, and newly added readable strings in the fact package.
- [Signal]: frontend module names, compressed bundle size movement, same-name module readable strings/classes; these show packaged UI or state direction, but do not prove source-level behavior or universal user visibility.

Platform markers are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`. The two platform packages have different version and release windows, so each conclusion is tied to its own platform evidence.

## ✨ Highlights

### [Confirmed][Windows only] Built-in plugins were reorganized from `browser-use` / `latex-tectonic` to `browser` / `latex`

The Windows built-in plugin list changes from `browser-use`, `chrome`, `latex-tectonic` to `browser`, `chrome`, `latex`. This is not just a filename rename: the fact package includes new plugin manifests, skill docs, default prompts, and largest-file details.

The `browser` plugin is version `26.519.22136`, with `341` files and total size `7.6 MB`. Its description covers opening, navigating, inspecting, clicking, typing, and screenshotting in the Codex in-app browser, especially for local targets such as `localhost`, `127.0.0.1`, `::1`, and `file://`. Its default prompt includes “Test my checkout flow on localhost”.

The `latex` plugin is version `0.2.0`, with `18` files and total size `49.7 MB`. It expands beyond the old `latex-tectonic` shape: the docs cover Tectonic-first simple builds, TeX Live or MacTeX fallback, `latex-doctor`, `latex-compile`, and optional Codex-managed TeX Live installation when no usable system installation is present. Its largest file is `bin/tectonic.exe` at `49.6 MB`.

### [Confirmed][Windows only] Browser-related app server runtime is now packaged, with large size impact

Windows adds `resources/plugins/openai-bundled/plugins/chrome/app-server-runtime/codex` at `247.7 MB`, plus `node` at `91.4 MB` and `node_repl` at `13.5 MB` in the same area. `resources/app.asar` grows from `136.7 MB` to `159.0 MB`, up `22.25 MB`; `Codex.exe` grows from `222.7 MB` to `226.6 MB`, up `3.92 MB`.

This indicates more browser/app-service runtime content is included in the Windows package in this sample. macOS has no matching added plugin inventory evidence in this batch.

### [Confirmed][Windows only] Chrome extension detection now summarizes across multiple profiles

Windows `check-extension-installed.js` changes from a single Preferences-path flow to a multi-profile flow. The new code adds `listChromeProfileDirectories` and `getChromeExtensionInstallStatusForProfile`, enumerates `Default` and `Profile N`, and reports installation, registration, enabled state, disabled reasons, and versions per profile.

The new logic also treats an unpacked extension path from `preferences.path` as installation evidence. CLI output shifts from “Checked Chrome profile” to “Checked Chrome user data directory”, “Selected Chrome profile”, “Installed in any profile”, and “Enabled in any profile”.

### [Confirmed][Windows only] LaTeX moves from a single-engine package toward a full workflow

The old Windows `latex-tectonic` plugin described bundled Tectonic compilation for LaTeX and TeX documents. The new `latex` plugin describes a workflow that tries bundled Tectonic for simple projects, falls back to TeX Live or MacTeX, and checks local LaTeX readiness.

The skill docs expose three capability lines:

- `latex-compile`: runs `python3 scripts/compile_latex.py /absolute/path/to/main.tex`, with options such as `--compiler tectonic`, `--compiler texlive`, `--engine xelatex`, and `--json`.
- `latex-doctor`: detects Tectonic, TeX Live, MacTeX, `latexmk`, `pdflatex`, `xelatex`, `lualatex`, `biber`, and `kpsewhich`.
- `texlive-runtime-installer`: detect-only by default; only after user confirmation does `--install-managed-full` run. The managed path is `~/.cache/codex-runtimes/codex-texlive/full`, with explicit safety notes that it does not use `sudo`, write `/Library/TeX`, write `/usr/local/texlive`, or modify shell startup files.

### [Confirmed][macOS only] Computer Use receives a small plugin-version bump and runtime binary growth

On macOS, the `computer-use` plugin manifest moves from `1.0.791` to `1.0.793`. The bundle file tree has no added or removed files: `1331 → 1331`, with `96` content changes.

The size movement is concentrated in Computer Use runtime binaries: `CUALockScreenGuardian` moves from `14.7 MB` to `14.9 MB`, up `0.17 MB`; `SkyComputerUseService` moves from `15.0 MB` to `15.1 MB`, up `0.14 MB`; `SkyComputerUseClient` moves from `10.1 MB` to `10.2 MB`, up `0.08 MB`.

### [Signal][macOS+Windows] Remote connections and device-control UI continue to evolve, with different platform landings

On macOS, `remote-connections-settings` adds text for failures while loading devices that can control the local computer, renewed ChatGPT authentication, and retrying the device load. Strings include `settings.remoteConnections.remoteControlClients.loadError.authRequired` and `Retry to load devices that can control this computer`.

On Windows, the same module changes more broadly. New strings include “Access and control other devices from this computer”, “Add SSH Connection”, “Add device”, “Add device to control this Mac remotely”, and “Add device to control this PC remotely”, plus accessible labels for the remote-connections view selector, SSH error details, refreshing signed-in devices, and refreshing discovered SSH connections.

Boundary: these are packaged frontend UI/state signals and do not prove the corresponding entry points are visible to every user.

### [Signal][macOS+Windows] Plugin pages, plugin details, and marketplace management text expand

On macOS, `plugin-detail-page` adds `Add to Codex`, `Adding to Codex`, `App enabled`, `App disabled`, `App unavailable`, `Add people`, and `Anyone at workspace with the link`, plus accessible text for a plugin share dialog and MCP server enablement toggle.

On Windows, `plugins-page` adds `Access is turned off by your admin`, `Built-in marketplaces are upgraded by Codex`, `Built-in marketplaces cannot be removed`, `Choose another plugin marketplace`, `Create a plugin for your team`, `Created by me`, `Curated by OpenAI`, `Disabled by admin`, and `Have a workflow in mind?`.

This is a high-visibility cross-platform plugin-management signal, but the macOS evidence comes from the plugin detail page, while the Windows evidence comes from the plugins page and new plugin inventory. It should not be flattened into a claim of identical UI.

### [Signal][macOS+Windows] Local conversation threads gain summary, review, diff-action, and attachment-related entry points

On macOS, `local-conversation-page` adds `Toggle summary` and `Toggle pinned summary`; `local-conversation-thread` adds `Open code review comments` and `localConversation.reviewComments.open`, with classes such as `model-authored` and `group/review-comments`.

On Windows, `local-conversation-thread` adds text for opening a Google Drive conversation resource card, opening the active heartbeat automation, opening review view from a completed turn diff card, and a disclosure chevron on pull request summary rows. `diff-unified` adds `Stage all`, `Stage file`, `Revert all`, `Revert file`, plus tooltips for staging, unstaging, and reverting files, hunks, and sections.

Together, these point to thread-level UI work around summaries, reviews, diff handling, external resource cards, and automation state.

### [Signal][Windows only] Codex Mobile, Appshot, floating avatar, and project-generation entry points appear across more UI

Windows new modules include `codex-mobile-page`, `codex-mobile-setup-dialog`, `codex-mobile-setup-flow`, `appshot-logo`, `appshot-window`, `appshots-settings`, `avatar-mascot-button`, and `avatar-overlay-analytics`. Same-name modules also add `Appshot shortcut failed/found target/had no target/received/started appshot`, plus `Appshot attachment` and `Alt text for appshot attachment in local conversation`.

`welcome-page` adds teen-onboarding example prompt text including `Build a game you can play in your browser.`, `Dodging Game`, and `Write my own prompt`. These are high-visibility Windows frontend signals; the macOS input does not provide equivalent new-module evidence.

### [Signal][Windows only] Usage, credits, and auto-reload settings become more complete

Windows `usage-settings` adds `Add credits`, `Auto-reload credits`, `Auto-reload is off`, `Auto-reload is on`, `Automatic reload`, `Automatically add credits when your balance runs low`, save-auto-reload and submit-add-credits button labels, and `https://developers.openai.com/codex/pricing`.

`app-main` and `composer` also add `Add credits`, `Add credits to continue using Codex now.`, and labels for upgrading from core usage surfaces. Evidence remains signal-level because it comes from frontend bundle strings and module changes.

### [Signal][Windows only] Hooks settings shift toward project-level trust review

Windows `hooks-settings` adds `Hooks can run outside of the sandbox so we ask you to review any recently installed or modified hooks`, `Banner shown when hooks require trust review`, `Projects with configured hooks will appear here`, `No hooks found`, `No projects found`, and project-row hook load issue counts.

It removes more event-enumeration-style strings such as `Before a tool executes`, `After a tool executes`, `Before Codex compacts the conversation`, and `After Codex compacts the conversation`. The overall signal is a Hooks page that emphasizes project discovery, load issues, and trust review.

### [Confirmed][Windows only] AppxManifest adds Office spreadsheet-like file associations

Windows AppxManifest adds file type associations for `.tsv`, `.xls`, `.xlsm`, and `.xlsx`. This is manifest-level evidence that the Windows package declares a broader file handling surface.

### [Confirmed][Windows only] Electron and key dependencies are upgraded

Windows `package.json` shows app-layer version `26.506.31421 → 26.519.22136` and `codexBuildNumber` `2620 → 3003`. `electron` moves from `41.2.0` to `42.1.0`; `better-sqlite3` moves from `^12.8.0` to `^12.9.0`.

Script changes include `dev` and `owl` moving to `scripts/prepare-dev.mjs`, plus new `forge:make:owl-shell` and `e2e:computer-use-native-pipe` scripts.

### [Confirmed][Windows only] Web sandbox preload adds a `navigate` port

Windows `.vite/build/sandbox-preload.js` adds `navigate` to the MCP apps sandbox guest message port list. The prior list started with `notifyMcpAppsHostContext`; the new version includes `navigate` in the same `portNames` set. The preload diff confirms the port exists, but the exact navigation semantics are not recoverable from this fact alone.

### [Signal][macOS+Windows] Syntax highlighting / grammar packages show broad language movement

On macOS, same-name frontend modules for `cpp`, `typescript`, `tsx`, `php`, `less`, `csharp`, `fortran-free-form`, `haxe`, `scss`, `python`, `ruby`, `tex`, `regexp`, `gdshader`, `sql`, and `json` add many syntax scope strings and classes.

On Windows, related evidence is more scattered, but several modules move in opposite directions versus macOS: macOS `pdf` shrinks from `0.4 MB` to `0.0 MB` while Windows `pdf` grows from `0.0 MB` to `0.4 MB`; macOS `general-settings` removes many settings strings while Windows `general-settings` adds similar strings. This looks more like cross-platform chunking or version-window skew than straightforward feature addition/removal.

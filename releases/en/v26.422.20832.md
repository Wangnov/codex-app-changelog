## ✨ Key Changes

### 1. Browser Use Has Been Repositioned as the Local Development Browser Automation Entry Point [Confirmed]

The display name of the `browser-use` plugin changed from `Browser` to `Browser Use`, and its description changed from the generic “Control browsers from Codex” to local target operations for the Codex in-app browser, specifically naming `localhost`, `127.0.0.1`, `::1`, and `file://`.

This means that when developers ask Codex to open, inspect, test, click, type into, or screenshot local pages, the recommended in-product entry point is now more clearly the in-app browser rather than macOS `open`, external Playwright, or general web browsing. Key evidence includes newly added default prompts in the plugin configuration: `Open localhost:3000`, `Test my latest changes in the browser`, plus newly added `composerIcon` and `logo` pointing to `./assets/browser.png`.

### 2. Browser Skill Documentation Tightens the Execution Path and API Boundaries [Confirmed]

The `browser/skill` documentation now requires the full skill document to be read before the first browser action; browser control can only be performed through the Node REPL `js` tool; and Playwright mentioned in the document refers only to the `tab.playwright` interface after browser-client initialization is complete.

This indicates that Browser Use invocation has been tightened at the artifact level: it is more like a dedicated in-app browser control channel than free switching among multiple browser automation backends. There are also interface-level changes: declarations related to `waitForEvent("filechooser")`, `downloadMedia`, and `PlaywrightFileChooser` were removed, while `FinalizeTabsOptions`, `FinalizeTabsKeep`, and `FinalizeTabStatus` were added, and `BrowserUserTabInfo` gained an `id` field.

### 3. New Local LaTeX/Tectonic Plugin [Confirmed]

The plugin marketplace manifest adds `latex-tectonic`, sourced from the local path `./plugins/latex-tectonic`, categorized as `Engineering`, with install strategy `AVAILABLE` and authentication strategy `ON_INSTALL`. The file tree also now includes `Contents/Resources/plugins/openai-bundled/plugins/latex-tectonic/bin/tectonic`, sized `50.3 MB`.

This means the desktop package now includes a local capability plugin for LaTeX/Tectonic. A possible visible effect is that Codex may later be able to handle or compile LaTeX projects more directly; however, this report only confirms that the plugin and binary entered the distribution package, and does not infer whether any specific UI has already been exposed.

### 4. Computer Use Plugin Patch Version Update, Underlying Service Size Increased [Confirmed]

The `computer-use` plugin version increased from `1.0.755` to `1.0.758`. At the same time, `SkyComputerUseService` grew from `11.9 MB` to `14.1 MB`, an increase of `2.23 MB`.

This shows that Computer Use was updated as an independent plugin version alongside the package update, with an accompanying increase in the service binary size. What can be confirmed here is the plugin version and packaged size change; specific functional changes cannot be reconstructed further.

### 5. Browser-Side “Comment with Codex” Copy Enters Many Localization Files [Confirmed]

Multilingual `native-menu-locales` files added or adjusted `browserSidebar.contextMenu.commentWithCodex`. For example, Simplified Chinese changed from having no such entry to `"browserSidebar.contextMenu.commentWithCodex":"评论"`; corresponding translations also appear in many language files, including Traditional Chinese, Japanese, Korean, Spanish, French, and German.

This points to a new user-visible browser-side context menu entry: “Comment with Codex.” Because the evidence comes from localization resources, it confirms that the menu copy has entered the package; however, whether it is shown by default in all scenarios still depends on runtime conditions, which the fact package does not provide.

### 6. New Frontend Modules Point Toward Settings, Authentication, PDF, and Event Pipelines [Signal]

After removing Vite hash filename noise, genuinely new modules include `browser-use-settings`, `experimental-features-settings`, `use-auth`, `pdf`, `product-event-signal`, `analytics-logger-provider`, `app-server-manager-signals`, `artifact-preview-page-navigation`, and others.

These module names point toward browser-use settings, experimental feature settings, authentication state, PDF handling, product event signals, and logging/reporting directions, but source-level intent cannot be reconstructed from the artifact, nor do they prove that these features are already exposed to users. They are recorded here only as indirect signals.

### 7. Development Startup Flow Adds a bare modifier monitor Build Step [Confirmed]

In `package.json`, the `dev` and `owl` scripts now add `pnpm exec node ./scripts/build-bare-modifier-monitor.mjs`. The file tree also adds `Contents/Resources/native/bare-modifier-monitor`, sized `0.1 MB`.

This indicates that the desktop package introduced a native component named `bare-modifier-monitor` and added a build step for it in the development startup flow. The fact package alone cannot confirm the specific user-facing functionality it corresponds to, but it is not merely a hash rename.

### 8. Version Metadata and Packaging Integrity Updated [Confirmed]

In `Info.plist`, `CFBundleShortVersionString` was updated from `26.417.41555` to `26.422.20832`, and `CFBundleVersion` was updated from `1858` to `2025`. `ElectronAsarIntegrity/Resources/app.asar/hash` was also updated from `eda8b73ad668dd304d3a68504c3c2882429418e6f9af71c4191bcffaf18bdbc6` to `2b5ed98dcfa233d7ed11349b82999a6ae8e045a51e83aaf02bcd7ecbbbc6b1f7`.

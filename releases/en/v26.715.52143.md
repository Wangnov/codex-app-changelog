# Codex Desktop 26.715.52143: macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

The clearest shared change is a native HID-topology watcher with associated failure paths. On the frontend, Windows shows a stronger concentration of skills, multi-agent, and internal debugging changes, while macOS shows a distinct direction around site publishing and sharing.

Both packages add a native HID-topology watcher, while Windows shows stronger signals around skills, multi-agent workflows, and debugging, and macOS shows a distinct site-sharing and publishing direction.

## About This Report

This report compares machine-extracted differences from reconstructed official signed distribution packages. Because this is a historical backfill, the macOS analysis compares two full official packages directly; both passed notarization validation and identify OpenAI OpCo, LLC as the developer. The Windows analysis compares two full official MSIX packages.

The comparison ranges are:

- [Confirmed][macOS only] 26.715.31925 build 5551 → 26.715.52143 build 5591.
- [Confirmed][Windows only] MSIX 26.715.4045.0 → 26.715.7063.0; the embedded application version also changes from 26.715.31925 to 26.715.52143, with build 5551 changing to 5591.

Evidence is divided into two levels:

- [Confirmed] covers plist data, manifests, `package.json`, plugin manifests, dependency lists, native files, and newly present readable strings. A string’s presence in a package is confirmed, but does not mean its associated UI is enabled for every user.
- [Signal] covers size, string, and style-set changes in matching minified frontend modules. These can indicate UI, state, layout, or chunking changes, but cannot reconstruct source code or independently prove a feature launch or removal.

Platform labels are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`.

The fact packages do not provide complete built-in plugin-list changes, non-hash frontend file additions and removals, CSP differences, or Windows AppxManifest differences. This report therefore makes no claim that those areas remained unchanged.

## ✨ Highlights

### Native HID-topology monitoring enters both packages

[Confirmed][macOS+Windows] Both platforms add a native `hid-topology-watcher.node` component. The file is 0.1 MB on macOS and 1.1 MB on Windows.

[Confirmed][macOS+Windows] Three associated failure paths also appear in the new main-process or frontend bundles:

- `Failed to reconcile HID topology`
- `Failed to watch HID topology`
- `HID topology watcher addon not found`

This confirms the presence of HID-topology watching, reconciliation, and missing-addon handling in the packages. The evidence does not identify the particular device class or user workflow that consumes it.

[Confirmed][macOS+Windows] The Computer Use runtime is unchanged at 365.7 MB on macOS and 287.5 MB on Windows. Neither platform adds a `cua_node` dependency. The HID change therefore appears as a separate native application resource, not an expansion of Computer Use runtime dependencies.

### Shared restructuring of chat navigation, history, and environment filters

[Signal][macOS+Windows] The `header` module grows by roughly 40 KB on both platforms and gains the same group of strings and accessibility labels, including `All chats`, `Chat history`, `Cloud chats`, `All environments filter option`, `App settings`, `Clear search`, a back-button label, and a more-actions button for the Chrome extension side-panel header.

Together, these signals point to chat history, cloud chats, environment filtering, recent-task search, and Chrome side-panel controls being consolidated into a new header-navigation structure. Because the evidence comes from matching minified modules, it does not prove that every entry is visible to every user.

### Windows gains signals for skill creation, search, and refresh flows

[Signal][Windows only] `skills-page` grows by 15,629 bytes and gains copy covering:

- `New skill` and a create-skill button;
- skill-list search;
- installed and recommended skill sections;
- loading and filtered empty states;
- reloading the skill list;
- a notice that newly installed skills require a refresh before composer use.

This is a comparatively complete skill-management UI signal spanning discovery, creation, post-install refresh, and filtering. The fact package contains no newly added skill plugin, skill documentation, or service entry point, so it does not confirm a new skill runtime capability.

### Windows introduces multi-agent home guidance

[Signal][Windows only] `home-announcements` grows by 26,379 bytes and gains multi-agent composer-banner copy, dismissal labels, and:

`Delegate work to subagents that work in parallel. Note: may increase token usage.`

The same module adds strings for Fast mode, rate-limit resets, gift credits, backend-driven home banners, and `Codex App Home Beacon Debug`. This confirms that the frontend module now carries multi-agent guidance and additional home-announcement states, but not that the banners are generally displayed.

[Signal][macOS only] The macOS module moves in the opposite direction, shrinking by 28,328 bytes and losing strings related to multi-agent, Fast mode, rate-limit reset, gift-credit, and backend-driven banners. “Removed” here only means that the strings left this matching module; it does not establish that the associated capabilities were removed from the macOS product.

### macOS shows site-publishing and access-control UI signals

[Signal][macOS only] `appgen-share-dialog` grows by 24,643 bytes and gains copy covering:

- adding people or groups;
- searching workspace users and groups;
- `Anyone in this workspace with the link`;
- `Anyone on the Internet`;
- `Can visit`;
- an in-progress save label;
- copying a live-site URL;
- opening a live site externally;
- publishing a site to the public internet;
- `Copied to clipboard` and `Copy link`.

Combined with scrolling-list, save-action, and accessibility-state styles, these strings form a substantial signal for site sharing, access-scope selection, and public publishing. They remain matching-module evidence and do not prove that public publishing is enabled by default.

### Windows substantially expands its debugging interface

[Signal][Windows only] `debug-modal` grows from near zero to about 0.2 MB, adding 226,002 bytes. New strings include `Account ID`, `Active browser tab`, `Active conversation`, `Activity grouping version`, `All processes`, `Ambient suggestion tasks`, `App Actions`, `Allow host`, `Allow host starting`, and `(command unavailable)`.

Numerous dialog, toggle, sorting, state-color, and backdrop-blur styles also enter the module. This indicates that the packaged Windows debug or diagnostic interface now covers more account, browser-tab, conversation, process, host-permission, and app-action state. It does not imply that these are normal end-user settings.

### Windows permission and general-settings modules undergo major splitting

[Signal][Windows only] `general-settings` shrinks by 92,351 bytes. Strings leaving the module concern default permissions, Full Access visibility, composer context usage, code sizing in chats and diffs, base ChatGPT UI sizing, hotkey capture, and ChatGPT macOS Dock-icon selection.

[Signal][Windows only] `permissions-mode-dropdown` shrinks by 25,480 bytes. Removed strings include `Approve for me`, `Ask for approval`, `Change permissions`, `Custom (config.toml)`, and permission descriptions for website access, sending data, using plugins, and editing external files.

These are high-visibility signals of settings and permission UI reorganization. Because the evidence only shows strings leaving their previous modules, it cannot establish that the settings were deleted; they may have moved, been rechunked, or been rewritten.

### Windows shows splitting signals in task onboarding, mentions, and checkout

[Signal][Windows only] `tasks` shrinks by 306,482 bytes. Content leaving the module covers connecting and installing apps, preparing a connected-app task, executing onboarding tasks, and animated status copy for generated CSV chart previews.

[Signal][Windows only] `at-mention-list` shrinks by 170,819 bytes. Removed strings concern ChatGPT conversations, Computer Use, custom agents, desktop apps, files and chats, Microsoft Excel and PowerPoint control, and MCP capability mentions.

[Signal][Windows only] `checkout-webview-page` shrinks by 13,227 bytes, with `Back to ChatGPT`, the embedded checkout-toolbar return button, and `https://pay.openai.com` clues leaving the module.

Together, these changes point to substantial Windows frontend rechunking around task onboarding, capability mentions, and embedded checkout. The evidence is insufficient to conclude that the associated entry points were removed.

### macOS plugin-detail, workbook, and animation-resource modules change shape

[Signal][macOS only] `plugin-detail-page` shrinks by 158,328 bytes. Strings concerning plugin share-target search, link access scope, an MCP-server enablement toggle, detail-page section navigation, and `App unavailable` leave the module.

[Signal][macOS only] `workbook-from-markdown` shrinks by 108,179 bytes, losing multiple errors concerning document closing, serialization, plugin lists, and find-and-replace operations.

[Signal][macOS only] `plugins-settings` loses `Manage plugins, skills, and MCPs` and the subtitle for the combined plugin-settings section. This only confirms a copy change in the previous module; it does not prove that plugin, skill, or MCP management was removed.

[Signal][macOS only] `animated-icon` grows by 11,729 bytes and gains several icon sizes, `web-search`, and a dotLottie WASM CDN address, indicating expanded animated-icon loading and sizing behavior.

### Built-in Browser and Chrome plugin versions stay aligned

[Confirmed][macOS+Windows] The `browser` and `chrome` built-in plugin manifests both move from version 26.715.31925 to 26.715.52143. The supplied diffs change only the version field; their descriptions remain unchanged. There is therefore no evidence of new commands, permissions, or service entry points for these two plugins.

[Confirmed][Windows only] The `computer-use` plugin manifest also moves from 26.715.31925 to 26.715.52143. Its description remains desktop-app control on Windows through Computer Use. The fact package shows no other manifest change for this plugin.

### Development build scripts and platform metadata are updated

[Confirmed][macOS+Windows] In both packages, the application version in `package.json` changes from 26.715.31925 to 26.715.52143 and `codexBuildNumber` changes from 5551 to 5591. `prepare:owl` now calls `prepare-dev.mjs --owl`, adding a dedicated parameter to the Owl development preparation path.

[Confirmed][macOS only] `CFBundleShortVersionString` changes to 26.715.52143, `CFBundleVersion` changes to 5591, and the Electron ASAR integrity hash is updated with the changed `app.asar`.

[Confirmed][Windows only] The outer MSIX version is 26.715.7063.0, while the embedded `package.json` version is 26.715.52143 with build 5591. This demonstrates that the Windows distribution version and shared application-code version are not identical in this batch.

## 🔎 Low-Level Signal Index

### Event and plugin error handling

[Signal][macOS only] The `esm` module gains `Callback Error`, `Context Cancel`, `Failed to deliver`, destination loading and unloading failures, `Not ready`, Promise timeout, and `plugin Error`. Previous event-bus connection, queue, and shim-dispatch strings leave the module. This suggests internal event or plugin-error-path reorganization but does not establish exact runtime behavior.

### Plugin-detail and sharing-module movement

[Signal][macOS only] Sharing and MCP-toggle clues decrease in the plugin detail page while the site-sharing dialog grows substantially. This is a weak correlated signal of redistributed UI responsibilities, not evidence that site sharing replaced plugin sharing.

### Concentrated Windows frontend rechunking

[Signal][Windows only] `debug-modal`, `skills-page`, and `home-announcements` grow while `tasks`, `general-settings`, `permissions-mode-dropdown`, `at-mention-list`, and the checkout page shrink. This indicates changed build boundaries and page responsibilities; bytes disappearing from one module do not equal product-capability removal.

### Short generic modules and build noise

[Signal][macOS only] `dist` loses only a few strings such as `dil.action`, `dil.error`, and `dil.load`. Their context is too limited to elevate them into a functional change.

[Signal][macOS+Windows] Large sets of additions and removals under `webview/assets` come from Vite hash renaming. They are not counted as genuine feature-file additions or removals.

## ⚖️ Boundaries and Disclaimer

This is an unofficial third-party report based on differences between official distribution packages. It is not an OpenAI release note.

No application binaries are redistributed. [Confirmed] means that the relevant file, configuration value, or string is present in the analyzed package; packaged copy does not mean that a capability is enabled for every account, region, or release channel.

[Signal] evidence comes from size, string, and style-set differences in minified frontend modules and does not constitute source-code fact. A shrinking module also does not independently prove feature removal, because code may have moved, been rechunked, or become conditionally loaded.

Corrections and additional evidence from complete plugin inventories, CSP data, AppxManifest changes, feature flags, or runtime observations are welcome.

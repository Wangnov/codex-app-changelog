# Codex Desktop 26.519.81530 macOS / Windows Comparison

> Third-party cross-platform changelog · unofficial

This batch looks like two offset package points from the same code line: plugin versions align, but visible frontend capability signals do not.  
One-liner: plugin manifests align across platforms, while UI-facing signals diverge sharply between macOS and Windows.

## About This Report

This report uses only the machine-extracted diff facts from official signed distribution packages. It does not use official release notes and does not infer behavior that is not present in the packages.

Evidence is split into two levels: [Confirmed] comes from plist, plugin manifests, readable strings, dependency lists, file trees, and concrete diffs; [Signal] comes from same-name frontend module size changes, module names, class names, and compressed bundle string direction. Platform tags are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`.

The compared ranges are macOS arm64 `26.519.41501` build `3044` → `26.519.81530` build `3178`, and Windows x64 MSIX `26.519.5221.0` → `26.519.11010.0`. The two platforms have different outer version numbers and package samples, so conclusions from one platform are not applied to the other.

This fact pack does not provide full built-in plugin inventory changes, Computer Use runtime totals, content security policy changes, new component/resource details, or AppxManifest changes. Those areas are handled only where facts are available.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Browser and Chrome bundled plugin versions move to 26.519.81530

On both platforms, the `browser` and `chrome` plugin manifests update their version from `26.519.41501` to `26.519.81530`. The diff does not show semantic changes to the plugin descriptions: Browser still points to the in-app browser, localhost, file URLs, screenshots, clicks, and typing; Chrome still points to remote URLs, authenticated pages, existing tabs, cookies, extensions, and Chrome Extension setup.

The platform difference is in the outer app version: macOS itself moves to `26.519.81530`, while the Windows outer version is `26.519.11010.0`; the embedded plugin manifest version still lands on `26.519.81530`.

### [Signal] [macOS+Windows] Terminal bottom-panel module expands significantly, with crash recovery and accessibility strings

On both platforms, the same-name `thread-page-bottom-panel-state` module grows from near-empty to about `0.3 MB`. New strings include a button label for reloading terminal rendering after a crash, a description for embedded terminal crashes, inability to enable accessibility before `Terminal.open`, buffer recycling limits, mouse event binding, keypad switching, and blocked link opening.

This is a clear cross-platform signal of work around terminal bottom-panel UI and state handling. It remains frontend bundle evidence, so it does not prove that every user can see a new terminal panel.

### [Signal] [macOS only] Browser-use settings expand on macOS, while the Windows sample removes the same family of signals

On macOS, `browser-use-settings` adds about `47 KB` and introduces strings for `Access history without asking`, adding allowed/blocked domains, allowed/blocked download domains, allowed/blocked upload domains, accessibility labels for expanding/collapsing data-clearing options, and a disabled toggle for unavailable plugins.

On Windows, the same-name module shrinks by about `47 KB` and removes strings in the same direction. In this input sample, browser permission/domain-rule settings are an added macOS signal and a removed or non-landed Windows signal.

### [Signal] [macOS only] Thread side-panel and bottom-panel entry points expand on macOS, while Windows removes similar entry-point signals

On macOS, `thread-app-shell-chrome` adds action labels for opening side chat, browser tab, review tab, terminal, timeline, and workspace file browser from the side panel. It also adds `Bottom panel`, `Browse project files`, and button labels for closing or adding tabs from the bottom panel.

On Windows, the same-name module shrinks by about `21 KB` and removes a set of those entry strings. The platform split is clear: the macOS sample shows richer multi-panel navigation signals, while the Windows sample does not show the same landing and instead removes related strings.

### [Signal] [macOS only] Skills page expands on macOS, while the Windows sample removes similar page strings

On macOS, `skills-page` adds strings for creating a skill, reloading the skills list, newly installed skills requiring refresh before composer use, filtered empty states, installed and recommended skills sections, search input, and loading skills.

On Windows, `skills-page` shrinks by about `9.9 KB` and removes part of the same skills-page string family. Since there is no new built-in plugin detail in the facts, this should be described as skills-management UI signal on macOS, not as a new skill capability surface.

### [Signal] [macOS only] Worktree initialization failure and recovery paths become more specific

On macOS, `worktree-init-v2-page` adds strings for creating a worktree, forking the current conversation, creation success followed by renderer failure to start the conversation, creation failure, retry, editing environment, and canceling to start a local conversation.

These strings indicate finer-grained UI signals for worktree initialization status, failure recovery, and environment settings entry points on macOS. The Windows fact pack does not list a corresponding module change.

### [Signal] [Windows only] General settings add font-size, work-mode, dictation, environment, and projectless-window signals

On Windows, `general-settings` grows by about `100 KB`. New strings cover code base size, Codex UI base size, work mode settings, Agent environment, hold-to-dictate hotkey, hotkey window, dictation dictionary entry, ambient suggestions, and the projectless default setting for Popout Window.

These are high-visibility settings signals, but they come from same-name frontend module change clues and should not be treated as source-level proof of enabled settings. The macOS fact pack does not list the same change.

### [Signal] [Windows only] Header adds recent tasks, cloud tasks, and environment-filter signals

On Windows, the `header` module adds strings for the recent tasks menu, All tasks, Cloud tasks, All environments filter option, clearing search, environment search failure, and environment listing failure, with state classes such as `aria-live` and `aria-atomic`.

This points to Windows-only UI signals in the header area for recent tasks, cloud tasks, and environment filtering. The macOS fact pack does not include a corresponding header change.

### [Signal] [Windows only] Thread side-panel tabs remove a set of CI/PR/edit-acceptance prompts

On Windows, `thread-side-panel-tabs` shrinks from `0.5 MB` to near-empty and removes strings about external log diagnosis, CI system access, credential/permission/token/MCP/tool access checks, an existing pull request, and accepting edits to stage them.

This looks like some side-panel prompts or workflow instructions moved out of this module. Because the evidence is only compressed frontend module change data, it does not prove that the underlying capabilities were removed overall.

### [Signal] [macOS+Windows] Markdown and Alert components share slimming/rearrangement signals

On both platforms, `markdown` shrinks by about `60 KB`, removing strings for no-alt image/video fallback, placeholders, file citation chip accessibility labels, KaTeX-related strings, and several style classes. `alert` shrinks by about `1.4 KB` on both platforms, removing validation background/border classes, `data-testid`, and layout classes, while adding an `http://www.w3.org/2000/svg` style/string signal.

This is a low-level rendering and alert-component rearrangement signal, not enough by itself to identify a user-visible feature change.

### [Confirmed] [Windows only] Windows package structure removes bwrap and adjusts several runtime binary sizes

The Windows file tree changes from `832` files to `831`, with `0` added, `1` removed, and `19` content-modified files. The removed file is `resources/codex-resources/bwrap`, sized `0.5 MB`.

Runtime binary size shifts are notable in the same Windows batch: `resources/codex.exe` and the Chrome plugin app-server-runtime `codex` both move from `247.7 MB` to `239.6 MB`, down `8.11 MB`; `resources/codex-windows-sandbox-setup.exe` moves from `8.3 MB` to `6.1 MB`, down `2.21 MB`; `resources/codex` moves from `219.5 MB` to `221.7 MB`, up `2.17 MB`; and `resources/codex-command-runner.exe` grows by `0.12 MB`.

### [Confirmed] [macOS only] macOS main binary and Chrome plugin runtime grow slightly

The macOS file count remains `1335 → 1335`, with `0` added, `0` removed, and `61` content-modified files. Both `Contents/Resources/codex` and the Chrome plugin app-server-runtime `codex` grow from `192.1 MB` to `193.8 MB`, up `1.70 MB`.

`app.asar` is shown as `159.0 MB → 159.0 MB`, but the Electron asar integrity hash in Info.plist changes from `91bd...c5c8` to `2cc5...c8b9`, confirming that the application-layer package content changed.

`thread-page-bottom-panel-state` expands on both platforms, while `thread-app-shell-chrome` expands on macOS and removes similar signals on Windows. `browser-use-settings` adds allowed/blocked domains, upload/download domains, history access, and data-clearing expand/collapse labels on macOS; the Windows sample removes strings in the same direction. Evidence boundary: readable strings entering or leaving the bundle are facts, but runtime enablement still requires live verification.

### Skills Management

`skills-page` adds create, refresh, search, empty states, and installed/recommended skills headings on macOS; Windows removes related signals. Evidence boundary: there is no new built-in plugin detail, so this is a skills-page UI signal, not a new skills capability surface.

### Settings, Tasks, and Environments

Windows `general-settings` and `header` point respectively to font sizes, dictation, environments, projectless popout, recent tasks, cloud tasks, and environment filters. Evidence boundary: this is Windows-only frontend signal; the macOS fact pack does not provide corresponding changes.

### Markdown, Alert, and Plugin Detail

Both platforms show common slimming signals in `markdown` and `alert`; `plugin-detail-queries` only changes by `+180 bytes` and removes `react.memo_cache_sentinel` on both platforms. Evidence boundary: these look closer to implementation rearrangement or build-output changes, with lower value than explicit command, setting, or page text.

### Debug Modal

On macOS, `debug-modal` shrinks by about `150 KB` and removes strings such as `Account ID`, `App Actions`, `App Server`, `App session ID`, and automatic worktree cleanup. The Windows fact pack does not list a same-name change. Evidence boundary: this only shows readable debug-modal content leaving or being rearranged in the macOS sample; it does not prove removal of the broader debugging capability.

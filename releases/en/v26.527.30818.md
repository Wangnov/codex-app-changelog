# Codex Desktop `26.527.30818`(build 3370)

> Third-party changelog · Unofficial · Based on reverse-engineered differences in official update packages

The most notable change in this release is the foundation of the desktop shell: package contents shift from Electron-related paths to a Codex Framework / Chromium model, while the frontend and plugin layers show new signals around app generation, browser sidebars, chart rendering, collaboration, and host connectivity.  
One-liner: the app shell appears to move toward a Chromium-based Codex Framework packaging model, with broader browser/app-host and project workflow surfaces.

## About this report

This is not an official changelog, but a **reverse-engineered diff report based on officially distributed artifacts**. This update is a historical backfill: both versions fall outside the official Sparkle delta window, so no delta package reconstruction was used. Instead, two official full packages were downloaded and compared through a structured bundle diff.

The comparison covers `26.519.41501` build `3044`, released on `2026-05-23`, through `26.527.30818` build `3370`, released on `2026-05-29`. Confidence comes from Apple notarization and OpenAI Developer ID code-signing verification: both the previous and latest `Codex.app` are `accepted`, both have `Notarized Developer ID` as the source, and both are signed by `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`.

This report uses two evidence levels:

- **[Confirmed]**: Direct evidence from Info.plist, package.json, plugin configuration, readable diffs, signing and notarization results, and similar sources.
- **[Signal]**: Indirect evidence from frontend module names or binary size changes. These point in a direction, but cannot reconstruct source-level intent from the artifact and do not mean the feature is necessarily available to users.

## ✨ Highlights

### 1. The desktop foundation shifts from Electron traces to a Codex Framework / Chromium model [Confirmed]

A new `Contents/Frameworks/Codex Framework.framework/Versions/148.0.7778.179/` appears in the package, with the main framework file sized at `248.3 MB`; meanwhile, the old `Contents/Frameworks/Electron Framework.framework/Versions/A/Electron Framework`, sized at `181.5 MB`, has been removed.

This means the underlying packaging model of the desktop app has clearly changed in this release. Users may not directly see a “framework name” change, but the underlying implementation of startup, windows, web rendering, crash handling, and system integration may all be affected by it.

Key evidence:

```diff
/NSPrincipalClass: 'AtomApplication' → 'BrowserCrApplication'
```

Info.plist also adds:

```text
/ChromiumBaseVersion = '148.0.7778.179'
/ChromiumBaseBundleVersion = '7778.179'
/CrProductDirName = 'com.openai.codex'
```

### 2. A new app shell and host connectivity channels begin to surface [Confirmed]

In `package.json`, the `owl` script was changed to run `build:dev-vite` first, and a new `build:dev-vite` script was added; the preload script adds channels related to `get-uses-owl-app-shell` and `connect-app-host`.

This points to the desktop app introducing or switching to a new app-shell path. At the user-visible layer, future communication between the window shell, WebView, and host process may become more centralized, and may also support more complex embedded app experiences.

Key evidence:

```diff

```


### 3. The browser plugin runtime path shifts from bundled app-server-runtime to external injection [Confirmed]

The Chrome plugin installation script no longer always constructs `codex`, `node`, and `node_repl` from the plugin directory’s `app-server-runtime`; instead, it allows runtime paths to be passed in through `appServerRuntimePaths`.

This indicates that the binding between the browser automation plugin and the main app runtime has become more flexible. The developer-visible implication is that browser host configuration no longer depends only on a fixed directory inside the plugin package, making it easier to reuse runtimes provided by the main app in the future.

Key evidence:

```diff
- let o=a(t,"app-server-runtime");return $({codexCliPath:a(o,"codex"), ... nodePath:a(o,"node"),nodeReplPath:a(o,"node_repl"), ...})
+ return $({browserClientPath:v(t,"scripts","browser-client.mjs"),...e})
```

And at the call site:

```diff
- appServerHostConfig:C(t,{channel:i.channel})
+ appServerHostConfig:S(e,{channel:s.channel,...t.appServerRuntimePaths})
```

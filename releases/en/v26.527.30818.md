# Codex Desktop 26.527.30818 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

The core of this batch is not a single feature, but a coordinated migration across the desktop shell, browser automation, plugin runtime, and workspace handoff layers.

One-liner: Codex Desktop 26.527 moves both platforms toward the Owl/Chromium shell and richer browser/runtime orchestration, while Windows gains a full bundled Computer Use plugin.

## About This Report

This report uses only the machine-extracted facts in the input packages. The macOS comparison range is 26.519.81530 build 3178 → 26.527.30818 build 3370; the Windows comparison range is 26.519.11010.0 → 26.527.3378.0. The two platforms have different version numbers and package layouts, so every conclusion is tied to its own platform facts.

Evidence is split into two levels. [Confirmed] means the evidence comes from plist, manifest, package.json, plugin inventory, skill docs, service entrypoints, dependency lists, readable strings, or concrete config diffs. [Signal] means it comes only from frontend module names, size movement, CSS/state class names, or compressed bundle direction.

Platform labels are [macOS+Windows], [macOS only], and [Windows only]. Same-name frontend module changes are treated as UI, state, or interaction signals, not as source-level facts or proof of user-visible rollout.

This batch does not provide Windows AppxManifest itemized changes or content security policy changes, so this report does not infer either category.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Desktop shell moves toward Owl/Chromium

On both platforms, `package.json` raises the version from `26.519.81530` to `26.527.30818`, raises the build from `3178` to `3370`, adds `build:dev-vite`, and changes the `owl` script so it runs the dev Vite build before `owl-shell.mjs`. `preload.js` adds `get-uses-owl-app-shell`, `usesOwlAppShell()`, and a `connect-app-host` message channel.

The macOS evidence is stronger: `Info.plist` changes `NSPrincipalClass` from `AtomApplication` to `BrowserCrApplication`, adds `ChromiumBaseVersion = 148.0.7778.179`, `ChromiumBaseBundleVersion = 7778.179`, and many Chromium type declarations. The file tree also removes `Electron Framework.framework` and adds `Codex Framework.framework/Versions/148.0.7778.179`. Windows also gains many Chromium-related files such as `chrome.dll`, `chrome_pwa_launcher.exe`, `elevation_service.exe`, and `notification_helper.exe`, while `Codex.exe` shrinks from 226.6 MB to 4.6 MB, indicating a major packaging reshuffle.

### [Confirmed] [Windows only] Full bundled Computer Use plugin added

The Windows bundled plugin list changes from `browser`, `chrome`, `latex` to `browser`, `chrome`, `computer-use`, `latex`. The new `computer-use` plugin has 187 files, totals 2.9 MB, carries version `26.527.30818`, has display name `Computer Use`, and short description “Control Windows apps from Codex”.

This is not just package-size movement. The manifest includes category `Productivity`, keywords `computer-use`, `desktop-control`, `windows`, `automation`, `node-repl`, `accessibility`, and default prompts about playing a playlist, building and running an existing Visual Studio project, and opening Notepad to clean up bullets. The skill doc says it automates Windows apps via SendInput and UI Automation, and captures windows via Windows.Graphics.Capture. The largest files include `node_modules/@oai/sky/bin/windows/codex-computer-use.exe` at 1.7 MB and `assets/app-icon.png` at 0.9 MB.

The macOS input sample has no corresponding bundled-plugin inventory addition, so the Windows Computer Use plugin surface cannot be projected onto macOS. Both platforms add main-bundle strings such as `Computer Use helper already has an active request` and `Computer Use native pipe client disconnected`, but those only prove related error paths exist in the package, not that macOS bundles the same plugin.

### [Confirmed] [macOS+Windows] Browser Use routing, sidebar webview, and Chrome native host pipelines expand

Both platforms add many Browser-related strings to the main/frontend bundles, including `Bound Browser Use session route to exact owner`, `Captured Browser Use session route`, `Registered Browser sidebar thread`, `Requested Browser sidebar webview attachment`, `Runtime attached Browser sidebar webview`, `No in-app Browser host is available for this browser-use session`, `Opening routed Browser page`, and `Synced exact Browser owner`. This shows observable package paths around Browser Use session routing, owner binding, sidebar webview attachment, and recovery.

Chrome native host handling also changes clearly. On macOS, `installManifest.mjs` stops constructing `codex/node/node_repl` paths from `app-server-runtime` and instead accepts `appServerRuntimePaths`. Both platforms add strings such as `Chrome native host resource sync requires resourcesPath`, `Invalid Chrome native host install request`, and `Failed to stop runtime marketplace Chrome native host`. On macOS, `browser` and `chrome` plugin versions both move from `26.519.81530` to `26.527.30818`; on Windows, the old large `chrome/app-server-runtime` files are removed.

### [Confirmed] [macOS+Windows] RPC, MessagePort, and streaming stub infrastructure enters the main package

Both platforms add many RPC and stream-related error strings: `An RPC stub cannot be used as a constructor.`, `Cannot serialize RPC stubs without an RPC session.`, `Cannot create pipes without an RPC session.`, `ReadableStream RPC stub was disposed without being consumed`, `Response body must be of type ReadableStream.`, `Peer closed MessagePort connection.`, and `Serialization exceeded maximum allowed depth.` `package.json` also adds the `capnweb` dependency.

This confirms infrastructure-level changes, but does not reconstruct full source semantics. Together with `preload.js` adding `connect-app-host` and the main package adding Browser sidebar and app host strings, it confirms that cross-context messaging, RPC stubs, and streaming pipelines are now packaged.

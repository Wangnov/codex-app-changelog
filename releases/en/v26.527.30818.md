The main theme of this batch is a coordinated reshaping of the desktop shell, browser automation host, and app-layer module structure, with different landing points on macOS and Windows.

## âś¨ Highlights

### Desktop shell naming shifts from Electron toward Chromium

[Confirmed] [macOS+Windows] Both platforms show a major runtime layout reshuffle. On macOS, many Chromium framework files were added under `Contents/Frameworks/Codex Framework.framework/Versions/148.0.7778.179/`, while many corresponding files under `Electron Framework.framework/Versions/A/` were removed. On Windows, new files include `chrome.dll`, `VkLayer_khronos_validation.dll`, `chrome_pwa_launcher.exe`, and `chrome_proxy.exe`, while `Codex.exe` shrank from `226.6 MB` to `4.6 MB`.

[Confirmed] [macOS only] macOS `Info.plist` adds `ChromiumBaseVersion = 148.0.7778.179`, `ChromiumBaseBundleVersion = 7778.179`, and `SCMRevision = 4687a88676f95bfbce9ec62314791faae363fea6-refs/branch-heads/7778@{#3197}`. `NSPrincipalClass` also changes from `AtomApplication` to `BrowserCrApplication`. This gives stronger Chromium version evidence than the Windows fact package provides.

### Browser automation host no longer pins app-server-runtime inside the plugin directory

[Confirmed] [macOS+Windows] Both platforms removed large `app-server-runtime` files under the Chrome plugin directory: macOS removed `plugins/chrome/app-server-runtime/codex`, `node`, and `node_repl`; Windows removed `resources/plugins/openai-bundled/plugins/chrome/app-server-runtime/codex`, `node`, and `node_repl`. At the same time, macOS root resources such as `Contents/Resources/codex`, `Contents/Resources/node`, and `Contents/Resources/node_repl` still change, and Windows root resources such as `resources/codex.exe` and `resources/codex` also change. This indicates the runtime was moved closer to the app resource root.

[Confirmed] [macOS only] The macOS `installManifest.mjs` diff confirms this path decoupling. The old logic derived `codex`, `node`, and `node_repl` from `app-server-runtime`; the new logic accepts `appServerRuntimePaths` and only composes `scripts/browser-client.mjs` inside the install config. This proves that, at least in the macOS package, Chrome Native Messaging host installation now supports injected runtime paths.

### App layer gets smaller while internal modules are heavily reorganized

[Confirmed] [macOS+Windows] `app.asar` gets smaller on both platforms. macOS `Contents/Resources/app.asar` drops from `159.0 MB` to `140.3 MB`, while Windows `resources/app.asar` drops from `159.0 MB` to `144.1 MB`. Internal file counts increase sharply on both sides: macOS `1577 â†’ 2774`, Windows `1531 â†’ 2722`.

[Signal] [macOS only] The macOS fact package lists frontend module names after removing Vite hash noise. The visible directions cluster around appgen, automation, Mermaid diagrams, workspace, MCP capability views, pull request status, browser-sidebar, and annotation areas. These are module-name signals only and do not prove that the corresponding capabilities are user-facing.

### Plugin versions move forward, with the LaTeX plugin advancing separately

[Confirmed] [macOS only] macOS plugin config shows the `chrome` and `browser` plugin versions moving from `26.519.81530` to `26.527.30818`, while the `latex` plugin moves from `0.2.0` to `0.2.2`. The Windows fact package does not include the corresponding plugin config diff, so the same plugin metadata change cannot be confirmed there.

[Confirmed] [macOS+Windows] Both platforms removed the skill document files at paths corresponding to `plugins/chrome/skills/chrome/SKILL.md` and `plugins/browser/skills/browser/SKILL.md`. Windows also removed `resources/plugins/openai-bundled/plugins/browser/skills/browser/agents/openai.yaml`.

### macOS adds authentication, file type, and scripting metadata

[Confirmed] [macOS only] `Info.plist` adds three capability flags under `ASWebAuthenticationSessionWebBrowserSupportCapabilities`: `CallbackURLMatchingIsSupported`, `EphemeralBrowserSessionIsSupported`, and `IsSupported`. This points to a change in macOS system-level web authentication and callback declarations.

[Confirmed] [macOS only] `Info.plist` adds multiple `UTImportedTypeDeclarations` and `UTExportedTypeDeclarations`, covering `crx`, `crwebloc`, GIF, HTML, XHTML, JavaScript, JPEG, MHTML, Ogg, PNG, SVG, text, WebM, WebP, PDF, AVIF, and more. It also adds platform metadata such as `NSAppleScriptEnabled = True`, `OSAScriptingDefinition = scripting.sdef`, and `NSBonjourServices[0] = _googlecast._tcp`. The Windows fact package does not include comparable manifest detail.

### Owl development/shell path is more visible in macOS readable config

[Confirmed] [macOS only] In `package.json`, the `owl` script changes from directly running `owl-shell.mjs run` to first running `build:dev-vite`, and `build:dev-vite = node ./scripts/build-dev-vite.mjs` is added. `preload.js` also adds `get-uses-owl-app-shell` and exposes `usesOwlAppShell()`. The fact package only proves these development/shell paths exist; it does not prove the default user interface switched.

### Windows package layout moves closer to a full Chromium distribution

[Confirmed] [Windows only] The Windows file tree expands from `831 â†’ 8648`, with `7827` added, `10` removed, and `95` content changes. Newly added files include `chrome.dll` at `306.2 MB`, DevTools-related files under `gen/third_party/devtools-frontend/...`, `elevated_tracing_service.exe`, `elevation_service.exe`, and `notification_helper.exe`. On macOS, the same broad direction appears as a renamed and versioned Framework directory; on Windows, it appears as many Chromium files entering the package directly.

[Confirmed] [Windows only] Windows `resources/native/windows-updater.node` grows from `0.2 MB` to `1.1 MB`, `resources/codex-windows-sandbox-setup.exe` grows from `6.1 MB` to `8.4 MB`, and several `node-pty` native components grow significantly. The macOS fact package has no corresponding Windows-native updater or sandbox installer evidence.

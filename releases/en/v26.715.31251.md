# Codex Desktop 26.715.31251 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

This batch touches the updater, plugin and skill management, usage limits, and pull-request filtering on both platforms; Windows also expands browser and site settings while several plugin-detail and panel capabilities move in opposite directions.

Windows expands browser and site controls, while shared updater, plugin-management, usage-limit, and pull-request changes land amid substantial cross-platform UI divergence.

## About This Report

This report compares two sets of official distribution artifacts:

- macOS arm64: `26.715.21425` (build 5488) → `26.715.31251` (build 5538).
- Windows x64 MSIX: `26.715.2305.0` → `26.715.3651.0`.

The platform versions are not aligned, so every conclusion is based on its respective fact package. Internal plugin versions change from `26.715.21425` to `26.715.31251` on both platforms and must not be confused with the outer Windows MSIX version.

Evidence levels:

- [Confirmed]: Derived from manifests, Info.plist, configuration files, dependency statistics, file-tree statistics, or newly added bundle strings. A string proves that wording, an error path, or an identifier entered the package; it does not prove universal user availability.
- [Signal]: Derived from changed same-name frontend modules, strings and style classes inside those modules, size differences, or frontend chunk composition. These indicate UI, state, or interaction direction but cannot reconstruct source semantics.

Platform markers are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`.

Neither top-level bundle tree adds or removes files, while app.asar contains extensive Vite hash renaming. That hash noise is excluded here. The fact packages do not provide complete bundled-plugin inventories, CSP differences, or Windows AppxManifest differences, so no additional conclusion is drawn about those categories.

## ✨ Highlights

### Windows Store update error paths enter both packages

[Confirmed] [macOS+Windows] Both new packages add the following strings to main-process or frontend bundles:

- `Downloaded MSIX has package identity `
- `Windows Store update operation failed`

This confirms that MSIX identity handling and a Windows Store update failure path are present in the shared packaged code. Their presence in the macOS package only proves that shared code was bundled; it does not prove that macOS executes this flow.

[Signal] [Windows only] `resources/native/windows-updater.node` grows from 1.2 MB to 1.8 MB, a 0.56 MB increase and the largest individual Windows component-size change in this batch. Size alone does not establish the implementation change.

### Browser and Chrome plugin versions align

[Confirmed] [macOS+Windows] The bundled `chrome` and `browser` plugin manifests both move from version `26.715.21425` to `26.715.31251`.

The capability descriptions visible in both manifests remain aligned:

- `chrome` covers automation that depends on existing Chrome state, including tabs, authenticated sessions, cookies, extensions, and ChatGPT Chrome Extension setup. Its description prioritizes purpose-built connectors, APIs, or CLIs.
- `browser` covers in-app browser operations such as opening, inspecting, navigating, testing, clicking, typing, and screenshotting local targets. Its aliases include `@browser`, `@browser-use`, and `browser-use`, with explicit coverage for localhost, loopback addresses, `file://`, and the current in-app browser tab.

[Confirmed] [Windows only] The `computer-use` plugin manifest also advances from `26.715.21425` to `26.715.31251`. Its description continues to specify controlling Windows desktop apps from ChatGPT through Computer Use.

No added bundled-plugin detail was provided, so these version updates cannot be interpreted as newly added plugins.

### Windows substantially expands browser settings

[Signal] [Windows only] `browser-use-settings` grows by about 95,488 bytes and adds a closely related set of settings strings:

- `Access history without asking`
- Browser download and history entry points
- Contact information, extension, password-manager, and site settings
- Expansion and collapse controls for individual browser-data clearing options
- `Add site permission`
- A disabled toggle for unavailable plugins

Together, these point to a more complete embedded-browser data, permission, and plugin settings surface, but do not prove that every option is enabled for every user.

[Signal] [Windows only] `header` adds `All chats`, `Cloud chats`, `Chat history`, `All environments filter option`, `App settings`, `ChatGPT for Chrome`, and `Chrome computer use settings`, indicating that chat scope, environment filters, and Chrome entry points are being integrated into header or navigation UI.

### Plugin, skill, and MCP management moves toward a unified surface

[Signal] [macOS+Windows] `plugins-settings` adds `Manage plugins, skills, and MCPs` and a corresponding combined-section subtitle on both platforms, indicating that the three extension categories are being organized within one settings area.

[Signal] [macOS only] `skills-page` adds a relatively complete set of skill-page strings:

- Page heading, search input, and loading state
- Installed and recommended skill sections
- A `New skill` button
- Skill-list reload
- A prompt to refresh newly installed skills before composer use
- Empty states for no skills and filtered results

This is a coherent signal for skill discovery, creation, and refresh UI, not merely a chunk-size change.

### Plugin details move in opposite directions

[Signal] [macOS only] `plugin-detail-page` grows by about 158,456 bytes and adds:

- An MCP server enablement toggle
- Plugin-detail section navigation and a back button
- Workspace user and group search
- Removal of selected sharing targets
- `Add people or groups`
- `Anyone at workspace with the link`
- `Anyone in this workspace with the link`
- `App unavailable`
- More-actions menus for app tools, skill previews, and skill cards

The related strings and state classes point to an expanded plugin configuration, sharing, tool, and skill-management surface.

[Signal] [Windows only] The same Windows module shrinks by about 156,015 bytes and removes the MCP toggle, section navigation, sharing-target, and `App unavailable` strings. This only shows that the chunk contracted or was repartitioned in this Windows build; it does not independently prove product-level removal.

### Workspace monthly usage limits gain a request-increase action

[Signal] [macOS+Windows] Both platforms add:

- `Button to request a workspace monthly usage limit increase`
- `Request limit increase`
- `settings.usage.limits.requestIncrease`

On macOS, the signal appears in a shared application/settings chunk; on Windows, it appears in `rate-limit-summary`. Both point to an action for requesting a higher workspace monthly usage limit.

### Pull requests gain a heading for explicit relationship filters

[Signal] [macOS+Windows] `pull-request-route` adds the following on both platforms:

- `Heading for pull requests matching an explicit relationship filter`
- `codex.pullRequests.inbox.section.results`

This points to a distinct heading or result section for explicitly filtered relationship results in the pull-request inbox.

### Site settings concentrate on Windows while sharing UI contracts on both platforms

[Signal] [Windows only] `appgen-settings-page` grows by about 45,588 bytes and adds strings for environment variables, secrets, site titles, deletion-confirmation slugs, custom domains, and DNS setup, including `Add custom domain`, `Add domain`, `Add secret`, and `Add a domain to show DNS setup records`.

[Signal] [macOS only] The same macOS module shrinks by about 46,791 bytes and removes the corresponding labels for environment variables, secrets, custom domains, site titles, and deletion confirmation. The platforms move in clearly opposite directions, although code splitting or migration may contribute.

[Signal] [macOS+Windows] `appgen-share-dialog` removes site-sharing target search, save-state, public-scope, URL-copy, and external-open strings on both platforms, including `Anyone on the Internet`. This indicates contraction or migration of the sharing dialog in both builds.

[Signal] [macOS only] `Sites tools list failed` also enters a shared application chunk, indicating that a site-tools-list failure state is present in the module. Because it comes from a same-name frontend-module diff, it remains a signal.

### Task sidebars and bottom panels expand on Windows and contract on macOS

[Signal] [Windows only] `thread-app-shell-chrome` grows by about 34,866 bytes and adds:

- Actions to open the browser, review tab, terminal, and workspace file browser from the task side panel
- An action to open a side chat from the chat panel
- `Bottom panel`
- A control for closing the thread bottom panel
- Controls for opening another tab from the bottom or side panel
- An empty state when the current chat has no side-panel actions

[Signal] [macOS only] The same macOS module shrinks by about 37,467 bytes and removes `Bottom panel` plus browser, review, terminal, file-browser, and side-chat action labels. This is another opposite-direction chunk change.

### Windows expands Markdown, image, and file-citation UI

[Signal] [Windows only] `markdown` grows by about 354,706 bytes and adds:

- An accessible label for file-citation chips with location information
- Button, loading-placeholder, and failure-fallback labels for Markdown images without alt text
- `A follow-up message is already awaiting confirmation`
- Coarse-pointer, hover, and dark-mode media queries
- A KaTeX display-related string

Several Markdown/TextMate syntax-scope names are removed at the same time. Overall, this suggests reorganization of Markdown rendering, image loading, file citations, and interaction states; the chunk diff alone cannot establish that the underlying parser was replaced.

### Event-bus connection and queue states enter a macOS frontend module

[Signal] [macOS only] `esm` grows by about 23,833 bytes and adds:

- `Connected to event bus`
- `Initializing event subscription for plugin`
- Emission to the client bus and internal event target
- Queuing while disconnected and emission after reconnection
- Skipping registration or emission when the client is disabled
- Retry exhaustion and shim-event dispatch failure

Together, these form a coherent plugin event-subscription, queueing, and failure-state surface. Older strings such as `Callback Error`, `Context Cancel`, `Failed to deliver`, `Not ready`, and timeout messages are removed.

### Input-validation error styling changes on both platforms

[Signal] [macOS+Windows] Composer or settings-related chunks on both platforms add `backdrop-blur-sm` and change the input-validation error background from `/20` to `/70`. This signals a change in the visual intensity or layering of validation errors.

### Runtime and Computer Use dependencies remain stable

[Confirmed] [macOS+Windows] Neither platform adds a `cua_node` dependency package.

[Confirmed] [macOS only] The Computer Use runtime remains at 365.7 MB, a 0 MB and 0% change.

[Confirmed] [Windows only] The Computer Use runtime remains at 287.5 MB, a 0 MB and 0% change.

[Confirmed] [macOS+Windows] `runtimeArchiveSha` changes in `owl-electron-app.json` on both platforms:

- macOS: `2d21…dae3` → `b6cf…d690`
- Windows: `9d26…4cbe` → `e0e1…fd85`

This proves that the packaged runtime archive content changed, but the fact packages do not expose its internal semantic differences.

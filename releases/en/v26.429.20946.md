> Third-party changelog · Unofficial · Based on reverse-engineered diffs of official update artifacts

## ✨ Highlights

### 1. Browser Use Adds DOM-Level Browser Operation APIs [Confirmed]

The Browser Use skill documentation adds a `dom_cua` interface, including capabilities to click, double-click, scroll, type, send key presses, and read visible DOM snapshots by visible DOM nodes.

This means that when Codex tests pages inside the built-in browser, it no longer relies only on coordinates or Playwright snapshots; it also exposes a control surface closer to “operate on page elements.” The key evidence is that the `Tab` interface adds `dom_cua: DomCUAAPI`, with methods such as `get_visible_dom()`, `click({ node_id })`, `type({ text })`, and `scroll({ node_id?, x, y })`.

### 2. Browser Use Positioning Expands from “Local Targets” to a More Complete Built-In Browser Testing Tool [Confirmed]

`browser/SKILL.md`, `agents/openai.yaml`, and `plugin.json` all revise their descriptions. The new version explicitly says it can be used to open and control the Codex built-in browser for testing tasks such as navigation, inspection, clicking, typing, and screenshots, mainly targeting local development pages and files.

From a user-visible perspective, the plugin description now reads more like a complete browser testing capability rather than a narrow entry point for `localhost`, `127.0.0.1`, `::1`, and `file://`. The default plugin prompt has also been narrowed from two examples to `"Test my checkout flow on localhost"`, and the brand color changes from `#0F766E` to `#013B7B`.

### 3. Screenshot Return Rules Are Clearer: Screenshots Users Need to See Should Be Embedded in Replies [Confirmed]

The Browser Use skill documentation adds an operational guideline: if a screenshot is intended for the user to see, it should be embedded inline in the Markdown response.

This affects how Codex presents results when using the built-in browser for visual verification: instead of merely saying “I took a screenshot,” it is more likely to include the screenshot as part of the result. The evidence comes from a new line in `browser/SKILL.md`: `If you take a screenshot that the user should see, include the image inline in your Markdown response.`

### 4. Browser Use Marketplace Category Changes from Engineering to Research [Confirmed]

In the bundled plugin marketplace manifest, Browser Use changes its `category` from `Engineering` to `Research`.

This does not necessarily mean the underlying capability changed, but it affects how the plugin is presented in the marketplace or category entry points. The evidence comes from `Contents/Resources/plugins/openai-bundled/.agents/plugins/marketplace.json`: `"category": "Engineering"` → `"category": "Research"`.

### 5. Computer Use Plugin Version Bumps and Adds Apple Events Permission Description [Confirmed]

The Computer Use plugin version increases from `1.0.758` to `1.0.770`, while the main app `Info.plist` adds `NSAppleEventsUsageDescription`: `Codex uses Apple Events to control Mac apps on your behalf.`

This indicates that the desktop app control capability now has a clearer permission description, which macOS may display in related authorization flows. At the same time, Computer Use-related binaries also changed size: `SkyComputerUseService` increases by `+0.38 MB`, while `SkyComputerUseClient` decreases by `-0.48 MB`. These size changes are treated only as supporting signals and are not used to infer specific functionality on their own.

### 6. Frontend Modules Point to New UI Directions Such as App Connections, OAuth, Automation, Remote Sessions, and Hotkey Windows [Signal]

After removing hash noise from `webview/assets` and `.vite/build`, the real newly added modules include names such as `app-connect-modal`, `app-connect-oauth`, `automations-page`, `remote-connections-page`, `hotkey-window-*`, `pull-requests-page`, `plugin-detail-page`, `file-preview-page`, `editor-diff-page`, and `scratchpad-page`.

These module names point toward app connections, OAuth, automation pages, remote connections, hotkey windows, Pull Request boards, plugin details, file previews, diff editing, and related surfaces. However, the artifacts do not allow source-level intent to be reconstructed, nor do they prove that these interfaces are available to all users.

### 7. Webview Preload Layer Adds MCP App Sandbox Message Channel [Confirmed]

`.vite/build/preload.js` adds listeners related to `codex_desktop:mcp-app-sandbox-host-message`, and forwards messages into the page context via `window.postMessage`.

This indicates that a new bridge path has been added between the app frontend and some form of MCP app sandbox host messaging. Because the evidence comes from a minified preload script, it only confirms that the channel exists; it does not support further inference about the specific product form.

### 8. Build Scripts Expand from Single SQLite Rebuild to Native Modules Rebuild, with a Windows Computer Use Packaging Script Appearing [Confirmed]

In `package.json`, multiple scripts change from `rebuild:sqlite` to `rebuild:native-modules`, and the corresponding script file name changes from `rebuild-sqlite.mjs` to `rebuild-native-modules.mjs`. A new `computer-use:windows:bundle` script is also added, pointing to `scripts/windows-computer-use-plugin-bundle.ts`.

This shows that the build flow is no longer named only around rebuilding the SQLite native module, but has expanded to a more general native modules rebuild. The new Windows Computer Use packaging script points to cross-platform plugin packaging work, but this is confirmed only at the build configuration level and does not mean macOS users directly received Windows capabilities.

### 9. Minor Revisions to Multilingual Menu Copy [Confirmed]

Several `native-menu-locales` files have copy changes, including `fa.json`, `mr-IN.json`, `ar.json`, `zh-TW.json`, `hr-HR.json`, `so-SO.json`, and `tl.json`.

These changes mainly affect localized wording for browser sidebar context menus, Intel / Apple Silicon launch prompts, and similar UI text. For example, Traditional Chinese `browserSidebar.contextMenu.forward` changes from “前進” to “下一頁”. These changes are visible copy refinements and do not indicate new functionality.

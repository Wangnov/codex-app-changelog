# Codex Desktop `26.429.20946` build 2312

> Third-party changelog · Unofficial · Based on reverse diffs of official update artifacts

At the artifact level, this release is mainly about desktop automation permissions, Browser Use plugin docs and APIs, plugin marketplace management, and several frontend surfaces being wired in.  
One-liner: this build expands automation surfaces and starts wiring more app-level workflows into the desktop shell.

## About This Report

This is not an official changelog. It is a **reverse diff report based on official distribution artifacts**. For this historical backfill, the fact pack states that the official appcast did not provide a delta package for `26.422.71525` → `26.429.20946`, so the comparison was performed by downloading and diffing two official full packages.

Evidence is split into two levels: **[Confirmed]** means direct human-readable evidence from Info.plist, package.json, plugin configuration, skill docs, main-process readable strings, service entries, or similar sources; **[Signal]** means evidence only from frontend module names, same-name frontend module string diffs, or binary size movement. Signals point in a direction, but cannot be reconstructed into source-level intent from the artifact alone.

The comparison covers macOS arm64. The previous version is `26.422.71525` build `2210`, released on `2026-04-29`; the latest version is `26.429.20946` build `2312`, released on `2026-04-30`.

## ✨ Highlights

### 1. Apple Events usage text was added, making Mac app-control permissions more explicit [Confirmed]

The new `Info.plist` adds `NSAppleEventsUsageDescription` with the value: `Codex uses Apple Events to control Mac apps on your behalf.`

This means Codex now declares the system permission purpose for controlling other macOS apps through Apple Events. The user-visible effect may be a clearer permission prompt when related capabilities are first triggered; for developers, this is a permission-boundary change around Computer Use-style desktop automation.

### 2. The Computer Use plugin was upgraded to `1.0.770` [Confirmed]

The bundled `computer-use` plugin’s `.codex-plugin/plugin.json` version changed from `1.0.758` to `1.0.770`.

Related component sizes also moved: `SkyComputerUseService` grew from `14.1 MB` to `14.5 MB`, while `SkyComputerUseClient` shrank from `10.5 MB` to `10.0 MB`. The version bump is direct evidence; the size changes only indicate implementation churn and do not identify exact behavior.

### 3. Browser Use was reframed from local-target browsing into broader in-app browser automation [Confirmed]

The Browser Use plugin description was rewritten. The old text emphasized `localhost`, `127.0.0.1`, `::1`, and `file://`; the new text says it can open and control the browser built into Codex, including navigation, inspection, clicking, typing, and screenshots.

The skill documentation also expands the interface with `dom_cua: DomCUAAPI`, covering `click`, `double_click`, `get_visible_dom`, `keypress`, `scroll`, and `type` operations against visible DOM node IDs. This means Codex’s in-app browser automation surface is no longer described only in coordinate or Playwright terms; it now includes a DOM-node-based interaction path.

Key evidence:
```diff
+  dom_cua: DomCUAAPI; // API for interacting with the tab via the dom based cua api
```

### 4. Browser Use history exposure was narrowed in the documented API [Confirmed]

In the Browser skill docs, `BrowserUser` changed from describing “tabs and history” to only “tabs”, and the `BrowserHistoryOptions` and `BrowserHistoryEntry` types were removed.

This shows that the skill API documentation no longer exposes a history-query shape. Compared with frontend settings strings such as “Ask before accessing history” and “Access history without asking”, the firmer conclusion is that the documented skill API exposure changed; the runtime policy cannot be fully confirmed from the artifact alone.

### 5. The preload layer gained an MCP app sandbox host message channel [Confirmed]

`.vite/build/preload.js` adds a listener for `codex_desktop:mcp-app-sandbox-host-message`, then forwards messages to the current origin via `window.postMessage`.

This is a cross-process communication change pointing to message bridging between an MCP app sandbox and the WebView/renderer side. Because this file is a minified preload artifact, the channel’s existence is confirmed, but the full protocol cannot be reconstructed.

Key evidence:
```diff
+e.ipcRenderer.on(t,(e,t)=>{let n=window.location.origin;n!==`null`&&window.postMessage(t,n,e.ports)})
```

### 6. Plugin marketplace management surfaced in the frontend, with one marketplace category change [Confirmed]

A bundled marketplace entry changed its `category` from `Engineering` to `Research`. The frontend and main-process strings also include marketplace-management copy such as `Manage-mode tab label for plugin marketplaces`, `Remove marketplace`, and `Actualizar todos los marketplaces`.

This indicates the marketplace is no longer only an install surface; management, removal, and upgrade language is now present. The category change is confirmed by configuration; exact UI availability still depends on runtime entry points and experiment gates.

### 7. package.json added browser backend, commands, Markdown AST, and Objective-C bridge dependencies [Confirmed]

The new `package.json` adds `browser-backend-common`, `commands`, `mdast-util-from-markdown`, `mdast-util-to-string`, and `objc-js`.

Together, these dependencies align with the release’s Browser Use, command handling, Markdown processing, and Apple-platform bridge directions. They prove the dependency surface expanded, but do not by themselves prove that the corresponding user-facing features are enabled by default.

### 8. Remote connection settings show richer SSH editing and availability-state language [Signal]

New modules include `remote-connections-page`, `remote-connections-onboarding-signals`, and `settings-host-dropdown`; the changed `remote-connections-settings` module adds strings such as `Add SSH connection`, `Edit SSH connection`, `Device connections`, `offline`, `online`, and `busy`.

These lines point toward a more detailed remote-connection management UI, possibly covering SSH connection editing, display-name validation, refresh, and state display. Because the evidence comes from frontend modules and string diffs, it is only a UI/state signal and cannot be reconstructed into source-level intent.

### 9. Automation, Heartbeat, and cloud/local thread-bridge pages are being laid out [Signal]

New frontend modules include `automation-dialog`, `automations-page`, `heartbeat-automation-eligibility`, `heartbeat-automation-permissions`, `heartbeat-automation-thread-bridge`, `send-to-cloud`, and `remote-conversation-page`.

This module cluster points toward automation, permission checks, thread bridging, and cloud send flows. The fact pack does not include readable configuration or skill docs confirming exact product behavior, so this should be treated only as evidence that related frontend surfaces are appearing, not that the feature is generally available.

### 10. PDF preview points from zoom-only controls toward annotation and presentation mode [Signal]

`pdf-preview-panel` adds strings such as `PDF annotation comment`, `Accessible label for a placed PDF annotation marker`, `Button label that exits PDF presentation mode`, `Next page`, and `Previous page`, while some `Zoom in` and `Zoom out` strings were removed.

This points to PDF artifact preview gaining annotation, page navigation, and presentation-mode interactions. Since the evidence is a same-name frontend module string diff, it confirms UI text and state direction only, not the underlying PDF implementation.

### 11. Personalization wording shifts from custom avatars to custom pets [Signal]

In `appearance-settings`, many strings such as `Custom avatars` and `Loading custom avatars` were replaced by `Custom pets`, `Loading custom pets`, `floating pet overlay`, and `pet picker`; the main-process added strings also include multilingual custom-pet and mascot wording.

This indicates visible personalization language is shifting from avatars toward pets/mascots. The conclusion comes from frontend text and module-change signals, so it describes UI naming and interaction direction, not the asset format or complete feature state.

### 12. The frontend shell was noticeably split into app, settings, thread, plugin, and workspace modules [Signal]

After hash-noise removal, new modules include `app-shell`, `onboarding-shell`, `settings-page`, `first-run`, `welcome-page`, `plugin-detail-page`, `plugins-page-section`, `local-conversation-page`, `remote-conversation-page`, and `worktree-init-v2-page`.

This shows a substantial WebView frontend restructuring across the app shell, login/first-run flow, settings, plugin detail, threads, and worktree initialization areas. Since these are frontend module-name signals, they indicate direction but should not be written as launched functionality.

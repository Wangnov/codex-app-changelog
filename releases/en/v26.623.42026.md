# Codex Desktop 26.623.42026 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

Thesis: this batch is not just an identical rebuild across platforms; package versions, frontend module direction, and plugin manifest changes need to be read per platform.  
One-liner: This batch looks like a shared codebase with platform-specific packaging and UI rollout divergence.

## About This Report

This report only uses machine-extracted facts from reverse-reconstructed official signed distribution packages. The macOS comparison is `26.623.41415 build 4505` to `26.623.42026 build 4514`; the Windows comparison is `26.623.5175.0` to `26.623.5546.0`.

Evidence has two levels. `[Confirmed]` comes from plist, manifest, plugin configuration, dependency lists, service entries, skill documentation, signing information, and readable strings outside same-name frontend-module clue sections. `[Signal]` comes from same-name frontend-module string diffs, style-class diffs, module names, and size changes. Platform ownership is marked as `[macOS+Windows]`, `[macOS only]`, or `[Windows only]`.

The evidence pass covered versions and channels, bundle file trees, bundled plugin manifests, Computer Use runtime totals, `cua_node` dependencies, `app.asar`, real frontend module additions and removals, same-name frontend-module clues, readable strings, Info.plist or AppxManifest, CSP, new components and resources, and key file diffs. This input does not provide CSP changes or a concrete Windows AppxManifest diff.

## ✨ Highlights

### [Confirmed][macOS+Windows] Browser and Chrome bundled plugin manifests moved to 26.623.42026

On both platforms, the `browser` and `chrome` plugin manifests update `version` from `26.623.41415` to `26.623.42026`. This is not a frontend inference; it comes directly from `.codex-plugin/plugin.json` diffs.

The platform difference is the outer package version. macOS itself is `26.623.42026`, build `4505` to `4514`; the Windows MSIX is `26.623.5546.0`, while the bundled plugin manifests still move to the `26.623.42026` app version line.

### [Confirmed][Windows only] The Computer Use plugin manifest was updated on Windows

On Windows, the `computer-use` plugin manifest updates `version` from `26.623.41415` to `26.623.42026`, with the description still pointing to “Control desktop apps on Windows from Codex through Computer Use.” The macOS input does not include a matching `computer-use` manifest diff.

This shows the Windows Computer Use plugin metadata participated in the version bump, but it does not imply runtime growth or new runtime dependencies.

### [Confirmed][macOS+Windows] Computer Use runtime and cua_node dependencies did not grow

The macOS Computer Use runtime total remains `377.4 MB → 377.4 MB`; Windows remains `298.9 MB → 298.9 MB`. Both platforms report `0` newly added `cua_node` packages.

This is an important negative fact: the batch shows plugin manifest and frontend-surface movement, but the provided facts do not support a claim that the Computer Use runtime added dependencies or materially expanded.

### [Signal][macOS+Windows] General settings gained more work-mode, permission, voice, and display surfaces

Both platforms’ `general-settings` module gained readable strings covering default permissions, work mode settings, code base size, hold-to-dictate hotkey capture, hotkey-window hotkey capture, dictation dictionary entries, the Popout Window projectless default toggle, ambient suggestions, color diff markers, and the Chrome theme contrast slider.

Because these come from same-name frontend-module clue sections, they are UI and settings direction signals, not proof that every entry is visible to all users. Platform-wise, this is one of the clearest cross-platform settings changes in the batch.

### [Signal][macOS only] macOS additionally shows Agent environment, default terminal location, default microphone, and detached code review setting clues

The macOS `pets-settings~appearance-settings~general-settings` module adds `Agent environment`, `Auto detect`, `Auto detect language option`, `Bottom panel`, `Default file open destination`, `Default microphone input option label`, `Default permissions`, `Default terminal location`, `Default to projectless chat`, and `Detached code review option label`.

These strings only appear in the macOS fact pack. They point to frontend rearrangement or expansion around local environment, default open destination, voice input, terminal panel placement, and code review mode.

### [Signal][macOS+Windows] The Workbook frontend module grew substantially, with workbook.help, named ranges, and Granola properties

On both platforms, the `workbook` module grows from a near-empty module to about `2.8 MB`: macOS adds `2,764,364 bytes`, and Windows adds `2,761,257 bytes`. New strings include `(unnamed range)`, prompts around `workbook.help(`, and many `data-granola-*` style or structural attributes.

This is a strong frontend capability signal around workbook parsing, rendering, or interaction support entering the package. It is still a same-name frontend-module clue, so it remains a `[Signal]`.

### [Signal][macOS only] Plugin detail moved toward sharing, MCP toggles, skill previews, and app tools menus on macOS

The macOS `plugin-detail-page` module adds strings for the plugin share dialog, searching workspace users and groups, an MCP server enablement toggle, section navigation, skill card actions, app tools modal, skill preview modal, `App unavailable`, and `Anyone in this workspace with the link`.

Windows moves in the opposite direction for the same module: it removes strings for plugin sharing, the MCP server enablement toggle, section navigation, `App unavailable`, and related text. This must not be written as a cross-platform enhancement; it is added in the macOS sample and removed in the Windows sample.

### [Signal][macOS only] AppGen and site sharing show public publish, copy link, and workspace-share clues

The macOS `appgen-share-dialog` adds site-sharing strings for adding people or groups, workspace links, public internet access, copying a live site URL, opening a live site externally, publishing to the public internet, `Can visit`, `Copied to clipboard`, and `Copy link`.

These are high-visibility frontend signals that AppGen site sharing and publishing flow text or interaction entries are present in the macOS package. The Windows fact pack has no matching addition.

### [Signal][macOS only] Projects, sidebar, Cloud tasks, and remote-connection navigation continue to expand

### [Signal][macOS only] Update recovery, download progress, rate-limit menus, and workspace usage entries appear in frontend strings

A separate macOS error-recovery module adds `Checking for updates`, `Downloading update`, `Installing update`, and `appUpdate.recovery.updateCodex`. Windows has no matching additions, so this should not be described as cross-platform update recovery.

### [Signal][Windows only] The Skills page gained create, reload, recommended, search, empty-state, and loading text

The Windows `skills-page` module adds strings for creating a new skill, reloading the skills list, newly installed skills requiring refresh before composer use, filtered empty states, installed skills, recommended skills, search input, and `Loading skills...`.

The macOS input does not include a same-name `skills-page` change entry. On Windows, this looks like a signal for fuller information structure or rearrangement in skill management.

Windows `profile-queries` adds profile streak and duration formatting strings, including day streak, hours, hours and minutes, minutes, minutes and seconds, and seconds. The real frontend add/remove section also removes the `profile-model` module on Windows.

Together, these facts suggest a reorganization in Windows profile stats query or display layers. macOS does not show the same addition; macOS instead has an `onboarding-page~profile` module shrinking sharply and adding `Profile photo upload failed`, which is a separate macOS-only signal.

### [Signal][macOS+Windows] Worktree setup failure continue, repair, and edit-environment copy was removed

### [Signal][macOS only] Electron worker bridge, process/spawn, cwd, git metadata, and permission error paths entered the macOS package

A large macOS same-name frontend module adds `Electron worker bridge is unavailable`, `Mismatched worker response method`, `Missing cwd`, `Missing git metadata`, `Unknown method: process/spawn`, `operation not permitted`, and `permission denied`.

These are frontend signals for runtime bridge, process spawning, working directory, git metadata, and permission error paths. The Windows fact pack has no matching addition.

### [Signal][macOS only] Syntax and document-rendering modules show broad grammar-token updates

On macOS, same-name frontend modules including `cpp`, `scss`, `yaml`, `postcss`, `tex`, `angular-html`, `json`, `wasm`, `lua`, `glsl`, `csv`, and `html-derivative` add many syntax token strings or style classes. `tsx` moves the other way, removing a set of TSX and regexp tokens.

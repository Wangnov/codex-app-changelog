# Codex Desktop 26.623.42026 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

This batch looks like the same code line landing at different packaging points: the macOS build carries a wider set of frontend route and UI composition changes, while the Windows package has fewer visible changes but shows skills-page and profile-stat signals.  
One-liner: this build advances shared settings and preview surfaces, while macOS carries broader AppGen/project/plugin UI movement and Windows shows narrower skills/profile signals.

## About This Report

This report uses only the machine-extracted facts in the input packages. It compares macOS arm64 `26.623.41415` build `4505` → `26.623.42026` build `4514`, and Windows x64 MSIX `26.623.5175.0` → `26.623.5546.0`.

Evidence is split into two levels: `[Confirmed]` covers plist, manifest, plugin lists, type/dependency lists, service entries, skill docs, configuration, and readable strings from the fact package outside the “same-name frontend module change clues” section; `[Signal]` covers frontend module names, size movement, and readable string/style-class diffs from compressed same-name bundles. Platform markers are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`.

This batch does not include a Content Security Policy change block or a new bundled plugin detail block. Plugin-layer visible changes are mainly version bumps in existing bundled plugin manifests.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Bundled plugin manifests follow the version line

Both platforms confirm that the bundled `chrome` and `browser` `.codex-plugin/plugin.json` manifests moved from `26.623.41415` to `26.623.42026`.

The `browser` plugin description still points to the Codex in-app browser for opening, inspecting, navigating, testing, clicking, typing, or screenshotting local web targets. The `chrome` plugin description still positions it as automation that depends on the user’s existing Chrome state. The confirmed fact here is the manifest version change, not newly added capability text in this batch.

Windows additionally confirms that the `computer-use` plugin manifest moved from `26.623.41415` to `26.623.42026`, with a description for controlling Windows desktop apps from Codex through Computer Use.

### [Confirmed] [macOS only] macOS app version, build, and asar integrity changed

The macOS `Info.plist` shows `CFBundleShortVersionString` changing from `26.623.41415` to `26.623.42026`, `CFBundleVersion` changing from `4505` to `4514`, and the `ElectronAsarIntegrity/Resources/app.asar/hash` changing as well.

In this input sample, the macOS bundle file count stayed `8875 → 8875`, with `0` added, `0` removed, and `67` content changes. The largest non-hash-noise size change is `Contents/Resources/app.asar`: `164.7 MB → 164.7 MB`, up `+0.04 MB`.

### [Confirmed] [Windows only] Windows bundled Computer Use plugin version changed, but runtime size did not

The Windows package confirms `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` moving from `26.623.41415` to `26.623.42026`, while its description remains about controlling Windows desktop apps from Codex.

At the same time, Windows `cua_node` total size stayed `298.9 MB → 298.9 MB`, with `0` new dependency packages. So the manifest version bump is confirmed, but the facts do not support a runtime dependency expansion.

At the runtime package and dependency level, this batch does not appear to be about expanding Computer Use dependencies.

Evidence levels are split into two tiers: **[Confirmed]** means directly human-readable evidence exists, such as Info.plist values, HTML, readable strings, dependency data, or signing verification output. **[Signal]** means the evidence comes only from frontend module names, changed strings in same-named frontend modules, or binary size movement; it points in a direction but cannot reconstruct source-level intent.

## ✨ Highlights

### 1. Plugin and skill pages move toward unified browsing and creation flows [Signal]

`plugins-page` gained a cluster of strings related to unified browsing, category selection, creation entry points, and detail-page navigation. This points to plugin and skill management becoming more unified, but the artifact cannot reconstruct source-level intent.

The likely user-visible surface includes a more unified way to browse plugins and skills, plus richer install modal, remove confirmation, detail-page back navigation, more-actions menu, and creation flows. Key signals include `Choose whether to browse plugins or skills`, `Choose a category`, `Create plugin`, `Create skill`, `Adding to Codex`, plus state classes `has-opened-plugin-creator-prefill-v1` and `has-opened-skill-creator-prefill-v1`.

### 2. The app entry preload switches from `logger` to `message-bus` [Confirmed]

`webview/index.html` clearly changes the module preload list: `logger-DGKjeFMx.js` is replaced by `message-bus-DcBNa_PD.js`. The de-noised module diff also lists `message-bus` as a real added module and `logger` as a removed module.

This suggests an adjustment in frontend cross-module communication or event-dispatch infrastructure. What is confirmed is that the entry HTML now preloads `message-bus`; the exact communication semantics cannot be recovered from the minified artifact.

```diff
- <link rel="modulepreload" crossorigin href="./assets/logger-DGKjeFMx.js">
+ <link rel="modulepreload" crossorigin href="./assets/message-bus-DcBNa_PD.js">
```

### 3. Automation thread and remote-connection surfaces appear in user-facing strings [Confirmed]

The new main-process / frontend bundle contains the readable string `Failed to set automation thread title`. The frontend `index` module also adds `Automation thread`, `Accessible label for the automation thread icon`, and strings related to a remote-connections banner.

This confirms that user-visible copy for automation threads and remote-connection prompts is present in the new artifact. Possible visible surfaces include a thread icon, a title-setting failure path, a remote-connections banner above the home composer, and an `Add Connections` entry point.

### 4. New window zoom menu strings [Confirmed]

The new bundle adds three readable strings: `Zoom In`, `Zoom Out`, and `Actual Size`.

These usually correspond to desktop app menu or view zoom controls. The fact package confirms that the strings exist in the new artifact, but not their exact menu placement or shortcuts.

### 5. New Raycast theme variants [Signal]

The de-noised frontend module additions include `raycast-dark` and `raycast-light`. The same-named module `use-resolved-theme-variant` also adds the strings `Raycast Dark` and `Raycast Light`.

This points to new or in-progress Raycast-style dark and light theme variants, but the artifact cannot reconstruct source-level intent or confirm whether the variants are exposed to all users.

### 6. macOS Bluetooth permission metadata moves to the newer key [Confirmed]

In `Info.plist`, the Bluetooth usage description moves from the removed `/NSBluetoothPeripheralUsageDescription` key to the newly added `/NSBluetoothAlwaysUsageDescription` key. The description text remains `This app needs access to Bluetooth`.

This is a macOS permission metadata change. The user-visible effect may be that the system permission prompt is backed by the newer plist key; the fact package does not show a Bluetooth feature change.

### 7. App Connect copy shifts toward a personalization toggle [Signal]

`control-group` adds `Accessible label for the personalization toggle in the app connect modal` and `settings.mcp.appConnectModal.personalizationToggle`, while removing `Continuous sync` and related sync title/description strings.

This points to the App Connect modal moving away from continuous-sync-oriented copy and toward a personalization toggle, but it remains a frontend-copy signal rather than a source-level fact.

### 8. Usage and billing prompt copy is replaced [Signal]

The `index` module adds `Banner message for non-owners on usage-based seat plans when spend cap blocks usage`, while removing several credits upsell banner strings for Plus, CBP, and exhausted-credit scenarios.

This points to a change in how usage limits are presented, especially for usage-based seat plans and spend-cap cases. These are frontend-copy signals only and do not establish a change to billing rules.

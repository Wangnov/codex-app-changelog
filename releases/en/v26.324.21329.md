## About This Report

This is not an official changelog. It is a **reverse-engineered diff report based on official distribution artifacts**. This release is a historical backfill: both versions are already outside the Sparkle delta-update window, so this comparison did not reconstruct from delta packages. Instead, both official full packages were downloaded and compared directly.

The comparison process was: download the previous official full package and the latest official full package → verify SHA-256 → extract `Codex.app` → run a structured diff. The fact package also includes notarization verification results from `codesign` / `spctl`: both versions show `CSSMERR_TP_CERT_REVOKED`, meaning that certificates for some early versions were later revoked. This affects the system’s security assessment for installation, but does not affect diff analysis of bundle contents.

This report uses two evidence levels:

- **[Confirmed]**: Direct human-readable evidence, such as `Info.plist`, HTML diff, and signing / notarization verification results.
- **[Signal]**: Indirect evidence, such as unobfuscated frontend module names or binary size changes. These point in a direction, but cannot recover source-level intent from the artifact.

Comparison scope: `26.323.20928` build `1173` (2026-03-24) → `26.324.21329` build `1214` (2026-03-25), on macOS arm64.

### 1. Frontend communication layer introduces `message-bus`, replacing `logger` in entry preload [Signal]

In the latest `webview/index.html`, the entry preload module changes from `logger-DGKjeFMx.js` to `message-bus-DcBNa_PD.js`; after removing build-hash noise, the real frontend module additions/removals also show `message-bus` added and `logger` removed.

This points to an adjustment in the frontend’s internal communication or event dispatch mechanism, but the artifact cannot recover source-level intent, nor can it prove that any corresponding feature has been exposed to users.

### 2. Bluetooth permission declaration migrates from the old key to the new key [Confirmed]

`Info.plist` adds `NSBluetoothAlwaysUsageDescription`, while the old `NSBluetoothPeripheralUsageDescription` is removed. The copy remains `This app needs access to Bluetooth`.

This means the app’s Bluetooth permission declaration has been updated at the metadata level. The user-visible implication is that if the system triggers a Bluetooth permission prompt, the prompt will be based on the new permission key. However, the fact package does not indicate that any new Bluetooth-related feature was added.

Key evidence:

```text
Added /NSBluetoothAlwaysUsageDescription = 'This app needs access to Bluetooth'
Removed /NSBluetoothPeripheralUsageDescription, old value was 'This app needs access to Bluetooth'
```

### 3. Frontend theme assets include `raycast-dark` / `raycast-light` [Signal]

After removing Vite hash noise, the real newly added frontend modules include `raycast-dark` and `raycast-light`.

This points toward changes in UI theme or color assets, but the artifact cannot recover source-level intent, nor can it prove that visible theme options have already been added to the user interface.

Key evidence:

```text

Removed modules: logger
```

### 4. The startup page no longer explicitly declares `color-scheme: light dark` [Confirmed]

The inline styles in `webview/index.html` remove `color-scheme: light dark;`.

This may affect the default behavior of browser-native controls or system theme adaptation. Based on the current evidence, we can only confirm that this style line was removed, and cannot infer the specific visual impact.

Key evidence:

```diff

```

### 5. The main app changes only slightly in size [Signal]

The top-level bundle file count remains `415 → 415`, with no files added or removed and `40` files modified. The largest size change is `Contents/Resources/codex`, from `146.5 MB` to `146.9 MB`, an increase of `0.37 MB`; `app.asar` goes from `42.2 MB` to `42.2 MB`, an increase of `0.07 MB`.

This points to a small maintenance update rather than a broad component replacement. However, binary size changes are only an indirect signal, and cannot recover source-level intent from the artifact.

# Codex Desktop `26.506.21252` build 2575

> Third-party changelog · Unofficial · Based on reverse diffs of official update artifacts

The main change in this build is a new Chrome automation plugin that can work with the user's existing browser state, alongside clearer routing rules for the in-app Browser plugin.  
One-liner: Codex Desktop adds a bundled Chrome automation plugin and tightens the in-app browser automation surface.

## About This Report

This is not an official changelog. It is a **reverse-diff report based on official signed distribution artifacts**. The comparison covers macOS arm64 from `26.429.61741` build 2429 to `26.506.21252` build 2575.

This entry is a historical backfill: the fact pack states that the official appcast did not provide a delta package for this version pair, so both official full packages were downloaded and compared directly. The findings come from the bundle file tree, Info.plist, package.json, bundled plugin manifests, skill docs, readable strings, and signature verification results.

Evidence is split into two levels: **[Confirmed]** means there is direct human-readable evidence such as a plugin manifest, skill document, package.json diff, Info.plist key, or newly present bundle string. **[Signal]** means the evidence only comes from frontend module names, changed strings in same-named frontend modules, or binary size deltas; it points in a direction, but cannot be reconstructed into source-level intent.

## ✨ Highlights

### 1. New Bundled Chrome Plugin [Confirmed]

The `openai-bundled` plugin set changed from `browser-use`, `computer-use`, and `latex-tectonic` to `browser-use`, `chrome`, `computer-use`, and `latex-tectonic`. The new `chrome` plugin is version `0.1.7`, displayed as `Chrome`, and categorized under `Productivity`.

This means Codex is beginning to include the user's existing Chrome state in desktop automation. The plugin description explicitly covers remote URLs, signed-in pages, existing Chrome tabs, cookies, extensions, and Codex Chrome Extension setup. Key evidence:

```diff
+      "name": "chrome",
+      "source": {
+        "source": "local",
+        "path": "./plugins/chrome"
+      },
+      "policy": {
+        "installation": "AVAILABLE",
+        "authentication": "ON_INSTALL"
+      },
+      "category": "Productivity"
```

The new plugin contains 345 files and totals 8.6 MB. Its default prompts include `file my expenses for me`, `review my messages on social media`, and `put in a PTO request in Workday`.

### 2. Chrome Automation Emphasizes User Authorization and Extension Connectivity [Confirmed]

`skills/chrome/SKILL.md` positions Chrome as the browser automation entry point for tasks that need the user's cookies, logged-in sessions, existing tabs, extensions, or remote authenticated sites. The document also requires a lightweight open-tabs check on the first Chrome-backed task, one retry after failure, and checks for Chrome installation, Chrome running state, extension presence, and native host manifest validity.

The user-visible implication is that Chrome support is not just generic web browsing. It connects through the Codex Chrome Extension to the user's existing browser environment. If Chrome is not running, Codex must ask before launching it. If the extension is missing, Codex should ask the user to confirm that the Codex Chrome Extension is installed and enabled. The skill doc also explicitly forbids falling back to applescript, bash, or other scripting methods after extension communication fails.

### 3. The Built-In Browser Plugin Is Renamed from “Browser Use” to “Browser” [Confirmed]

The `browser-use` plugin version moved from `0.1.0-alpha1` to `0.1.0-alpha2`. Its display name changed from `Browser Use` to `Browser`, and the description now refers to the `in-app browser`. The alias set adds `@browser` while retaining `@browser-use`.

This makes the two browser surfaces clearer: `Browser` is for the Codex in-app browser and local development targets, while the new `Chrome` plugin is for the user's existing Chrome state. Key evidence:

```diff
-    "displayName": "Browser Use",
+    "displayName": "Browser",
...
-  "version": "0.1.0-alpha1",
+  "version": "0.1.0-alpha2",
```

### 4. Browser Skill Docs Rewrite the Runtime Entry Point and Usage Rules [Confirmed]

`browser-use/skills/browser/SKILL.md` changed substantially. The setup example no longer passes `backend = "iab"` directly into `setupAtlasRuntime`; it now calls `setupAtlasRuntime({ globals: globalThis })` and retrieves the browser with `agent.browsers.get("iab")`. The API description shifts from `agent.browser.*` to `agent.browsers.*`.

This affects how Codex agents drive the in-app browser. The doc also adds rules requiring screenshots intended for the user to be rendered inline with Markdown image syntax, discouraging broad per-element reads on large pages, and preferring snapshots or focused checks. This points to stricter cost control and clearer user-visible output for browser automation.

### 5. Computer Use Gets a Minor Version Bump and Declares a Skills Directory [Confirmed]

The `computer-use` plugin version moved from `1.0.770` to `1.0.780`, and its plugin config now includes `"skills": "./skills/"`.

This suggests Computer Use is not only receiving binary changes, but is also participating in capability routing through skill docs. Related binaries also changed slightly: `SkyComputerUseService` grew from 14.5 MB to 14.7 MB, and `SkyComputerUseClient` grew from 10.0 MB to 10.2 MB.

### 6. Computer Use Settings Show Chrome Extension and Download/Upload Permission Controls [Signal]

The same-named `computer-use-settings` frontend module adds strings such as `Always allow`, `Always ask`, `Any App`, `Ask before downloading files`, `Ask before uploading files`, `Browser extension not connected`, `manage Google Chrome settings`, and `reinstall/remove the Google Chrome extension`.

These signals point to Computer Use settings gaining Chrome extension connection state and permission controls for asking before website downloads or uploads. Because this comes from a same-named frontend module diff, it only indicates UI, state, or interaction direction and cannot be reconstructed into source-level intent.

### 7. Plugin Marketplace Management Moves Beyond a Fixed Built-In List [Confirmed]

The package and readable frontend strings together show stronger marketplace management signals. `marketplace.json` adds `chrome` as a local plugin source. The same-named `plugins-page` frontend module adds copy such as `Add marketplace`, `Add a plugin marketplace from a GitHub repo, Git URL, or local folder`, `Disable plugin`, and `Built by OpenAI`.

The `marketplace.json` diff is [Confirmed]. The `plugins-page` strings are [Signal]. Taken together, the change points to a plugin page that can enable or disable plugins, add more marketplace sources, and distinguish OpenAI-built plugins, while the source-level implementation details remain unconfirmed.

### 8. External Agent Migration Becomes More Specific [Confirmed]

`package.json` adds the `external-agent-migration` dependency. The same-named `external-agent-import` frontend module adds copy including `Import from other AI apps`, `Apps found`, `Claude Code`, `Claude Cowork`, and `Bring over your setup, projects, and recent chats`.

The user-visible direction is an import flow that moves beyond generic project or tool import toward bringing setup, projects, and recent chats from other AI apps, explicitly naming Claude Code and Claude Cowork. The dependency addition is [Confirmed]; the detailed UI flow copy comes from a same-named frontend module and is [Signal].

### 9. Markdown Rendering and Preview Surfaces Continue to Expand [Signal]

The `markdown` module grew from 0.1 MB to 0.3 MB and adds strings for `ARIA label for rendered mermaid diagrams`, markdown image buttons, image fallbacks, and file citation chips with line information. Real frontend module additions after hash-noise filtering also include `image-preview-dialog`, `notebook-preview-panel`, `diff-unified`, `file-diff`, and `review-file-tree-pane`.

These signals point toward richer Markdown, image preview, file citation, diff, and review UI surfaces. They come from frontend module and same-named module diffs, so they do not confirm that the features are fully exposed.

### 10. macOS Now Declares CSV Document Viewing Association [Confirmed]

Info.plist adds a CSV document type: extension `csv`, MIME type `text/csv`, UTI `public.comma-separated-values-text`, role `Viewer`, and `LSHandlerRank` set to `Alternate`.

This means macOS can identify Codex as an alternate viewer for CSV files. The evidence comes directly from added Info.plist keys.

### 11. LaTeX Plugin Adjusts Positioning and Visual Assets [Confirmed]

The `latex-tectonic` plugin version moved from `0.1.0` to `0.1.1`. Its category changed from `Engineering` to `Research`, and it adds both `composerIcon` and `logo`, pointing to `./assets/tex.svg`.

This is not a major capability change, but it affects how the plugin is categorized and presented in the UI.

### 12. Automation, Remote Connections, Hooks, Skills, and Commands Keep Expanding [Signal]

New frontend modules include `automation-schedule`, `automation-shared`, `hooks-settings`, `install-hook`, `skills-page`, `remote-connections-settings`, `thread-side-panel-browser-tab-state`, `open-config-toml-button`, and `workspace-directory-tree`. Same-named modules also add strings such as `Authorize remote control to load signed-in devices`, `Connect via SSH`, `Choose a discovered connection or add one manually`, `Button label for creating a new skill`, and `codex.command.forceReloadSkills`.

These signals point to continued expansion across automation, remote control authorization, SSH connections, hooks management, skill refresh, and browser-tab state in side panels. They are frontend signals only and cannot confirm full product intent from the artifact.

# Codex Desktop `26.506.21252`(build 2575)

> Third-party changelog · Unofficial · Based on reverse-engineered differences in official update packages

The main theme of this release: Codex’s browser automation entry point has become more formalized, the plugin marketplace adds a Chrome plugin, and the system level now declares Codex as a viewer for CSV files.  
English one-liner: this build tightens browser automation, expands bundled plugins, and adds CSV document handling.

## About this report

This is not an official changelog, but a **reverse-engineered diff report based on officially distributed artifacts**. This update is a historical backfill: both versions fall outside the official Sparkle incremental update window, so the comparison uses two official full packages, the previous version and the latest version, rather than reconstructing from an incremental package.

In terms of trustworthiness, both Apps passed Apple notarization and OpenAI Developer ID signature verification. The `spctl` results are both `accepted`, the source is `Notarized Developer ID`, and the signing source is `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`.

Evidence is divided into two levels:

- **[Confirmed]**: Direct human-readable evidence, such as `Info.plist`, plugin `plugin.json`, skill documentation, `package.json`, and signature and notarization results.
- **[Signal]**: Evidence from frontend module names or binary size changes that points in a certain direction, but cannot be restored from the artifact into source-level intent.

Comparison scope: `26.429.61741` build `2429`, released on `2026-05-05`; to `26.506.21252` build `2575`, released on `2026-05-07`. The platform is macOS arm64.

## ✨ Key Changes

### 1. The built-in browser plugin was renamed from “Browser Use” to “Browser” and upgraded to `0.1.0-alpha2` [Confirmed]

The `browser-use` plugin version was upgraded from `0.1.0-alpha1` to `0.1.0-alpha2`, the display name changed from `Browser Use` to `Browser`, and the alias `@browser` was added. The description also changed from “recommended for testing” to using Browser to open local targets after significant frontend changes when the target is clear.

This means the built-in Codex browser is no longer packaged merely as a somewhat experimental “use the browser” capability, but as a clearer and higher-priority entry point for inspecting and operating local web pages.

Key evidence:

```diff
- "version": "0.1.0-alpha1"
+ "version": "0.1.0-alpha2"

- "displayName": "Browser Use"
+ "displayName": "Browser"
```

### 2. The browser automation API entry point changed to `agent.browsers.*`, with emphasis on using the `iab` browser [Confirmed]

The browser skill documentation changed from explicitly passing `backend = "iab"` to initializing the runtime first, then obtaining the browser object via `agent.browsers.get("iab")`. The call examples in the documentation also migrated heavily from `agent.browser.*` to `browser.*`.

The visible change is that Codex now has a clearer internal abstraction for “browser instances”: first select the built-in browser, then use that browser object to open tabs, name sessions, take screenshots, and operate pages.

Key evidence:

```diff
- await setupAtlasRuntime({ globals: globalThis, backend });
+ await setupAtlasRuntime({ globals: globalThis });
+ globalThis.browser = await agent.browsers.get("iab");
```

### 3. A local Chrome plugin entry point was added, but visible capability is still governed by marketplace configuration [Confirmed]

The plugin marketplace configuration added a `chrome` plugin, sourced from the local path `./plugins/chrome`, with installation policy `AVAILABLE`, authentication method `ON_INSTALL`, and category `Productivity`. The file tree also now contains Chrome plugin-related `extension-host` and `browser-client.mjs` files.

This shows that this Codex release has started bringing Chrome-related capabilities into the built-in plugin system. However, the artifact only proves that “the plugin entry exists and is installable”; it does not prove that it is enabled by default or which specific operations are exposed to users.

Key evidence:

```json
{
  "name": "chrome",
  "source": { "source": "local", "path": "./plugins/chrome" },
  "policy": { "installation": "AVAILABLE", "authentication": "ON_INSTALL" },
  "category": "Productivity"
}
```

### 4. The Computer Use plugin was upgraded and now declares a skills directory [Confirmed]

The `computer-use` plugin version was upgraded from `1.0.770` to `1.0.780`, and `"skills": "./skills/"` was added. Computer Use had already declared that it requires `accessibility` permission; the new addition in this release is the skills directory entry point.

This means desktop operation capability is not only receiving an underlying service upgrade, but also starting to enter Codex’s plugin description and orchestration system through skill documentation.

Key evidence:

```diff
- "version": "1.0.770"
+ "version": "1.0.780"
+ "skills": "./skills/"
```

### 5. Codex now declares system-level support for viewing CSV files [Confirmed]

`Info.plist` adds a set of `CFBundleDocumentTypes` configuration declaring the file extension `csv`, MIME type `text/csv`, content type `public.comma-separated-values-text`, role `Viewer`, and rank `Alternate`.

This means macOS can recognize Codex as an alternate viewer for CSV files. The artifact does not show a specific CSV preview interface, so this only confirms a change at the system file association level.

Key evidence:

```text
CFBundleTypeExtensions[0] = csv
CFBundleTypeMIMETypes[0] = text/csv
CFBundleTypeRole = Viewer
LSHandlerRank = Alternate
```

### 6. The LaTeX plugin was upgraded to `0.1.1` and moved to the Research category [Confirmed]

The `latex-tectonic` plugin version was upgraded from `0.1.0` to `0.1.1`, its category changed from `Engineering` to `Research`, and `composerIcon` and `logo` were added, both pointing to `./assets/tex.svg`.

This looks more like an improvement to plugin presentation and categorization: the capability description remains compiling LaTeX and TeX documents with the bundled Tectonic engine, but its positioning in the plugin marketplace is now more oriented toward research scenarios.

Key evidence:

```diff
- "version": "0.1.0"
+ "version": "0.1.1"
- "category": "Engineering"
+ "category": "Research"
+ "composerIcon": "./assets/tex.svg"
+ "logo": "./assets/tex.svg"
```

### 7. Frontend module names point to more complete settings, skills, plugins, diff viewing, and workspace interfaces, but should not be treated as source-level conclusions [Signal]

After removing Vite hash noise, new modules include names such as `account-settings`, `hooks-settings`, `skills`, `plugin-install-store`, `plugins-availability`, `diff-unified`, `file-diff`, `workspace-directory-tree`, `notebook-preview-panel`, and `image-preview-dialog`.

These module names point to expanded experiences around settings pages, skills management, plugin installation state, file diff viewing, workspace file trees, and previews. However, the artifacts cannot be restored into source-level intent, and they do not prove that all of these interfaces have been opened to all users.

Key evidence: the denoised newly added modules listed in the artifact include `skills`, `plugin-install-store`, `diff-unified`, `file-tree-search-input`, and `workspace-directory-tree`.

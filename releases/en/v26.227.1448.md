# Codex Desktop `26.227.1448` build 747

> Third-party changelog · Unofficial · Based on reverse diffing official update artifacts

The clearest changes in this build are around the hotkey window, app-server connection startup, sandbox permission messaging, and macOS Bluetooth permission metadata.  
One-liner: this build appears to refine hotkey-window setup, app-server transport handling, and permission messaging.

## About This Report

This is not an official changelog. It is a **reverse-diff report based on official distribution artifacts**. The comparison covers macOS arm64 from `26.226.940` build 720 to `26.227.1448` build 747.

Because the official appcast did not provide a delta package for this historical version pair, this report compares the two official full packages directly. The evidence comes from the bundle file tree, Info.plist, package.json diff, newly added readable strings in main/frontend bundles, and string/style diffs from same-name frontend modules.

Evidence is split into two levels: **[Confirmed]** means there is direct human-readable evidence such as Info.plist, package.json, readable strings, or similar artifacts; **[Signal]** means the evidence is indirect, such as frontend module names, same-name module string/style changes, or binary size changes. Signal items point in a direction but cannot reconstruct source-level intent from the shipped artifact.

## ✨ Highlights

### 1. Hotkey window runtime state and shortcut validation text were added [Confirmed]

The new main-process bundle contains several hotkey-window-related strings, including `Failed to apply hotkey window runtime state`, `Hotkey Window hotkey is not configured.`, `Shortcut must include Cmd/Ctrl or Alt.`, `Shortcut must include a non-modifier key.`, and `Shortcut must include exactly one non-modifier key.`.

This means the shipped artifact added or strengthened paths around hotkey-window configuration, runtime-state application failure handling, and shortcut validation. The same-name frontend module also adds clues such as `Configured hotkey`, `hotkey-window`, `data-hotkey-window-curtain`, and `data-hotkey-window-floating-ui`, but those are only [Signal] evidence for UI/interaction direction.

### 2. app-server connection and transport disposal paths gained new handling [Confirmed]

The main-process bundle adds `Starting app-server connection` and `Error while disposing app-server transport`.

This shows visible new logging or error handling around app-server connection startup and transport disposal. The likely user/developer-visible surface is connection state, failure diagnosis, or background-service robustness, but the artifact does not allow reconstruction of full source-level intent.

### 3. Windows sandbox / WSL entry points continue to shift toward permission-state messaging [Signal]

Same-name frontend modules remove many strings such as `Windows Sandbox`, `runInWsl`, `windowsDocumentation`, and `windowsInstallWslBanner`, while adding `Codex automatically runs commands in a sandbox`, `Codex has full access over your computer (elevated risk)`, `Codex is in read-only mode and requires approval to edit files and run commands.`, and `Button text to continue using read-only mode instead of enabling Windows sandbox`.

These clues point to UI copy and flow changes around sandbox, read-only mode, and full-access permission states, but they cannot establish source-level intent. The presence of Windows-related strings in a macOS arm64 artifact also must not be read as evidence that macOS gained Windows functionality.

### 4. macOS Bluetooth permission metadata changed from Always to Peripheral [Confirmed]

Info.plist adds `/NSBluetoothPeripheralUsageDescription = This app needs access to Bluetooth` and removes the old `/NSBluetoothAlwaysUsageDescription`, whose previous value was also `This app needs access to Bluetooth`.

This is a direct macOS permission-metadata change. The visible effect is that the Info.plist key used for system permission prompts changed, but the fact package does not identify the concrete feature path that triggers Bluetooth access.

### 5. Build and test scripts now explicitly rebuild sqlite, and TypeScript commands moved to tsgo [Confirmed]

In `package.json`, `compile` changes from `tsc -b` to `pnpm exec tsgo -b`, and `tsc` changes from `tsc -noEmit` to `pnpm exec tsgo --noEmit`. The `dev`, `forge:package`, `forge:make`, `forge:publish`, `test`, `test:quiet`, and `playwright:agent:repl` scripts now all prepend `pnpm run rebuild:sqlite`, while the `postinstall`-level `pnpm run rebuild:sqlite` was removed.

This moves sqlite native rebuilds from a post-install action into explicit prerequisites for development, packaging, testing, and Playwright entry points. For developers, this is a clearer build precondition; for end users, it is mostly an engineering-process change behind the packaged app.

### 6. Localization resources add coverage for MCP tool output, patch failures, and plan suggestions [Signal]

Multiple locale modules add strings such as `codex.mcpTool.contentBlock.annotationsLine`, `codex.mcpTool.rawOutputHeading`, `codex.mcpTool.rawOutputTriggerTooltip`, `codex.mcpTool.resourceLink.reading`, `composer.aboveSuggestion.plan.*`, `codex.applyResultsDialog.notGitRepo`, and `codex.unifiedDiff.patchFailureDetailsIntroReapply`.

These clues point to expanded localization coverage for MCP tool output, resource-reading state, plan suggestions, non-Git repository handling, and patch failure details. They are still same-name frontend-module diffs, so they do not constitute source-level facts.

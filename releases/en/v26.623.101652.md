# Codex Desktop 26.623.101652 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

The core change in this batch is that the embedded Codex app advances to 26.623.101652 / build 4674 on both platforms, while macOS and Windows differ sharply in package-level versioning and how much frontend change is visible.  
One-liner: Codex desktop advances the embedded app and bundled browser automation plugins, with broad macOS UI signals and a much narrower Windows diff.

## About This Report

This report uses only machine-extracted diffs from reverse-reconstructed official signed distribution packages. The macOS sample compares 26.623.81905 build 4598 → 26.623.101652 build 4674. The Windows MSIX sample compares 26.623.11225.0 → 26.623.13972.0, while its embedded `package.json` and plugin manifests show the app version moving from 26.623.81905 / build 4598 to 26.623.101652 / build 4674.

There are two evidence levels: [Confirmed] comes from plist, manifest, `package.json`, plugin manifests, dependency lists, service entrypoints, signing information, explicit config, and readable strings outside the “same-name frontend module change clues” section. [Signal] comes from same-name frontend module change clues, module names, size changes, CSS class names, and compressed bundle direction clues. Platform scope is marked as `[macOS+Windows]`, `[macOS only]`, or `[Windows only]`.

The comparison covers macOS arm64 Sparkle full packages and Windows x64 MSIX full packages. Hash renames under webview/assets and `.vite/build` are treated as noise, not feature additions or removals.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Embedded app version advanced to 26.623.101652 / build 4674

On both platforms, `package.json` changes `version` from `26.623.81905` to `26.623.101652`, and `codexBuildNumber` from `4598` to `4674`.

The packaging layer differs by platform. On macOS, `Info.plist` also updates `CFBundleShortVersionString` to `26.623.101652` and `CFBundleVersion` to `4674`. On Windows, the version/channel block reports the MSIX package version as `26.623.11225.0` → `26.623.13972.0`, which is not the same numbering system as the embedded app version.

### [Confirmed] [macOS+Windows] Bundled Browser and Chrome plugin versions refreshed with the app

On both platforms, the bundled `browser` and `chrome` `.codex-plugin/plugin.json` versions move from `26.623.81905` to `26.623.101652`. The diff does not show semantic changes to the plugin descriptions, but the manifest version update is a confirmed package fact.

The `browser` plugin still describes the in-app browser, `@browser` / `@browser-use`, and local target opening, inspection, navigation, testing, clicking, typing, and screenshots. The `chrome` plugin still describes automation that depends on the user’s existing Chrome state, including tabs, logged-in sessions, cookies, extensions, and Codex Chrome Extension setup.

### [Confirmed] [Windows only] Computer Use plugin manifest version refreshed

Windows additionally shows `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` updating from `26.623.81905` to `26.623.101652`. Its description says it controls Windows desktop apps from Codex through Computer Use.

The macOS fact package does not include an equivalent `computer-use` plugin manifest diff. It only reports unchanged `cua_node` runtime totals.

### [Confirmed] [macOS+Windows] Computer Use runtime size and dependency count did not change

macOS `cua_node` totals `377.4 MB → 377.4 MB`, `+0 MB`, `+0%`. Windows totals `298.9 MB → 298.9 MB`, also `+0 MB`, `+0%`. Both platforms report `0` newly added `cua_node` packages.

Within this input sample, the Computer Use runtime file total and dependency set do not show expansion. The Windows plugin manifest version refresh does not imply a runtime size change.

### [Signal] [macOS only] Large-scale frontend route and page chunk reorganization

Inside macOS `app.asar`, internal files move from `2652` to `2635`. After removing hash noise, many real frontend modules are added or removed. New modules include `settings-page`, `page`, `first-run~new-thread-panel-page`, `debug-window-page~debug-modal~home-ambient-suggestions-content`, and many combinations involving `remote-conversation-page`, `new-thread-panel-page`, `projects-index-page`, `worktree-init-v2-page`, `pull-requests-page`, and `appgen`.

This evidence only shows bundle splitting, routing, or shared chunk structure changes. It does not prove any feature is enabled for all users. Together with same-name module strings, though, it points to touched areas around settings, first run, remote conversations, projects, AppGen, plugin detail, debug windows, and the new thread panel.

### [Signal] [macOS only] Onboarding and settings entrypoints carry more developer docs and browser layout editing signals

`app-initial~app-main~onboarding-page` shrinks from `5.5 MB` to `3.5 MB`. New strings include browser tweak editor dropdown labels for flex direction, alignment, and distribution. New class/string entries include links to `https://developers.openai.com/apps-sdk`, `/codex/app/automations`, `/codex/app/chrome-extension`, `/codex/app/computer-use#locked-use`, `/codex/app/local-environments`, `/codex/concepts/sandboxing#how-you-control-it`, `/codex/config-basic`, `/codex/guides/agents-md/#create-global-guidance`, and `/codex/hooks`.

These are signals inside a same-name frontend module. They do not prove the links or settings are visible in stable UI, but they show onboarding or a related shared chunk carrying more developer docs, automation, Chrome extension, Computer Use, local environments, sandboxing, hooks, and global guidance entrypoints.

# Codex Desktop 26.707.72221 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

This batch updates the Chrome and in-app browser plugins on both platforms, while Windows and macOS respectively expose expanded automation documentation and broad application-layer UI restructuring.

Both platforms refreshed the Chrome and in-app browser plugins, while Windows added automation capability documentation and macOS showed broad UI restructuring signals.

## About This Report

This report compares two pairs of official full distribution packages:

- macOS arm64: `26.707.71524` build 5263 → `26.707.72221` build 5307.
- Windows x64 MSIX: `26.707.9564.0` → `26.707.9981.0`. The Windows fact pack incorrectly labels its platform field as macOS arm64; this report classifies it according to the input’s stated Windows x64 MSIX sample.
- Platform version numbers and release timing may be offset. Every conclusion therefore comes from that platform’s own package pair, without projecting findings from one platform onto the other.
- Both macOS packages passed Apple notarization validation and were attributed to OpenAI OpCo, LLC’s Developer ID.
- The Windows input provides SHA-256 values for both packages but no MSIX signature-validation output.

Evidence uses two levels:

- [Confirmed] covers manifests, plist data, plugin descriptions, skill or capability documents, dependency lists, service entry points, signature results, and packaged readable content outside the same-name frontend-module clue section.
- [Signal] covers frontend module names, compressed-bundle sizes, string differences inside same-name modules, and style-class changes. It shows that code entered or left a build chunk, not that a feature is enabled or that source code was added or removed.

Platform markers are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`.

Evidence inventory:

- Versions, channels, Bundle trees, Computer Use runtime totals, `cua_node` dependencies, and `app.asar` statistics were checked separately for both platforms.
- Both inputs provide Chrome and Browser plugin manifests; Windows also provides a Computer Use manifest. Neither input includes a complete built-in-plugin inventory delta.
- Windows adds six Chrome capability documents. macOS adds no Bundle files.
- macOS provides real frontend-module additions/removals and same-name module clues. Windows provides no corresponding module detail.
- macOS provides Info.plist and HTML entry-point changes. Windows provides no AppxManifest, CSP, or added-readable-string section.
- The macOS CSP appears only as unchanged context in the `index.html` diff; the evidence does not show a policy change.
- No complete main-process string additions, CSP-change section, or full new-component inventory is present. Those omissions are not interpreted as proof of no change.

## ✨ Highlights

### Chrome and in-app browser plugins were refreshed together

[Confirmed] [macOS+Windows]

The `chrome` and `browser` plugin manifests on both platforms move from `26.707.71524` to `26.707.72221`. Their packaged descriptions continue to define two distinct automation paths:

- The Chrome plugin targets existing Chrome state, including tabs, authenticated sessions, cookies, extensions, and ChatGPT Chrome Extension setup. It says reauthentication or explicit approval is required before using Chrome as a fallback when a purpose-built connector, API, or CLI fails.
- The Browser plugin targets the in-app browser and covers opening, inspecting, navigating, testing, clicking, typing, and screenshots, explicitly including localhost, loopback addresses, `file://`, and the current in-app browser tab.
- Both platforms contain tiny changes to the Chrome and Browser copies of `browser-client.mjs`, although each remains 1.0 MB after rounding. No readable diff is supplied to identify the behavioral change.

This confirms that the plugin builds were updated in the batch. A manifest version change alone does not establish availability for every account or task.

### Windows added Chrome automation capability documentation

[Confirmed] [Windows only]

The Windows Bundle adds six documents under the Chrome plugin:

- Tab browser authentication: `browserAuth.md`
- Bot detection: `botDetection.md`
- Chrome DevTools Protocol: `cdp.md`
- Page assets: `pageAssets.md`
- Browser viewport: `viewport.md`
- Browser visibility: `visibility.md`

These are complete packaged document additions, not size-only signals. Their filenames confirm that these capability topics entered the distribution, but the fact pack does not include their contents, commands, defaults, or rollout conditions.

The macOS Bundle adds no files and reports no content changes for these six documents.

### The Windows Computer Use plugin advanced, with no runtime growth

[Confirmed] [Windows only]

The Windows `computer-use` plugin manifest moves from `26.707.71524` to `26.707.72221`. Its description remains “Control desktop apps on Windows from ChatGPT through Computer Use.”

At the same time:

- Windows `cua_node` remains 287.5 MB.
- No Windows `cua_node` dependencies were added.
- The evidence confirms a plugin-version update, not an expanded Computer Use runtime or dependency set.

The macOS `cua_node` total likewise remains unchanged at 365.7 MB with no added dependencies. The macOS fact pack does not list a corresponding Computer Use plugin-manifest change.

The mutually reinforcing modules and strings strongly indicate a new frontend build shape for site configuration, publishing, and sharing, but do not establish universal rollout.

## ⚖️ Boundaries and Disclaimer

[Confirmed] means only that the relevant configuration, document, manifest, string, or file is present in the specified distribution.

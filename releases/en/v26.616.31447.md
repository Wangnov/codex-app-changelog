# Codex Desktop 26.616.31447 macOS / Windows Comparison

> Third-party cross-platform changelog · unofficial

The most important change in this batch: both platforms update the Browser and Chrome bundled plugin version markers, while Windows also changes how Computer Use loads the Windows Sky runtime.  
One-liner: this batch is mostly a packaged-runtime and UI-surface alignment update, with a Windows-only Computer Use loader fix and clear cross-platform baseline skew.

## About This Report

This report only uses machine-extracted facts from reconstructed official signed packages: macOS arm64 from `26.616.30709 build 4108` to `26.616.31447 build 4133`, and Windows x64 MSIX from `26.616.3309.0` to `26.616.3767.0`. The two platforms have different version baselines, so every conclusion is labeled by platform.

Evidence levels:  
[Confirmed] means the evidence comes from plist, manifest, plugin manifest, script source diff, dependency list, signing verification, file statistics, or readable facts outside the “same-name frontend module change clues” section.  
[Signal] means the evidence comes from same-name frontend module strings, class names, module names, or size changes. It indicates UI or interaction direction inside the package, but does not prove the feature is visible to all users.

Platform labels: `[macOS+Windows]` means both fact packs contain same-direction evidence; `[macOS only]` / `[Windows only]` means the evidence only appears on that platform.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Browser and Chrome bundled plugin versions were synchronized to 26.616.31447

Both platforms show `openai-bundled/plugins/chrome/.codex-plugin/plugin.json` changing `version` from `26.616.30709` to `26.616.31447`. Both also show `openai-bundled/plugins/browser/.codex-plugin/plugin.json` changing `version` from `26.616.30709` to `26.616.31447`.

The plugin description text itself does not show a semantic change: Chrome still targets the user’s existing Chrome state, tabs, logged-in sessions, cookies, extensions, and Codex Chrome Extension setup; Browser still covers the Codex in-app browser, localhost / file URLs, screenshots, clicks, typing, and local target inspection.

Platform difference: the macOS fact pack only lists Browser and Chrome manifest version changes; Windows also includes Computer Use plugin and script changes.

### [Confirmed] [Windows only] Computer Use plugin version updated, with a Windows Sky runtime loader change

On Windows, `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` shows the `computer-use` plugin version changing from `26.616.30709` to `26.616.31447`. Its description says it controls Windows desktop apps from Codex through Computer Use.

The stronger change is in `computer-use-client.mjs`: it no longer statically imports `WindowsComputerUseClientBase` at the top level. Instead, when `platform() === "win32"`, it reads `NODE_REPL_NODE_MODULE_DIRS`, locates node_modules, and dynamically imports `@oai/sky/.../computer_use_client_base.js`. If that path is unavailable, it throws `Windows Computer Use Sky runtime is unavailable`.

This is a substantive Windows packaging/runtime boundary change: the loader now depends on the privileged Node REPL module directory instead of fixed dependency resolution. The macOS fact pack has no corresponding script diff, so this cannot be projected onto macOS.

### [Signal] [macOS+Windows] The @ mention list is expanding into capability and object entry points

Both platforms’ `at-mention-list` module adds the same readable strings, including `Computer use`, `Custom agents`, `Desktop apps`, `Files and chats`, `Live agents`, `MCP servers`, `Mac apps`, plus MCP server mention scope, user section, and agents/chats loading text. New class names include `mcp-capability`, `mcp-capability-mentions`, and `mcp-servers`.

This is a strong UI signal: @ mention candidates in the composer are being organized around not only files or chats, but also Computer Use, custom agents, MCP servers, and desktop app capabilities. Because the evidence comes from same-name frontend module clues, it does not prove every entry point is enabled or backend-backed for all users.

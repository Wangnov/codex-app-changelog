# Codex Desktop `26.409.20454` build 1462

> Third-party changelog · Unofficial · Based on reverse diffing official update artifacts

The main theme of this build is a runtime upgrade plus continued build-out of local sidecar, remote connection, browser panel, and Projectless-session surfaces.  
One-liner: Electron moved to 41.2.0 while app-server, remote control, browser panel, and projectless-thread surfaces gained visible wiring.

## About This Report

This is not an official changelog. It is a **reverse diff report based on official distribution artifacts**. The comparison covers macOS arm64 from `26.406.40811` build 1457 to `26.409.20454` build 1462. Both builds are dated `2026-04-11`.

This is a historical backfill: the fact package states that the official appcast did not provide a delta package for this version pair, so two official full ZIP packages were downloaded and compared directly. Bundle diffs, `Info.plist`, `package.json`, frontend entry HTML, signature checks, and notarization results all come from machine-extracted facts.

Evidence is split into two levels: **[Confirmed]** means there is direct human-readable evidence, such as config diffs, permission keys, dependency versions, main-process strings, or entry files. **[Signal]** means the evidence is indirect, such as frontend module names, same-name bundle string diffs, or size changes; it points in a direction, but cannot reconstruct source-level intent.

## ✨ Highlights

### 1. Electron upgraded to `41.2.0` [Confirmed]

`package.json` shows Electron moving from `40.0.0` to `41.2.0`, while `better-sqlite3` moves from `^12.4.6` to `^12.8.0`.

This means both the desktop runtime and the local SQLite binding were updated. User-visible impact may sit around windows, WebView behavior, native-module compatibility, and performance boundaries. Key evidence:

```diff
-    "electron": "40.0.0",
+    "electron": "41.2.0",
-    "better-sqlite3": "^12.4.6",
+    "better-sqlite3": "^12.8.0",
```

### 2. Local app-server sidecar lifecycle handling became more explicit [Confirmed]

The new main-process bundle contains multiple sidecar lifecycle strings covering start, stop, restart scheduling, exit handling, recovery failure, and missing-binary paths.

This indicates that the local app-server sidecar has more complete runtime state management and error handling. Key evidence includes `Starting local app-server sidecar`, `Stopping local app-server sidecar`, `Local app-server sidecar exited`, `Local app-server sidecar restart failed`, and `Sidecar binary enabled but command not found`.

### 3. Remote Control entered the settings surface [Signal]

The same-name `remote-connections-settings` frontend module gained strings such as “Enable Remote Control on this Computer” and “Allows other signed-in Codex clients to connect to this computer”, along with related settings keys.

This points to a local setting that allows other signed-in Codex clients to connect to this machine, but the artifact does not allow reconstruction of the full enablement conditions, permission model, or rollout state. Key signals:

```text
Enable Remote Control on this Computer
Allows other signed-in Codex clients to connect to this computer
settings.remoteControlConnections.localRemoteControl.label
```

### 4. Remote connection states now cover version mismatch and restart paths [Signal]

The `onboarding-login-content` module gained copy for remote Codex being too old, update required, connecting, error, unavailable, unauthenticated, and restart-available states.

This points to remote connection flow moving from a simpler SSH entry toward more detailed Codex-to-Codex connection state handling, but the frontend artifact cannot confirm the full protocol or backend behavior. Key signals include `Update required`, `Update now`, `Label shown when remote connection needs a newer Codex version`, and `Action label shown when remote Codex has a newer installed version and can be restarted`.

### 5. Browser panel and browser-region capabilities continued to expand [Confirmed]

New main-process strings include `Toggle Browser Panel`, `Selected browser region`, `browser region`, `Browser sidebar runtime message received without owner window`, and `Invalid browser sidebar runtime message received`. The `vscode-api` module also gained `codex.command.toggleBrowserPanel` and `codex.command.openAvatarOverlay`.

This shows continued work on the browser sidebar, browser-region selection, and related command entry points. Because the strings and command names are direct readable evidence in the artifact, the direction is confirmed; exact UI completeness still cannot be fully reconstructed from minified output.

### 6. GitHub PR comment and diff-comment paths appeared [Confirmed]

The new main-process bundle contains strings such as `Failed to load GitHub user`, `Failed to post pull request comment`, `Failed to post pull request diff comment`, `Failed to reply to pull request comment`, and `Repository is required for PR diff comments`.

This shows that the app now contains paths for GitHub PR comments, diff comments, replies, and repository validation. The user-visible implication is possible local integration around PR review workflows, though the full success-path interaction cannot be confirmed from strings alone.

### 7. Projectless sessions and ambient suggestions became more structured [Confirmed]

New main-process strings include `Recent Projectless Codex threads with the same cwd`, `For Projectless chat, ground each suggestion in the user`, and `this Projectless chat`. The de-noised frontend module list also includes `projectless-thread`.

This indicates that sessions without an explicit project context are being modeled separately and connected to recent threads, suggestion generation, and the current cwd. The `agent-settings` same-name module also shifts older project-level copy toward broader ambient suggestion wording, which is a UI/settings signal.

### 8. Update installation now warns about interrupting local sessions [Signal]

The `plugins-page` module gained update-confirmation dialog copy stating that installing an update will quit Codex and interrupt active local sessions on the machine.

This points to a protective confirmation step in the update flow for active local sessions, but because the evidence comes from same-name frontend module diffs, it only establishes UI copy and interaction direction. Key signal:

```text
Codex will quit to install the update, which will interrupt active local sessions on this machine
Title for the confirmation dialog shown before installing an app update while a local session is active
```

### 9. macOS permission declaration shifted from Bluetooth to audio capture [Confirmed]

`Info.plist` added `NSAudioCaptureUsageDescription = This app needs access to audio capture` and removed the previous `NSBluetoothPeripheralUsageDescription`.

This is a user-visible system-permission change: the new build declares audio capture access and no longer declares Bluetooth peripheral access. Key evidence:

```text
Added /NSAudioCaptureUsageDescription = 'This app needs access to audio capture'
Removed /NSBluetoothPeripheralUsageDescription
```

### 10. Code-highlighting syntax assets were reorganized [Signal]

Several language modules show clear size and scope-string changes, including added scopes in `tsx`, `less`, `python`, `java`, `yaml`, `shellscript`, `ruby`, `r`, and `lua`, while `typescript`, `markdown`, `scss`, and `haxe` shrink.

This points to a reorganization of code-highlighting or syntax assets, but the artifact cannot reconstruct source-level intent and does not prove that any language support was added or removed. The likely user-visible surface is code block rendering, token classification, or theme styling.

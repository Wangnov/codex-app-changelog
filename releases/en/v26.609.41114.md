# Codex Desktop 26.609.41114 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

This batch mainly changes the Computer Use runtime, Sites skill guidance, database recovery paths, browser/terminal/plugin-management UI signals, and platform-specific shell resources.  
One-liner: This build expands the local Computer Use stack, tightens Sites deployment guidance, and surfaces new recovery, browser, terminal, plugin, and workspace UI paths across platforms.

## About This Report

This report only uses the two reverse-diff fact packages extracted from official signed distributions: macOS arm64 from `26.608.12217` build `3722` to `26.609.41114` build `3888`, and Windows x64 MSIX from `26.608.1337.0` to `26.609.4994.0`. The two platforms are different packages of the same Codex codebase, but their version and packaging numbers do not fully align, so every conclusion is platform-scoped.

Evidence levels:

- [Confirmed]: plist, manifest, CSP, skill docs, type declarations, package.json, dependency lists, plugin manifests, service entrypoints, signing data, or newly readable strings outside the “same-name frontend module change clues” section.
- [Signal]: frontend module names, size changes, same-name frontend-module readable strings/classes, or compressed bundle direction clues. These show that related UI or state paths entered the package, but do not prove that every user can see them.

Platform tags are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`. Hash-only renames under webview/assets and `.vite/build` are treated as noise.

## ✨ Highlights

### [Confirmed][macOS+Windows] The Computer Use runtime grew substantially, adding image, PDF, OCR, and browser automation dependencies

The `cua_node` runtime grew sharply on both platforms. macOS went from `213.5 MB` to `365.8 MB`, up `152 MB` or `+71%`. Windows went from `120.6 MB` to `285.5 MB`, up `165 MB` or `+137%`.

The new dependency direction is highly aligned: `sharp@0.34.5`, platform-specific sharp/libvips packages, `@napi-rs/canvas@0.1.91`, platform canvas backends, `pdfjs-dist@5.4.624`, `tesseract.js@7.0.0`, `tesseract.js-core@7.0.0`, `playwright@1.57.0`, `playwright-core@1.57.0`, `pixelmatch@7.1.0`, `pngjs@7.0.0`, `jpeg-js@0.4.4`, `bmp-js@0.1.0`, and related packages now ship in the app. The confirmed capability surface is that the local runtime now bundles libraries for image processing, PDF parsing, OCR, pixel comparison, and browser automation.

The platform split is in native prebuilt packages: macOS adds `@img/sharp-darwin-arm64`, `@img/sharp-libvips-darwin-arm64`, `@napi-rs/canvas-darwin-arm64`, and `fsevents`; Windows adds `@img/sharp-win32-x64` and `@napi-rs/canvas-win32-x64-msvc`. Windows adds `27` packages, while macOS adds `29`.

### [Confirmed][macOS+Windows] `@oai/sky` upgraded to `0.4.10`, and Linux full-desktop calls now pass options through

On both platforms, `@oai/sky` moved from `0.4.8-202606020558-pr-981870-f570e0604110` to `0.4.10`. `Options.d.ts` adds `mouse_size_px?: number`, documenting configurable mouse pointer size in screenshots.

Linux target calls for `click`, `drag`, `move`, `scroll`, `press_key`, `type_text`, and `get_screenshot` now pass `{ options, input }` or `{ options }` instead of only `input`. `sky_linux.js` also adds `--mouse-size-px` argument handling. `create_client.js` changes from `get_screenshot:t()` to `get_screenshot:t(i)`, so screenshot capture receives the same options object.

### [Confirmed][macOS+Windows] Local database recovery and backfill paths entered main-process strings

Both platforms add the same readable strings: `Back Up and Rebuild`, `Codex rebuilt its local database.`, `Rebuilding conversation history`, `Database path: `, `Backup folder: `, `state db backfill`, `timed out waiting for state db backfill`, `Failed to show SQLite backfill progress window`, `Failed to show app-server SQLite recovery confirmation`, `Queued app-server SQLite recovery confirmation`, and `Showing app-server SQLite recovery confirmation`.

This confirms package-level paths for SQLite backfill, recovery confirmation, backup/rebuild, and progress windows. It does not prove that the recovery flow is visible to every user, but the runtime strings are present.

### [Confirmed][macOS+Windows] Browser profile import, cookie selection, and non-importable states entered the package

New strings include `Browser profile is no longer importable`, `Owl browser profile importer is unavailable`, `Cookie domain selection requires cookie import`, and `Select cookies, passwords, or both to import`. This confirms package content for browser profile import, cookie/password selection, and non-importable error states.

Windows also has [Signal]-level changes in `browser-use-settings`, including `Access history without asking`, `Add allowed domain`, `Add allowed download domain`, `Add allowed upload domain`, and embedded browser/site/password/extension settings labels. The macOS fact package does not list the same module change, so this is Windows-only as a UI signal.

### [Confirmed][macOS+Windows] The Sites plugin upgraded to `0.1.15`, shifting publishing responsibility toward connector descriptions

On both platforms, the `sites` plugin moved from `0.1.14` to `0.1.15`. Its long description now includes `games` among examples and changes wording to “such as ... with Sites.”

`sites-building/SKILL.md` adds D1 prepared-statement rules: application queries and runtime initialization should use `env.DB.prepare(...).run()`, multiple semicolon-delimited statements should not be combined in one `prepare()`, and multi-statement operations should be separately prepared and run with `batch([...])`. This is a concrete skill-instruction change.

`sites-hosting/SKILL.md` was heavily rewritten. It moves from a full local flow for creating/reusing hosting config, saving versions, deploying to production, checking status, and managing access controls toward “local preparation of deployable Sites source and build artifacts.” It explicitly states that Sites connector and tool descriptions are the source of truth for site creation, environment variables, source credentials, versions, deployments, and access control.

### [Confirmed][macOS+Windows] CSP now allows Mapbox connections

On both platforms, `webview/index.html` and `webview/avatar-overlay-composition-surface.html` add `https://api.mapbox.com` and `https://events.mapbox.com` to `connect-src`. This confirms that the main webview and avatar overlay composition surface now permit connections to Mapbox API and events domains.

This only proves the network-policy allowance, not that a map UI is visible.

### [Signal][macOS+Windows] Multi-panel thread UI: bottom panel, side panel, and terminal/browser/review tab paths expanded

Both platforms add the same frontend modules: `thread-side-panel-active-signals`, `windows-tabs-open`, `windows-tabs-open-schema`, `run-app-action-in-primary-window`, and `interrupted-turn-state`. The same-name module `thread-page-bottom-panel-state` grows to `0.3 MB` and adds strings for reloading terminal rendering after a crash, terminal panel crash description, `Opening link blocked as opener could not be cleared`, and xterm buffer/accessibility errors.

`app-main` adds `Browser tab could not be opened`, `Review tab could not be opened`, `Terminal tab could not be opened`, and `Terminal tab is unavailable for this thread`. The macOS fact package also shows `thread-app-shell-chrome` strings for `Bottom panel` and actions that open side chat, browser, review, terminal, timeline, and workspace file browser. Together, these point to thread-level multi-tab, bottom-panel, terminal-panel, and side-panel UI changes.

The shared signals are bottom panel and tab-open failure text. macOS has additional explicit `thread-app-shell-chrome` side/bottom action labels in the fact package; Windows does not list that same module change.

### [Signal][macOS+Windows] Feedback and reporting expanded from simple ratings into detailed report-reason flows

On both platforms, `plan-summary-item-content` adds `Accessible label for Codex thread report reason options`, `Back button for Codex thread report reason selection`, `Bad response`, `Body shown after a Codex thread safety or legal report is submitted`, `Body shown when a Codex thread report reason needs an external form`, `Button label for opening an external Codex report form`, and `Error text shown when Codex thread report submission fails`. Artifact feedback also splits into missed template, incorrect content, and incorrect topic coverage.

This indicates new bundle signals for report-reason selection, post-submit states, external forms, and error handling. It remains a frontend-module signal, not proof that every entrypoint is enabled for every user.

### [Signal][macOS+Windows] Tool activity summaries shifted toward skills, integrations, MCP, code search, and file reads

Both platforms’ `local-conversation-thread` adds collapsed activity-summary text for loaded skill definitions, named integrations, named MCP tool call sources, running named MCP tool calls, completed/in-progress code searches, and completed/in-progress file reads. Removed strings focus on auto-review reviewer-agent states and `Auto-review approved/denied`.

This suggests the conversation-thread activity display is moving away from auto-review statistics toward general summaries for skills, integrations, MCP tool calls, code searches, and file reads.

### [Signal][macOS+Windows] Plugin detail and sharing UI expanded, with different details per platform

Both platforms’ `plugin-detail-page` shows plugin sharing and installation signals such as `Add plugin`, `Add people or groups`, workspace user/group search, and plugin share dialog empty states or accessibility labels.

The platform split is clear: macOS shows an MCP server enablement toggle, section navigation, skill preview modal actions, skill card action menus, removal of selected workspace share targets, and `App unavailable`. Windows shows uploading a newer local plugin version to an existing share, `Shared version is out of date`, `Shared version updated`, local-vs-shared version status text, and developer/version labels. In short, both packages changed the plugin-management surface, but Windows emphasizes local plugin share-version updates, while macOS emphasizes detail-page navigation, MCP toggles, and skill preview actions.

### [Signal][macOS+Windows] Sites live-site sharing gained workspace and internet access options

Both platforms’ `appgen-share-dialog` adds `Anyone in this workspace with the link`, `Anyone on the internet`, `Can visit`, `Copy link`, `Copied to clipboard`, `Button label for copying a live site URL`, `Button label for opening a live site externally`, and workspace user/group search plus saving state.

This points to live-site sharing UI with access scopes, link copying, external opening, and collaborator selection.

### [Signal][macOS+Windows] Voice, realtime waveform, and avatar overlay states entered frontend modules and localization keys

Both platforms add `transcribe-audio` and `use-realtime-waveform` modules. Localization keys add `avatarOverlay.startRealtimeVoice`, `avatarOverlay.showActivity`, `avatarOverlay.hideActivity`, and `avatarOverlay.realtime.createThreadError`. This signals new packaged paths for audio transcription, realtime waveform, and avatar overlay realtime voice/activity states.

### [Confirmed][macOS only] A Dock tile plugin and multiple icon resources were added

macOS `Info.plist` adds `/CodexAppIconBaseName = icon` and `/NSDockTilePlugIn = CodexDockTilePlugin.plugin`. The bundle adds `Contents/PlugIns/CodexDockTilePlugin.plugin`, signed with identifier `com.openai.codex.dock-tile-plugin` and authority `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`.

It also adds `Contents/Resources/icon.png`, `icon-dev-dark.png`, `icon-chatgpt.png`, and `icon-dev-light.png`. Combined with the new main-process string `Preferred macOS Dock icon`, this confirms a macOS Dock tile plugin plus Dock icon preference resources/text.

### [Confirmed][macOS only] macOS permission descriptions dropped trailing periods

In `Info.plist`, `NSAppleEventsUsageDescription`, `NSCameraUsageDescription`, and `NSMicrophoneUsageDescription` remove trailing periods while keeping the same meaning: Apple Events for controlling Mac apps on the user’s behalf, camera for video input, and microphone for voice input.

### [Confirmed][Windows only] The Computer Use plugin version moved with the app and retains its Windows description

In Windows, `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` moves from `26.608.12217` to `26.609.41114`, with the description still saying “Control desktop apps on Windows from Codex through Computer Use.” The macOS fact package does not include the corresponding computer-use plugin manifest diff, so this is scoped to Windows facts.

### [Signal][Windows only] Skills page and @mention list expose more entry categories

Windows `skills-page` adds strings for creating a new skill, reloading the skills list, the skills page header, installed/recommended skill sections, search input, empty states, and a message that newly installed skills need refresh before composer use.

Windows `at-mention-list` adds `Computer use`, `Custom agents`, `Desktop apps`, `Live agents`, `MCP servers`, `Mac apps`, current MCP server mention scope, and file fuzzy-search loading/empty strings. This points to an @mention list covering Computer Use, agents, desktop apps, MCP servers, and file search. The macOS fact package does not list these same-name module changes.

## 🔎 Low-Level Signal Index

### New frontend module index [Signal][macOS+Windows]

After hash noise removal, both platforms add the same modules: `annotation-mode-button`, `announcement-atoms`, `appgen-gating`, `automatic-approval-review`, `constants`, `extension-info`, `gradient`, `iconResolver`, `interrupted-turn-state`, `phone`, `plugin-detail-queries`, `robot`, `run-app-action-in-primary-window`, `thread-side-panel-active-signals`, `transcribe-audio`, `use-realtime-waveform`, `windows-tabs-open`, `windows-tabs-open-schema`, and `workspace-message-announcement-state`.

These module names point to annotation, announcements, appgen gating, approval review, extension info, phone/robot, side-panel active signals, audio transcription, realtime waveform, window tabs, and workspace announcements entering the frontend package. Module names alone do not prove UI availability.

### Removed frontend module index [Signal][macOS+Windows]

Both platforms remove `browser-sidebar-prompt-context`, `process-manager-target`, `window-app-action-helpers`, and `xterm-display-helpers`. Given the new bottom-panel terminal signals, the `xterm-display-helpers` removal looks more like internal restructuring than removal of terminal capability.

### Markdown and citation experience [Signal][macOS only]

macOS `markdown` adds accessibility text for file citation chips, extensionless file citations, markdown image/video fallback states, image placeholders, `Add to chat`, and selected-text overlays. The fact package only lists this module change for macOS, so it is scoped as a macOS signal.

### Workspace announcements and low-credit owner notification [Signal][macOS+Windows]

Both platforms’ `app-main` adds `Workspace announcement`, `Title for a Codex workspace announcement modal`, `Notify owner`, low-Codex-credit owner notification button text, and `Unread chats` as a command-menu group. Older work-plugin/mobile announcement marketing strings were removed. This points to workspace announcements, unread chats, and credit notifications entering the main UI bundle.

### Remote platform and shell/terminal strings [Confirmed][macOS+Windows]

Both platforms add `Remote platform probe did not return an OS`, `fluent terminal`, `git bash`, `tera term`, `windows powershell`, `windows terminal`, and `invalid transport message`. These confirm packaged paths for remote platform probing and multiple terminal names, though the exact selection logic cannot be recovered from strings alone.

### Localization key expansion [Signal][macOS+Windows]

Multiple locale bundles repeatedly add `Google Docs`, `Google Sheets`, `Google Slides`, `appgenPage.refresh`, `automation.updateDirective.open`, `codex.commandMenu.unreadThreadsGroup`, referral invite consent/eligibility keys, stories tooltip keys, and avatar overlay realtime voice/activity keys. Many `assistantMessage.autoReviewStats.*` keys are removed. This aligns with the activity-summary and workspace/automation/appgen directions, but localization keys remain frontend signals.

### Removed command/install-script strings [Confirmed][macOS+Windows]

The removed readable strings include `tar -xzf `, `ln -sfn `, `rm -rf `, `set -eu`, `trap cleanup EXIT INT TERM`, and `Codex archive checksum did not match expected digest`. Because the fact package only provides string-level changes, this does not prove the whole installation mechanism was removed; it only confirms those literals no longer appear in the compared readable-string set.

# Codex Desktop 26.623.41415 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

This batch looks less like a single feature drop and more like a coordinated packaging and frontend reshaping pass around local environments, Browser Use, Appgen, settings, and runtime status display.  
One-liner: this batch refreshes local-worktree recovery, browser automation surfaces, Appgen/settings wiring, and platform-specific plugin packaging signals.

## About This Report

This report is based on reverse-reconstructed diffs of official signed distribution packages: macOS arm64 compares `26.623.31921` build `4452` to `26.623.41415` build `4505`; Windows x64 compares `26.623.4041.0` to `26.623.5175.0`, while its bundled plugin manifests move from `26.623.31921` to `26.623.41415`.

Evidence is split into two tiers. `[Confirmed]` covers plist, manifest, CSP, plugin configuration, readable strings, file-level diffs, and dependency lists. `[Signal]` covers same-name frontend modules, size changes, module names, CSS/state classes, and compressed bundle directionality. Platform scope is marked as `[macOS+Windows]`, `[macOS only]`, or `[Windows only]`.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Browser and Chrome bundled plugins now report version 26.623.41415

On both platforms, the `browser` and `chrome` plugin manifests update `version` from `26.623.31921` to `26.623.41415`. Their descriptions do not show semantic changes: Browser remains the in-app browser / browser-use plugin for opening, inspecting, clicking, typing, and screenshotting local targets such as `localhost`, `127.0.0.1`, `file://`, and the current in-app browser tab; Chrome remains automation for tasks depending on the user’s existing Chrome state.

The platform difference is that Windows also includes a `computer-use` plugin manifest version update; the macOS input sample does not include an equivalent Computer Use plugin manifest diff.

### [Confirmed] [Windows only] Computer Use plugin version updated

On Windows, `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` updates the `computer-use` version from `26.623.31921` to `26.623.41415`, with the description still saying it controls Windows desktop apps through Computer Use.

This is Windows-only plugin-level confirmed evidence. The macOS sample only provides Browser and Chrome plugin version changes.

### [Signal] [macOS+Windows] Worktree initialization failures now have clearer recovery paths

Both platforms’ `worktree-init-v2-page` add the same user-visible strings: `Continue anyway`, `Edit environment`, `Fix this project`, `Fix worktree setup`, plus button descriptions for continuing after local environment setup failure, opening local environment settings, starting a repair thread, and canceling worktree setup to start a local conversation.

This suggests a more recoverable UI flow when worktree initialization fails: continue, edit environment, start a repair thread, or switch to a local conversation. Because this comes from same-name frontend module clues, it remains a `[Signal]`, not proof that the full behavior is available.

### [Signal] [macOS+Windows] Appgen site-sharing copy was removed from the share dialog bundle

Both platforms’ `appgen-share-dialog` remove strings such as `Add people or groups`, `Anyone in this workspace with the link`, `Anyone on the Internet`, copy live site URL, open live site externally, and related accessibility labels.

This is a cross-platform signal that the Appgen sharing UI was reduced or reorganized. It only proves those strings no longer appear in the same-name bundle diff; it does not by itself prove the feature was removed.

### [Signal] [macOS+Windows] General settings appear to have been split or moved

On both platforms, `general-settings` shrinks substantially and removes strings for the disabled default permissions toggle, work mode settings, base code size adjustment, hold-to-dictate hotkey, hotkey window hotkey, and dictation dictionary entry.

This looks more like settings content being moved, split, or lazily loaded than a simple deletion. macOS also adds module combinations such as `settings-page`, `projects-index-page~appearance-settings~general-settings`, and `pets-settings~appearance-settings~general-settings`; the Windows input does not show an equivalent large added-module list.

### [Signal] [Windows only] Browser Use settings expand around history, downloads, site settings, and allowed domains

Windows `browser-use-settings` grows from about `0.0 MB` to `0.1 MB` and adds strings including `Access history without asking`, embedded browser downloads/history/settings breadcrumbs/contact info/extension/password manager/site settings, show/hide browser data clearing options, disabled toggles for unavailable plugins in settings, and `Add allowed domain`.

This is a high-visibility Windows-only Browser Use settings signal, pointing to browser history, downloads, site settings, password manager, extension settings, allowed domains, and history access permissions.

### [Signal] [macOS only] Any-website Browser Use access confirmation enters the request panel

macOS `pending-request-item-panel` adds `Allow Codex to interact with any website without asking?`, `Allow any website`, confirm/cancel labels for allowing Browser Use on any website, a risk warning, and a title for confirming access to any website through an external browser connection.

These strings show that an “allow any website” confirmation and warning path entered the macOS package. They are readable strings in the package, but because they come from same-name frontend module clues, this report treats them as `[Signal]` and does not infer default visibility or default enablement.

### [Signal] [macOS only] Browser context and Chrome extension side-panel prompts enter conversation-related bundles

On macOS, a `worktree-init-v2-page~remote-conversation-page~new-thread-panel-page` combined module grows from `0.3 MB` to `1.5 MB` and adds context strings such as “the user has selected text on the page,” “Chrome extension side panel open,” “in-app browser open with,” and “in-app browser open.”

These strings point to context awareness around browser state, selected text, the Chrome extension side panel, and the in-app browser. They are still frontend module clues and should not be treated as proof of runtime behavior being available.

### [Signal] [macOS only] Local conversations and environment-state indicators were rewired into the frontend graph

macOS adds multiple `local-conversation-page` related modules, including `app-initial~app-main~local-conversation-page`, `remote-conversation-page~thread-app-shell-chrome~local-conversation-page`, `hotkey-window-detail-layout~local-conversation-page`, and `worktree-init-v2-page~hotkey-window-worktree-init-page~local-conversation-page`. These are macOS-side signals that the UI more explicitly distinguishes Codex Cloud, local git worktree, and local-running conversations.

### [Signal] [macOS only] External agent import and provider settings appear in settings-related bundles

macOS `pets-settings~appearance-settings~general-settings` adds many external-agent onboarding/import strings, including provider toggle, provider list, import options, projects import, recent chats import, tools and setup import, importing/imported row states, `Apps found`, and agent count in import tooltips.

This suggests an external agent import or migration flow inside macOS settings-related frontend code. Since the source is a same-name frontend module diff, it does not prove that the flow is visible to every user.

### [Signal] [macOS only] Appgen, project index, publication terms, and onboarding chunks were heavily reorganized

After ignoring hash noise, macOS adds many module combinations centered on `appgen-page`, `appgen-settings-page`, `appgen-library-page`, `appgen-publication-terms-route`, `projects-index-page`, `new-thread-panel-page`, `onboarding-page`, `plugin-detail-page`, `pull-requests-page`, and `worktree-init-v2-page`.

This is a macOS-only large-scale frontend routing and chunk graph signal. It is more active than the Windows sample, but because most evidence is module names and preload graph changes, it should not be written as concrete feature availability.

### [Signal] [macOS only] Automations page grows substantially and adds model applicability, AGENTS.md, and referral accessibility strings

macOS `automations-page` grows from `0.9 MB` to `1.3 MB`, adding `(Does not apply to current model)`, `AGENTS.md already exists here. ### [Signal] [macOS only] Debug modal expands around app actions, active context, and unpackaged-build controls

macOS `debug-modal` grows from about `0.0 MB` to `0.2 MB`, adding `Account ID`, `Active browser tab`, `Active conversation`, `Ambient suggestion chats`, `App Actions`, `Allow host`, `Allow host starting`, `Already current`, and `Allow automatic worktree cleanup in this unpackaged build.`

This is a macOS debug-window expansion signal covering account, browser tab, conversation, ambient suggestions, App Actions, host allow state, and an unpackaged-build worktree cleanup toggle.

### [Signal] [macOS+Windows] Pet-related state is still being reorganized, with different evidence per platform

Windows `pet-install-state` adds CSS/state classes `pet-install` and `pet-install-preview`. macOS adds `app-initial~app-main~pets-settings`, `avatar-overlay-page~avatar-overlay-native-page~pets-settings`, and several pet/avatar preload changes in settings-related modules.

Both platforms show pet-related frontend reorganization, but the evidence differs: Windows has state classes, while macOS has module graph and preload changes. This does not prove a new pet feature.

### [Signal] [macOS only] Code highlighting grammar assets were reorganized

On macOS, several grammar modules show large additions or removals. `typescript`, `php`, `c`, `haxe`, `java`, `stylus`, and `gdshader` add many scope strings; `javascript`, `less`, `python`, `scss`, `postcss`, and `katex` remove many strings or shrink; `diff` adds scopes such as `markup.changed.diff`, `meta.diff.header.git`, and `meta.diff.range.unified`.

This looks like a reorganization of syntax highlighting resources or code display dependencies. It may affect code-reading presentation, but the evidence remains bundle-level signal.

### [Confirmed] [macOS only] CSP shows no semantic policy change, only entry resource and preload graph changes

In macOS `webview/index.html`, the CSP remains `default-src 'none'`, with images allowed from `self/app/blob/data/https`, frame/child sources pointing to `web-sandbox.oaiusercontent.com`, workers from `self/blob`, scripts from `self` plus a fixed sha256 and `wasm-unsafe-eval`, and connect sources including OpenAI/Mapbox and ChatGPT websocket endpoints. The diff shows entry script and modulepreload resource changes.

The Windows input does not provide AppxManifest or CSP changes, so no corresponding conclusion is made.

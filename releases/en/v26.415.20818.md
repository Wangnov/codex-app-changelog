# Codex Desktop `26.415.20818` (build 1727)

> Third-party changelog · Unofficial · Based on reverse-engineered differences in the official update packages

From the artifact layer, this release is less about minor fixes and more about further packaging local execution, browser automation, Computer Use, and document-style preview capabilities into the desktop app.

One-liner: this build expands bundled local tooling and surfaces early signals for richer artifact previews and computer-use workflows.

## About This Report

This is not an official changelog. It is a **reverse-engineered diff report based on officially signed distribution artifacts**. This is a historical backfill: both versions have already fallen outside the official Sparkle incremental-update window, so the comparison uses two official full packages directly rather than reconstructing an incremental package.

The comparison range is from `26.409.20454` build `1462`, released on `2026-04-11`, to `26.415.20818` build `1727`, released on `2026-04-16`. The trust root comes from Apple notarization and OpenAI Developer ID code signing. `Codex.app` in both versions passed `spctl` notarization verification, with source `Notarized Developer ID` and signing identity `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`.

This report uses two evidence levels:

- **[Confirmed]**: Direct human-readable evidence, such as `Info.plist`, `package.json`, build script diffs, signing / notarization results, or explicitly added bundle files.
- **[Signal]**: Indirect evidence such as frontend module names or binary size changes. These point in a direction, but cannot be traced back from artifacts to source-level intent, nor do they mean the feature is necessarily available to users.

### 6. New warning logic for running the Intel build on Apple Silicon Macs [Confirmed]

The startup script `.vite/build/bootstrap.js` added detection logic: when the app is on macOS, packaged, running as `x64`, and the process is running through Rosetta, it shows a warning that the Intel build is running on an Apple Silicon Mac, recommending that the user quit and install the Apple Silicon build, or continue using the Intel build.

Although the comparison platform for this report is `macOS arm64`, this startup logic is a clear human-readable diff. It improves the launch experience when users have downloaded the wrong architecture build.

Key evidence:

```text
message: "{appName} is running the Intel build on an Apple Silicon Mac"
detail: "This build works through Rosetta, but the Apple Silicon build launches faster and performs better..."
buttons: "Quit" / "Continue Anyway"
```

### 7. Build scripts added Owl and Windows MSIX-related entries [Confirmed]

`package.json` added `owl`, `build:owl`, `owl:ensure`, and `owl:package` scripts, added the `electron-windows-msix` and `fs-extra` dependencies, added a linked `browser-common` dependency, and removed `socks-proxy-agent`.

This looks more like a build and packaging system change than an individual user-facing feature. For developers, it indicates that the same desktop code package has begun carrying more cross-platform packaging and browser-common build entry points. However, the artifact package does not provide source-level documentation for Owl, so this report does not expand on its specific purpose.

### 8. Theme-change subscription logic was consolidated into a shared listener set [Confirmed]

In `.vite/build/preload.js`, system theme-change subscriptions changed from “each subscription attaching its own `ipcRenderer.on` listener” to maintaining a `Set` in the preload layer, with a single system theme update event dispatching to all subscribers.

This is a lower-level but understandable stability change: it reduces duplicate event listeners and may make theme-change notifications more centralized and controllable. The artifact package only shows a change in implementation approach, and is not enough to prove that any specific user-facing issue was fixed.

Key evidence:

```diff
+ y=new Set;
+ e.ipcRenderer.on(s,(e,t)=>{_=t,y.forEach(e=>{e()})});
+ subscribeToSystemThemeVariant:e=>(y.add(e),()=>{y.delete(e)})
```

# Codex Desktop 26.602.30954 (macOS / Windows Comparison)

> Third-party cross-platform changelog · Unofficial

The main thread in this batch is a synchronized update to browser control, Chrome integration, and the Sites build/deploy workflow in OpenAI Codex.  
One-liner: this release mostly tightens browser automation guidance and Sites deployment packaging across macOS and Windows, with platform-specific packaging differences around Computer Use.

## About This Report

This report is based on reverse diffs of official signed distribution packages: the macOS side compares full Sparkle packages, while the Windows side compares MSIX packages. The macOS comparison is 26.601.21317 build 3511 → 26.602.30954 build 3575; the Windows comparison is 26.601.2237.0 → 26.602.3474.0. The release date is taken from the provided fact bundle: 2026-05-30.

Evidence is split into two levels. [Confirmed] means the change is backed by readable files such as plist, plugin.json, package.json, skill docs, template config, or dependency manifests. [Signal] means the evidence is limited to frontend module names or size changes; it points to a direction but should not be treated as source-level intent. Platform markers mean: [macOS+Windows] has evidence on both platforms, [macOS only] appears only in the macOS package, and [Windows only] appears only in the Windows package.

## ✨ Highlights

### Browser and Chrome automation docs were reorganized [Confirmed] [macOS+Windows]

Both platforms significantly shrink the Browser and Chrome skill docs, while moving many operational details into on-demand documentation entries such as `api.md`, `playwright.md`, `confirmations.md`, and `api-troubleshooting.md`. Chrome also gains `chrome-troubleshooting.md`; the Windows fact bundle also lists `file-management.md`. The old `browser/docs/capabilities/tab/cdp.md` is removed on both platforms.

The platform difference is the depth of evidence: macOS includes concrete diffs showing `control-in-app-browser` shrinking from 0.1 MB to 0.0 MB, and the Chrome skill description now says to prefer purpose-built connectors, APIs, or CLIs; if authentication is missing or expired, the user should reauthenticate or explicitly approve Chrome as a fallback. Windows file statistics and added/removed file lists show the same direction, but the Windows fact bundle does not include the line-by-line skill doc diff.

### Browser runtime entry now emphasizes Node REPL and complete runtime docs [Confirmed] [macOS+Windows]

The macOS diff explicitly shows Browser and Chrome skills now loading `browser-client.mjs` through the Node REPL `js` tool, then calling `agent.browsers.get("iab")` or `agent.browsers.get("extension")`. The docs also require reading the full `browser.documentation()` before interacting with the browser, and using tool discovery when the `js` tool is missing.

The Windows package also adds Browser/Chrome API and Playwright docs, and the same three skill docs shrink in size. That confirms the documentation split and runtime-entry cleanup as a cross-platform change, though the Windows bundle does not expose enough line-level text to reconstruct the exact wording.

### Sites expands from “create and deploy” into a broader website build/hosting workflow [Confirmed] [macOS+Windows]

Both platforms update the Sites plugin metadata: the `sites` plugin moves from 0.1.10 to 0.1.12, the description changes to “Build and deploy websites with Sites,” keywords add build, website, landing page, portfolio, dashboard, KPI dashboard, portal, tracker, hub, and internal tool, the brand color changes from `#10A37F` to `#0C79D8`, and default prompts shift from a pizza landing page, sudoku game, and image-generation site to a new-hire portal, sudoku leaderboard, and customer-support trends dashboard.

There is little platform divergence here: macOS provides the full plugin.json diff, while Windows shows the same Sites plugin/template files changing and lists the new `sites-vite-plugin.ts` files in the Sites templates.

### Sites build artifacts are now packaged through a Vite plugin [Confirmed] [macOS+Windows]

The Sites templates on both platforms remove the manual `_appgen_meta` copy logic from `package.json` build scripts, reducing build to `vinext build`. Template `vite.config.ts` adds the `sites()` Vite plugin. The hosting skill docs also move deployment metadata paths from `dist/_appgen_meta/appgarden.json` and `dist/_appgen_meta/drizzle/` to `dist/.openai/hosting.json` and `dist/.openai/drizzle/`.

This suggests the Sites build/deploy contract is being centralized around the template plugin and the `.openai` directory structure. The Windows added-file list explicitly includes `build/sites-vite-plugin.ts` for both templates; the macOS diff clearly shows `vite.config.ts` importing and using `sites()`, but the added-file excerpt does not list the plugin file name.

### Computer Use changed on both platforms, but not in the same shape [Confirmed] [macOS+Windows]

On macOS, the `computer-use` plugin version moves from 1.0.799 to 1.0.809, and native component growth is visible: `SkyComputerUseService` grows by +0.74 MB, `CUALockScreenGuardian` by +0.71 MB, and `SkyComputerUseClient` by +0.18 MB. macOS also adds `SkysightSummarizer.md` and several signing/config-related files.

Windows also changes computer-use files, but differently: `@oai/sky/bin/windows/codex-computer-use.exe` grows by only +0.01 MB, while multiple `tslib` files and two files under `objc-js@1.5.0` are removed. In short, both platforms touched Computer Use, but macOS looks like a native component upgrade, while Windows more clearly shows dependency pruning plus a small binary change.

### New frontend feature-direction signals appear in the app layer [Signal] [macOS only]

The macOS `app.asar` module names add `pricing-plan-page`, `rate-limit-reset-modal`, `github-details`, `share`, `comment`, `composer-controller`, `local-remote-selection`, `selectable-remote-connections-signal`, `project-context-signal`, and `bundled-plugin-auto-install-preference`, among others. Removed modules include `device-connections`, `file-preview-page`, `mcp-capability-file-viewer-frame`, and `use-automation-history-items`.

These names only signal possible directions around pricing/plan UI, rate-limit reset UI, GitHub details, sharing, comments, remote connection selection, project context, and bundled plugin auto-install preferences. The fact bundle explicitly warns that presence does not mean user-facing availability. The Windows fact bundle does not list module names, so this report cannot confirm whether the same module names are present there.

### Development scripts and lint entrypoints changed [Confirmed] [macOS only]

The macOS `package.json` splits `owl` into `prepare:owl` plus `owl`, and changes `lint` / `lint:fix` from direct `oxlint` calls to `node ../scripts/run-package-oxlint.mjs --include-top-level-ts-scripts`. This is a development/build-chain change and does not directly imply a user-visible feature.

The Windows fact bundle does not include the corresponding `package.json` diff, so this is not marked as cross-platform.

### Small localization text fixes [Confirmed] [macOS only]

The macOS package changes `native-menu-locales/mn.json` and `native-menu-locales/bg-BG.json`, including text around the About dialog and build information. The Windows fact bundle does not provide corresponding localization diffs.

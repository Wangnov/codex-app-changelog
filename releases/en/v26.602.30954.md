# Codex Desktop 26.602.30954 macOS / Windows Comparison

> Third-party cross-platform changelog · unofficial

The main story in this batch is not a single feature, but a broader move toward fuller browser, site-building, remote-environment, and collaboration surfaces in Codex.  
One-liner: Codex 26.602 tightens browser/plugin runtime docs, expands Sites workflows, and surfaces broader remote-control, sharing, task, and handoff UI signals across platforms.

## About This Report

This report uses only the supplied reverse-engineered diff facts from officially signed distribution packages: macOS moves from 26.601.21317 build 3511 to 26.602.30954 build 3575, while Windows moves from 26.601.2237.0 to 26.602.3474.0. Platform versions and packaging cadence differ, so each conclusion is tagged by platform.

Evidence is split into two levels. [Confirmed] means plist, manifest, plugin metadata, skill docs, package.json, dependency metadata, type declarations, service entrypoints, or newly observed readable strings. [Signal] means frontend module names, same-name module readable string/style-class deltas, or compressed bundle directionality. Platform tags use `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`.

Hashed webview/assets and .vite/build renames are treated as build noise, not feature additions/removals. The input does not provide a content-security-policy diff, so this report does not claim any CSP change.

## ✨ Highlights

### [Confirmed] [macOS+Windows] Browser and Chrome plugins moved toward a documentation-first runtime surface

The Browser and Chrome skill docs were heavily rewritten. Long embedded operating instructions were replaced with a Bootstrap flow plus `agent.documentation.get(...)` documentation entrypoints. New documentation topics include `api-troubleshooting`, `confirmations`, `playwright`, and `screenshots`; Chrome also adds `chrome-troubleshooting` and `file-management`.

Both platforms add `browser/docs/api.md`, `chrome/docs/api.md`, `browser/docs/playwright.md`, `chrome/docs/playwright.md`, confirmation and troubleshooting docs, and remove `browser/docs/capabilities/tab/cdp.md`. This indicates a split from monolithic skill files into a separate documentation surface, not just a size reduction.

The Chrome plugin description is also tightened. It still covers tasks depending on Chrome state, including tabs, logged-in sessions, cookies, and extensions, but now explicitly prefers purpose-built connectors, APIs, or CLIs. If the preferred path only has missing or expired authentication, the user should reauthenticate or explicitly approve Chrome as a fallback.

### [Confirmed] [macOS+Windows] Sites shifts from “create sites” toward broader website build and deployment workflows

The Sites plugin moves from 0.1.10 to 0.1.12, and its description changes from “Create, save, deploy, and inspect sites hosted by OpenAI” to “Build and deploy websites with Sites.” Keywords now include `build`, `website`, `landing page`, `portfolio`, `dashboard`, `KPI dashboard`, `portal`, `tracker`, `hub`, and `internal tool`. Default prompts also shift from a pizza landing page/image generation website toward a new-hire portal, a sudoku leaderboard, and a customer-support trends dashboard.

The `sites-building` skill adds a Fit Check: use the bundled vinext starter for unconstrained projects, and use Sites while preserving project structure when `.openai/hosting.json` exists. Template `vite.config.ts` files add the `sites()` Vite plugin, while template build scripts simplify from manually copying `_appgen_meta` to just `vinext build`. `sites-hosting` updates deployment metadata paths from `dist/_appgen_meta/appgarden.json` / `dist/_appgen_meta/drizzle/` to `dist/.openai/hosting.json` / `dist/.openai/drizzle/`, and says deployable site work should not stop at a local build unless the user explicitly asks to keep it local.

Platform difference: both platforms contain the metadata and skill-doc changes. The Windows added-file list additionally names the two template `build/sites-vite-plugin.ts` files; the macOS added-file excerpt does not list them.

On macOS, Info.plist changes `CFBundleShortVersionString` from 26.601.21317 to 26.602.30954, `CFBundleVersion` from 3511 to 3575, and updates the app.asar integrity hash. The root `package.json` also moves from 26.601.21317 to 26.602.30954, with `codexBuildNumber` from 3511 to 3575.

The Browser and Chrome plugin versions move to 26.602.30954 on both platforms. Computer Use moves from 1.0.799 to 1.0.809 on macOS; on Windows, its plugin version moves from 26.601.21317 to 26.602.30954. Windows also shows @oai/sky moving from `0.4.5-26566467949-4daa3290021d` to `0.4.9-26913854413-62c399f4cf2c`.

### [Confirmed] [Windows only] Computer Use @oai/sky dependency and screenshot data type changed in the Windows package

Windows package metadata shows @oai/sky upgraded to `0.4.9-26913854413-62c399f4cf2c` and its direct `tslib` dependency removed. The file tree also removes multiple `node_modules/tslib` files. The macOS facts only report “0 new cua_node packages” and do not provide the same @oai/sky package diff.

Windows @oai/sky docs and type declarations change Screenshot `data_url` comments from “Base64-encoded PNG data URL” to “Base64-encoded JPEG data URL.” The Linux screenshot implementation also changes its `to_data_url` call to include `image/jpeg`. This proves the Windows-distributed cross-platform sky JS code and docs changed; it does not prove the native macOS screenshot path changed.

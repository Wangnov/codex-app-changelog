# Codex Desktop `26.602.30954` (build 3575)

> Third-party changelog · Unofficial · Based on reverse-engineered diffs from official update packages

The main direction of this release is clear: Codex Desktop significantly consolidates the workflow guidance for Browser, Chrome, and Sites, while also updating the bundled Computer Use component version.  
One-liner: browser automation guidance is consolidated, Sites becomes more deployment-oriented, and Computer Use receives a bundled component update.

## About this report

This is not an official changelog, but a **reverse-engineered diff report based on officially distributed artifacts**. This entry is a historical backfill: both versions are outside the official Sparkle delta window, so the evidence package uses a direct comparison between two official full packages rather than reconstructing from delta packages.

The comparison covers `26.601.21317` build `3511`, released on `2026-06-03`, through `26.602.30954` build `3575`, released on `2026-06-04`. The platform is `macOS arm64`.

In terms of confidence, both compared artifacts come from official full packages, with Apple notarization and the OpenAI Developer ID as the trust root. The evidence package shows that `Codex.app` from both the previous and latest versions passed `spctl` notarization verification, with source `Notarized Developer ID` and signing origin `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`.

This report uses two evidence levels:

- **[Confirmed]**: Direct human-readable evidence, such as `Info.plist`, `package.json`, plugin configuration, skill document diffs, or signature verification results.
- **[Signal]**: Indirect evidence, such as frontend module names or binary size changes. These only indicate that something points in a certain direction, but cannot recover source-level intent from the artifact.

## ✨ Highlights

### 1. Browser and Chrome automation guidance has been substantially consolidated around “runtime documentation first” [Confirmed]

Browser and Chrome skill documentation has changed from lengthy built-in procedures to shorter Bootstrap instructions, requiring topic-specific documentation to be retrieved through `agent.documentation.get("<name>")` or `browser.documentation()`.

This means browser automation guidance is shifting from “hard-coding many rules into skill documents” toward “reading the full capability documentation exposed by the current runtime at startup.” The developer-visible change is that guidance around debugging, screenshots, confirmations, Playwright, and related topics has been split into dedicated documentation entry points, while the main skill documents now act more like routing and initialization instructions.

Key evidence:

```diff
+Start with the directions in the Bootstrap section below. Use `await agent.documentation.get("<name>")` when you need information about the specific topic they cover:
+nodeRepl.write(await browser.documentation());
```

The Chrome side also adds a clearer selection boundary: use Chrome only when the task depends on existing Chrome state, or when the user explicitly requests Chrome. If a connector, API, or CLI authentication fails, the user should first be asked to fix authentication, or explicitly approve Chrome as a fallback.

```diff
+Prefer purpose-built connectors, APIs, or CLIs. If one fails due to missing or expired authentication, ask the user to reauthenticate or explicitly approve Chrome as a fallback.
```

### 2. The Browser plugin adds multiple API and operation docs, while the old CDP doc is removed [Confirmed]

New files include Browser and Chrome `docs/api.md`, `docs/playwright.md`, and `docs/confirmations.md`; Chrome also adds `docs/chrome-troubleshooting.md`. At the same time, Browser removes `docs/capabilities/tab/cdp.md`.

This points to a more documented and more layered browser automation interface: capability guidance is no longer mainly packed into `SKILL.md`, but distributed across documentation files for API, Playwright, confirmations, troubleshooting, and related topics. Note that the evidence package only proves changes in documentation and distributed content; it does not prove that underlying browser capabilities themselves were added or removed.

Key evidence:

- Added: `Contents/Resources/plugins/openai-bundled/plugins/browser/docs/api.md`
- Added: `Contents/Resources/plugins/openai-bundled/plugins/browser/docs/playwright.md`
- Added: `Contents/Resources/plugins/openai-bundled/plugins/chrome/docs/chrome-troubleshooting.md`
- Removed: `Contents/Resources/plugins/openai-bundled/plugins/browser/docs/capabilities/tab/cdp.md`

### 3. Sites positioning expands from “creating and hosting sites” to a fuller website build and deployment workflow [Confirmed]

The Sites plugin version increases from `0.1.10` to `0.1.12`, and the description changes to “Build and deploy websites with Sites.” Keywords also expand from the narrower `deploy`, `sites`, and `games` to use cases such as `website`, `landing page`, `portfolio`, `dashboard`, `KPI dashboard`, `portal`, `tracker`, `hub`, and `internal tool`.

This means Sites is now described more as a tool for general website and internal tool building/publishing, rather than merely “create, save, deploy, and inspect sites.” The default prompts also change from a restaurant landing page, a Sudoku game, and an image generation website to an employee onboarding portal, a Sudoku leaderboard, and a customer support trends dashboard.

### 4. Sites build artifact packaging shifts from script concatenation to a Vite plugin [Confirmed]

In both Sites templates, `package.json` simplifies the `build` script from a long `vinext build && rm -rf dist/_appgen_meta ...` chain to `vinext build`. At the same time, both templates add a `sites()` plugin in `vite.config.ts`, imported from `./build/sites-vite-plugin`.

This indicates that metadata packaging logic required for Sites-compatible deployment is moving from manual copy steps in template scripts into a Vite plugin. The direct impact for developers is a cleaner template build command, while the hosting documentation also adjusts placement rules for `.openai/hosting.json` and `drizzle` migration files.

Key evidence:

```diff
-"build": "vinext build && rm -rf dist/_appgen_meta && mkdir -p dist/_appgen_meta && if [ -f .openai/hosting.json ]; then cp .openai/hosting.json dist/_appgen_meta/appgarden.json; fi && if [ -d drizzle ]; then cp -R drizzle dist/_appgen_meta/drizzle; fi",
+"build": "vinext build",
```

```diff
+import { sites } from "./build/sites-vite-plugin";
...
+    sites(),
```

The artifact paths in the hosting documentation also change from `_appgen_meta` to `.openai`:

```diff
-   `dist/_appgen_meta/appgarden.json`
-   `dist/_appgen_meta/drizzle/**`
+   `dist/.openai/hosting.json`
+   `dist/.openai/drizzle/**`
```

### 5. The Sites deployment workflow now more strongly emphasizes “do not stop at a local build” [Confirmed]

The `sites-hosting` skill document adds a requirement: unless the user explicitly asks to keep the result local, deployable site work should not stop at only completing a local build. The agent should continue to deployment, or report the concrete blocker and ask for the next action needed.

This affects the default completion standard when agents execute website tasks: for a “deployable site,” a successful `build` is no longer treated by the documentation as the natural endpoint. Deployment, or an explicit blocker report, is the endpoint.

Key evidence:

```diff
+Do not finish deployable site work with only a local build unless the user explicitly asks to keep the work local. Deploy the site, or report the concrete blocker and ask for the action needed to continue.
```

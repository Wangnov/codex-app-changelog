# Codex Desktop 26.608.12217(macOS / Windows Comparison)

> Third-party cross-platform changelog · Unofficial

The main story in this batch is runtime and plugin-layer reorganization: both platforms introduce `cua_node` and update browser automation and Sites hosting documentation, while the macOS and Windows packages still differ in visible packaging details.  
One-liner: Codex desktop moves more computer-use runtime into `cua_node`, refreshes automation/Sites plugin docs, and shows platform-specific packaging drift.

## About This Report

This report is based on reverse diffs of official signed distribution packages: full Sparkle packages on macOS and x64 MSIX packages on Windows. The macOS comparison is `26.602.71036` build `3685` to `26.608.12217` build `3722`; the Windows comparison is `26.602.9276.0` to `26.608.1337.0`. The batch release date from the fact pack is `2026-06-09`.

Evidence is split into two levels: [Confirmed] means readable evidence such as plist, package.json, dependency lists, CSP, skill docs, or plugin manifests; [Signal] means only module names, file movement, or size changes are visible, enough to indicate direction but not enough to reconstruct source intent.

Platform markers mean: `[macOS+Windows]` appears in both fact packs with the same direction; `[macOS only]` or `[Windows only]` appears only in that platform’s facts. Vite hash churn under `webview/assets` and `.vite/build` is treated as noise.

## ✨ Highlights

### Runtime Node Moved Into `cua_node`

[Confirmed] [macOS+Windows]  
Both platforms add a `cua_node` tree and remove the previous top-level Node/REPL binaries: macOS removes `Contents/Resources/node` and `node_repl`, then adds `Contents/Resources/cua_node/bin/node` and `node_repl`; Windows removes `resources/node.exe` and `node_repl.exe`, then adds `resources/cua_node/bin/node.exe` and `node_repl.exe`.

The platform difference is clear in dependency count: macOS adds 6 `cua_node` packages, while Windows adds 3. Both platforms share `@oai/sky@0.4.8-202606020558-pr-981870-f570e0604110`, `corepack@0.34.6`, and `npm@11.9.0`; macOS additionally has `node-addon-api@8.8.0`, `node-gyp-build@4.8.4`, and `objc-js@1.5.0`.

### computer-use Dependency Packaging Was Repositioned

[Confirmed] [Windows only]  
The Windows package explicitly shows many files removed from `resources/plugins/openai-bundled/plugins/computer-use/node_modules/@oai/sky/...`, while `@oai/sky` appears under `resources/cua_node/bin/node_modules/`, including `codex-computer-use.exe`. This indicates that Windows moved computer-use related dependencies from the plugin-local node_modules tree into the `cua_node` runtime tree.

macOS also adds `Contents/Resources/cua_node/lib/node_modules/@oai/sky/...`, but the fact pack does not list a corresponding large removal from the old computer-use plugin node_modules path, so the “moved out of the plugin tree” claim is kept Windows-only.

### Browser and Chrome Automation Docs Were Tightened

[Confirmed] [macOS+Windows]  
The readable macOS diff shows that the in-app Browser and Chrome control skills now require the initial `browser.documentation()` read to use the direct call `nodeRepl.write(await browser.documentation());`, and explicitly prohibit assigning it to a variable, measuring length, slicing, truncating, summarizing, or proactively paging it. Chunked reads are allowed only if the tool output explicitly reports truncation. The Windows fact pack also lists the same Browser/Chrome skill documentation files as changed.

This is a stronger agent-tool usage constraint, not evidence of a new UI. Its purpose appears to be more reliable full API documentation reads for automation skills.

### `domSnapshot()` Now Mentions iframe Body Expansion

[Confirmed] [macOS+Windows]  
The macOS `browser/docs/api.md`, `chrome/docs/api.md`, and multi-tab API docs change the `PlaywrightAPI.domSnapshot()` description from returning the current DOM snapshot to returning it with expanded iframe body content when available. The Windows fact pack also lists the corresponding API docs as changed.

This means browser automation may see a more complete page structure, especially on iframe-heavy pages. The fact pack confirms the API documentation change, not implementation internals.

### Sites Plugin API Names and Access-Control Guidance Were Updated

[Confirmed] [macOS only]  
The macOS Sites skill docs rename multiple connector operations to more explicit Sites terms: for example, `get-environment` to `get-environment-variables`, `update-environment` to `update-environment-variables`, `create-project` to `create-site`, `create-project-version` to `save-site-version`, `deploy-project-version` to `deploy-site-version`, and `deployment-status` to `get-deployment-status`. The docs also clarify that `project_id` is the persisted identifier for a Site.

Access-control guidance changed too: access controls should be used only when the user asks to change site access; `workspace_all` should be described to users as “Anyone in this workspace can visit”, and `custom` as “Private”. The macOS Sites plugin version moves from `0.1.12` to `0.1.14`. The Windows fact pack does not list Sites docs or plugin-version changes, so Windows sync cannot be confirmed.

### WebSocket Connection Allowlist Expanded

[Confirmed] [macOS only]  
The macOS `webview/index.html` Content Security Policy (CSP, the browser policy that restricts what a page can connect to) adds `wss://chatgpt.com`, `wss://ws.chatgpt-staging.com`, and `wss://ws.chatgpt.com` to `connect-src`.

The Windows fact pack does not include a CSP diff, so this cannot be confirmed on Windows for this batch.

### macOS Camera Permission Text Became More Specific

[Confirmed] [macOS only]  
The macOS `Info.plist` `/NSCameraUsageDescription` changes from “This app needs access to the camera” to “Codex needs access to your camera for video input.” This appears only in the macOS fact pack and is a system permission prompt text change.

### Avatar Overlay and Personalization Signals

[Signal] [macOS only]  
The macOS non-hash frontend module additions include `avatar-overlay-featuregate-page`, `avatar-overlay-native-frame`, `avatar-overlay-native-page`, `avatar-overlay-notifications`, `avatar-overlay-pill-dismiss-button`, `avatarOverlayCompositionSurface`, and `custom-avatars-query`; it also adds `Contents/Resources/native/avatar-overlay.node`.

These names strongly point toward avatar overlay or custom-avatar capabilities, but the fact pack does not include source logic or a product entry point, so user availability cannot be determined. The Windows fact pack does not provide a comparable module-name list and does not list `avatar-overlay.node`, so this remains macOS-only signal.

### Thread, Mention, and Link Rendering Reshuffle

[Signal] [macOS only]  
macOS adds modules such as `inline-mention-content`, `mention-icons`, `rich-link`, `external-link-favicon`, `web-search-favicon-icon`, `thread-browser-panel-tabs`, `thread-scroll-controller-context-value`, `thread-user-message-navigation-rail`, `sidebar-thread-row-signals`, and `threads-create`; it removes modules such as `home-project-recent-chats`, `local-task-row`, `pull-request-status`, `pull-request-visual-state`, and `sidebar-task-pr-chip-signals`.

These changes point to frontend restructuring around thread lists, mentions, link previews, or PR status display, but they remain module-level signals. The Windows fact pack only provides app.asar file counts, not module names, so the same conclusion cannot be drawn there.

### Build Scripts and Runtime Archive Updated

[Confirmed] [macOS only]  
The macOS `package.json` version changes from `26.602.71036` to `26.608.12217`, and `codexBuildNumber` changes from `3685` to `3722`. The `compile`, `tsc`, `lint`, and `lint:fix` scripts gain a `node ./scripts/ensure-owl-electron-types.mjs` pre-step, while `build:ci` changes to `node ./scripts/build-ci.mjs`. `owl-electron-app.json` also changes its `runtimeArchiveSha`.

The Windows fact pack does not include a package.json diff, so the same script changes cannot be confirmed there.

### Native Menu Localization Tweaks

[Confirmed] [macOS only]  
Several macOS `native-menu-locales/*.json` files update wording around version, build information, and release dates. For example, `zh-CN` changes `codex.aboutDialog.buildInfoLabel` from “构建信息” to “版本和构建信息”. The fact pack lists changes in `hi-IN`, `el-GR`, `fr-CA`, `ru-RU`, `mn`, `zh-CN`, `bs-BA`, `it-IT`, `sw-TZ`, `is-IS`, `kk`, `de-DE`, `lt`, `nl-NL`, `nb-NO`, `sv-SE`, and `vi-VN`.

The Windows fact pack has no corresponding locale diff, so synchronization cannot be confirmed.

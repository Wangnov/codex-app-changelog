# Codex Desktop 26.608.12217(macOS / Windows comparison)

> Third-party cross-platform changelog · Unofficial

The main story in this batch is the move of the Computer Use-related Node runtime into `cua_node`, alongside browser automation and Sites workflow documentation updates.  
One-liner: this batch reorganizes the packaged computer-use runtime and tightens browser/Sites automation contracts, with macOS carrying a few extra app-surface signals.

## About This Report

This report is based on reverse-reconstructed diffs of two official signed distribution packages: macOS arm64 from `26.602.71036` build `3685` to `26.608.12217` build `3722`, and Windows x64 MSIX from `26.602.9276.0` to `26.608.1337.0`. The release date is the one provided by the fact package: `2026-06-09`.

Evidence is split into two levels. [Confirmed] means the claim comes from readable evidence such as plist, package.json, plugin manifests, skill docs, API docs, CSP, dependency lists, or file trees. [Signal] means it comes only from frontend module names or size changes; it can indicate direction, but it does not reconstruct source intent. Platform tags mean `[macOS+Windows]` for both platforms, `[macOS only]` or `[Windows only]` when the current fact package shows it only on that platform.

## ✨ Highlights

### [Confirmed][macOS+Windows] Computer Use / Node runtime moved into `cua_node`

Both platforms add a `cua_node` directory and move the Node executables that used to sit at the resource root into it. macOS adds `Contents/Resources/cua_node/bin/node` and `node_repl`, while removing `Contents/Resources/node` and `node_repl`. Windows adds `resources/cua_node/bin/node.exe` and `node_repl.exe`, while removing `resources/node.exe` and `node_repl.exe`.

The platform difference is dependency count: macOS adds 6 `cua_node` packages, while Windows adds 3. Both share `@oai/sky@0.4.8-202606020558-pr-981870-f570e0604110`, `corepack@0.34.6`, and `npm@11.9.0`; macOS additionally adds `node-addon-api@8.8.0`, `node-gyp-build@4.8.4`, and `objc-js@1.5.0`, meaning the macOS package carries extra Node-API / native prebuild loading / Objective-C bridge dependencies in this batch.

### [Confirmed][macOS+Windows] Computer Use `@oai/sky` packaging was reorganized

The Windows fact package shows many old files removed from `resources/plugins/openai-bundled/plugins/computer-use/node_modules/@oai/sky/...`, while the new package contains corresponding runtime files and multi-platform binaries under `resources/cua_node/bin/node_modules/@oai/sky/...`. macOS also adds Windows, Linux, and macOS-related files under `Contents/Resources/cua_node/lib/node_modules/@oai/sky/...`.

This looks more like a packaging layout change than a simple user-visible feature addition: both platforms consolidate Computer Use dependencies into `cua_node`. Windows clearly cleans out the old plugin-local `node_modules/@oai/sky` path, while the macOS package does not list an equivalent old-path removal set.

### [Confirmed][macOS+Windows] Browser and Chrome automation docs became stricter

The in-app browser and Chrome skill docs on both platforms now require the first `browser.documentation()` read to use the direct call `nodeRepl.write(await browser.documentation());`. They also explicitly forbid assigning the docs to a variable, checking length, slicing, paging, or emitting only excerpts before that first read. Chunked reads are allowed only if the tool output itself reports truncation.

The same batch updates browser / Chrome API docs: `domSnapshot()` is no longer described only as returning the current DOM string; it now says it includes expanded iframe body content when available. This is a real automation contract change for browser tasks that depend on DOM snapshots.

### [Confirmed][macOS+Windows] Sites skill docs switched to newer connector operation names

The Sites skill docs replace several project-named connector operations with site-named ones: `create-project` → `create-site`, `create-project-version` → `save-site-version`, `deploy-project-version` → `deploy-site-version`, and `deployment-status` → `get-deployment-status`. Environment operations also move from `get-environment` / `update-environment` to `get-environment-variables` / `update-environment-variables`.

The same doc now states that `project_id` is the persisted identifier for a Site, and changes access-control wording: use access controls only when the user asks to change access, refer to `workspace_all` as “Anyone in this workspace can visit,” and refer to `custom` as “Private.”

### [Confirmed][macOS only] macOS adds camera permission copy and WebSocket CSP allowances

macOS changes `NSCameraUsageDescription` in `Info.plist` from a generic camera-access string to “Codex needs access to your camera for video input.” This is an explicit permission prompt text change.

macOS `webview/index.html` also adds `wss://chatgpt.com`, `wss://ws.chatgpt-staging.com`, and `wss://ws.chatgpt.com` to the CSP (Content Security Policy) `connect-src`. The Windows fact package does not include a corresponding CSP diff, so this report can only mark it as observed on macOS and not confirmed on Windows.

### [Signal][macOS only] Avatar overlay and related frontend entries appear by module name

After removing Vite hash noise, the macOS frontend build adds modules named `avatar-overlay-featuregate-page`, `avatar-overlay-native-frame`, `avatar-overlay-native-page`, `avatar-overlay-notifications`, `avatar-overlay-pill-dismiss-button`, and `avatarOverlayCompositionSurface`, plus `custom-avatars-query` and `experimental-feature-visibility`.

These names only show that the macOS package contains frontend build artifacts pointing toward avatar overlay / custom avatars / feature visibility. They do not prove the feature is enabled. The Windows fact package provides app.asar statistics but no equivalent module-name list, so Windows cannot be judged on the same signal.

### [Confirmed][macOS only] Build scripts add owl type preparation and a CI build wrapper

The macOS `package.json` shows `compile`, `tsc`, `lint`, and `lint:fix` now running `node ./scripts/ensure-owl-electron-types.mjs` first. `build:ci` changes from inline cleanup plus `forge:package` to `node ./scripts/build-ci.mjs`. `owl-electron-app.json` also changes its `runtimeArchiveSha`.

These are packaging / development workflow changes confirmed on macOS. The Windows fact package does not provide a matching `package.json` diff, so this is marked macOS only.

### [Confirmed][macOS only] Native menu localization copy was lightly revised

Several macOS `native-menu-locales/*.json` files update About dialog translations around version, build information, and release date wording, including `zh-CN`, `hi-IN`, `el-GR`, `fr-CA`, `ru-RU`, and `de-DE`. These are copy refinements rather than new interaction surfaces.

The Windows fact package does not list matching locale diffs.

### [Signal][macOS only] Frontend module structure continues to shift

macOS adds module names such as `thread-browser-panel-tabs`, `thread-user-message-navigation-rail`, `thread-scroll-controller-context-value`, `sidebar-thread-row-signals`, `home-artifact-templates`, `rich-link`, `inline-mention-content`, and `web-search-favicon-icon`. It also removes modules such as `pull-request-status`, `pull-request-readonly-comment`, `workspace-dependencies-feature`, `automation-queries`, and several settings search modules.

These names suggest frontend restructuring around thread navigation, browser panels, artifact templates, rich links, mention content, and related areas, but they do not prove visible product behavior. Windows lacks the module-name list, so no equivalent judgment can be made.

# Codex Desktop `26.608.12217` (build 3722)

> Third-party changelog · Unofficial · Based on reverse-engineered diffs from official update packages

This release is not merely a version-number bump: the package adds a new `cua_node` runtime directory, tightens the browser automation documentation, and systematically renames interfaces in the Sites plugin.  
English one-liner: runtime packaging, browser inspection, and Sites deployment flows all moved in a more explicit direction.

## About This Report

This is not an official changelog, but a **reverse-engineered diff report based on official distribution artifacts**. This entry is a historical backfill: both versions fall outside the official Sparkle delta window, so the evidence package uses a direct comparison of two official full packages rather than reconstructing from a delta package.

The comparison covers `26.602.71036` build `3685` to `26.608.12217` build `3722`. Both versions were released on `2026-06-09`, for the macOS arm64 platform.

Confidence comes from Apple notarization and the OpenAI Developer ID signature: both the previous and latest versions pass `spctl` notarization verification with the result `accepted`, the source `Notarized Developer ID`, and the signing identity `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`.

## ✨ Key Changes

### 1. Local Node Runtime Reorganized as `cua_node`, with Computer Use Dependencies Added [Confirmed]

The package removes the old `Contents/Resources/node` and `Contents/Resources/node_repl`, and adds `Contents/Resources/cua_node/bin/node` and `Contents/Resources/cua_node/bin/node_repl`. At the same time, `cua_node` adds 6 packages, including `@oai/sky@0.4.8-202606020558-pr-981870-f570e0604110`, `npm@11.9.0`, `corepack@0.34.6`, and `objc-js@1.5.0`.

This means the desktop app’s local JavaScript capability is no longer just a bundled Node runtime; it has been organized into a more dedicated `cua_node` runtime directory. Windows and Linux binaries for `@oai/sky` also appear there, pointing toward more complete cross-platform runtime support for Computer Use.

Key evidence:

```text
Added: Contents/Resources/cua_node/bin/node (119.1 MB)
Added: Contents/Resources/cua_node/bin/node_repl (15.2 MB)
Removed: Contents/Resources/node (119.1 MB)
Removed: Contents/Resources/node_repl (15.2 MB)
```

### 2. Documentation Reading Rules for Browser / Chrome Automation Are Stricter [Confirmed]

The skill documentation for both Browser and Chrome now requires the first read of `browser.documentation()` to be called directly “in one go”:

```text
nodeRepl.write(await browser.documentation());
```

It also explicitly forbids assigning it to a variable first, checking its length, slicing it, summarizing it, or proactively paginating it. Chunked reading is allowed only if the tool output explicitly reports truncation.

The developer-visible impact is that browser automation workflows now emphasize reading the complete API documentation, reducing the chance that the model misuses an interface based on partial documentation. This change appears in the built-in browser, multi-tab browser, and Chrome plugin skill documentation.

### 3. `domSnapshot()` Now Says It Attempts to Expand iframe Content [Confirmed]

In the API documentation for the Browser and Chrome plugins, the description of `domSnapshot()` changes from “Return a snapshot of the current DOM as a string” to “Return a snapshot of the current DOM as a string, including expanded iframe body content when available.”

This means embedded page content may now be included in readable snapshots during automated webpage inspection. Snapshot visibility may improve for scenarios such as debugging login pages, embedded consoles, and sandbox preview pages.

Key evidence:

```diff
- domSnapshot(): Promise<string>; // Return a snapshot of the current DOM as a string.
+ domSnapshot(): Promise<string>; // Return a snapshot of the current DOM as a string, including expanded iframe body content when available.
```

### 4. Sites Plugin Renames “project” Concepts to “site”, with Deployment Documentation Closer to the Actual Interface [Confirmed]

The Sites plugin is upgraded from `0.1.12` to `0.1.14`. Multiple interface names in the skill documentation are replaced: `create-project` becomes `create-site`, `create-project-version` becomes `save-site-version`, `deploy-project-version` becomes `deploy-site-version`, and `deployment-status` becomes `get-deployment-status`.

This is not merely a wording change. It unifies the key steps of creation, version saving, deployment, and status querying under the “site” concept. The documentation also adds that, in connector APIs and hosting configuration, `project_id` is the persistent identifier for a Site.

Key evidence:

```text
Sites plugin version: 0.1.12 → 0.1.14
create-project → create-site
create-project-version → save-site-version
deploy-project-version → deploy-site-version
deployment-status → get-deployment-status
```

### 5. Sites Access Control Defaults Are Described More Conservatively, and User-Facing Copy Is Clearer [Confirmed]

The Sites skill documentation changes “Use access controls” to “Use access controls only when the user asks to change who can access the deployed site.” It also requires `workspace_all` to be described to users as “Anyone in this workspace can visit”, and `custom` as “Private”.

This means access control is no longer described as a step that must be involved by default when deploying a site; instead, it is triggered by user intent. The user-facing permission explanations are also clearer.

Key evidence:

```diff
- Use access controls when the user asks to change who can access the deployed site.
+ Use access controls only when the user asks to change who can access the deployed site.
```

### 6. Frontend Adds Module Signals for Avatar Overlays, Mention Content, Thread Navigation, and More [Signal]

After filtering out Vite hash-file noise, newly added modules include `avatar-overlay-featuregate-page`, `avatar-overlay-native-frame`, `avatar-overlay-notifications`, `custom-avatars-query`, `inline-mention-content`, `mention-icons`, `thread-browser-panel-tabs`, and `thread-user-message-navigation-rail`.

These module names point toward avatar overlays, avatar queries, inline mentions, thread browsing / navigation, and related directions. However, artifacts cannot be reconstructed into source-level intent, and they do not prove that these capabilities are already available to users.

### 7. WebSocket Connection Targets Add ChatGPT Domains [Confirmed]

The CSP `connect-src` in `webview/index.html` now allows 3 additional WebSocket addresses:

```text
wss://chatgpt.com
wss://ws.chatgpt-staging.com
wss://ws.chatgpt.com
```

This shows that the Webview layer now permits connections to these real-time channels. The evidence package can only prove that CSP allows them; it cannot further prove the specific feature entry point or invocation path.

### 8. Camera Permission Description Changes from Generic Access to Video Input [Confirmed]

In Info.plist, `NSCameraUsageDescription` changes from the generic “This app needs access to the camera” to the more specific “Codex needs access to your camera for video input.”

This affects the text users see when macOS prompts for camera permission. The change more clearly explains camera access as being for “video input”.

Key evidence:

```text
NSCameraUsageDescription:
This app needs access to the camera
→ Codex needs access to your camera for video input.
```

### 9. Version / Build Info Copy in the Localized “About” Window Is Tweaked [Confirmed]

Multiple `native-menu-locales` localization files update the wording for version, build information, and release date in the About window. In Chinese, `codex.aboutDialog.buildInfoLabel` changes from “构建信息” to “版本和构建信息”.

This is a user-visible but low-risk copy revision, mainly affecting the “About Codex” window and wording in some localized languages.

Key evidence:

```text
zh-CN: 构建信息 → 版本和构建信息
```

### 10. macOS Cleanup Flow from DMG Installation to Applications Adds Retries [Confirmed]

In `.vite/build/bootstrap.js`, the logic handling source DMG unmounting and cleanup adds retry-related constants: `25` attempts, with an interval of `250` each time. The failure log also changes from a single unmount failure to failure after multiple retries.

This points to a more robust installation cleanup flow: when the app starts from a DMG and attempts to move itself to Applications, system unmounting of the image may not always succeed immediately, so the new version retries multiple times.

Key evidence:

```text
C = 25
w = 250
Failed to detach app source DMG after retries
```

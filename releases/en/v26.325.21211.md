# Codex Desktop `26.325.21211` build 1255

> Third-party changelog · Unofficial · Based on reverse diffs of official update artifacts

The move from `26.324.21641` to `26.325.21211` is small in raw size, but the direction is clear: startup and update plumbing are being reshaped for multiple build flavors and Windows updater paths, while remote control, heartbeat automation, permission modes, and connector settings show visible movement.  
English one-liner: this build appears to reorganize startup/update plumbing while surfacing signals for heartbeat automation, remote control, permissions, and connector privacy flows.

## About This Report

This is not an official changelog. It is a reverse-diff report based on official distribution artifacts. This entry is a historical backfill: the official appcast did not provide a delta package for this version pair, so the comparison was made directly between two official full packages.

The comparison covers macOS arm64, from `26.324.21641` build `1228` to `26.325.21211` build `1255`. Both builds are dated `2026-03-26`.

Evidence is grouped into two levels:

- **[Confirmed]**: direct human-readable evidence such as readable config, diffs, Info.plist values, package.json, CSP, service entries, signature identifiers, dependency lists, and newly observed main-process bundle strings.
- **[Signal]**: indirect evidence from frontend module names, changed readable strings in same-name frontend modules, style/state classes, or binary size shifts. These point in a direction but cannot reconstruct source-level intent.

Noise from hashed filenames under `webview/assets` and `.vite/build` has been ignored. The app.asar internal count of `598 added / 600 removed` is dominated by Vite hash renames and is not treated as real feature churn.

## ✨ Highlights

### 1. Startup Now Uses Build Flavor, Isolated userData Paths, and Windows AppUserModelId [Confirmed]

The new `.vite/build/bootstrap.js` no longer initializes only around macOS packaged state. It resolves a product/build flavor, sets the app name, computes a flavor-specific `userData` path, and sets AppUserModelId on Windows.

The practical implication is clearer isolation between desktop runtime variants. In particular, the `Agent` runtime can use a separate `agent/<run_id>` data directory via `CODEX_ELECTRON_AGENT_RUN_ID`.

Key evidence:

```diff
+n.app.setName(e.n(s)),n.app.setPath(`userData`,i(s)),process.platform===`win32`&&n.app.setAppUserModelId(e.a(s));
```

### 2. The Updater Is Moving From a Single Sparkle Script Toward a Cross-Platform Abstraction [Confirmed]

`package.json` removes the `build:sparkle-native` script and adds `codexSparklePublicKey`. New main-process strings include Sparkle addon paths as well as Windows updater, Windows update manifest, package identity, package URL, and updater unavailable paths.

The user-visible angle is that macOS still carries the Sparkle feed, but update initialization, checking, and installation failure paths are being folded into a broader updater abstraction. Windows updater error messages are now present in the product artifact.

Key evidence:

```diff
-    "build:sparkle-native": "node ./scripts/build-sparkle-native.mjs",
+  "codexSparklePublicKey": "rhcBvttuqDFriyNqwTQJR3L4UT1WjIK4QxtwtwusVic="
```

New strings include:

```text
Failed to load native Windows updater addon
Windows update manifest is missing required fields.
Windows update package URL must use https transport.
failed to initialize sparkle addon
```

### 3. Heartbeat Automation Reaches the Main Process and Frontend State Layer [Confirmed]

The main process now contains multiple `Heartbeat automation` strings covering feature disabled, not due yet, missing thread, renderer/thread state blocked, timeout, busy state, and active heartbeat cases. The frontend `index` module also adds accessibility copy around heartbeat interval, automation type, and pinned thread selection.

This points to an automation mechanism for periodically checking or surfacing thread attention. The strings prove that these states and error paths exist in the artifact, although the full trigger conditions and UI flow cannot be fully reconstructed from minified output.

Key evidence:

```text
Heartbeat automation skipped: not due yet
Heartbeat found something worth your attention.
That thread already has an active heartbeat.
Timed out waiting for heartbeat automation.
```

### 4. Remote Control Environment Support Expands [Confirmed]

The new main process includes strings for remote control websocket payloads, JSON-RPC transport, desktop app-server client authentication, ChatGPT login forwarding, VS Code remote open, and apply patch helpers. The frontend adds the real module `use-remote-connection-states`, and the settings module adds copy for `Remote control environments`.

This suggests remote control environments are moving beyond SSH connection error handling into environment listing, connection, authentication, and capability negotiation. The artifact explicitly contains “not supported yet” paths for ChatGPT login forwarding, VS Code remote open, and apply patch helpers, which means those boundaries are modeled too.

Key evidence:

```text
connect remote control environments
load remote control environments
Remote control transport expected JSON-RPC objects.
Remote control transport requires a desktop app-server client for authentication.
```

### 5. Permission Modes Become More Granular: Default, Custom, and Full Access [Signal]

The same-name `external-agent-config-utils` and `agent-settings` modules add permission-mode copy such as `Default permissions`, `Custom (config.toml)`, `Full access`, `Guardian approvals`, and `Full access is disabled by requirements.toml`.

This points to clearer agent permission configuration UI, likely separating global defaults, project overrides, custom `config.toml`, and full access controls. Because this comes from changed frontend modules, it only indicates UI, state, and configuration direction; it is not source-level fact.

Key evidence:

```text
Default permissions mode
Custom config.toml settings
Enable full access?
Full access is disabled by requirements.toml
```

### 6. Connector and App Settings Add Privacy, Permission, and Risk Copy [Signal]

The `plugins-page` module grows from `0.2 MB` to `0.3 MB` and adds connector privacy, site risk, permission scope, personalization toggle, app actions menu, load failure, and empty-state copy. At the same time, `control-group` removes some of the same copy, suggesting that this UI may have moved out of a shared control and into the plugins page.

This points to a connector flow that puts more emphasis on privacy boundaries and risk messaging, but it remains a same-name frontend module signal and should not be treated as source-level fact.

Key evidence:

```text
Connectors may introduce risk
ChatGPT is strictly limited to permissions you
By default, we do not train on your data. Data from this connector may be used to provide you relevant and useful information.
```

### 7. Markdown and Syntax Highlighting Assets Are Redistributed Across Language Modules [Signal]

The `markdown` module grows from `0.0 MB` to `0.7 MB`. Modules such as `javascript`, `csharp`, `yaml`, `html`, `r`, `graphql`, and `tex` add many grammar scope strings, while `wasm`, `typescript`, and `less` shrink.

This points to repackaging or expansion around Markdown rendering, math rendering, and code highlighting resources. The evidence is mainly frontend module size and grammar-string churn, so it does not prove exactly which languages or rendering capabilities became user-visible.

### 8. SQLite Rebuild Is Added to the Development Startup Path [Confirmed]

In `package.json`, the `dev` script now runs `pnpm run rebuild:sqlite`. That step was already present in `test`, `forge:package`, `forge:make`, and related paths.

This reduces native module drift between development startup and packaged/tested environments.

Key evidence:

```diff
-    "dev": "pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
+    "dev": "pnpm run rebuild:sqlite && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
```

### 9. Frontend Module Boundaries Add VS Code, Windows Sandbox, and Toast Signals [Signal]

After removing hash noise, the new real frontend modules include `vscode-api`, `windows-sandbox-queries`, `toast-signal`, `use-platform`, `use-remote-connection-states`, and `code-theme`. Removed modules include `playwright`, `product-event-signal`, `use-os-info`, and `with-window`.

These module names point toward platform abstraction, VS Code APIs, Windows sandbox queries, remote connection state, and toast signaling. Module presence does not mean the feature is exposed to users, and it cannot reconstruct source-level intent.

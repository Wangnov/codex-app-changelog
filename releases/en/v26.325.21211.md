# Codex Desktop `26.325.21211` (build 1255)

> Third-party changelog · Unofficial · Based on reverse-diffing official update artifacts

Judging by file size, this is a relatively small update, but there are readable changes in the startup entry point and frontend module boundaries.  
One-liner: a small desktop update with clearer product/runtime bootstrapping and signals around platform and remote-connection surfaces.

## About This Report

This is not an official changelog, but a **reverse-diff report based on official distribution artifacts**. This pass is a historical backfill: the fact pack states that both versions have fallen outside the official Sparkle delta window, so two official full packages were downloaded and compared directly instead of reconstructing from delta packages.

The comparison covers the macOS arm64 platform, from `26.324.21641` build `1228` to `26.325.21211` build `1255`; both versions were released on `2026-03-26`.

In terms of confidence, both artifacts come from official full-package URLs and include SHA-256 hashes. Notarization verification returned `CSSMERR_TP_CERT_REVOKED` for both. The fact pack specifically notes that this affects whether the version can be safely installed by the system, but does not affect differential analysis of the bundle contents.

This report uses two evidence levels:

- **[Confirmed]**: Direct evidence from Info.plist, package.json, HTML / CSP, readable diffs, and similar sources.
- **[Signal]**: Evidence from frontend module names or binary size changes; it can only indicate a directional signal and cannot reconstruct source-level intent from the artifact.

## ✨ Highlights

### 1. The startup entry point now sets app name and data directory by product form [Confirmed]

The startup logic in `bootstrap.js` no longer initializes only by current platform and packaging state. It now adds product-name resolution, app-name setting, user-data path setting, and Windows AppUserModelId setting.

This means the desktop app now more explicitly distinguishes different runtime forms or product contexts at startup. User-visible effects may appear as isolation across different instances, data directories, or system-level app identities.

Key evidence:

```diff
+n.app.setName(e.n(s)),n.app.setPath(`userData`,i(s)),process.platform===`win32`&&n.app.setAppUserModelId(e.a(s));
```

The new user-data path logic also reads `CODEX_ELECTRON_AGENT_RUN_ID` and, in Agent scenarios, places the path under `agent/<run id>`:

```diff
+let a=process.env.CODEX_ELECTRON_AGENT_RUN_ID?.trim()||null;
+return r===e.u.Agent&&a!=null?(0,t.join)(i,`agent`,a):i
```

### 2. Update-flow configuration is consolidated: Sparkle native build script removed, public key configuration added [Confirmed]

In `package.json`, the `build:sparkle-native` script was removed, while a new `codexSparklePublicKey` field was added. The `dev` script also now runs `rebuild:sqlite` first.

This indicates that update-related configuration has moved from a script-based form for locally building the Sparkle native module toward carrying the Sparkle public key in package configuration. Meanwhile, the development startup chain places more emphasis on rebuilding the SQLite native module.

Key evidence:

```diff
-    "build:sparkle-native": "node ./scripts/build-sparkle-native.mjs",
+  "codexSparklePublicKey": "rhcBvttuqDFriyNqwTQJR3L4UT1WjIK4QxtwtwusVic="
```

And:

```diff
-    "dev": "pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
+    "dev": "pnpm run rebuild:sqlite && pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development electron-forge start",
```

### 3. Frontend module signals point to platform adaptation, remote connection state, and Windows sandbox [Signal]

After removing Vite hash noise, the truly new modules include `use-platform`, `use-remote-connection-states`, `windows-sandbox-queries`, `vscode-api`, `toast-signal`, and `code-theme`.

These module names point toward platform detection, remote connection state, Windows sandbox, VS Code API, toast messaging, and code theme areas. However, source-level intent cannot be reconstructed from the artifact, and this does not prove that the related features have already been exposed to users.

Key evidence:

```text
新增模块: code-theme, toast-signal, use-platform, use-remote-connection-states, vscode-api, windows-sandbox-queries
```

### 4. Frontend preload entry shifts from `app-scope` to `vscode-api` [Signal]

In the modulepreload list of `webview/index.html`, the old `app-scope` entry was removed, and the new version includes `vscode-api`. This is a change in the dependency shape of the frontend entry point.

It suggests that the webview side may be abstracting application context or editor integration boundaries, but source-level intent cannot be reconstructed from the artifact.

Key evidence:

```diff
-    <link rel="modulepreload" crossorigin href="./assets/app-scope-BIXChNbS.js">
+    <link rel="modulepreload" crossorigin href="./assets/vscode-api-BtEtzXmw.js">
```

### 5. The native `sparkle.node` file was removed from the unpacked directory [Confirmed]

The bundle file tree shows that `Contents/Resources/app.asar.unpacked/native/sparkle.node`, about `0.1 MB`, was removed. Together with the removal of the `build:sparkle-native` script in `package.json`, this is direct evidence of a layout change in update-related native components.

This does not mean that “auto-update was removed”: the same `package.json` still retains `codexSparkleFeedUrl` and adds `codexSparklePublicKey`. A more accurate description is that the Sparkle-related implementation or packaging method changed.

Key evidence:

```text
移除: Contents/Resources/app.asar.unpacked/native/sparkle.node (0.1 MB)
```

### 6. Security policy keeps a very narrow network connection surface [Confirmed]

In the new `webview/index.html`, the CSP still limits `connect-src` to itself and two OpenAI-related domains: `https://ab.chatgpt.com` and `https://cdn.openai.com`.

From this entry file, the explicit network connection allowlist for the webview has not expanded to additional domains.

Key evidence:

```html
connect-src 'self' https://ab.chatgpt.com https://cdn.openai.com;
```

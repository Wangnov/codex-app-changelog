# Codex Desktop `26.519.22136`(build 3003)

> Third-party changelog · Unofficial · Based on reverse-engineered diffs of official update packages

The clearest changes in this Codex release focus on the Chrome automation plugin: there is readable evidence for installation, detection, tab takeover, and cleanup semantics.  
English one-liner: Chrome automation now has clearer multi-profile diagnostics and a more self-contained host setup.

## ✨ Highlights

### 1. Chrome Plugin Version Now Aligns with the Desktop App, with Additional Extension Host Configuration [Confirmed]

The Chrome plugin’s `.codex-plugin/plugin.json` version changed from `0.1.7` to `26.519.22136`, matching the desktop app’s short version. More importantly, `installManifest.mjs` no longer writes only the Native Messaging host manifest; it also writes `extension-host-config.json`, containing runtime configuration such as `channel: "prod"`, `codexCliPath`, `nodePath`, `nodeReplPath`, `browserClientPath`, `proxyHost: "127.0.0.1"`, and `proxyPort: 0`.

This means the Chrome extension host installation path now carries more explicit context. For future troubleshooting of issues such as “the extension is installed but the host is unavailable,” the artifact layer now provides more complete path and configuration evidence. Newly added files support this as well: the Chrome plugin directory now includes `app-server-runtime/codex`, `app-server-runtime/node`, and `app-server-runtime/node_repl`.

Key evidence: `"version": "0.1.7"` → `"version": "26.519.22136"` in `plugin.json`; new logic in `installManifest.mjs` that writes `extension-host-config.json`.

### 2. Chrome Extension Installation Detection Expanded from a Single Profile Directory to Multi-Profile Scanning [Confirmed]

The behavior of `check-extension-installed.js` changed clearly: the old version checked only the selected Chrome profile, while the new version scans every usable `Default` or `Profile X` directory and returns a complete `profiles` array in its JSON output, along with `selectedProfileDirectory`.

This is more visible for users with multiple Chrome profile directories: the tool can now distinguish between “the extension is installed in some profile directory” and “the profile directory Chrome will actually launch with is usable.” The top-level `installed`, `enabled`, and exit code still reflect the selected profile directory, allowing automation diagnostics to determine more accurately whether the profile directory currently intended for use is ready.

Key evidence: the skill docs changed from “checks whether the selected Google Chrome profile” to “checks every usable Google Chrome profile,” and explicitly state that the JSON output includes `profiles` and `selectedProfileDirectory`.

### 3. Chrome Tab Claiming and Cleanup Semantics Changed to “Release Control Afterward, Preserve Position Where Possible” [Confirmed]

The Chrome skill docs state that `browser.user.claimTab(tab)` now gains control without moving the tab into the agent tab group, and returns a controllable `Tab`. The old docs said it would move the selected Chrome tab into the current agent tab group.

Tab finalization semantics also changed: `deliverable` tabs are no longer described as being moved to a shared `✅ Codex` tab group, but as being “left open after the current browser session releases them”; `handoff` tabs likewise emphasize releasing control and leaving them in place. This suggests automation will make fewer changes to users’ existing tab grouping. At least as defined in the skill docs, handoff behavior now leans more toward “release control” than “regroup.”

Key evidence: in the skill docs, `Claiming moves...` changed to `Claiming gives... without moving...`; `deliverable` changed from “move to shared ✅ Codex tab group” to “left open after the current browser session releases them.”

### 4. Browser and Chrome Automation Skill Docs Synchronously Adjusted Interaction Constraints [Confirmed]

Both the Browser and Chrome skill docs removed the restriction “do not use `tab.playwright.waitForTimeout(...)`,” while the comment for `FinalizeTabsOptions.keep` changed from “tabs to keep open” to “explicitly retained tab disposition after cleanup.”

This only proves a documentation-level constraint changed. It does not imply that the full upstream Playwright API is available. The user-visible point mainly affects developers and automation workflow authors: descriptions of tab cleanup and waiting behavior are now less absolute.

Key evidence: the same `waitForTimeout` prohibition was removed from both `plugins/chrome/skills/chrome/SKILL.md` and `plugins/browser/skills/browser/SKILL.md`.

### 5. Frontend Module Names Indicate Continued Expansion of Main UI, Plugins, Workspaces, and Review Capabilities [Signal]

After filtering out Vite hash noise, the real newly added frontend modules listed in the fact pack point in several directions: app shell and window management, such as `app-shell-tab-controller`, `app-window`, and `thread-app-shell-chrome`; plugin and MCP capabilities, such as `plugin-auth`, `plugin-config-edits`, `mcp-capability-view-page`, and `use-plugin-install-flow`; Git and review workflows, such as `git-branch-switcher`, `diff-stats`, `parse-diff`, `review-mode-content`, and `pull-request-open`; as well as modules related to the input box, suggestions, queued messages, and pinned threads.

These module names suggest that the interface and workflows continue to be split out and expanded, but source-level intent cannot be reconstructed from the artifact, nor do they prove these capabilities are available to all users.

Key evidence: the fact pack’s list of newly added modules “after removing hash noise” includes the module names above; it also explicitly states that “module names are not obfuscated and can indicate direction, but appearance ≠ user availability.”

### 6. Size Growth Is Concentrated Mainly in app.asar, the codex Binary, and Chrome Runtime-Related Components [Signal]

File-level statistics show `Contents/Resources/app.asar` increasing from 147.7 MB to 159.0 MB, up 11.33 MB; `Contents/Resources/codex` increasing from 190.9 MB to 192.1 MB, up 1.13 MB. The Chrome plugin also adds relatively large `app-server-runtime/codex`, `app-server-runtime/node`, and `app-server-runtime/node_repl` files.

This indicates that packaged content and runtime components grew in this release, directionally echoing the strengthened Chrome extension host configuration. But binary size changes alone are only signals and cannot reconstruct source-level intent from the artifact.

Key evidence: the size change table and added-file excerpt in the Bundle file tree overview.

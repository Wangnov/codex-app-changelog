# Codex Desktop 26.519.22136 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

## ✨ Highlights

### Chrome Plugin Version Aligned With Desktop Version, With Bundled App Server Runtime Config
[Confirmed] [macOS only]

The macOS `chrome` plugin version changed from `0.1.7` to `26.519.22136`. The same package also adds `plugins/chrome/app-server-runtime/codex`, `node`, `node_repl`, and `docs/capabilities/tab/pageAssets.md`; `codex` is 192.1 MB, `node` is 118.4 MB, and `node_repl` is 15.1 MB.

`installManifest.mjs` no longer only writes the native messaging host manifest. The new version also writes `extension-host-config.json` with `schemaVersion: 1`, `channel: "prod"`, `codexCliPath`, `browserClientPath`, `nodePath`, `nodeReplPath`, `proxyHost: "127.0.0.1"`, and `proxyPort`. This shows the Chrome plugin package expanding from extension host installation into bundled App Server runtime configuration.

The Windows sample has identical SHA-256 before and after, with zero file, app.asar, or content changes, so this input cannot confirm the same change on Windows.

### Chrome Extension Install Check Now Scans Multiple Chrome Profiles
[Confirmed] [macOS only]

`check-extension-installed.js` changed from checking the selected profile to listing and checking every usable `Default` / `Profile X`. JSON output now includes a `profiles` array, `selectedProfileDirectory`, and `userDataDirectory`; each profile reports `installed`, `registered`, `enabled`, `disabled`, `versions`, paths, and disable reasons.

The skill documentation was updated accordingly: top-level `installed`, `enabled`, and exit code still reflect the selected profile, so troubleshooting can decide whether the Chrome profile that will actually launch is ready. The debug path now scans all usable profiles by default, with `CODEX_CHROME_USER_DATA_DIR` or `CODEX_CHROME_PREFERENCES_PATH` available to narrow the scope.

### Browser Tab Cleanup Semantics No Longer Emphasize Moving Pages Into a Codex Group
[Confirmed] [macOS only]

The Chrome skill documentation changed “claiming moves the chosen Chrome tab into the current agent tab group” to “the current browser session gets control without moving it into an agent tab group.” The `finalize` language also changed from deliverable tabs moving to a shared Codex tab group to being left open after control is released.

Both Browser and Chrome skill docs changed `FinalizeTabsOptions.keep` from “Tabs to keep open” to “Explicit tab dispositions to preserve after cleanup,” and describe `FinalizeTabsKeep.tab` as the tab object to preserve. This is a concrete documentation change in browser automation lifecycle semantics.

### App Startup Path Gains a New Conditional Gate
[Confirmed] [macOS only]

`.vite/build/bootstrap.js` now wraps `sparkleManager.initialize()` and `runMainAppStartup()` inside `if(await t.g())`. The fact package cannot recover the full meaning of `t.g()`, but it does confirm that main app startup is now behind a newly added condition in the macOS app bundle.

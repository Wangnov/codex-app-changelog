# Codex Desktop 26.527.31326 macOS / Windows Comparison

> Third-party cross-platform changelog · unofficial

The main confirmed change is bundled plugin version synchronization, while the two platform bundles show noticeably different frontend UI signals.  
One-liner: shared bundled plugin version bumps, with macOS surfacing more task/browser/Computer Use UI signals while Windows removes several matching UI traces in this sample.

## About This Report

This report uses only the provided fact packs: macOS compares 26.527.30818 build 3370 → 26.527.31326 build 3390, while Windows compares 26.527.3378.0 → 26.527.3686.0. The platform releases and version numbers are not perfectly aligned, so every conclusion is attributed to its own platform facts rather than projected across platforms.

Evidence has two levels. [Confirmed] means plist, manifest, plugin list, dependency list, service entry, readable string, or key file diff evidence. [Signal] means same-name frontend module, size delta, CSS class, or compressed bundle directional evidence only.

Platform labels are [macOS+Windows], [macOS only], and [Windows only]. webview/assets and .vite/build hash renames are treated as build noise per the fact packs and are not counted as functional additions or removals.

## ✨ Highlights

### [Confirmed][macOS+Windows] Bundled Browser and Chrome plugins synchronized to 26.527.31326

On both platforms, the `browser` plugin manifest version changes from `26.527.30818` to `26.527.31326`. Its description continues to cover the in-app browser, localhost / 127.0.0.1 / file:// targets, screenshots, clicking, typing, testing, and the rule that explicit @browser / @browser-use requests should not be silently satisfied with macOS `open` or generic web browsing.

On both platforms, the `chrome` plugin manifest also moves from `26.527.30818` to `26.527.31326`. Its description remains focused on automation for remote URLs, authenticated or profile-dependent pages, existing Chrome tabs, cookies, extensions, and Codex Chrome Extension setup.

Platform difference: macOS only shows `browser` and `chrome` manifest version changes. Windows also shows a `computer-use` plugin version change and an owl shell installation record change.

### [Confirmed][Windows only] Windows Computer Use plugin synchronized, with refreshed owl shell install metadata

In the Windows sample, `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` changes version from `26.527.30818` to `26.527.31326`. Its description says it controls Windows desktop apps from Codex through Computer Use.

The same Windows sample updates `.installed-owl-shell.json` `installedAt` from `2026-05-29T16:39:46.972Z` to `2026-05-29T21:44:41.484Z`. Its `cacheKey` points to `owl-b7c292defc716f761f309794f6e96fd278525d4e-codex-win32-x64.zip`, and `archiveSha` is `dd57bb72e98b5fbfa35820db4c05bcaa67c0f6a17372696485d741fc0993d85b`.

Platform difference: the macOS input does not provide Computer Use plugin manifest or owl shell record changes, so the same change cannot be claimed for macOS.

### [Signal][macOS only] Task menu, cloud tasks, and environment filtering signals expanded

The macOS `header` module grows by 27,874 bytes and adds readable strings for the recent tasks menu, `All tasks`, `Cloud tasks`, `All environments filter option`, clearing search, environment search failure, and environment listing failure. Added class and state names such as `aria-live`, `aria-atomic`, and multiple layout gaps point toward a searchable, filterable task/environment entry with error states.

Platform difference: the Windows `header` module moves in the opposite direction and removes the same recent tasks, cloud tasks, environment filter, and clear search strings. In this Windows sample, that UI direction is a removal signal.

### [Signal][macOS only] @ mention menu gains Computer Use, Custom agents, and desktop app capability groups

The macOS `at-mention-list` module adds `Computer use`, `Custom agents`, `Desktop apps`, `Live agents`, `Mac apps`, `No results`, plus app capability mention server / scope section header text. Together these point to capability mentions in the composer, potentially organized by current app capability, external agents, or desktop app capability.

Platform difference: the Windows same-name module removes `Computer use`, `Custom agents`, `Desktop apps`, `Live agents`, and `Mac apps`, so this sample does not support saying Windows gained the same entry. It only supports saying the corresponding Windows package signal was removed.

### [Signal][macOS only] Browser Use settings expand toward history access, domain allow/block rules, and upload/download domains

The macOS `browser-use-settings` module grows by 61,356 bytes and adds strings including `Access history without asking`, accessible labels for embedded extension / password manager / site settings, and `Add allowed domain`, `Add blocked domain`, `Add allowed download domain`, `Add blocked download domain`, and `Add allowed upload domain`.

This indicates more granular browser permission or site policy setting signals in the macOS package, especially around history access, extension settings, password manager settings, site settings, and upload/download domain allow/block rules.

Platform difference: the Windows input does not provide a same-name settings module change, so the same settings cannot be inferred for Windows.

### [Signal][macOS only] Markdown rendering and citation interactions expand across images, videos, file citations, and selected text

The macOS `markdown` module grows by 63,388 bytes and adds strings for markdown image buttons, image loading placeholders, image failure fallbacks, video fallbacks, no-alt-text accessible labels, extensionless file citation chips, file citation chips with line information, `Add to chat`, and a button label for adding currently selected overlay text to chat.

These signals point to media loading states, citation chip accessibility, and direct selected-text-to-chat interaction inside markdown content. Because they come from same-name frontend module change clues, this is treated as UI/interaction signal rather than proof that the full feature is visible to all users.

Platform difference: the Windows input does not provide a `markdown` module change.

### [Signal][macOS only] Thread side panel adds CI diagnosis workflow prompts and PR / staging copy

The macOS `thread-side-panel-tabs` module grows by 291,755 bytes and adds strings such as `A pull request already exists for this branch`, `Accept edits to stage them`, plus CI handling instructions: first check whether installed skills or tools can access the CI system, then check credentials, permissions, tokens, MCP, or tool access, stop and tell the user what is missing when blocked, otherwise fetch external logs, diagnose, and make the smallest safe fix.

The readable strings themselves are confirmed to be inside the package, but because they are surfaced through same-name frontend module change clues, this item is classified overall as [Signal]. It strongly points to the thread side panel carrying PR, staging, and CI repair context.

Platform difference: the Windows input does not provide a `thread-side-panel-tabs` change.

### [Signal][macOS+Windows] Thread side panel bottom panel / new tab action traces removed

Both macOS and Windows `thread-app-shell-chrome` modules remove `Bottom panel`, action labels for opening side chat, browser tab, review tab, timeline tab, terminal, workspace file browser, and `(max-width: 920px)`. Both also remove class or state names including `aria-pressed`, `data-thread-side-panel-new-tab-action-label`, and several panel layout classes.

Platform difference: both platforms move in the same direction as a removal signal, but this is only frontend bundle evidence and does not prove product-level removal.

### [Signal][macOS+Windows] App generation share dialog traces removed

Both platforms remove site share dialog strings from `appgen-share-dialog`, including user/group search, saving button loading label, `Add people or groups`, workspace link, internet link, copying a live site URL, and opening a live site externally. Several layout classes are removed as well.

Platform difference: both platforms move in the same direction as share dialog UI copy removal signals.

### [Signal][macOS only] Plugin detail page removes add, enablement, and sharing copy

The macOS `plugin-detail-page` module shrinks by 130,713 bytes and removes `Add to Codex`, `Adding to Codex`, `App enabled`, `App disabled`, MCP server enablement toggle, section navigation, and workspace link sharing accessible labels.

Platform difference: the Windows input does not provide a `plugin-detail-page` change. On macOS this is a plugin detail page UI copy removal signal, not proof that plugin system capability was removed.

### [Signal][Windows only] Debug modal removes debugging, account, host, and worktree cleanup copy

The Windows `debug-modal` module shrinks by 171,302 bytes and removes strings including `(command unavailable)`, `Account ID`, `Active browser tab`, `Active conversation`, `Allow automatic worktree cleanup in this unpackaged build.`, `Allow host`, `Allow host starting`, and `Already current`. It also removes modal, status color, and backdrop classes.

Platform difference: the macOS input does not provide a `debug-modal` change.

### [Signal][Windows only] Worktree initialization failure recovery copy removed

The Windows `worktree-init-v2-page` module shrinks by 5,023 bytes and removes strings for continuing after local environment setup failure, opening local environment settings, starting a repair thread, canceling worktree setup and starting a local conversation, `Continue anyway`, and progress copy for creating a worktree / forking a conversation.

Platform difference: the macOS input does not provide a `worktree-init-v2-page` change.

macOS adds `aria-label`, `aria-expanded`, `aria-haspopup`, `aria-pressed`, `aria-live`, `aria-atomic`, and related class or state clues across thread side panel, markdown, browser settings, mention list, and header modules. Boundary: these are style/state signals that suggest UI structure changes, not source implementation facts.

Windows removes similar aria and layout classes across debug modal, mention list, header, thread shell, and share dialog modules. Boundary is the same.

### Browser device toolbar

macOS `thread-side-panel-tabs` adds accessible labels for browser device toolbar width, height, zoom select, and close mode. Boundary: this is package-included UI string evidence, but it does not by itself prove the device toolbar is available to all users.

### File citation and media preview

macOS `markdown` adds line information citation chip, extensionless file citation chip, markdown image/video fallback, placeholder, preview trigger, and selected text overlay signals. Boundary: readable strings prove the copy entered the package; module context only provides UI direction signal.

### Plugin and sharing entry contraction

macOS `plugin-detail-page` removes Add to Codex, app enablement status, MCP enablement toggle, and workspace link sharing copy. Both platforms remove site sharing copy from `appgen-share-dialog`. Boundary: this describes bundled copy changes, not permission model or backend capability changes.

### Windows local environment and debug panel contraction

Windows `debug-modal` and `worktree-init-v2-page` both remove debugging, host, worktree cleanup, local environment repair, and continue-flow copy. Boundary: this is a grouped Windows frontend module removal signal and cannot be generalized to all local environment flows.

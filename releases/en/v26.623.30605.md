# Codex Desktop 26.623.30605 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

This batch looks like capability-surface consolidation: Sites, browser automation, scheduled tasks, PR workflows, remote devices, and worktree paths all get denser across both packages, while Windows shows extra local-system and settings surfaces.  
One-liner: Sites, browser automation, scheduled tasks, PR review, remote devices, and worktree flows moved forward cross-platform, with Windows showing additional local-system and settings surfaces.

## About This Report

The comparison range is macOS arm64 `26.616.81150 build 4306` → `26.623.30605 build 4415`, and Windows x64 `26.616.10790.0` → `26.623.3245.0`. Both come from the supplied fact packages, but their version and build numbers differ, so they are cited separately.

Evidence levels: [Confirmed] means plist, manifest, skill docs, plugin metadata, service entry files, dependency manifests, readable string literals, or readable diffs. [Signal] means same-name frontend module deltas, module names, CSS/state classes, or size direction. Platform tags are `[macOS+Windows]`, `[macOS only]`, and `[Windows only]`.

Hashed `webview/assets` and `.vite/build` renames are treated as build noise. The fact packages do not include a CSP diff section. The Windows package also does not include an AppxManifest diff section, so no claim is made there.

## ✨ Highlights

### [Confirmed][macOS+Windows] Sites plugin upgraded with terms and ChatGPT sign-in template support

The Sites plugin moves from `0.1.16` to `0.1.21`. Its description, long description, and `termsOfServiceURL` now point to the ChatGPT Sites Terms and add responsibility language for website content.

The `vinext-starter` README adds Optional Dispatch-Owned ChatGPT Sign-In: `getChatGPTUser()`, `requireChatGPTUser(returnTo)`, `chatGPTSignInPath(returnTo)`, and `chatGPTSignOutPath(returnTo`. It also says `/signin-with-chatgpt`, `/signout-with-chatgpt`, and `/callback` are Dispatch-owned, and that SIWC establishes identity but not workspace membership.

The Sites hosting skill adds plan continuity, preview handoff, and deployment thumbnail rules: hosting should continue the `sites-building` plan, attempt `open_in_codex` preview handoff before deployment, and treat `public/screenshot.jpeg` as an optional deployment thumbnail that is not created, refreshed, or deleted by default.

### [Confirmed][macOS+Windows] Browser / Chrome automation docs and browser context menus expanded

The Browser plugin manifest version changes from `26.616.81150` to `26.623.30605` on both platforms. Playwright docs in both Browser and Chrome plugins were structurally adjusted, with several sections promoted from `###` to `##`, while preserving guidance on snapshot discipline, locator uniqueness, and the limited Playwright API.

Native menu locale diffs add browser context menu items for opening in an external browser, opening in a browser, and opening a link in a new tab. Update confirmation strings also move from hardcoded Codex to `{appName}`. Main-process strings add Browser page snapshots, navigation-history restore, PiP metadata, sidebar storage ownership, `Quick annotate`, `Selected text`, `Tab persistence is only available with Chrome or multi-tab IAB.`, and WebMCP JSON-serialization constraints.

### [Confirmed][macOS+Windows] Built-in plugin metadata and visual assets changed

The LaTeX plugin moves from `0.2.3` to `0.2.4`; its logo switches from `latex-logo.svg` to `assets/latex.png`, and `logoDark` is added. Sites and Browser manifest changes are visible on both platforms.

The Windows fact package additionally shows Chrome and Computer Use plugin manifest versions moving from `26.616.81150` to `26.623.30605`. Windows also adds Browser / Chrome `api.json`, `unified-skill.md`, `browserAuth.md`, `installed-browsers.js`, `open-chrome-window.js`, `check-extension-installed.js`, and `check-native-host-manifest.js`, while removing older `api.md`, split multi-tab docs, and some troubleshooting docs.

### [Signal][macOS+Windows] Automations are shifting toward Scheduled tasks

The `automations-page` module adds strings for scheduled task/template search, a task detail right sidebar, switching between tasks and templates, pausing/resuming/deleting scheduled tasks, a new scheduled task dropdown, creating tasks from the side panel, and personalizing templates by chatting with Codex. Older automation overview, automation not found, and automation started strings decrease.

`local-conversation-page` and `local-conversation-thread` also replace “heartbeat automation” entry points with “scheduled task” entry points. These are same-name frontend module signals, so they do not prove all entry points are visible to users.

### [Confirmed][macOS+Windows] PR / GitHub validation paths grew, with frontend reviewer interaction signals

New main-process strings include `At least one reviewer is required`, GitHub user and pull request search failures, pull request head commit verification failure, missing or invalid PR state/URL/head revision, account mismatch, and head changed during diff acquisition. These confirm more PR and reviewer error paths are bundled.

The same-name frontend modules add [Signal] evidence for reviewer pickers, reviewer search, approved/pending/requested-changes statuses, fixing all failing PR checks, and fixing or removing all PR comments. Both platforms contain these signals.

### [Confirmed][macOS+Windows] Thread catalog, worktree migration, and app-server file I/O paths entered bundled errors

New strings include local thread catalog scan states, blocking prune during active scan, refusing to bind disposed catalog hosts, repairing missing thread cwd metadata, failure to move a thread to a worktree, app-server file read/write failures, ExecutionHost requirements, and unsupported exclusive copy across worker RPC or app-server hosts.

Windows `split-items-into-render-groups` adds [Signal] entries such as `move-thread-to-local`, `move-thread-to-worktree`, `worktree-create-managed`, `prepare-host-transfer`, `worktree-init`, and `Branch is already checked out in another worktree`, making the Windows frontend worktree/thread handoff surface more explicit.

### [Signal][macOS+Windows] Windows sandbox and limited-access onboarding strings are bundled cross-platform

Both packages’ `onboarding-page` modules add strings for an illustrative Windows UAC prompt, continuing Windows setup without administrator access, continuing in read-only mode or limited access without completing Windows sandbox setup, and “ChatGPT can suggest what to do next by searching project files and connected apps.” The feature target is Windows, but the strings exist in both macOS and Windows packages, so the package-level platform tag is `[macOS+Windows]`.

### [Confirmed][Windows only] Windows added native local-system integration files

Windows adds `resources/native/windows-account.node` at about `0.8 MB`, plus `resources/icon.ico`. `resources/owl-electron-app.json` changes `packagedFrom` from a GitHub Actions path to `D:\codex-apps-workspace\...`, and updates `runtimeArchiveSha`. `reading_mode_gdocs_helper_manifest.json` moves from `149.0.7827.115` to `149.0.7827.157`.

### [Signal][Windows only] Composer, project setup, PDF, browser settings, usage, and referral UI signals cluster on Windows

Windows-only same-name module deltas show several high-visibility UI directions: Composer adds Plan mode placeholder text, `/init` creation of `AGENTS.md`, skipping when it already exists, Windows sandbox readiness retry, safety buffering help links, and selected-text comment/edit actions. Local project setup adds source folder picking plus project marker colors and icons. The `pdf` module grows from `0.0 MB` to `0.4 MB` and adds PDF password and load-failure strings. Browser-use settings add history, downloads, password manager, site settings, blocked/allowed domains, and “Access history without asking.” Rate-limit and referral modules add credits, usage resets, invite tracking, and all-time/current-month periods. Home announcements add gift credits, Fast mode, multi-agent composer, and remote connections banners.

These are frontend module signals, not source-level confirmed behavior.

### [Signal][macOS+Windows] Remote connections move from pairing-copy toward signed-in device and secure transport states

`remote-connections-settings` adds strings for a remote Codex app-server version tooltip, online devices requiring a newer version, waiting for device, initializing secure connection, confirming connection, and signing in to ChatGPT before refreshing to add a device. Older manual pairing dialog and pairing-code strings decrease. Both platforms contain the signal.

### [Signal][macOS+Windows] Skills / Plugins wording moves from Add toward Install and imported setup

`plugins-page` adds Skills page sections, install plugin, install recommended skill, try installed plugin in chat, Finish setup, Imported plugins, and official app connectors suggested from imported external agent setup. Older Add skill, Adding plugin, and connected recommended skills strings decrease. This points to unified install language for plugins and skills, but remains a frontend signal.

### [Signal][macOS+Windows] Cloud Library upload and AppGen Library type filtering are expanding

`appgen-library-page` adds cloud Library file preview, a hidden upload input, `Add anything`, file upload, drag-and-drop to add any file to the library, MIME fallback text, and a generated images tab. Windows additionally has `Library tab that shows output files`; macOS has unavailable states for cloud Library files and thumbnails. The shared direction is Library moving from “continue chat/site settings” toward uploads, previews, and type filters.

### [Confirmed][macOS only] macOS packaging, Chromium baseline, and native resources changed

macOS `Info.plist` changes `CFBundleShortVersionString` from `26.616.81150` to `26.623.30605`, `CFBundleVersion` from `4306` to `4415`, and `ChromiumBaseVersion` from `149.0.7827.115` to `149.0.7827.157`; `ElectronAsarIntegrity` hash and `SCMRevision` also change. It adds `CodexDmgBackgroundFileName = codex-dmg-background.png` and `Contents/Resources/icon-codex-dark-color.png`.

The macOS file tree replaces the Chromium Framework version directory from `149.0.7827.115` to `149.0.7827.157`; `app.asar` grows by `+8.88 MB`. Computer Use app binaries such as `SkyComputerUseService`, `CUALockScreenGuardian`, and `SkyComputerUseClient` grow by about `+1.33 MB` to `+1.40 MB`, while total cua_node size remains `377.4 MB → 377.4 MB`. macOS also removes `realtime-voice-wake-word` and `WakeSingleHead.mlpackage` files; this only proves those native files disappeared from the macOS package, not that every voice feature was removed.

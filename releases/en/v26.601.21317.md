# Codex Desktop 26.601.21317 (macOS / Windows Comparison)

> Third-party cross-platform changelog · Unofficial

This batch looks like a cross-platform application-layer packaging refresh: macOS exposes additional runtime and plugin metadata, while Windows currently shows package-level and app.asar-level changes only.  
One-liner: A small cross-platform application-bundle refresh, with macOS exposing additional runtime and bundled-plugin version evidence.

## About This Report

This report is based on reverse diffs of official signed distribution packages: macOS via Sparkle-related full packages, Windows via MSIX packages. The comparison range is macOS `26.601.20914` build `3497` → `26.601.21317` build `3511`, and Windows `26.601.1994.0` → `26.601.2237.0`. The batch date is recorded as `2026-05-30` from the supplied fact packet.

Evidence has two levels: [Confirmed] means readable evidence such as plist, plugin metadata, dependency lists, or SHA-256 values; [Signal] means size, file-count, or frontend build-output movement only, which can show direction but not source intent. `[macOS+Windows]` means both platforms have comparable evidence, while `[macOS only]` and `[Windows only]` mean the change is only confirmed in that platform’s packet. `webview/assets` and `.vite/build` hash renames are treated as noise, not feature additions or removals.

## ✨ Key Changes

### Slight Application Archive Size Reduction [Signal][macOS+Windows]

Both platforms show the main application archive shrinking by `0.33 MB`: macOS `Contents/Resources/app.asar` goes from `141.6 MB` to `141.3 MB`, and Windows `resources/app.asar` goes from `145.4 MB` to `145.1 MB`. This confirms application-layer package movement, but size alone does not identify a specific feature change.

### One Fewer Internal App-Layer File [Signal][macOS+Windows]

Inside app.asar, macOS moves from `2804` to `2803` files, with `428` added, `429` removed, and `13` modified. Windows moves from `2752` to `2751` files, with `428` added, `429` removed, and `12` modified. The pattern is highly similar across platforms, but the fact packet explicitly marks heavy `webview/assets` churn as Vite build hash-renaming noise, so this should not be reported as a new frontend feature.

### No Expansion in cua_node Dependencies [Confirmed][macOS+Windows]

Both platforms report `0` newly added `cua_node` packages. This is a shared negative change: at least from the dependency-list evidence, this batch does not add to that native dependency group.

### macOS Package Metadata and Runtime Fingerprint Updated [Confirmed][macOS only]

On macOS, `Info.plist` shows the version moving from `26.601.20914` to `26.601.21317`, the build moving from `3497` to `3511`, and the `ElectronAsarIntegrity` app.asar hash changing from `8293044a58d98dd06f740b67b9cc68e26b0b74c476baa48caae1161cdd130356` to `97e10d2004ddc2988ee1a145a5950be9e90e3a4c7c9957961e278490b47c4fd3`.

The macOS `owl-electron-app.json` also changes `runtimeArchiveSha` from `9084289a644128f65ba44b97caa360ad87cc27291951f6bca141742ca4bb7035` to `c6a2b52fed12c736be826f39d524cc764b3bd232ee0727c92e44626e662e06fe`. The Windows packet does not provide an equivalent readable runtime-manifest diff, so the corresponding Windows state cannot be confirmed.

### Bundled browser/chrome Plugin Versions Follow the Main Version [Confirmed][macOS only]

In the macOS packet, the `.codex-plugin/plugin.json` version fields for `browser` and `chrome` both move from `26.601.20914` to `26.601.21317`. The readable diff only proves plugin metadata version alignment with the main app version; it does not prove a plugin capability change. The Windows packet does not include corresponding plugin metadata diffs, so this is recorded as `[macOS only]`.

### No Readable Windows-Only Functional Evidence [Confirmed][Windows only]

The supplied Windows packet contains SHA-256 values, outer file-tree stats, app.asar stats, and the `cua_node` added-package count, but it does not list readable AppxManifest, skill documentation, type declaration, package.json, or plugin metadata diffs. Therefore this report does not claim any `[Windows only]` functional change.

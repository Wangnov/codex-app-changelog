# Codex Desktop 26.601.21317 macOS / Windows Comparison

> Third-party cross-platform changelog · unofficial

The common thread in this batch is productization around plugin entry points and thread-tool wording, plus a new packaged owl runtime on both platforms.  
One-liner: this cross-platform build shifts plugin discovery toward work plugins, refreshes thread-tool wording, and updates the packaged owl runtime.

## About This Report

This report only uses the machine-extracted facts in the input packages: macOS compares `26.601.20914 build 3497` → `26.601.21317 build 3511`, while Windows compares `26.601.1994.0` → `26.601.2237.0`. The two platform samples have different version strings and release-package contexts, so every item below is attributed to its own platform evidence rather than projected across platforms.

Evidence has two levels: `[Confirmed]` comes from manifests, plist fields, package configuration, service entries, skill descriptions, dependency lists, signing information, or readable strings outside the “same-name frontend module change clues” section; `[Signal]` comes from same-name frontend module names, size changes, readable-string diffs, and class-name diffs. Platform scope is marked as `[macOS+Windows]`, `[macOS only]`, or `[Windows only]`.

The evidence inventory covered version/channel data, Bundle file trees, bundled-plugin list changes, cua_node dependency changes, app.asar summaries, real frontend module additions/removals, same-name frontend module clues, newly readable main-process and frontend strings, Info.plist / AppxManifest changes, content security policy changes, new components/resources, and concrete key-file diffs. The input did not provide Computer Use runtime totals, CSP changes, AppxManifest changes, or new bundled-plugin details, so this report does not invent those conclusions.

## ✨ Highlights

### [Signal][macOS+Windows] Work-plugin announcement replaces the single Sales plugin announcement

Both platforms’ `app-main` added `work-plugins-announcement` copy, including labels for adding a plugin, already-added state, body copy, and phrases such as `Build deal materials faster`, `Create polished campaign visuals`, `Execute deals with confidence`, and `Prepare sales work faster`. The new short descriptions explicitly mention Creative Production, Investment Banking, and Product Design, while class names also include `data-analytics@openai-curated-remote`, `product-design@openai-curated-remote`, and `public-equity-investing@openai-curated-remote`.

Both platforms also removed `Add Sales plugin`, `Meet your new AI sales assistant`, the Sales plugin announcement title and primary-action key, and migrated the old `has-seen-sales-plugin-announcement` state toward `has-seen-work-plugins-announcement`. This points to a frontend announcement surface moving from a single Sales plugin toward a broader work-plugin set, but it remains same-name frontend module evidence and does not prove that all users can see the announcement.

### [Signal][macOS+Windows] The debug modal expands around App Server, sessions, browser tabs, and account state

The `debug-modal` grew from near 0 MB to about 0.2 MB on both platforms: +177,618 bytes on macOS and +175,508 bytes on Windows. New readable strings include `(command unavailable)`, `Account ID`, `Active browser tab`, `Active conversation`, `Allow host`, `Allow host starting`, `Already current`, `Ambient suggestion chats`, `App Actions`, and `App Server`. Windows additionally includes `App session ID`.

This points to a wider debug surface, especially around the app server, active conversation, browser tab, account state, and host-allow flow. The platform delta is small, with Windows adding one session-ID string.

### [Signal][macOS+Windows] Thread dynamic-tool copy removes “in the background”

Both platforms’ `app-server-dynamic-tools` removed `in the background` from thread-tool descriptions. The changed descriptions cover archiving/unarchiving, listing recent threads, pinning/unpinning, renaming a thread, and sending a follow-up prompt to an existing thread.

This is readable copy inside the frontend bundle and suggests “background thread operation” wording has been softened into ordinary thread-operation wording. Because it comes from same-name frontend module diffs, it is classified as a signal.

### [Signal][macOS+Windows] Chat creation status also drops “background chat”

Both platforms’ `split-items-into-render-groups` replaced `Creating background chat`, `Created background chat`, and related in-progress/completed labels with `Creating chat`, `Created chat`, and labels for creating a Codex chat.

This matches the dynamic-tool wording change: visible status text is moving away from “background chat” toward the more neutral “chat.” The evidence is still a frontend module signal.

### [Confirmed][macOS+Windows] Browser and Chrome bundled-plugin versions move to 26.601.21317

Both platforms’ `browser/.codex-plugin/plugin.json` changed `version` from `26.601.20914` to `26.601.21317`. The description still points to the Codex in-app browser, local addresses, file URLs, screenshots, clicking, typing, and testing.

Both platforms’ `chrome/.codex-plugin/plugin.json` also changed from `26.601.20914` to `26.601.21317`. Its description still targets remote URLs, authenticated/profile-dependent pages, existing Chrome tabs, cookies, extensions, and Codex Chrome Extension setup. This is manifest-level confirmed evidence, but the input does not show new capabilities for these plugins, only version metadata changes.

### [Confirmed][Windows only] Computer Use plugin version moves to 26.601.21317

Windows `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` changed from `26.601.20914` to `26.601.21317`, with a description for controlling Windows desktop apps from Codex.

The macOS key-file diff does not include a corresponding Computer Use plugin manifest change. That does not prove the plugin is absent on macOS; it only means this confirmed plugin-version evidence appears in the Windows fact package.

### [Signal][macOS only] Browser-use settings add site, download, upload, password-manager, and history-access entries

macOS `browser-use-settings` grew by 62,967 bytes and added readable strings including `Access history without asking`, embedded browser settings breadcrumbs, extension settings, password manager settings, site settings, show/hide controls for individual browser data-clearing options, a disabled toggle for unavailable plugins, plus `Add allowed domain`, `Add allowed download domain`, `Add allowed upload domain`, and `Add blocked domain`.

The Windows input does not list a corresponding module change. On macOS, this is a concentrated signal that browser-use permissions and site-rule configuration expanded, but it does not prove the settings are visible to every user.

### [Signal][macOS only] The @ mention menu adds Computer Use, agents, MCP servers, and Mac apps clues

macOS `at-mention-list` added `Computer use`, `Custom agents`, `Desktop apps`, `Live agents`, `MCP servers`, `Mac apps`, `No results`, plus copy for the current MCP server mention scope, file-search loading state, and empty-query state.

The Windows same-name module moves in the opposite direction: those strings were removed and the module shrank by 29,315 bytes. This is a clear platform split in the @ mention menu clues: macOS looks like expansion, while Windows looks like removal of the same set of entry strings.

### [Signal][Windows only] Plugin detail page removes add, share, enablement, and App unavailable clues

Windows `plugin-detail-page` shrank by 112,310 bytes and removed strings for included-resource action menus, removing a workspace share target, the MCP server enablement toggle, section navigation, `Add plugin`, `Adding plugin`, `Anyone in this workspace with the link`, and `App unavailable`, along with a set of detail-page class names.

The macOS input does not list a same-name module change. This means the Windows sample shows a clear contraction in plugin-detail-page bundle clues, but size and string diffs alone do not prove final interaction removal.

### [Signal][macOS+Windows] Plugin availability adds an admin-disabled state

Both platforms’ `plugins-availability` added `Disabled by admin`, a primary-button-label description for when an app is unavailable because an admin disabled it, and `settings.mcp.appConnectModal.disabledByAdminButton`. These strings entering frontend modules indicate an admin-disabled state in plugin connection/availability paths.

### [Signal][macOS+Windows] Home announcements and AppGen share-dialog clues contract

Both platforms’ `home-announcements` removed dismiss labels for the Fast mode home banner, backend-driven Codex app home banner, multi-agent composer banner, and remote connections banner, plus `Add Connections` and related banner body copy.

Both platforms’ `appgen-share-dialog` removed site-share dialog strings for searching workspace users/groups, removing selected users, the save-while-saving button, `Add people or groups`, workspace link, internet link, copying a live site URL, and opening a live site externally. These are same-name frontend module diffs, so they should be read as visible-interface direction signals, not confirmed feature removals.

### [Confirmed][macOS+Windows] owl runtime package changes, while app.asar shrinks by about 0.33 MB

macOS `Contents/Resources/owl-electron-app.json` changed `runtimeArchiveSha` from `9084289a...` to `c6a2b52f...`. Windows `resources/owl-electron-app.json` changed from `8ee5ac24...` to `76a82b58...`, and `.installed-owl-shell.json` also updated the archive URL, installedAt, and archiveSha.

The top-level app.asar shrank by 0.33 MB on both platforms: macOS `141.6 MB` → `141.3 MB`, Windows `145.4 MB` → `145.1 MB`. This is confirmed runtime and package-size evidence; the input does not include a runtime content diff, so no functional inference is added.

### [Confirmed][macOS only] Info.plist version and app.asar integrity hash update

macOS `CFBundleShortVersionString` changed from `26.601.20914` to `26.601.21317`, `CFBundleVersion` from `3497` to `3511`, and `ElectronAsarIntegrity/Resources/app.asar/hash` was updated. Both versions’ notarization checks were accepted, with source `Notarized Developer ID` and origin OpenAI OpCo, LLC.

The Windows input does not provide AppxManifest changes, so this report does not report Windows manifest-field changes.

## ✨ Highlights

### 1. Bluetooth permission metadata switched from Always to Peripheral [Confirmed]

Info.plist removed `NSBluetoothAlwaysUsageDescription` and added `NSBluetoothPeripheralUsageDescription`. Both use the same text: `This app needs access to Bluetooth`.

This means the app’s macOS Bluetooth permission metadata changed. From the artifact alone, we can confirm the key-name change, but not whether the underlying Bluetooth behavior changed.

Key evidence:

```diff
- /NSBluetoothAlwaysUsageDescription = "This app needs access to Bluetooth"
+ /NSBluetoothPeripheralUsageDescription = "This app needs access to Bluetooth"
```

### 2. Task-start failure messages were split out, especially cloud requirements loading failures [Signal]

The same-name `index` frontend module gained readable strings related to starting threads, cloud tasks, cloud requirements loading, and task resume failures, including `Error starting thread. Please log out and sign in again`, `composer.localTaskError.cloudRequirements.v2`, and `localTaskRow.resumeError.v2`.

These clues point to a cleanup of error presentation around task creation, startup, and resume paths, with “cloud requirements could not be loaded” exposed as a more specific failure reason. This is still inferred from same-name frontend module strings, so it cannot establish source-level intent.

### 3. Error-copy keys moved from conversation/task wording toward thread/task v2 [Signal]

Removed strings include `Error starting conversation`, `Error creating task`, `Failed to resume task`, and `composer.cloudTaskCreationError`; added strings include `Error starting thread...`, `composer.cloudTaskCreationError.v2`, `composer.cloudTaskError.v2`, and `composer.localTaskError.v2`.

This points to v2 variants of the relevant error states and localization keys, with frontend copy leaning more heavily on the thread concept. Because this is an indirect frontend-artifact signal, it does not prove a change in the underlying model or task-system protocol.

### 4. Bundle structure stayed essentially stable [Confirmed]

The outer bundle file count remained `415 → 415`, with `0` added files, `0` removed files, and `40` files changed. Major component sizes also barely moved: `Contents/Resources/app.asar` remained `41.7 MB`, shown as `-0.00 MB`.

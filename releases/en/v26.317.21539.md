# Codex Desktop `26.317.21539` build 1088

> Third-party changelog · Unofficial · Based on reverse diffs of official update artifacts

This build moves from `26.313.41514` to `26.317.21539`, with visible changes around startup failure handling, plugin management, local environment setup, PR / GitHub UI strings, and code-highlighting resource layout.  
One-liner: a small desktop update focused on startup fallback handling, plugin UI wording, environment setup affordances, and bundled grammar/resource reshuffling.

## About This Report

This is not an official changelog. It is a **reverse-diff report based on official distribution artifacts**. For this historical backfill, the official appcast did not provide a delta package for `26.313.41514` → `26.317.21539`, so both official full packages were downloaded and compared directly.

Evidence has two levels: **[Confirmed]** means there is direct human-readable evidence, such as `Info.plist`, `package.json`, HTML, startup scripts, readable strings, or signing verification output. **[Signal]** means the conclusion comes only from frontend module names, same-name module string diffs, or binary size changes, and cannot be reconstructed into source-level intent.

The comparison scope is macOS arm64: previous version `26.313.41514` build `1043`, released on `2026-03-17`; latest version `26.317.21539` build `1088`, released on `2026-03-19`.

## ✨ Key Changes

### 1. Startup Failure Now Has an Update / Quit Fallback Dialog [Confirmed]

The new `.vite/build/bootstrap.js` no longer only loads the main entrypoint. It now includes an error dialog path when the main app fails during startup. The dialog text includes `Codex failed to start.` and can offer `Install Update`, `Check for Updates`, or `Quit` depending on update state.

This means users may see a clearer recovery path if the main desktop app fails during startup, instead of a silent exit or logs only. Key evidence includes the new startup script branches `install-update`, `check-for-updates`, `quit`, and the log string `Desktop bootstrap failed to start the main app`.

### 2. The Plugins Page Appears to Move Toward Clearer Plugin Management [Signal]

The same-name `skills-page` frontend module adds strings such as `All plugins`, `Choose a plugin marketplace`, `Enable plugin`, `Disable plugin`, `Install plugin`, and `Failed to load plugins`, while removing older wording such as `All Apps`, `All skills`, `New app`, and `No apps found`.

This points toward a reshaped plugin directory and plugin enablement UI, but the artifact does not reveal source-level intent. The likely user-visible direction is a Plugins page more focused on marketplaces and enablement state rather than mixed Apps / Skills / Apps navigation.

### 3. Local Environment Settings Add Signals for Setup Environment Variables [Signal]

The `local-environments-settings-page` module adds `Available environment variables`, `Setup script environment variables`, `Source workspace path`, `New worktree path`, and `Button label that opens the setup env vars popover`.

This points to UI for viewing or using environment variables in local environment setup scripts, but the artifact does not reveal source-level intent. For developers, the visible direction is that source workspace and worktree paths may be exposed more explicitly to setup flows.

### 4. PR / GitHub Workflows Get More Accessibility and Failure-State Coverage [Confirmed]

The new main-process / frontend bundle adds readable strings `No git repository found for pull request board` and `gh.pr.board.failed`. The `index` module also adds text such as `Accessible label for a link that opens a pull request comment on GitHub` and `ARIA label for the hidden pull request board columns button`.

This shows new artifact-level coverage for PR board, PR comment links, and related failure states. In parallel, older strings such as `Failed to start PR comments workflow` and `Failed to start PR fix` were removed, indicating that related error paths or wording changed.

### 5. Markdown, Diff, and Multi-Language Highlighting Resources Were Significantly Reshuffled [Signal]

Several language modules show large size and syntax-token changes: `cpp` grows from `0.0 MB` to `0.6 MB`, `javascript` grows to `0.2 MB`, and `php` grows to `0.1 MB`, while `markdown` shrinks from `0.5 MB` to `0.1 MB` and `browser` shrinks from `0.7 MB` to `0.0 MB`.

These signals point to syntax highlighting or editor grammar resources being split, replaced, or repackaged, but the artifact does not reveal source-level intent. New tokens include `heading.1.markdown`, `markup.inserted.diff`, `comment.block.cpp`, `constant.other.description.jsdoc`, and `comment.block.documentation.phpdoc.php`.

### 6. The Frontend Adds `loading-page` and `popover` Modules [Signal]

After removing Vite hash noise, the real newly added frontend modules are `loading-page` and `popover`; removed modules are `arrow-left`, `regenerate`, and `sentry-tracing.electron`.

This points to frontend structural changes around a loading page and popover component, but module presence does not mean a feature is enabled for all users, and it cannot be reconstructed into source-level intent. `webview/index.html` also shows the new preload entries `popover-BZJE0yel.js` and `loading-page-Dr8JlrqP.js`.

### 7. Crash Minidump and Profiling Paths Add New Readable Strings [Confirmed]

The new bundle adds strings including `Could not delete minidump`, `Failed to load minidump`, `Timed out waiting for minidump to stop being modified`, `No Sentry client available, profiling is not started`, and `ProfilingIntegration is not available`.

This shows new readable handling text for desktop crash dump and profiling-related error paths. The likely visible impact is in diagnostics, telemetry, or error-reporting behavior rather than ordinary UI features.

### 8. The Build Toolchain Moves from Vite Beta to Stable [Confirmed]

In `package.json`, the dev dependency changes from `vite: 8.0.0-beta.15` to `vite: 8.0.0`; the package version changes from `26.313.41514` to `26.317.21539`, and `codexBuildNumber` changes from `1043` to `1088`.

This is a lower-level build-chain change that users normally would not notice directly, but it helps explain why many `webview/assets` hash filenames changed and should not be interpreted as feature additions or removals.

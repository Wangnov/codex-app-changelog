## ✨ Highlights

### 1. New actionable fallback dialog when startup fails [Confirmed]

The desktop startup entry `.vite/build/bootstrap.js` changed from simply loading the main process file to showing an error dialog when the main app fails to start, with action buttons offered based on update status.

This means that when users encounter a main-app startup failure, the process no longer merely exits abnormally; the new version can at least provide fallback paths such as “Install Update”, “Check for Updates”, or “Quit”. Key evidence includes the newly added button labels:

```diff
+"install-update":`Install Update`,"check-for-updates":`Check for Updates`,quit:`Quit`
```

The error dialog text also includes:

```diff
+message:`${t.app.getName()} failed to start.`
+detail:r instanceof Error?r.message:`The main desktop app failed during startup.`
```

### 2. Startup flow now integrates Sparkle update state [Confirmed]

The new startup entry initializes `sparkleManager`, and when startup fails, decides whether to show “Install Update” or “Check for Updates” based on `getIsUpdateReady()` and updater state.

The user-visible implication is: when the desktop app is damaged, the main startup flow fails, and an update is already available, the app has a chance to guide the user directly into installing the update instead of leaving them only in an error state. Key evidence:

```diff
+await a.initialize();
+let{desktopSentry:i,sparkleManager:a}=n;
+i.installUpdatesIfAvailable();
+a?.checkForUpdates();
```

### 3. Frontend loading-page and popover modules appear [Signal]

After removing Vite build hash noise, the fact package identifies the real newly added frontend modules as `loading-page` and `popover`. This points toward loading-page and overlay / popover capability areas, but source-level intent cannot be recovered from the artifact, and it does not prove the functionality has been enabled for all users.

The possible user-visible impact is that interface infrastructure for certain waiting, transition, or popover interactions may be changing. Key evidence comes from the de-noised module list:

### 4. Build tool Vite upgraded from beta to stable [Confirmed]

`package.json` shows that the development dependency Vite was upgraded from `8.0.0-beta.15` to `8.0.0`.

This looks more like stabilization of the underlying build pipeline than a directly user-facing new feature; however, it affects the packaging shape of frontend resources, so the large number of hash filename changes under `webview/assets` in this comparison should be treated as build artifact renaming noise. Key evidence:

```diff

+    "vite": "8.0.0",
```

### 5. Main binary size increased notably [Signal]

`Contents/Resources/codex` increased from `106.4 MB` to `116.9 MB`, a gain of `10.51 MB`. This points to a significant underlying binary content change, but size alone cannot identify the specific functionality, nor can it recover source-level intent from the artifact.

Any user-visible impact would need to be verified against actual runtime behavior; within the scope of this fact package, it can only be recorded as a binary-layer change signal.

# Codex Desktop 26.715.21425 macOS / Windows Comparison

> Third-party cross-platform changelog · Unofficial

The shared themes are the Ultra permission flow and multi-agent entry points. Platform priorities diverge: macOS focuses on browser and thread-panel UI, while Windows focuses on plugins, sharing, and permission management.

Ultra/full-access handling and multi-agent entry points changed on both platforms, while macOS emphasizes browser and panel UI and Windows emphasizes plugins, sharing, and permissions.

## About This Report

This report compares two sets of officially signed distribution artifacts:

- macOS arm64: `26.715.21316` build 5484 → `26.715.21425` build 5488.
- Windows x64 MSIX: `26.715.2236.0` → `26.715.2305.0`.

The platform version numbers are not aligned. Every conclusion is derived independently from its platform-specific comparison range. The Windows fact package incorrectly labels its platform/architecture line as macOS arm64; this report follows the input definition of the sample as Windows x64 MSIX without altering any version numbers.

These are historical full-package comparisons, not incremental-package analyses. Both macOS packages passed Apple notarization verification and were signed by OpenAI OpCo, LLC. The fact package provides no Windows signature-verification result.

Evidence uses two levels:

- [Confirmed]: Directly supported by plist data, plugin manifests, signature verification, or other readable configuration facts.
- [Signal]: Derived from strings, style sets, module names, or compressed-size changes in same-name frontend modules. It indicates that related code entered, left, or was reorganized within a bundle, not that the feature is enabled for every user.

The labels `[macOS+Windows]`, `[macOS only]`, and `[Windows only]` identify the samples containing the evidence. They do not imply that another platform could not have had or receive the capability at another time.

The supplied fact packages list no directory-level additions or removals of bundled plugins and provide no new bundled-plugin detail section, CSP differences, standalone new main-process strings, Windows AppxManifest changes, or new bundle files. This report does not interpret missing evidence as proof that those areas were unchanged.

## ✨ Highlights

### Ultra Can Continue with Full Access

[Signal] [macOS+Windows]

Both platforms added `Use Ultra with Full access?`, `Use Full access`, and button copy for using Ultra with either restricted permissions or Full access. The corresponding confirmation flow offers separate paths to continue with approval or continue with Full access.

The removed copy described a previous flow in which selecting Ultra switched away from Full access, or retaining Full access canceled Ultra. macOS additionally removed `Unavailable with Ultra reasoning` and descriptions saying Full access was unavailable under Ultra.

This is a strongly aligned interaction signal: Ultra and Full access appear to move from a mutually exclusive switch toward a warned choice. Because the evidence comes from frontend-module differences, it does not confirm account-level permission policy or rollout status.

### Multi-Agent Composer Banner Enters Home Code on Both Platforms

[Signal] [macOS+Windows]

The `home-announcements` module on both platforms added body and dismissal copy for a multi-agent composer banner. The strings explicitly refer to:

- Delegating work to subagents running in parallel;
- Potentially increased token usage;
- A `multi-agent` state and a banner-seen state.

The same module also added dismissal labels for Fast mode, Codex gift credits, rate-limit resets, and backend-driven home banners, plus copy for the gift-credit purchase flow and the rate-limit reset modal. These elements entered the frontend packages, but display may still depend on account state, backend configuration, or experiments.

### Bundled Browser and Chrome Plugin Versions Are Synchronized

[Confirmed] [macOS+Windows]

On both platforms, the bundled `browser` and `chrome` plugin manifest versions changed from `26.715.21316` to `26.715.21425`. There were no directory-level plugin additions or new dependencies.

Their existing manifest descriptions remain unchanged:

- `browser` targets the in-app browser and covers opening, inspecting, navigating, testing, clicking, typing, and taking screenshots of local web targets, including localhost, loopback addresses, `file://`, and the current in-app browser tab.
- `chrome` targets automation that depends on the user's existing Chrome state, including tabs, logged-in sessions, cookies, extensions, and ChatGPT Chrome Extension setup. Its description continues to prioritize purpose-built connectors, APIs, or CLIs.

The confirmed change is plugin-version metadata synchronization, not the first introduction of these capabilities.

### macOS Expands Embedded-Browser Settings, History, and Downloads

[Signal] [macOS only]

`browser-use-settings` grew by approximately 98,875 bytes and added a mutually reinforcing set of UI labels:

- Access history without asking;
- Browser downloads and history entry points;
- Back and forward controls;
- Contact information, extension, password-manager, and site settings;
- Controls for expanding or hiding individual browser-data clearing options.

New states such as `aria-controls`, `aria-expanded`, and `aria-labelledby` also point to an expandable settings UI with more complete accessibility labeling. The Windows fact package contains no corresponding module change.

### Thread Side and Bottom Panels Move in Opposite Directions

[Signal] [macOS+Windows]

On macOS, `thread-app-shell-chrome` grew by approximately 37,467 bytes and added:

- `Bottom panel`;
- An action for closing the thread bottom panel;
- Actions for opening another tab from the bottom or side panel;
- Actions for opening browser, review, terminal, or workspace-file-browser views from a task side panel;
- An action for opening a side chat;
- An empty state when no side-panel actions are available;
- A `max-width: 920px` responsive condition.

On Windows, the same-name module shrank by approximately 34,866 bytes and removed that responsive condition, `Bottom panel`, and the browser, review, terminal, workspace-file-browser, and side-chat action labels.

This is not a simple synchronized change. Within their independent comparison ranges, macOS brought these strings into the module while Windows moved them out. It may reflect module splitting, loading changes, or platform-specific UI routing; it does not prove that Windows removed the underlying features.

### Windows Plugin Details Add Sharing and MCP Management UI

[Signal] [Windows only]

`plugin-detail-page` grew by approximately 156,015 bytes. New strings cover several high-visibility interactions:

- Searching for workspace users and groups;
- Adding or removing plugin share targets;
- Access for workspace members who have the link;
- An MCP server enablement toggle;
- Plugin-detail section navigation and a back button;
- More-actions menus for app tools, skill previews, and skill cards;
- An `App unavailable` state.

This evidence indicates that the plugin-detail bundle is taking on richer sharing, skill-action, and MCP enablement UI. It does not confirm that every workspace has the same options enabled.

### Windows Expands Site Publishing and Sharing

[Signal] [Windows only]

`appgen-share-dialog` grew by approximately 23,410 bytes and added:

- Workspace user and group search;
- Adding and removing site share targets;
- A saving state;
- Workspace-link and public-Internet access;
- A `Can visit` permission;
- Publishing a site to the public Internet;
- Copying a live-site URL, opening it externally, and a `Copied to clipboard` state.

Together, these strings point to access control and link-distribution flows for published sites. At the same time, Windows `appgen-settings-page` removed strings for environment variables, secrets, custom domains, site titles, and deletion-slug confirmation. Because those elements may have moved to another chunk, the evidence does not establish that the settings were removed from the product.

### Windows Permission Menu Adds Auto-review, Custom, and Capability Details

[Signal] [Windows only]

`permissions-mode-dropdown` grew by approximately 25,480 bytes and added:

- `Approve for me` and `Ask for approval`;
- `Change permissions`;
- `Custom (config.toml)`;
- An Auto-review option and its unavailable state;
- An explanation for Full access being unavailable;
- “Always ask” behavior for external-file editing and Internet use;
- A capability covering websites, data transmission, and enabled plugins;
- Capability titles and descriptions for the Full access confirmation dialog.

This signals a more granular permission-mode presentation, bringing network access, external files, plugin use, and custom configuration into the same selection surface. Actual permission enforcement cannot be confirmed from frontend strings.

### Windows Computer Use Plugin Metadata Advances

[Confirmed] [Windows only]

The bundled Windows `computer-use` plugin manifest version changed from `26.715.21316` to `26.715.21425`. Its description remains control of Windows desktop applications from ChatGPT through Computer Use.

The runtime stayed at `287.5 MB`, and `cua_node` added no dependency packages. The confirmed change is therefore plugin-version metadata, not Computer Use runtime expansion or a dependency upgrade.

The macOS Computer Use runtime likewise had no size or dependency change, but its fact package does not list a corresponding `computer-use` manifest update.

### Windows Embedded Checkout Adds a Back-to-ChatGPT Action

[Signal] [Windows only]

`checkout-webview-page` grew by approximately 12,752 bytes and added `Back to ChatGPT`, an explanation for returning to ChatGPT within the Codex app, `https://pay.openai.com`, and a page-title update state.

This places return navigation and overlay state in the payment WebView module. It does not establish that checkout is available to every user.

### Home and History Navigation Code Contracts on Both Platforms

[Signal] [macOS+Windows]

The `header` module on both platforms removed the same set of strings, including `All chats`, environment filtering, app settings, back navigation, recent-task search clearing, chat history, Chrome extension side-panel actions, and `ChatGPT for Chrome`.

The aligned removal is more consistent with shared frontend code splitting or migration. Compressed-module differences alone cannot establish that these navigation features disappeared from the product.

### Each Platform Shows Additional Page-Migration Signals

[Signal] [macOS only]

On macOS, `skills-page` removed strings for creating and reloading skills, refreshing after installation, empty states, the page title, and the installed-skills section. `plugins-settings` removed `Manage plugins, skills, and MCPs`. This points to an information-architecture reorganization around skills and plugin settings, not confirmed removal of skill management.

[Signal] [Windows only]

On Windows, `tasks` shrank by approximately 306,482 bytes and removed onboarding copy for connecting and installing apps, preparing a connected-app task, executing onboarding tasks, and showing a CSV chart example. This is a strong signal of conversational-onboarding migration or trimming, not confirmed removal of connected-app functionality.

## 🔎 Low-Level Signal Index

### macOS Chunk Loading Relationships

[Signal] [macOS only]

After excluding Vite hash noise, new chunk names continue to connect `app-main`, the new-thread panel, onboarding, AppGen library/settings, plugin detail, pull requests, artifact tabs, notebook preview, business checkout, settings command menus, and the avatar overlay. Many older combined chunks were removed at the same time, making this more useful as evidence of shared-chunk boundary changes than as a list of newly introduced features.

### Windows Window, Avatar Overlay, and RPC Modules

[Signal] [Windows only]

Windows added three genuine module names: `app-window-focus-signal`, `avatar-overlay-open-state-signal`, and `await-rpc-request`. They point respectively to window focus, avatar-overlay open state, and RPC request waiting, but the available strings and configuration do not support more specific user-facing conclusions.

### macOS Events and Error States

[Signal] [macOS only]

The `esm` module added `Callback Error`, `Context Cancel`, `Context canceled`, `Failed to deliver`, destination load and unload failures, `Not ready`, Promise timeout, and `plugin Error`. Older event-bus connection, queued-emission, and shim-dispatch logs were removed. This signals a reorganization of error paths and event infrastructure.

### macOS Page and Device Context

[Signal] [macOS only]

`middleware` added Page/Screen loaded events and field names for device, IP, referrer, timezone, email, user agent, integrations, and options. They entered the frontend bundle, but the fact package is insufficient to establish collection timing, destination, or whether they are active.

### macOS Home and Profile Styling

[Signal] [macOS only]

`profile` added only a `gap-5` class, a low-value layout signal. In `home-announcements`, the yellow chart color, banner-seen states, and icon sizing reinforce the home-banner changes described above.

### Evidence Areas Without a Product Conclusion

[Confirmed] [macOS+Windows]

Neither platform added or removed top-level bundle files, and neither added `cua_node` dependencies. The fact packages list no directory-level new plugins, CSP changes, or new-resource details. macOS contains only Info.plist version and asar-integrity changes in this category; no AppxManifest difference was supplied for Windows.

# Codex Desktop `26.406.31014` build 1395

> Third-party changelog · Unofficial · Based on reverse diffs of official update artifacts

This build looks broader than a small patch: the artifacts show visible movement around background suggestions, heartbeat automation, browser sidebar comments, tray menus, and projectless chat location settings.

## About This Report

This is not an official changelog. It is a **reverse-diff report based on official distribution artifacts**. This entry is a historical backfill: the official appcast did not provide a delta package for `26.325.31654` → `26.406.31014`, so the comparison was performed directly between the two official full packages.

Evidence is split into two levels: **[Confirmed]** means direct human-readable evidence exists in the artifact, such as `package.json`, `Info.plist`, CSP text, readable strings, or configuration diffs; **[Signal]** means the evidence comes only from frontend module names, same-name frontend module diffs, or size changes, pointing in a direction but not reconstructing source-level intent.

The comparison covers macOS arm64. The previous version is `26.325.31654` build `1272`, released on `2026-03-28`; the latest version is `26.406.31014` build `1395`, released on `2026-04-08`.

## ✨ Highlights

### 1. Background suggestions and project-aware suggestion settings were added [Confirmed]

The new build contains settings strings such as `Ambient suggestions`, `Enable ambient suggestions`, and `Generate project-aware suggestions on window focus and show the latest 3 pending rows above the home composer`, along with error paths like `Failed to refresh ambient suggestions` and `Failed to update ambient suggestions enabled state`.

The user-visible implication is that Codex is turning “generate project-aware suggestions on window focus and show the latest 3 pending rows above the home composer” into a configurable capability. The key evidence comes from `agent-settings` and newly added main-process bundle strings.

### 2. Heartbeat automation expands into a thread-level automation surface [Confirmed]

The new build adds strings such as `Accessible label for opening the heartbeat automation attached to the current thread`, `Created automation in the app.`, `Updated automation in the app.`, `Failed to delete heartbeat automation for archived thread`, `post a heartbeat message`, and `send a heartbeat`.

This points to heartbeat automation being attachable to threads, with create, update, and archived-thread cleanup paths. Compared with the removed previous string `Heartbeat found something worth your attention.`, this build appears to move heartbeat behavior from a result notification toward a manageable automation object.

### 3. Browser sidebar comments and browser bridge paths were expanded [Confirmed]

The main-process bundle adds `Button label to cancel creating an inline browser comment`, `Button label to revisit the annotated browser location`, `browserSidebarCommentRuntime.*`, `Timed out waiting for browser bridge response`, `Unknown browser bridge error`, `chrome bridge install complete`, and `chrome bridge install failed`.

The visible direction is an inline comment flow inside the browser sidebar, including cancel, close, delete, and revisit actions. Underneath it, the artifact also contains browser bridge timeout, unknown-error, and Chrome bridge install-result paths. `app-server-manager-hooks` also adds `Browser bridge request failed` and `features.browser_use`.

### 4. Tray menu and quick chat entry points are taking shape [Confirmed]

The new build adds `Failed to set up tray`, `trayMenu.newChat`, `trayMenu.openApp`, `trayMenu.openHotkeyWindow`, `trayMenu.pinnedThreads`, `trayMenu.projectlessThreads`, `trayMenu.recentThreads`, `Quick Chat`, and `Open Quick Composer`, along with new resources `codexTemplate.png` and `codexTemplate@2x.png`.

This indicates a concrete tray menu surface with entries for opening the app, starting a new chat, opening the hotkey window, pinned threads, projectless threads, and recent threads. The new template image resources also fit a macOS tray icon resource shape, although the artifact does not expose the full UI behavior.

### 5. Projectless new chats can choose a starting location [Confirmed]

The settings surface adds `New chat location`, `Where should new chats without a project start?`, `Use Codex threads folder`, `Use home folder`, and `Label for choosing where non-project chats run`.

This means projectless chats now have a visible setting rather than only an implicit location: new chats without a project can start in either the Codex threads folder or the home folder.

### 6. Pull request board and PR update paths keep expanding [Confirmed]

The new build adds `Accessible label for the pull request auto-merge toggle`, `Accessible label for the pull request board saved view toggle`, `Accessible label for the pull request board search field`, `Failed to load pull request diff`, `Failed to update pull request`, `No git repository found for PR updates`, and `No git repository found for pull request diff`.

This shows PR-related UI is not limited to display-only paths: the artifact includes diff loading, PR update, auto-merge toggle, saved view, and search field paths. Because the evidence comes from bundle strings, it confirms those paths exist, but not that every entry point is enabled for every user.

### 7. Remote connections now show an update-required state for old versions [Signal]

`app-server-connection-state` adds `Update required`, `appServer.error.unsupportedVersion`, and `Error shown when an app-server connection is rejected because the remote Codex version is too old`.

This is a same-name frontend module diff, so it points to an “update required” state when a remote Codex version is too old, but it cannot reconstruct source-level intent or the full handshake behavior from the artifact.

### 8. The frontend moves to React 19 and adds a shared-object snapshot bridge [Confirmed]

`package.json` adds `react` and `react-dom` at `^19.2.0`, plus `@types/react` and `@types/react-dom` using `catalog:react-19`. `preload.js` adds `codex_desktop:get-shared-object-snapshot`, `shared-object-set`, `shared-object-updated`, and `getSharedObjectSnapshotValue`.

This means the new frontend runtime explicitly carries React 19, and the Electron preload bridge now includes read/write/sync paths for shared-object snapshots. For users this is mostly infrastructure, but it can support state synchronization between windows or between the main and renderer processes.

### 9. The user data directory can be overridden by environment variable [Confirmed]

`bootstrap.js` adds logic for `CODEX_ELECTRON_USER_DATA_PATH`: if present, the resolved value is used as `userData`; otherwise the app continues using a build-flavor-specific directory under appData.

This is practically useful for testing, isolated runs, or multi-instance setups because Codex’s user data directory can now be supplied through an environment variable. The capability is visible in a readable diff, so it is direct evidence.

### 10. Syntax highlighting assets were redistributed [Signal]

Across same-name frontend modules, `markdown` shrinks from `0.7 MB` to `0.1 MB`, while `typescript`, `c`, `haxe`, `scss`, `vue`, `angular-html`, `gdscript`, `regexp`, and `glsl` gain many syntax scope strings. `javascript`, `php`, `csharp`, `java`, and `yaml` show corresponding string removals.

These clues point to syntax-highlighting assets being split or rearranged, but this is only a build-artifact signal. It cannot be treated as source-level intent, and it should not be read as real removal of language support.

# Codex 桌面版 `26.608.12217`(build 3722)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版没有只表现为单纯的版本号更新：包内新增了 `cua_node` 运行时目录，浏览器自动化文档更严格，Sites(站点托管与部署能力)插件接口也做了成体系的命名调整。  
English one-liner: runtime packaging, browser inspection, and Sites deployment flows all moved in a more explicit direction.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次是历史回填：两个版本均超出官方 Sparkle(应用更新框架)增量窗口，因此事实包采用两个官方全量包直接比较，而不是增量包重建。

对比范围是 `26.602.71036` build `3685` 到 `26.608.12217` build `3722`，两个版本发布日期均为 `2026-06-09`，平台为 macOS arm64(苹果芯片版)。

可信度来自 Apple(苹果)公证与 OpenAI Developer ID(开发者身份签名)：上一版与最新版均通过 `spctl` 公证验证，结论为 `accepted`，来源为 `Notarized Developer ID`，签名主体为 `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`。

证据等级分为两档：

- **【实证】**：来自 Info.plist(应用元信息文件)、CSP(内容安全策略)、技能文档、插件配置、package.json(包配置文件)、依赖清单、公证 / 签名结果等人类可读证据。
- **【信号】**：来自前端模块名或二进制体积变化，只能说明“指向某方向，但无法从产物还原到源码级意图”。

## ✨ 重点变化

### 1. 本地 Node 运行时被重组为 `cua_node`，并补入 Computer Use 相关依赖 【实证】

包内移除了旧的 `Contents/Resources/node` 和 `Contents/Resources/node_repl`，新增了 `Contents/Resources/cua_node/bin/node` 与 `Contents/Resources/cua_node/bin/node_repl`。同时，`cua_node` 新增 6 个包，包括 `@oai/sky@0.4.8-202606020558-pr-981870-f570e0604110`、`npm@11.9.0`、`corepack@0.34.6`、`objc-js@1.5.0` 等。

这意味着桌面端的本地 JavaScript(脚本运行环境)能力不再只是简单携带 Node(服务端 JavaScript 运行时)，而是被整理进一个更专门的 `cua_node` 运行时目录；其中还出现了 Windows 与 Linux 的 `@oai/sky` 二进制文件，指向更完整的跨平台 Computer Use(让模型操作电脑 / 浏览器的能力)运行支持。

关键证据：

```text
新增: Contents/Resources/cua_node/bin/node (119.1 MB)
新增: Contents/Resources/cua_node/bin/node_repl (15.2 MB)
移除: Contents/Resources/node (119.1 MB)
移除: Contents/Resources/node_repl (15.2 MB)
```

### 2. 浏览器 / Chrome 自动化的文档读取规则更严格 【实证】

Browser(内置浏览器自动化插件)与 Chrome(谷歌浏览器自动化插件)的技能文档都要求首次读取 `browser.documentation()` 时必须“一次性”直接调用：

```text
nodeRepl.write(await browser.documentation());
```

并明确禁止先赋值、查看长度、切片、摘要或主动分页。只有工具输出明确报告被截断时，才允许分块读取。

这对开发者可感知的影响是：浏览器自动化流程更强调完整读取 API(应用程序接口)文档，减少模型基于片段文档误用接口的机会。该变化出现在内置浏览器、多标签浏览器以及 Chrome 插件技能文档中。

### 3. `domSnapshot()` 现在声明会尽量展开 iframe 内容 【实证】

浏览器与 Chrome 插件的 API 文档中，`domSnapshot()` 的说明从“返回当前 DOM(网页文档对象模型)字符串快照”，变为“返回当前 DOM 字符串快照，并在可用时包含展开后的 iframe(嵌入网页框架)正文内容”。

这意味着自动化检查网页时，内嵌页面内容有机会被纳入可读快照。对调试登录页、嵌入式控制台、沙盒预览页这类场景，快照可见性可能更好。

关键证据：

```diff
- domSnapshot(): Promise<string>; // Return a snapshot of the current DOM as a string.
+ domSnapshot(): Promise<string>; // Return a snapshot of the current DOM as a string, including expanded iframe body content when available.
```

### 4. Sites 插件从“project”命名转向“site”命名，部署流程文档更贴近真实接口 【实证】

Sites 插件版本从 `0.1.12` 升至 `0.1.14`。技能文档中多处接口名被替换：`create-project` 改为 `create-site`，`create-project-version` 改为 `save-site-version`，`deploy-project-version` 改为 `deploy-site-version`，`deployment-status` 改为 `get-deployment-status`。

这不是简单改字，而是把创建、保存版本、部署、查询状态这些关键步骤统一到了 “site” 语义下。文档还补充说明：在连接器 API 和托管配置中，`project_id` 是 Site 的持久标识符。

关键证据：

```text
sites 插件版本: 0.1.12 → 0.1.14
create-project → create-site
create-project-version → save-site-version
deploy-project-version → deploy-site-version
deployment-status → get-deployment-status
```

### 5. Sites 访问控制的默认表达更保守，用户可见文案也更清晰 【实证】

Sites 技能文档把“使用访问控制”改成了“仅当用户要求改变访问者时才使用访问控制”。同时，文档要求把 `workspace_all` 面向用户称作 “Anyone in this workspace can visit”，把 `custom` 称作 “Private”。

这意味着部署站点时，访问控制不再被描述为默认必须介入的步骤，而是按用户意图触发；同时面向用户的权限解释更明确。

关键证据：

```diff
- Use access controls when the user asks to change who can access the deployed site.
+ Use access controls only when the user asks to change who can access the deployed site.
```

### 6. 前端新增头像覆盖层、提及内容、线程导航等模块信号 【信号】

去掉 Vite(前端构建工具)哈希文件噪音后，新增模块中出现了 `avatar-overlay-featuregate-page`、`avatar-overlay-native-frame`、`avatar-overlay-notifications`、`custom-avatars-query`、`inline-mention-content`、`mention-icons`、`thread-browser-panel-tabs`、`thread-user-message-navigation-rail` 等。

这些模块名指向头像覆盖层、头像查询、内联提及、线程浏览 / 导航等方向，但无法从产物还原到源码级意图，也不能证明这些能力已经对用户开放。

### 7. WebSocket 连接目标新增 ChatGPT 域名 【实证】

`webview/index.html` 的 CSP `connect-src` 新增允许 3 个 WebSocket(双向网络连接)地址：

```text
wss://chatgpt.com
wss://ws.chatgpt-staging.com
wss://ws.chatgpt.com
```

这说明 Webview(内嵌网页视图)层现在允许连接这些实时通道。事实包只能证明 CSP 放行，不能进一步证明具体功能入口或调用路径。

### 8. 摄像头权限说明从通用表述改为视频输入场景 【实证】

Info.plist 中 `NSCameraUsageDescription` 从通用的 “This app needs access to the camera” 改为更具体的 “Codex needs access to your camera for video input.”

这会影响 macOS 弹出摄像头权限申请时用户看到的说明。变化更明确地把摄像头访问解释为“视频输入”。

关键证据：

```text
NSCameraUsageDescription:
This app needs access to the camera
→ Codex needs access to your camera for video input.
```

### 9. 多语言“关于”窗口里的版本 / 构建信息文案被微调 【实证】

多个 `native-menu-locales` 本地化文件更新了关于窗口中的版本、构建信息、发布日期表达。中文里 `codex.aboutDialog.buildInfoLabel` 从“构建信息”改为“版本和构建信息”。

这属于用户可见但低风险的文案修订，主要影响“关于 Codex”窗口和部分本地化语言的措辞。

关键证据：

```text
zh-CN: 构建信息 → 版本和构建信息
```

### 10. macOS 从 DMG 安装到 Applications 的清理流程增加重试 【实证】

`.vite/build/bootstrap.js` 中，处理来源 DMG(磁盘镜像安装包)卸载与清理的逻辑加入了重试相关常量：`25` 次、每次间隔 `250`。失败日志也从单次卸载失败，变为“多次重试后仍无法卸载”。

这指向一个更稳健的安装清理流程：当应用从 DMG 启动并尝试移动到 Applications(应用程序目录)后，系统卸载镜像可能不总是立即成功，新版会多次尝试。

关键证据：

```text
C = 25
w = 250
Failed to detach app source DMG after retries
```

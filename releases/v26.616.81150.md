# Codex 桌面版 26.616.81150 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

本批次没有看到新的运行时依赖、CSP、资源树或明确前端功能面变化，最强证据集中在版本元数据和内置插件 manifest 的版本同步。

The strongest confirmed change in this batch is cross-platform package and bundled-plugin version alignment, with Windows additionally showing the computer-use plugin manifest update.

## 关于这份报告

这份报告基于官方签名分发包的逆向重建 diff。macOS 对比范围为 `26.616.71553 build 4265` → `26.616.81150 build 4306`；Windows 对比范围为 `26.616.9593.0` → `26.616.10790.0`。

证据等级分为两档：【实证】来自 plist、manifest、依赖清单、插件清单、服务入口、技能文档、可读配置或事实包中的非前端线索；【信号】来自前端模块名、压缩 bundle 方向、样式类名或体积变化线索。本批次事实包未提供“同名前端模块改动线索”和“新增可读字符串”区块，因此没有把前端 UI 或文案变化写成确认事实。

平台归属使用 `[macOS+Windows]`、`[仅 macOS]`、`[仅 Windows]`。两个平台来自同一套 Codex 代码的不同打包，但版本号和文件统计分别引用各自事实，不互相套用。

## ✨ 重点变化

### 【实证】[macOS+Windows] 应用版本元数据进入新批次

macOS 的 `Info.plist` 显示 `CFBundleShortVersionString` 从 `26.616.71553` 更新到 `26.616.81150`，`CFBundleVersion` 从 `4265` 更新到 `4306`，`ElectronAsarIntegrity/Resources/app.asar/hash` 也随之变化。

Windows 包的版本对比为 `26.616.9593.0` → `26.616.10790.0`。事实包未提供 AppxManifest 具体 diff，因此 Windows 侧只能确认包级版本对比，不能补写 manifest 字段变化。

### 【实证】[macOS+Windows] browser 内置插件版本同步到 26.616.81150

两平台的 `browser` 插件 manifest 均显示版本从 `26.616.71553` 更新到 `26.616.81150`。插件描述仍指向 Codex in-app browser，用于打开、检查、导航、测试、点击、输入或截图本地目标，包括 `localhost`、`127.0.0.1`、`::1`、`file://` URL 和当前 in-app browser 标签页。

这属于内置插件清单级别的确认变化。事实包只显示版本字段变化，没有显示服务入口、技能说明或默认提示新增。

### 【实证】[macOS+Windows] chrome 内置插件版本同步到 26.616.81150

两平台的 `chrome` 插件 manifest 均显示版本从 `26.616.71553` 更新到 `26.616.81150`。描述继续覆盖依赖用户现有 Chrome 状态的自动化场景，包括标签页、登录会话、cookies、扩展和 Codex Chrome Extension 设置。

这同样是插件 manifest 级确认变化。事实包没有显示 chrome 插件新增能力面，只能确认随包版本同步。

### 【实证】[仅 Windows] computer-use 内置插件版本同步到 26.616.81150

Windows 的 `computer-use` 插件 manifest 显示版本从 `26.616.71553` 更新到 `26.616.81150`，描述为通过 Computer Use 从 Codex 控制 Windows 桌面应用。

macOS 事实包没有对应的 `computer-use` 插件 manifest diff，因此不能说 macOS 在本输入样本中也发生了同项插件 manifest 变化。

### 【实证】[macOS+Windows] Computer Use 运行时体量与 cua_node 依赖未变

macOS 的 Computer Use 运行时总量保持 `377.4 MB → 377.4 MB`，变化为 `+0 MB, +0%`；Windows 保持 `298.9 MB → 298.9 MB`，变化为 `+0 MB, +0%`。两平台的 `cua_node` 新增依赖包数量均为 `0`。

这说明本批次事实包没有显示 Computer Use 运行时扩容或依赖新增。Windows 虽有 `computer-use` 插件 manifest 版本更新，但运行时总量和依赖清单没有同步增长。

### 【实证】[macOS+Windows] Bundle 文件树整体稳定，变更集中在少量内容改动

macOS bundle 文件数保持 `8827 → 8827`，新增 `0`、移除 `0`、内容改动 `68`。Windows 文件数保持 `9697 → 9697`，新增 `0`、移除 `0`、内容改动 `15`。

两平台最大组件 `app.asar` 体积都显示 `+0.00 MB`：macOS 为 `Contents/Resources/app.asar`，`155.8 MB → 155.8 MB`；Windows 为 `resources/app.asar`，`159.6 MB → 159.6 MB`。这支持“没有大规模资源或运行时重排”的判断，但不等同于没有代码改动。

### 【信号】[macOS+Windows] app.asar 内部出现构建产物层面的增删，但事实包提醒不要按 hash 噪音判断功能

macOS `app.asar` 内部文件数保持 `3115 → 3115`，新增 `10`、移除 `10`、改动 `10`。Windows `app.asar` 内部文件数保持 `3078 → 3078`，新增 `10`、移除 `10`、改动 `11`。

事实包明确说明 `webview/assets` 下大量增减是 Vite 构建 hash 重命名，属于噪音。因此这些增删只能作为构建产物变化信号，不能写成新增前端功能。

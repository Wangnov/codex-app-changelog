# Codex 桌面版 26.616.81150 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

本批次从包内事实看，更像一次内置插件元数据与打包版本同步，而不是大规模功能面扩展。

One-liner: a small cross-platform packaging/plugin metadata sync, with Windows additionally updating the bundled computer-use plugin manifest.

## 关于这份报告

本报告只使用两份官方签名分发包逆向重建后的机器提取事实：macOS arm64 为 `26.616.71553` build `4265` → `26.616.81150` build `4306`，Windows x64 MSIX 为 `26.616.9593.0` → `26.616.10790.0`。

证据分两档：【实证】来自 plist、manifest、插件清单、服务入口、依赖清单、技能文档、可读字符串或关键文件 diff；【信号】来自前端模块名、体积变化、压缩 bundle 方向线索。平台归属使用 `[macOS+Windows]`、`[仅 macOS]`、`[仅 Windows]`。

本输入样本中未提供新增内置插件明细、前端模块真实增删清单、同名前端模块改动线索、新增可读字符串、内容安全策略变化、AppxManifest 变化或新增组件与资源的具体条目，因此这些类别不在正文里扩展为功能结论。

## ✨ 重点变化

### 【实证】[macOS+Windows] Browser 内置插件随应用版本同步

两平台的 `browser` 插件 manifest 都从 `26.616.71553` 更新到 `26.616.81150`。插件描述仍指向 Codex 内置浏览器自动化场景，包括打开、检查、导航、测试、点击、输入、截图本地 web 目标，以及处理 `localhost`、`127.0.0.1`、`::1`、`file://` 和当前 in-app browser 标签页。

平台差异：macOS 路径为 `Contents/Resources/plugins/openai-bundled/plugins/browser/.codex-plugin/plugin.json`；Windows 路径为 `resources/plugins/openai-bundled/plugins/browser/.codex-plugin/plugin.json`。本输入只显示版本字段变化，没有显示 browser 插件能力文本变化。

### 【实证】[macOS+Windows] Chrome 内置插件随应用版本同步

两平台的 `chrome` 插件 manifest 都从 `26.616.71553` 更新到 `26.616.81150`。包内描述仍指向依赖用户现有 Chrome 状态的自动化能力，包括标签页、登录态、cookie、扩展和 Codex Chrome Extension 设置；也保留了优先使用专用连接器、API 或 CLI，认证失败时要求用户重新认证或明确批准 Chrome 兜底的说明。

平台差异：macOS 和 Windows 都有该 manifest 版本同步。本输入没有显示 Chrome 插件描述或入口能力变化。

### 【实证】[仅 Windows] Windows 额外同步 Computer Use 插件 manifest

Windows 包内 `resources/plugins/openai-bundled/plugins/computer-use/.codex-plugin/plugin.json` 的版本从 `26.616.71553` 更新到 `26.616.81150`，描述为通过 Computer Use 从 Codex 控制 Windows 桌面应用。

平台差异：macOS 本批事实包没有列出 `computer-use` 插件 manifest diff；Windows 有。因此只能写作 Windows 样本中的独有 manifest 变化，不能推断 macOS 没有对应能力或没有同步。

### 【实证】[仅 macOS] macOS 应用版本与 asar 完整性哈希更新

macOS `Info.plist` 显示 `/CFBundleShortVersionString` 从 `26.616.71553` 更新到 `26.616.81150`，`/CFBundleVersion` 从 `4265` 更新到 `4306`，`/ElectronAsarIntegrity/Resources/app.asar/hash` 也随 `app.asar` 内容更新。

平台差异：Windows 输入没有提供 AppxManifest 变化，因此本报告不写 Windows manifest 字段变化。

### 【实证】[macOS+Windows] 应用层 app.asar 内容有少量变化，但文件数量保持不变

macOS `app.asar` 内部文件为 `3115 → 3115`，新增 `10`、移除 `10`、改动 `10`。Windows `app.asar` 内部文件为 `3078 → 3078`，新增 `10`、移除 `10`、改动 `11`。

平台差异：两平台都有少量内部文件替换或改动，但 macOS 与 Windows 的内部文件总数和改动数不同。输入明确提示 `webview/assets` 下大量增减属于 Vite 构建 hash 重命名噪音，不能据此判断新功能。

### 【实证】[macOS+Windows] Computer Use 运行时总量未变化，cua_node 依赖无新增

macOS Computer Use 运行时总量为 `377.4 MB → 377.4 MB`，变化 `+0 MB`、`+0%`。Windows 为 `298.9 MB → 298.9 MB`，变化同为 `+0 MB`、`+0%`。两平台 `cua_node` 依赖新增包数量均为 `0`。

平台差异：运行时总量绝对值不同，macOS 样本更大；但本批在各自平台内都没有总量增长，也没有新增 `cua_node` 包。

### 【实证】[macOS+Windows] 包级文件树没有新增或移除文件，只有内容改动

macOS bundle 文件数为 `8827 → 8827`，新增 `0`、移除 `0`、内容改动 `68`。Windows 文件数为 `9697 → 9697`，新增 `0`、移除 `0`、内容改动 `15`。

平台差异：两平台都没有包级新增/移除文件，但 macOS 内容改动文件数明显多于 Windows。两平台最大体积变化组件都指向 `app.asar`，且表内变化为 `+0.00 MB`。

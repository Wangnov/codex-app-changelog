# Codex 桌面版 `26.415.20818`(build 1727)

> 第三方变更日志 · 非官方 · 基于官方更新包逆向差异

这一版从产物层看，重点不是小修小补，而是把本地运行、浏览器自动化、Computer Use(让模型操作桌面 / 浏览器的能力)以及文档类预览能力进一步打包进桌面应用。

One-liner: this build expands bundled local tooling and surfaces early signals for richer artifact previews and computer-use workflows.

## 关于这份报告

这不是官方变更日志，而是一份**基于官方签名分发产物的逆向差异报告**。本次为历史回填：两个版本均已超出官方 Sparkle(应用自动更新框架)增量窗口，因此采用两个官方全量包直接比较，而不是增量包重建。

对比范围为 `26.409.20454` build `1462`，发布于 `2026-04-11`，到 `26.415.20818` build `1727`，发布于 `2026-04-16`。信任根来自 Apple(苹果)公证与 OpenAI Developer ID(开发者身份)代码签名；两个版本的 `Codex.app` 均通过 `spctl` 公证验证，来源为 `Notarized Developer ID`，签名主体为 `Developer ID Application: OpenAI OpCo, LLC (2DC432GLL2)`。

本文使用两档证据等级：

- **【实证】**：来自人类可读的直接证据，例如 `Info.plist`、`package.json`、构建脚本差异、签名 / 公证结论、明确新增的 bundle 文件。
- **【信号】**：来自前端模块名或二进制体积变化等间接证据。它们指向某方向，但无法从产物还原到源码级意图，也不代表功能一定已对用户开放。

## ✨ 重点变化

### 1. 打包体积显著上升，本地运行组件被明显加重 【实证】

`Contents/Resources/app.asar` 从 `44.4 MB` 增至 `111.5 MB`，增加 `67.14 MB`；`Contents/Resources/codex` 从 `151.4 MB` 增至 `166.9 MB`，增加 `15.45 MB`。同时新增了 `Contents/Resources/node`，大小为 `118.4 MB`，以及 `Contents/Resources/node_repl`，大小为 `4.6 MB`。

这意味着桌面应用在这一版开始随包携带更重的本地 Node.js(本地 JavaScript 运行时)相关能力。可感知层面上，这通常指向更强的本地执行、插件或调试 / 自动化支撑，但仅凭产物不能确认具体交互入口。

关键证据：

```text
Contents/Resources/app.asar: 44.4 MB → 111.5 MB (+67.14 MB)
Contents/Resources/codex: 151.4 MB → 166.9 MB (+15.45 MB)
新增 Contents/Resources/node: 118.4 MB
新增 Contents/Resources/node_repl: 4.6 MB
```

### 2. Computer Use 作为独立打包组件进入应用包 【实证】

应用包中新增了 `Contents/Resources/plugins/openai-bundled/plugins/computer-use/Codex Computer Use.app` 相关文件，其中主服务二进制 `SkyComputerUseService` 为 `11.9 MB`，配套客户端 `SkyComputerUseClient` 为 `10.5 MB`，并包含图标与资源文件。

这是一条强实证：Computer Use 不只是前端模块名，而是以独立 `.app` 组件形式随 Codex 桌面版分发。它意味着桌面端正在为“模型控制本机或浏览器上下文”准备更完整的本地执行层。不过，事实包没有说明功能是否默认开启、如何授权、或哪些用户可见。

关键证据：

```text
新增 Codex Computer Use.app/Contents/MacOS/SkyComputerUseService: 11.9 MB
新增 Codex Computer Use.app/Contents/SharedSupport/SkyComputerUseClient.app/.../SkyComputerUseClient: 10.5 MB
新增 computer-use/assets/app-icon.png: 0.9 MB
```

### 3. 浏览器自动化插件资产进入包内 【实证】

新增文件中出现 `browser-use` 与 `chrome` 插件脚本：`browser-use/scripts/browser-client.mjs` 和 `chrome/scripts/browser-client.mjs` 均为 `0.6 MB`。此外，`chrome/skills/chrome/SKILL.md` 与 `browser-use/skills/browser/SKILL.md` 也进入包内。

这说明桌面应用包开始携带面向 Browser(浏览器)与 Chrome(谷歌浏览器)控制的插件资产。可感知点可能是更丰富的网页打开、点击、截图或浏览器会话控制能力，但事实包只能证明这些插件文件被打包，不能证明入口已开放。

关键证据：

### 6. Intel 版本在 Apple Silicon Mac 上运行时新增提示逻辑 【实证】

启动脚本 `.vite/build/bootstrap.js` 新增了检测逻辑：当应用处于 macOS、打包状态、架构为 `x64`，且进程通过 Rosetta(苹果转译层)运行时，会弹出警告，提示 Intel(英特尔架构)构建在 Apple Silicon Mac(苹果芯片 Mac)上运行，建议退出并安装 Apple Silicon 构建，或继续使用 Intel 构建。

虽然本报告对比平台是 `macOS arm64`，但这段启动逻辑是明确的人类可读差异。它改善的是用户拿错架构版本时的启动体验。

关键证据：

```text
message: "{appName} is running the Intel build on an Apple Silicon Mac"
detail: "This build works through Rosetta, but the Apple Silicon build launches faster and performs better..."
buttons: "Quit" / "Continue Anyway"
```

### 7. 构建脚本新增 Owl 与 Windows MSIX 相关条目 【实证】

`package.json` 中新增了 `owl`、`build:owl`、`owl:ensure`、`owl:package` 脚本，并新增 `electron-windows-msix`、`fs-extra` 依赖，同时加入 `browser-common` 链接依赖，移除了 `socks-proxy-agent`。

这更像是构建与打包体系变化，而非单独用户功能。对开发者而言，它说明同一个桌面代码包开始携带更多跨平台打包与浏览器通用逻辑的构建入口；但事实包没有提供 Owl 的源码说明，因此不扩展解释其具体用途。

关键证据：

```diff
+ "owl": "pnpm --dir .. run generate:third-party-notices && cross-env NODE_ENV=development node ./scripts/owl-shell.mjs run"
+ "build:owl": "cross-env PNPM_YES=true node ./scripts/owl-shell.mjs package"
+ "electron-windows-msix": "2.0.4"
+ "fs-extra": "11.3.2"
+ "browser-common": "link:../../../lib/browser_use/browser-common"
- "socks-proxy-agent": "^8.0.5"
```

### 8. 主题变化订阅逻辑被收敛为共享监听集合 【实证】

`.vite/build/preload.js` 中，系统主题变化订阅从“每个订阅单独挂载 `ipcRenderer.on` 监听器”，改为在 preload(页面预加载脚本)层维护一个 `Set`，由单个系统主题更新事件分发给所有订阅者。

这属于偏底层但可理解的稳定性改动：它减少了重复事件监听器，可能让主题变化通知更集中、更可控。事实包只显示实现方式变化，不足以证明修复了某个具体用户问题。

关键证据：

```diff
+ y=new Set;
+ e.ipcRenderer.on(s,(e,t)=>{_=t,y.forEach(e=>{e()})});
+ subscribeToSystemThemeVariant:e=>(y.add(e),()=>{y.delete(e)})
```

## 关于这份报告

这不是官方变更日志，而是一份**基于官方分发产物的逆向差异报告**。本次事实包采用两个官方全量包直接比较，full-pair(全量包成对比较)：`26.609.30741` → `26.609.41114`。官方 appcast(更新清单) 对这一对版本提供了增量包，但本流程主动选择完整全量包以覆盖完整 bundle(应用包) 内容，因此不引用官方增量包自带的 EdDSA(一种数字签名算法) 验签结论。

可信度方面，本报告只使用机器提取出的结构化差异事实。事实包给出了两个官方全量包的 SHA-256(哈希摘要)；同时，spctl(系统安全评估工具) 公证验证输出在两个版本上均为 `bundle format unrecognized, invalid, or unsuitable`，因此这里不把公证通过写成事实。

证据等级分两档：

- 【实证】：来自人类可读的直接证据，例如 Info.plist(应用信息清单) 键值、可读字符串、配置文件 diff(差异)、插件 manifest(清单文件)。
- 【信号】：来自间接证据，例如压缩后的前端 bundle 新增字符串或体积变化；它们指向某方向，但无法从产物还原到源码级意图。

## ✨ 重点变化

### 1. 新增本地 SQLite 数据库恢复 / 回填流程文案 【实证】

新版主进程 / 前端 bundle 中新增了一组围绕 SQLite(轻量级本地数据库) 恢复、备份、回填进度与会话历史重建的可读字符串，包括 `Back Up and Rebuild`、`Codex rebuilt its local database.`、`Rebuilding conversation history`、`This may take several minutes — keep the app open`。

这意味着桌面端已经加入或至少暴露了本地状态数据库异常后的恢复确认、备份重建、进度展示相关用户路径。可感知点可能是：当本地数据库需要修复时，用户会看到“备份并重建”、数据库路径、备份目录、会话历史重建进度等提示。

### 2. Browser 与 Chrome 内置插件跟随应用版本更新 【实证】

内置 Browser(应用内浏览器自动化插件) 与 Chrome(谷歌浏览器自动化插件) 的 `plugin.json` 版本号从 `26.609.30741` 更新到 `26.609.41114`，描述文本未显示能力层面的变化。

这说明这两个随应用打包的自动化插件参与了本次版本同步，但事实包没有显示新增命令、权限或说明文字变化，因此不应解读为插件能力扩展。

关键证据：

```diff
+  "version": "26.609.41114",
```

### 3. 应用主体内容有少量变动，但整体体积基本稳定 【实证】

bundle 文件数保持 `8298 → 8298`，新增 `0`、移除 `0`、内容改动 `61`。`Contents/Resources/app.asar` 从 `150.7 MB` 到 `150.7 MB`，仅 `+0.01 MB`。

这表明本版不是大规模能力包更新，更像是应用层逻辑与文案的小步修正。事实包中的 app.asar 内部 `新增 295 / 移除 295` 主要受 Vite(前端构建工具) hash 文件名重命名影响，不能当作真实功能增删。

### 4. Computer Use 运行时与 cua_node 依赖没有可见扩张 【实证】

Computer Use(让模型操作桌面 / 浏览器的能力) 运行时 `cua_node` 总量保持 `365.8 MB → 365.8 MB`，变化为 `+0 MB, +0%`；`cua_node` 新增依赖为 `0` 个。

这意味着本次更新没有表现出运行时能力包扩容，也没有新增底层依赖包的直接证据。可感知变化更集中在应用层数据库恢复路径，而不是 Computer Use 运行时。

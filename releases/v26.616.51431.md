# Codex 桌面版 26.616.51431 macOS / Windows 对照

> 第三方跨平台变更日志 · 非官方

本批次的核心差异集中在 Windows MSIX：Computer Use 运行时和 `@oai/sky` 升级，加入浏览器后端依赖、应用策略审批、工具级遥测和应用专属指令注入；macOS arm64 输入样本为同一包对同一包比较，无内容变化。

Windows carries the substantive runtime changes in this batch, while the macOS sample is byte-identical.

## 关于这份报告

本报告只使用输入事实包中的机器提取事实。macOS 对比范围是 `26.616.51431 build 4212` 到同一 `26.616.51431 build 4212`；Windows 对比范围是 `26.616.5445.0` 到 `26.616.6631.0`。两个平台版本号和发布时间可能错位，因此所有结论分别按平台归属标注。

证据等级分两档：【实证】来自 manifest、插件清单、依赖清单、类型声明、技能/文档差异、服务入口和事实包中非“同名前端模块改动线索”的可读字符串；【信号】仅来自模块名、体积方向或压缩 bundle 线索。本批输入未提供前端模块真实增删、同名前端模块改动线索、主进程和前端 bundle 新增可读字符串、CSP 变化、新增组件与资源的独立区块；可展开的高价值变化主要来自 Windows 的 `cua_node`、依赖清单和关键文件 diff。

平台标记含义为：[macOS+Windows] 两平台均有证据；[仅 macOS] 只在 macOS 事实包中出现；[仅 Windows] 只在 Windows 事实包中出现。

## ✨ 重点变化

### 【实证】[仅 Windows] Computer Use 运行时升级到 `cua-node 0.0.2`

Windows 的 `resources/cua_node/manifest.json` 从 `win32/x64` 命名改为 `windows/x64`，`target` 从 `win32-x64` 改为 `windows-x64`。`node_repl_archive_path` 从 `20260609.1` 更新到 `20260619.1`，运行时归档从 `cua-node-0.0.1-20260609232601-73ffe8b8261e-win32-x64.zip` 更新到 `cua-node-0.0.2-20260620010411-120c650ffb83-windows-x64.zip`。

运行时总量从 `285.6 MB` 增至 `298.9 MB`，增加 `13 MB`，约 `+5%`。`node_repl.exe` 从 `13.5 MB` 增至 `14.0 MB`。

macOS 输入样本中 `cua_node` 总量保持 `377.4 MB → 377.4 MB`，没有文件内容变化。

### 【实证】[仅 Windows] 新增 CDP 浏览器后端包

Windows 新增 `@oai/cdp-browser-backend@0.2.0`，描述为 “Browser Use backend that talks to a Chrome DevTools Protocol endpoint.” 体积最大的新增文件包括：

- `@oai/cdp-browser-backend/dist/main.cjs`，`4.2 MB`
- `dist/skill/scripts/browser-client.mjs`，`1.0 MB`
- `classic-level` 的多平台 native prebuild，包括 `darwin-x64+arm64`、`win32-x64`、`linux-x64`、`linux-arm64`、`android-arm64` 等

这是一组目录级新增依赖，证据强度高于单纯包体增长。它说明 Browser Use 相关后端能力已经进入 Windows 分发包，但事实包没有给出用户界面入口或可见开关，不能断言所有用户已可直接使用。

### 【实证】[仅 Windows] `@oai/sky` 从 `0.4.13` 升级到 `0.4.19`

Windows 的 `@oai/sky/package.json`、`.modules.yaml` 和 `.pnpm/lock.yaml` 均显示 `@oai/sky` 从 `0.4.13` 升级到 `0.4.19`。新版本增加对 `@statsig/js-client` 的依赖，锁文件中新增 `@statsig/js-client@3.33.2` 和 `@statsig/client-core@3.33.2`。

同时，旧的可选依赖链 `objc-js@1.5.0`、`node-addon-api@8.8.0`、`node-gyp-build@4.8.4` 从锁文件相关位置移除。事实包只能证明依赖清单变化，不能推断遥测开关、实验分流策略或运行时行为已经对用户开放。

### 【实证】[仅 Windows] Computer Use 增加应用策略查询与风险化审批

Windows 的 `MacComputerUseClient` 类型声明新增 `getAppPolicy(appOrArgs)`，返回 `MacAppPolicyResult`，包含：

- `decision: "allowed" | "denied" | "forbidden"`
- `allowPersistentApproval`
- `target.bundleIdentifier`
- `target.displayName`
- `target.appPath`
- `target.risk: "high" | "low"`
- `warningSubtitle`

`client.js` 新增 `ComputerUseIPCAppPolicyRequest`，默认 `apiVersion` 从一串哈希样式值变为 `CodexComputerUseIPC-1`。`computer-use-policy.js` 在执行工具前调用 `getAppPolicy`，并根据结果生成审批：`denied` 会报组织策略阻止，`forbidden` 会报安全原因禁止。审批元数据也从原来的按输入 app 显示，改为按 `bundleIdentifier` 和 `displayName` 展示，并根据 `allowPersistentApproval` 决定是否允许 `always` 持久批准。

这表明 Windows 包内的 Computer Use 审批链路从简单 app 字符串确认，升级为带策略结果、风险等级、持久化限制和安全拒绝路径的审批模型。

### 【实证】[仅 Windows] Computer Use 工具调用开始携带工具名并接入遥测

Windows 的 `withComputerUsePolicy` 类型签名从接收输入对象改为接收 `ComputerUseToolName` 和输入对象。多个工具调用文件把工具名显式传入审批包装器，包括：

- `get_app_state`
- `click`
- `drag`
- `perform_secondary_action`
- `press_key`
- `scroll`
- `select_text`
- `set_value`
- `type_text`

`list_apps.js` 新增 `logComputerUseToolCalled({ toolName: "list_apps" })`。`create_client.js` 新增 `logComputerUseClientCreated()`，并创建一个 `Set` 传入 `get_app_state` 链路。`computer-use-policy.js` 还出现 `logComputerUseApprovalRequested` 和 `logComputerUseApprovalResolved`。

这些都是包内实证：工具级名称、审批请求、审批结果和客户端创建事件已进入 Windows 运行时代码。

### 【实证】[仅 Windows] 应用专属指令可注入到 Computer Use 的窗口状态文本

Windows 的 `AppState` 类型注释从“描述可见元素和索引的 accessibility text”改为“首次访问且可用时，会带有应用专属 guidance 前缀”。`MacWindowAppState` 新增 `appSpecificInstructions?: string | null`，`get_app_state` 和 `window_result` 签名增加 `appsWithDeliveredInstructions: Set<string>`。

`window_result.js` 会读取 `appSpecificInstructions`，若存在且尚未对该 app 交付过，则把内容包在 `<app_specific_instructions>` 标签中并前置到窗口文本。代码中还出现 `com.apple.iWork.Numbers` 的例外集合，命中时不注入。

这说明 Windows 包内已经有“每个 app 首次访问时注入专属指令”的运行时机制。事实包没有提供具体指令正文，因此不能扩展为某个应用的完整行为说明。

### 【实证】[仅 Windows] Computer Use 错误类型扩展

Windows 的 `errors.d.ts` 和 `errors.js` 新增四个服务器错误码：

- `couldNotGetSenderPID: -10017`
- `ambiguousApp: -10018`
- `couldNotGetBootstrapPort: -10019`
- `screenLocked: -10020`

这些错误路径与进程识别、应用歧义、bootstrap port 和锁屏状态相关，说明 Computer Use 服务端错误面在 Windows 包内被扩展。

### 【实证】[仅 Windows] 内置插件版本号同步到 `26.616.51431`

Windows 的内置插件清单中，`computer-use`、`chrome`、`browser` 的 `.codex-plugin/plugin.json` 版本从 `26.616.41845` 更新到 `26.616.51431`。描述文本在 diff 中没有显示为新增能力正文变化。

可确认的是插件 manifest 版本同步；不能仅凭版本号断言插件功能发生同等范围变化。

### 【实证】[仅 Windows] Browser / Chrome / Computer Use 三个内置插件仍在包内

Windows diff 明确触及三个 openai-bundled 插件：

- `computer-use`：描述为通过 Computer Use 控制 Windows 桌面应用
- `chrome`：描述为使用用户现有 Chrome 状态进行自动化，包含标签页、登录会话、cookies、扩展和 Codex Chrome Extension setup
- `browser`：描述为 Codex in-app browser，面向 localhost、`127.0.0.1`、`::1`、`file://` 和当前 in-app browser tab 的打开、检查、导航、测试、点击、输入和截图

本批变化中它们的实证变化是版本号同步；browser 后端能力的新增证据来自 `@oai/cdp-browser-backend` 依赖，而不是这三个 manifest 描述本身的改写。

### 【实证】[仅 macOS] 本输入样本在 macOS 无内容变化

macOS 上一版与最新版均为 `26.616.51431 build 4212`，两个全量包 URL 相同，SHA-256 均为 `766454cc6d98cb7c82cbd2169198abba8939b57f50b8f0188b4c639de1d61f91`。Bundle 文件数 `8827 → 8827`，新增 `0`、移除 `0`、内容改动 `0`；`app.asar` 内部文件 `3115 → 3115`，新增 `0`、移除 `0`、改动 `0`；`cua_node` 总量 `377.4 MB → 377.4 MB`。

因此只能说：本输入样本在 macOS 平台无内容变化。不能扩展为 macOS 没有获得相关版本变化，因为样本本身是同一版本同一包的比较。
